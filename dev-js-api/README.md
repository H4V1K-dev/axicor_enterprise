# Архитектура AxiCAD — Шпаргалка Разработчика

Быстрая шпаргалка по устройству AxiCAD для быстрого онбординга и предотвращения "размывания" архитектурных границ.

---

## 0. Фундаментальные Правила (Железобетонные границы)

Каждый разработчик обязан соблюдать следующие принципы:

1. **Contract-First**: Никаких "магических чисел" и захардкоженных размеров в коде. Размеры шардов, сокетов, емкости и параметры слоев должны браться из скомпилированного контракта (`contract_generated.py` / `model.toml` / `placement.json`).
2. **out= Mandate (Оптимизация Hot Loop)**: Все методы, вызываемые в цикле анимации или частых расчетах (например, интерполяция, расчет гизмо, коллизии), обязаны принимать параметр `out=` для записи результата, исключая аллокацию новых объектов в памяти.
3. **Зависимости строго вниз по слоям**:
   * **UI / Панели** -> **Режимы (Modes)** -> **Сцена / SceneBuilder** -> **Хранилище (Store)**.
   * Никакого прямого обращения UI к Three.js мешам в обход `store.set('placementData')` или шины событий.
4. **Атомарность мутаций и выделения**:
   * Любые действия интерактивного изменения координат пишутся в `store.set('placementData')`.
   * При смене выделения в [selection.js](./js/editor/selection.js) обновляй ключи стейта (`selectedShardKey`, `selectedSocketKey`) **до** очистки старых элементов управления. Это предотвращает отправку ложных промежуточных событий "пустого выделения", сбрасывающих активные режимы.
5. **Undo/Redo (Timeline Branching)**:
   * Все интерактивные действия (создание, удаление, перемещение, изменение размеров, перелинковка) должны регистрироваться как именованные экшены (`move`, `resize`, `create`, `delete` и т.д.) через `HistoryManager`. 
   * Не изменяй кэш `placementData` напрямую без регистрации в истории.
   * При откате или применении экшена из прошлого "будущее" этой ветки истории инвалидируется и отсекается.
6. **Блокировка ввода (modalActive)**:
   * При открытии модальных окон (Настройки, подтверждение удаления, Hub) в стор выставляется флаг `store.set('modalActive', true)`.
   * `ModeManager` и обработчики кликов на сцене обязаны проверять этот флаг и блокировать клавиатурные хоткеи / рейкастинг сцены.

## 1. Стек и Структура файлов
- **Фронтенд**: Vanilla JS + Three.js + Vanilla CSS.
- **Бэкенд**: Python (HTTP-сервер `server/server.py` + `persistence.py` для сохранения оверрайдов и запуска плейсера).
- **Ключевые файлы**:
  - [main.js](./js/main.js) — точка входа, инициализация и загрузка данных проекта.
  - [viewer.js](./js/viewer.js) — инициализация сцены, камеры, рендерера и `OrbitControls` Three.js.
  - [scene_builder.js](./js/scene_builder.js) — отрисовка мешей шардов, сокетов (со слоями и цветами орбит) и линий связей.
  - [editor.js](./js/editor.js) — управление режимами редактирования и хоткеями.
  - [store.js](./js/store/store.js) — реактивное микро-хранилище состояния (`store.get()`/`store.set()`).
  - [history_manager.js](./js/store/history_manager.js) — гибридный Undo/Redo стек с Timeline Branching.
  - [collision_manager.js](./js/editor/collision_manager.js) — унифицированный резолвер коллизий и рейкастинга для выделения объектов.

## 2. Потоки данных и Реактивность
1. Любое интерактивное изменение (drag, resize, ввод в инспектор) -> меняет свойства Three.js объектов и пишет в `store.set('placementData')`.
2. UI-панели слушают шину событий (`store.on(...)` или `event_bus.js`) и перерисовываются автоматически.
3. При сохранении топологии:
   - Отсылается POST-запрос на `/api/save`.
   - Бэкенд сохраняет `layout_overrides.json` в `projects/local/{project}/`.
   - Запускается скрипт компилятора (`placer.py` -> `router.py`), генерируя свежие `placement.json` и `routes.json`.
   - Клиент получает сигнал `RELOAD_REQ` и перерисовывает 3D-сцену.

## 3. Гибридный Undo/Redo & Ветвление (Timeline Branching)
- **Стеки**:
  - `globalStack` (линейная история с указателем `globalIndex`).
  - `objectHistory = { [key]: [...] }` (персональные истории изменений по каждому объекту).
- **Инвалидация будущего (Timeline Branching)**:
  - При переходе к старому глобальному состоянию (или при новом действии из состояния отмены) всё будущее в `globalStack` отсекается и рекурсивно удаляется из локальных историй (`objectHistory`).
  - При локальном откате объекта все будущие действия этого объекта удаляются из его локального стека и точечно удаляются из `globalStack` по их ID (с корректировкой `globalIndex`).
- **Холодное хранение**:
  - При сохранении layout-а история сериализуется в опциональный файл `projects/local/{project}/history_cache.json`.
  - При старте проекта клиент загружает его через `deserializeHistory()`.

## 4. UI и Стили (Дизайн-система)
Все стили живут в [editor.css](./css/layouts/editor.css) и переиспользуют CSS-переменные:
- **Glassmorphism**: `--ax-bg-glass` и `--ax-border-glass` для окон и кнопок.
- **Типографика**: Шрифт *Outfit* (Google Fonts), цвета текстов `--ax-text` и `--ax-text-faint`.
- **Цвета экшенов**:
  - Красный: `--red` (`DEL`) — удаление.
  - Зеленый: `--green` (`RES`) — ресайз.
  - Желтый: `--yellow` (`DIS`) — разрыв связей.
  - Голубой: `--blue` (`NEW`/`MOV`) — создание/перемещение.
- **Стена Блюра**: Модальное окно `.confirm-overlay` имеет `backdrop-filter: blur(8px)` and `pointer-events: auto`. Оно полностью блокирует raycasting сцены, Ctrl-выделение и тултипы.
- **Линтинг и стандарты CSS**:
  * **🚫 Запрет на анимацию сокращенных свойств (shorthand properties)**: В свойстве `transition` запрещено анимировать сокращенные свойства (например, `background`, `border`, `margin`, `padding`). Всегда указывай конкретные свойства (например, `background-color`, `border-color`) во избежание снижения производительности рендеринга и ошибок линтеров.
  * **🚫 Обязательное дублирование стандартных свойств**: При использовании вендорных префиксов (например, `-moz-appearance: textfield`) обязательно дублируй их стандартной директивой (`appearance: textfield`) для обеспечения кроссбраузерности и соответствия стандартам валидации.

## 5. Горячие клавиши (Binds)
- `Q` — перейти в режим выделения (`SelectMode`).
- `W` — перейти в режим перемещения (`TranslateMode` с отображением Gizmo).
- `T` — перейти в режим изменения размеров (`ResizeMode` для шардов и сокетов).
- `E` — перейти в режим создания сокета (`AddSocketMode` для выбранного шарда).
- `R` — перейти в режим создания шарда (`AddShardMode`).
- `Control` (удержание) — временный переход в режим выделения (`SelectMode`) для быстрого выбора объектов.
- `Delete` — удалить выделенный объект (открывает кастомную модалку с 5 сценариями подтверждения).
- `Ctrl + Z` / `Ctrl + Y` — глобальные отмена / повтор действий из истории.
- `ПКМ` (в режимах создания) — отменить размещение объекта и выйти из режима.
- `Escape` — сброс превью истории, отмена текущего драга, закрытие модалок или снятие выделения.

## 6. Позиционирование по оси Z и валидация сокетов
- **Перемещение по оси Z**: Сокеты поддерживают перемещение по глубине шарда (ось Z в локальных координатах). 
- **Авто-линковка к слоям**: При перемещении сокета по Z автоматически рассчитывается его положение относительно слоев шарда, и сокет линкуется к соответствующему слою (название записывается в `entry_z`). Граничные положения привязываются к `"top"` и `"bottom"`.
- **Пропорциональное масштабирование**: При изменении высоты шарда в `ResizeMode` все сокеты на нем пропорционально сдвигаются по Z (`oz = old_oz * new_height / old_height`), не вылетая за новые границы шарда.
- **Валидация Z-коллизий**: Если сокеты перекрываются по X/Y проекции на одном шарде, валидатор проверяет расстояние между ними по Z на соответствие воздушному зазору: `air_gap = (perimeter / 4) / 10`. При нарушении выводится критическая ошибка топологии.

## 7. WebGL ViewCube (CubeGizmo.glb)
- **WebGL сценка**: Инициализируется в [viewcube.js](./js/ui/viewcube.js) поверх выделенного `<canvas>` в контейнере `#viewcube-container`.
- **Синхронизация**: Ортогональная камера ViewCube копирует кватернион и позицию основной камеры на каждый кадр в цикле `animateViewer()`.
- **Рейкастинг и центроиды**: Клики и ховеры обрабатываются с помощью `THREE.Raycaster` по мешам загруженного файла `CubeGizmo.glb`. Вектор целевого взгляда камеры рассчитывается динамически по центроиду геометрии выбранного меша, что избавляет от необходимости жесткого прописывания векторов.
- **Ортографический / Перспективный режимы**:
  * Клик на плоскость (грань) переводит камеру в ортокамеру для точного выравнивания. Клик на ребро или угол возвращает перспективный режим.
  * `OrbitControls` сбрасывает камеру обратно в перспективный режим только при начале вращения (`controls.state === 0` или `3`). Зум (колесо) и панорама (ПКМ) сохраняют орто-проекцию.

## 8. Унифицированный резолвер коллизий (X-ray выделение)
Все интерактивные режимы (`SelectMode`, `TranslateMode`, `InspectMode`) для определения выбранного объекта под курсором обязаны вызывать функцию `resolveRaycastHit(raycaster)` из [collision_manager.js](./js/editor/collision_manager.js).
- **Стек коллизий**: Метод собирает до 5 ближайших пересечений луча и сортирует кандидатов по расстоянию. Это предотвращает джиттер (дрожание фокуса) и гарантирует выбор правильного элемента.
- **Приоритет выбора**: Узлы трасс (Route Control Points) -> Сокеты -> Шарды -> Линии трасс.
- **X-ray выделение сокетов**: Если родительский шард выделен и прозрачен, сокеты внутри него могут быть выбраны сквозь геометрию шарда. Сокеты на невыделенных шардах экранируются их непрозрачными стенками.
- **Предотвращение Raycast Fighting (z-fighting)**: Введен порог `epsilon = 0.08 * VIS_SCALE`. Если сокет находится на грани или внутри шарда, он не перекрывается геометрией этого шарда при проверке дистанции.
- **Backface Culling для сокетов**: Если сокет повернут от камеры (нормаль плоскости сокета сонаправлена с лучом камеры, `dot(normal, direction) > 0`), он игнорируется при рейкастинге, что позволяет кликать сквозь шард только на лицевые элементы.

### 🚫 Запрет top-level side-effects в ES-модулях
**Никогда** не вызывай `scene.add()`, `renderer.*`, `camera.*` или любой метод
на объектах из `viewer.js` на **верхнем уровне модуля** (вне функций).

Все экспорты `viewer.js` (`scene`, `camera`, `renderer`, `controls`) — это
`let`-переменные, которые равны `undefined` до вызова `initViewer()` в `main.js`.
Если модуль X делает `scene.add(...)` на top-level, а любой другой модуль
статически импортирует что-то из X — **всё приложение молча умирает** (белый экран,
консоль пустая, ES modules глушат каскадные ошибки).

**Плохо:**
```js
// route_editor.js — ЛОМАЕТ ВСЁ
import { scene } from '../viewer.js';
export const group = new THREE.Group();
scene.add(group); // 💀 scene === undefined при загрузке модуля
```

**Хорошо:**
```js
// route_editor.js — ленивая инициализация
import { scene } from '../viewer.js';
export const group = new THREE.Group();
let added = false;
function ensureInScene() {
  if (!added && scene) { scene.add(group); added = true; }
}
export function showEditor() {
  ensureInScene(); // вызываем только когда scene гарантированно жив
  // ...
}
```

### Правило: статический import = загрузка top-level кода
Если модуль A статически импортирует модуль B (`import { x } from './B.js'`),
весь top-level код B **выполнится немедленно**. Поэтому:
- Top-level допустим только для: объявлений переменных, классов, чистых функций
- Любые side-effects (DOM-манипуляции, scene.add, fetch, addEventListener) — только внутри функций
