# Axicor Visualizer & SDK — Road Map & TODO

Этот список задач сформирован на основе опыта работы с динамическими скриптами (например, генерация 56 шардов и 720 связей). Задачи направлены на повышение стабильности, UX и производительности системы.

---

## Основные задачи

- [x] **0. Реинициализация Git (Git Reinitialization):** Удаление вложенного `.git` в SDK и инициализация единого репозитория в корне с правильным `.gitignore`.
- [ ] **1. Асинхронные уведомления с прогресс-баром (Progress Stack)**
  - [ ] Разработать UI-систему уведомлений (в правом нижнем углу экрана), складывающихся «бутербродом».
  - [ ] Реализовать отправку тяжелых фоновых задач (компиляция связей, расчет пинов) в субпроцессы (worker threads или фоновые API-запросы).
  - [ ] Блокировать интерактивные кнопки в UI (перекомпиляция, сохранение), пока идут активные фоновые расчеты.
  - [ ] Добавить анимированные прогресс-бары для длительных операций в стиле UE (Shader Compilation).

- [ ] **2. Обработка ошибок в стеке (Error Handling Stack)**
  - [ ] Доработать стек сбора ошибок в бэкенд-компиляторе (`placer.py`, `router.py`, `model_loader.py`).
  - [ ] Разработать UI-панель/консоль для отображения ошибок компиляции с подсветкой синтаксиса и точной строкой падения в скрипте.
  - [ ] Предотвращать падение веб-сервера при синтаксических или логических ошибках в пользовательских Python-скриптах.

- [ ] **3. Генеративные алгоритмы размещения и подстройки**
  - [ ] Реализовать автоподстройку размеров и пропорций матриц на основе их связности (density-based scaling).
  - [ ] Доработать алгоритм размещения (`placer.py`), чтобы он учитывал оптимальную длину связей (minimize wire length).
  - [ ] Реализовать автоматическое разделение на «стаки» и группы для сложных генеративных схем (сгенерированных через циклы `for`).
  - [ ] **[ГОРЯЧАЯ ТЕМА] Автоопределение архитектуры по коннектомам:**
    - [ ] Реализовать автоматическое определение количества уровней (орбит) на основе анализа графа связности.
    - [ ] Научить алгоритм автоматически группировать шарды по департаментам, распределять связи и определять физические размеры шардов под компоновку матриц.
    - [ ] Реализовать импорт реальных биологических матриц коннектомов (например, типа `marmosetbrain`) с автогенерацией всей 3D-сцены с нуля.

- [ ] **4. Блокировка инпута в ProjectHub**
  - [ ] Перехватывать и гасить все глобальные хоткеи редактора (клавиатурные события), пока открыто окно хаба.
  - [ ] Заблокировать фоновое выделение, перемещение камеры и клики по Three.js сцене сквозь оверлей.

- [ ] **5. Страница загрузки (Loading Screen) и пайплайн инициализации**
  - [ ] Создать плавный сплэш-экран (Loading Screen) между закрытием Project Hub и инициализацией Three.js сцены.
  - [ ] Выводить пошаговый статус загрузки (например, *«1/3 Парсинг Python-скрипта...»*, *«2/3 Расчет позиций шардов...»*, *«3/3 Роутинг связей...»*) в прогресс-уведомлениях.

- [ ] **6. Оптимизация перформанса и логика инструментов редактирования**
  - [ ] Оптимизировать рендер линий (связей) в Three.js при больших объемах (сотни и тысячи кривых Безье). Использовать инстансинг или батчинг мешей для связей.
  - [ ] Проработать режимы редактирования и работу инструментов (выделение группы, добавление/удаление связей вручную) в условиях высокой плотности сцены.

- [ ] **7. Локальный автономный рантайм (Offline Distribution)**
  - [ ] Реализовать сборку редактора в переносимый ZIP-архив (портативный локальный веб-сервер + фронтенд).
  - [ ] Интегрировать логику: при первом запуске (или пустом локальном кэше) отдавать пользователю ZIP-архив для скачивания.
  - [ ] Все вычисления и запуск Python-скриптов должны происходить на локальном железе пользователя (безопасность исходного кода + масштабируемость по ресурсам).

- [ ] **8. Реструктуризация C/CUDA/HIP кода (GPU Kernels)**
  - [ ] Вынести CUDA/HIP ядра (например, `core_math.h`, `physics.cu`) из Rust-крейта `compute-cuda` в выделенную общую директорию `compute-kernels/` на верхнем уровне.
  - [ ] Адаптировать `build.rs` в Rust-крейтах для компиляции из внешней общей папки.

- [ ] **9. Организация единой документации (Centralized Docs)**

  - [ ] Распланировать правильную структуру документации (архитектура, спецификации SDK, доки редактора, физика).
  - [ ] Перенести туда все разбросанные `.md` файлы и спецификации из кодовых папок.
