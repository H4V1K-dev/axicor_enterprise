# Спецификация Selection Engine для AxiCAD (Selection Engine Spec)

> Этот документ формально описывает архитектуру, интерфейсы и политики управления выделением объектов (Selection Engine) в 3D-редакторе AxiCAD. Selection Engine преобразует ввод пользователя и результаты пространственных запросов в стабильное состояние выделения и предоставляет контекст для инструментов, инспектора свойств и визуальной подсветки.

## Status: Draft

---

## 1. Назначение и границы (Scope & Non-goals)

`Selection Engine` управляет логикой выбора трехмерных и плоских объектов на сцене, отслеживанием фокуса мыши (hover) и формированием стека выделенных сущностей (multi-selection).

### Вне зоны ответственности (Non-goals)
- **Не рендерит подсветку**: Движок не занимается графическим отображением контуров выделения, рамок выбора или подсветок в WebGL/Three.js. Он только поставляет состояние выделения в Store/UI.
- **Не производит геометрические расчеты пересечений**: Движок не выполняет трассировку лучей (raycasting) или поиск объектов в R-Tree. Эти операции запрашиваются у [Geometry & Spatial Service](geometry-spatial-service-spec-ru.md).
- **Не изменяет TOML напрямую**: Состояние выделения не влияет на биологическую модель проекта. Движок не производит запись в TOML-файлы конфигураций.
- **Не является Tool System (Системой инструментов)**: Движок не решает, какую операцию совершить при клике или перетаскивании (drag). Он лишь предоставляет контекст выделения (`SelectionContext`) для [Tool System](tool-system.md).

## 2. Место в архитектуре (Architecture Placement)

Selection Engine связывает низкоуровневый ввод пользователя и геометрию сцены со стабильным состоянием приложения:

```
[ Pointer/Keyboard Input ]
           │
           ▼ (Screen coords / click)
┌──────────────────────┐
│   Viewport Adapter   │
└──────────┬───────────┘
           │ (Pick Intent / Ray direction / Abstract intents)
           ▼
┌──────────────────────────────────────┐
│ Geometry & Spatial Service Ray Query │◄──── [ 3D Scene / Spatial Index ]
└──────────┬───────────────────────────┘
           │ (Hit Stack)
           ▼
┌──────────────────────┐
│   Selection Engine   │◄──── [ Selection Policies / Workspace Profile ]
└──────────┬───────────┘
           │ (next SelectionState)
           ▼
┌──────────────────────┐
│  External Controller │ ──► Writes to Session UI Store / Project JSON
└──────────┬───────────┘
           ├─────────────────► [ Property Inspector ]
           ├─────────────────► [ Tool System ]
           └─────────────────► [ Rendering Pipeline (Highlight) ]
```

### Потоки данных:
1. **Ввод пользователя**: События мыши/пера/клавиатуры преобразуются во вьюпорте (Viewport Adapter) в трехмерный луч (`RayCast`) и абстрактные интенты модификаторов ввода.
2. **Пространственный запрос (Ray Query)**: Spatial-сервис возвращает упорядоченный стек пересечений (Hit Stack).
3. **Selection Engine**: Является чистым вычислительным слоем. Он принимает текущее состояние, стек попаданий, политику (`SelectionPolicy`) и возвращает вычисленное новое стабильное состояние (`next SelectionState`).
4. **Внешний контроллер (Controller / Command Adapter)**: Принимает вычисленный `SelectionState` и выполняет его запись в реактивное сессионное хранилище (Session UI Store) или в файл проекта.
5. **Session UI Store**: На изменение состояния реагируют Property Inspector, Tool System и Rendering Pipeline.

> [!NOTE]
> **Сохранение и персистентность выделения (SelectionState Persistence)**:
> Состояние выделения не является биологическим состоянием модели, но сохраняется в файле проекта `axicad.project.json` в качестве UX-метаданных для удобства продолжения работы инженера при перезапуске проекта. Для предотвращения чрезмерных записей на каждый клик мыши применяется политика задержки записи (`debounced persistence`) либо принудительное сохранение при нажатии "Save Project".

## 3. Основные сущности и контракты (Core Entities)

```typescript
// Концептуальный интерфейс (Conceptual interface, не является прямым API реализации)

type UUID = string; // Концептуальный UUID сущности из Store
type TypedEntityPath = string; // Типизированный путь к сущности, например, "departments.Sensory.shards.L2"

/** Абстрактное намерение модификации выделения и ввода (Input Intents) */
export type PickModifierIntent = 
  | 'replace'                 // Заменить текущее выделение новым объектом
  | 'add-to-selection'        // Добавить объект к текущему стеку (additive)
  | 'subtract-from-selection' // Исключить объект из текущего стека
  | 'toggle-selection'        // Инвертировать выделение объекта (toggle)
  | 'precision-mode'          // Режим высокой точности манипуляции
  | 'axis-lock'               // Блокировка движения по оси
  | 'navigation-orbit'        // Вращение камеры
  | 'navigation-pan'          // Панорамирование камеры
  | 'navigation-zoom'         // Масштабирование камеры
  | 'confirm'                 // Подтверждение действия
  | 'cancel'                  // Отмена действия
  | 'cycle-pick'              // Перейти к следующему объекту в стеке попаданий
  | 'pick-parent'             // Выбрать родительский контейнер вместо текущего объекта
  | 'pass-through';            // Пропустить луч сквозь внешнюю оболочку

/** Способ фильтрации нелицевых полигонов при рейкасте */
export type BackfaceFilteringPolicy =
  | 'frontFaceOnly'         // Игнорировать любые элементы, повернутые нормалью от камеры
  | 'allowBackface'         // Разрешить выбор с любой стороны
  | 'preferFrontFace'       // Возвращать задние грани только если нет передних
  | 'transparentPassThrough'; // Пропускать луч сквозь задние грани прозрачных оболочек

/** Причина недоступности объекта для выбора */
export type SelectionDisabledReason =
  | 'hidden'
  | 'locked'
  | 'not-selectable-in-mode'
  | 'invalid-target'
  | 'generated-preview-only';

/** Категория геометрической поверхности, на которую пришелся хит */
export type SurfaceKind = 'solid' | 'transparent-shell' | 'thin-plane' | 'handle' | 'grid';

/** Цель выделения, описывающая тип и путь к объекту */
export interface SelectionTarget {
  readonly entityPath: TypedEntityPath;
  readonly entityId: UUID;
  readonly entityKind: string; // e.g. "shard", "socket", "tract"
}

/** Стек выделения, содержащий упорядоченный список целей */
export interface SelectionStack {
  readonly targets: SelectionTarget[];
  readonly activeIndex: number; // Индекс главного выделенного элемента (-1 при пустом)
}

/** Ограничение области выделения */
export interface SelectionScope {
  readonly allowedKinds: string[]; // e.g. ["shard", "socket"]
  readonly allowedParentPaths?: TypedEntityPath[]; // Ограничение выбора внутри конкретного департамента
}

/** Описание единичного пересечения луча с объектом */
export interface SelectionHit {
  readonly hitId: string;
  readonly entityPath: TypedEntityPath;
  readonly entityId: UUID;
  readonly entityKind: string;
  readonly distance: number; // Дистанция от начала луча
  readonly hitPoint: [number, number, number]; // Физические координаты µm
  readonly normal: [number, number, number]; // Нормаль поверхности в точке хита
  readonly surfaceKind: SurfaceKind;
  readonly localCoordinates?: any; // Локальные координаты (например, воксели U/V для сокета)
  readonly confidence: number; // Вес/приоритет попадания (0.0 .. 1.0)
  readonly selectable: boolean; // Доступен ли объект для выбора
  readonly disabledReason?: SelectionDisabledReason; // Причина блокировки выбора
}

/** Намерение пользователя совершить выбор */
export interface PickIntent {
  readonly screenCoords: { x: number; y: number };
  readonly rayOrigin: [number, number, number];
  readonly rayDirection: [number, number, number];
  readonly modifierIntents: PickModifierIntent[]; // Перечень интентов (допускается комбинация, например ['additive', 'cyclePick'])
}

/** Параметры и фильтры, накладываемые при выборе */
export interface SelectionPolicy {
  readonly backfacePolicy: BackfaceFilteringPolicy;
  readonly scope: SelectionScope;
  readonly selectionPriorityOrder: string[]; // Список приоритетов типов (от высокого к низкому)
  readonly allowLockedEntities: boolean; // Разрешено ли выбирать locked-объекты
}

/** Текущее состояние наведения курсора (эфемерное состояние вьюпорта) */
export interface HoverState {
  readonly hoveredEntity?: TypedEntityPath;
  readonly hoveredId?: UUID;
  readonly hitPoint?: [number, number, number];
  readonly normal?: [number, number, number];
}

/** Полное состояние выделения (Editor UI Selection State) */
export interface SelectionState {
  readonly selectionStack: SelectionStack;
  readonly hover: HoverState;
  readonly activeFocus?: SelectionTarget; // Сущность, на которой сфокусирован инспектор/активный инструмент
  readonly cycleState?: {
    readonly lastHitStackSignature: string;
    readonly currentIndex: number;
  };
}

/** Полный контекст выделения для Tool System */
export interface SelectionContext {
  readonly selectionState: SelectionState;
  readonly workspaceMode: string; // Текущий режим редактора (Composition, Connectome и т.д.)
  readonly activePolicy: SelectionPolicy;
}
```

## 4. Типизированные пути сущностей (Typed Entity Paths)

Каждый объект, который может быть выбран на сцене, адресуется в Selection Engine через пару `(entityId, entityPath)`, где `entityPath` — типизированный путь (`TypedEntityPath`). Это позволяет однозначно связать выбор с элементами Store.

Поддерживаемые типы сущностей (kinds):
1. **model**: Глобальный уровень проекта.
2. **composition-level**: Слой сборки в Composition Workspace (высотные пояса).
3. **department**: Логический департамент, объединяющий шарды.
4. **shard**: Объемный воксельный шард, содержащий нейроны.
5. **shard-face**: Отдельная грань шарда (используется для размещения сокетов).
6. **socket**: Площадка ввода-вывода на грани шарда.
7. **socket-sample**: Конкретная дискретная ячейка сэмплинга внутри сокета (порт).
8. **tract**: Трасса межкабельного соединения.
9. **tract-segment**: Отдельный прямолинейный участок или узел (vertex) трассы тракта.
10. **generated-preview**: Сгенерированный временный объект предпросмотра (например, предварительная трасса тракта при автороутинге).
11. **diagnostic-marker**: Визуальный маркер ошибки на сцене (например, область коллизии).
12. **runtime-probe / runtime-path (Перспектива)**: Эфемерные объекты симуляции (зонды, пути прохождения сигнала при отладке). Runtime-spike сам по себе не является канонической выделяемой сущностью (`selectable target`) и отображается как read-only derived визуализация.

## 5. Стек попаданий и политики фильтрации (Hit Stack & Filtering)

При клике во вьюпорте Spatial-сервис возвращает стек попаданий `SelectionHit[]`, упорядоченный по физическому расстоянию от камеры к объекту. Задача Selection Engine — применить политики фильтрации к этому стеку.

### Фильтрация задних граней (Backface / Normal-facing Filtering)

Для предотвращения выбора задних стенок полупрозрачных или сплошных оболочек (например, когда пользователь пытается выбрать шард *внутри* департамента, кликая по передней поверхности, а выбор срывается на заднюю внутреннюю стенку департамента) применяется фильтрация по нормалям.

```
Камера
                   │
                   │ луч
                   ▼
   ┌───────────────────────────────────────┐
   │  Передняя грань оболочки              │ ──► Валидный хит (сохраняется)
   │  (нормаль навстречу лучу)             │
   ├───────────────────────────────────────┤
   │                                       │
   │   Внутренний объект: шард             │ ──► Валидный хит (сохраняется)
   │                                       │
   ├───────────────────────────────────────┤
   │  Задняя грань оболочки                │ ──► Фильтруется / отбрасывается
   │  (нормаль сонаправлена лучу)          │
   └───────────────────────────────────────┘
```

### Алгоритм фильтрации нормалей:
- Вычисляется скалярное произведение вектора направления луча $\vec{D}$ и нормали поверхности в точке пересечения $\vec{N}$:
  $$\text{dot} = \vec{D} \cdot \vec{N}$$
- Если $\text{dot} > 0$, нормаль направлена от камеры (задняя грань).
- В зависимости от активной политики `BackfaceFilteringPolicy`:
  - **`frontFaceOnly`**: Если $\text{dot} > 0$, пересечение немедленно удаляется из стека попаданий.
  - **`allowBackface`**: Все попадания сохраняются в стеке без ограничений.
  - **`preferFrontFace`**: Попадания с $\text{dot} > 0$ перемещаются в конец стека (получают низший приоритет по сравнению с передними гранями).
  - **`transparentPassThrough`**: Для объектов с `surfaceKind = 'transparent-shell'` (полупрозрачные границы департаментов) все задние грани отбрасываются, а передние пропускают выбор внутрь. Детерминированное поведение пропуска описывается ниже.

*Особое правило для тонких поверхностей*: Для объектов с `surfaceKind = 'thin-plane'` (двусторонние тонкие площадки сокетов, плоскости привязки или тракты) проверка нормалей игнорируется, так как они считаются двусторонними и должны выбираться под любым углом.

## 6. Приоритеты выделения (Selection Priority)

Выбор объекта не может основываться исключительно на физическом расстоянии (distance). Иначе мелкие управляющие элементы (gizmos, handles) или тонкие сокеты было бы невозможно выделить на фоне массивных шардов.

Selection Engine сортирует отфильтрованный стек попаданий по приоритету сущностей (`selectionPriorityOrder`).

### Канонический приоритет (от высокого к низкому):
1. **Explicit handles / gizmos**: Интерактивные маркеры перемещения, вращения, изменения размеров (всегда перехватывают выбор).
2. **Active tool targets**: Объекты, на работу с которыми настроен активный в данный момент инструмент (например, если выбран инструмент "Socket Tool", сокеты получают наивысший приоритет над шардами).
3. **Sockets / Tract handles**: Контакты ввода-вывода и узлы трассировки кабелей.
4. **Shards**: Биологические шарды.
5. **Departments**: Логические департаменты (группы шардов).
6. **Composition levels / containers**: Уровни сборки, высотные рамки.
7. **Generated preview**: Элементы предпросмотра генерации.
8. **Background grid**: Фоновая сетка сцены (используется для привязки координат).

Приоритеты динамически перестраиваются в зависимости от:
- Активного режима рабочей области (Workspace Mode).
- Состояния блокировки объекта (`lock` флаг) и его видимости (`visibility` флаг).

## 7. Окклюзия и прозрачность (Occlusion & Transparency)

Сложные сцены AxiCAD содержат вложенные полупрозрачные оболочки (например, полупрозрачные контейнеры департаментов, внутри которых размещаются десятки шардов).

Для комфортной работы без постоянного скрытия внешних оболочек применяются следующие политики:

1. **Прозрачный пропуск (Pass-through transparent shell)**:
   - Пропуск сквозь полупрозрачную оболочку департамента к внутренним шардам не является спонтанным. Он определяется детерминировано на основе текущего состояния циклического перебора (`cycleState`).
   - Повторный клик в ту же точку распознается внешним контроллером по следующим правилам:
     - Клик пришелся в ту же экранную ячейку (screen pixel/cell с допуском 2-3 пикселя);
     - Идентична сигнатура стека попаданий (`hit stack signature`, рассчитываемая как хэш отсортированного списка `hitId` кандидатов);
     - Временной интервал между кликами соответствует окну взаимодействия, обрабатываемому внешним вьюпортом.
   - При совпадении условий внешний контроллер передает в Selection Engine интент `cyclePick` или `passThrough`, сдвигая указатель выбора на следующий элемент в стеке попаданий (например, на шард внутри департамента).
2. **Циклический перебор (Cycle through selection stack)**:
   - Если луч пересекает несколько объектов, интент `cyclePick` последовательно перебирает объекты в отфильтрованном стеке попаданий.
3. **Интент родительского выбора (Parent/Container picking)**:
   - Интент `pickParent` автоматически переносит выбор с дочернего элемента (шарда) на его родительский контейнер (департамент), если он существует в иерархии Store.

## 8. Множественное выделение (Multi-selection)

Selection Engine вычисляет новый стек выделения в зависимости от переданных интентов в `PickIntent.modifierIntents` (поддерживается комбинация нескольких интентов одновременно, например, `['additive', 'cyclePick']` или `['pickParent', 'toggle']`):

- **Выбор с заменой (`replace`)**: Предыдущий стек очищается, создается новый стек, содержащий один выбранный объект (он становится активным).
- **Аддитивный выбор (`additive`)**: Новая цель добавляется в `SelectionStack.targets`. Активный индекс смещается на нее.
- **Инвертируемый выбор (`toggle`)**: Если цель уже есть в стеке, она удаляется из него; если её нет — добавляется.

### Ограничения области выбора (SelectionScope):
Для предотвращения логического хаоса в Store запрещено смешивать в одном стеке выделения сущности разных категорий:
- Нельзя одновременно выделить канонический биологический шард (`shard`) и эфемерные диагностические маркеры ошибок (`diagnostic-marker`).
- Нельзя смешивать в мультиселекте канонические сокеты (`socket`) и симуляционные зонды (`runtime-probe`).
- При попытке смешать несовместимые типы Selection Engine автоматически отфильтровывает кандидатов согласно правилам `SelectionScope`.

## 9. Hover vs Active vs Focus (Dirty Behavior)

Selection Engine оперирует тремя уровнями интерактивных состояний, каждое из которых имеет свои правила жизненного цикла:

1. **Hover State (Наведение)**:
   - Является чисто эфемерным состоянием вьюпорта (ephemeral viewport state).
   - Не вызывает обновление тяжелых компонентов UI и транзакций Store, обрабатывается локально во вьюпорте для высокой производительности (60 FPS).
   - Никогда не помечает проект как грязный (`dirty`).
2. **Active Selection (Активное выделение)**:
   - Состояние выделенной группы объектов (`SelectionStack`).
   - Сохраняется внешним контроллером в сессионное хранилище (Session UI Store).
   - Влияет на `project_json_dirty` только при включении опции кэширования выделения (для восстановления UI при перезапуске редактора). В канонические TOML-файлы не пишется никогда.
3. **Focus / Inspector Target (Фокус свойств)**:
   - Сущность, назначенная целевым объектом инспектора свойств или активного инструмента.
   - Фокус не накладывает блокировок или запретов на редактирование, а лишь указывает инспектору, чьи свойства выводить на панель. Изменения свойств инициируют мутации через команды (которые уже выставляют флаги `bio_toml_dirty` или `project_json_dirty`).

## 10. Интеграция с инструментами (Interaction with Tools)

Selection Engine является пассивным вычислителем. Он подготавливает структуру `SelectionContext` и возвращает её внешнему контроллеру для передачи в Tool System.

- **Геометрический контекст**: Контекст содержит точные геометрические параметры попадания: координаты точки хита в микрометрах `hitPoint`, вектор нормали `normal` для расчета направления привязки (snapping) и локальные координаты (например, UV-ячейку сокета).
- **Ограничения выбора (Lock & Selectability)**:
  - Флаги блокировки (`locked`), видимости (`visible`) или доступности (`selectable`) определяются на основе данных из Store, метаданных проекта или активного профиля рабочей области (`WorkspaceProfile`).
  - [Constraint Engine](constraint-engine-spec-ru.md) может поставлять геометрические или биологические факты несоответствия (`invalid/blocked/readiness facts`), но не владеет флагами пользовательской блокировки элементов.
  - Selection Engine объединяет флаги из Store, правила активного `WorkspaceProfile` и факты от Constraint Engine для вычисления доступности выбора конкретного кандидата.

## 11. Профили выделения по рабочим пространствам (Workspace Profiles)

В зависимости от активного режима редактора (Workspace Mode), применяются специализированные профили правил:

### 11.1 Composition Workspace Profile
- **Основная цель**: Размещение и компоновка макроструктуры сети (департаментов и шардов).
- **Разрешенные типы**: `composition-level`, `department`, `shard`.
- **Приоритеты**: Департаменты и шарды имеют равный приоритет; сокеты и тракты полностью игнорируются для кликов.
- **Политика нормалей**: `frontFaceOnly` для сплошных шардов, `transparentPassThrough` для департаментов.

### 11.2 Connectome Workspace Profile
- **Основная цель**: Проектирование связей, сокетов, пинов и трассировка трактов.
- **Разрешенные типы**: `socket`, `socket-sample`, `tract`, `tract-segment`, `shard` (только как подложка).
- **Приоритеты**: Гизмо привязки -> Сокеты -> Тракты -> Шарды (с низким приоритетом).
- **Политика нормалей**: `preferFrontFace` (сохраняется возможность выбора скрытых за шардами сокетов при циклическом клике).

### 11.3 Shard / Neuron Lab Workspace Profile (Перспектива)
- **Основная цель**: Распределение типов нейронов по биологическим слоям внутри отдельного шарда.
- **Разрешенные типы**: `shard-face`, `biological-layer`, `neuron-placement`.
- **Приоритеты**: Слои шарда -> Внутренние сомы.

### 11.4 Growth & Inference Profile (Перспектива)
- **Основная цель**: Анализ симуляции в реальном времени.
- **Разрешенные типы**: `runtime-probe`, `runtime-path`, `diagnostic-marker`. (Спайки симуляции не являются каноническими `selectable targets` и инспектируются через связанный сегмент тракта, зонд или кадр таймлайна).
- **Приоритеты**: Маркеры ошибок -> Активные синапсы / зонды. Каноническая геометрия шардов блокируется для случайного изменения.

## 12. Детерминированность и стабильность (Determinism)

1. **Детерминизм выбора**:
   - При одинаковом состоянии Store, положении камеры и координатах клика, Selection Engine должен возвращать абсолютно одинаковый `SelectionState` на любой платформе.
2. **Стабильное разрешение неопределенностей (Tie-breakers)**:
   - Если два или более объектов находятся на одинаковом расстоянии от луча (или разница расстояний попадает в пределы машинной погрешности float $\le 10^{-5}$ µm), выбор разрешается по стабильным правилам сортировки:
     1. Приоритет типа сущности (`selectionPriorityOrder`).
     2. Лексикографический порядок путей (`entityPath`).
     3. Стабильный хэш UUID сущности.
3. **Защита от дрожания (Hysteresis / Sticky Hover)**:
   - Для предотвращения быстрого переключения фокуса (дрожания hover) при микродвижениях мыши на стыке объектов вводится зона гистерезиса (Hysteresis Zone) радиусом в 2-3 пикселя на экране. Фокусировка на текущем объекте сохраняется («прилипает»), пока курсор не выйдет за пределы этой зоны.

## 13. Диагностика и недоступные элементы (Disabled Selection)

Если луч пересек объект, который виден на сцене, но не может быть выделен по правилам текущего профиля или Store, Selection Engine возвращает у этого кандидата статус `selectable = false` и причину `disabledReason`.

Возможные причины недоступности выбора:
- `hidden`: Объект скрыт в дереве видимости.
- `locked`: Объект заблокирован пользователем от изменений (флаг `locked` в Store/metadata).
- `not-selectable-in-mode`: Объект принадлежит другому рабочему пространству (например, попытка выбрать сокет в Composition режиме).
- `invalid-target`: Объект поврежден или содержит критические ошибки структуры данных, манипуляции с ним временно ограничены.
- `generated-preview-only`: Объект является эфемерным превью автотрассировки и не может быть выбран как самостоятельная сущность.

Selection Engine **не генерирует `DiagnosticItem` напрямую**, а поставляет эти причины в Tool System и UI для отображения всплывающих подсказок (tooltip: *"Объект заблокирован"*).

## 14. Зафиксированные архитектурные решения (Resolved Architecture Decisions)

1. **Персистентность `SelectionState` (Resolved)**: Состояние выделения сохраняется в файле проекта `axicad.project.json` (debounced / on project save) наряду с позициями камер и UX-метаданными.
2. **Интерфейс циклического выбора (Cycle-Pick UI) (Resolved)**: Вьюпорт предоставляет компактный мини-оверлей рядом с курсором со списком стека пересечений (Hit Stack), позволяя инженеру прокликивать насквозь и точно выбирать глубокие объекты.
3. **Глубина выделения (Deep Selection) в MVP (Resolved)**: В MVP поддерживается глубокое выделение как сокет-пинов/сэмплов (`socket samples/pins`), так и контрольных пространственных фреймов трактов (`wireframe control frames/boxes`).
4. **Выделение артефактов симуляции и спайков (Resolved)**: Спайк (`runtime spike`) сам по себе в MVP **не является** канонической выделяемой сущностью (`selectable entity`). Он трактуется как активный фронт активности аксона с угасающим хвостом, визуализируется и инспектируется через связанный сегмент тракта/зонд/кадр таймлайна. Runtime-артефакты остаются read-only derived визуализацией.
5. **Абстрактные интенты и модель модификаторов (Resolved / Blender-like base)**: Назначение клавиш ориентировано на Blender-like модель взаимодействия. Сырые клавиши ОС транслируются платформенным адаптером в абстрактные интенты (`add-to-selection`, `precision-mode`, `axis-lock`, `confirm`, `cancel` и др.). Точные биндинги уточняются по мере UX-тестирования.
6. **Графический оптимизационный статус GPU Picking Buffer (Resolved)**: Аппаратный буфер выбора на стороне GPU (`Offscreen Picking Buffer`) выступает исключительно графической оптимизацией скорости поиска кандидатов и не является источником истины (*source of truth*). Selection Engine полностью сохраняет за собой владение политиками выборки, пространственной фильтрацией, проверкой прав доступа в Store (`disabledReason`) и окончательным резолвингом целей в Hit Stack.
