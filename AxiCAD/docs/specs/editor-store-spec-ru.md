# Спецификация реактивного хранилища AxiCAD (Editor Store Spec)

> Этот документ формально описывает внутреннюю реактивную модель состояния (Store) 3D-редактора AxiCAD. Модель адаптирует плоские декларативные файлы TOML DSL Axicor к требованиям интерактивного редактирования, обеспечивая согласованность ссылок, отслеживание изменений (Undo/Redo), интеграцию диагностик и отделение биологических данных от метаданных визуализации.

## Status: Draft

---

## 1. Назначение документа

Editor Store Spec регламентирует структуру данных, индексы памяти, логику импорта/экспорта и мутаций внутри ядра 3D-редактора AxiCAD.

### Вне зоны ответственности (Non-goals)
- Документ не вводит новые биологические параметры Axicor и не изменяет TOML-схему (см. `toml-schema-spec-ru.md`).
- Документ не описывает структуру бинарных файлов сборщика Baker.
- Документ не навязывает конкретную реализацию UI-компонентов или рендеринга Three.js.

## 2. Базовые принципы (Core Principles)

Внутреннее состояние редактора строится на следующих архитектурных правилах:

- **Изоляция биологических данных (Clean Domain Payload)**: Биологические параметры TOML хранятся в изолированном поле `data` записей Store. Визуальные метаданные редактора (положение камер, выделение, раскрытые папки дерева) никогда не записываются в TOML-файлы.
- **Стабильная идентичность (UUID)**: Каждая сущность в памяти редактора идентифицируется уникальным UUID (`id`), который генерируется при импорте и остается стабильным на протяжении сеанса работы (сессии проекта).
- **Имя (Name) как путь**: Имя сущности (`name` в TOML) является биологическим идентификатором в рамках контейнера и компонентом пути, но не служит первичным ключом в Store. Изменение `name` сущности автоматически пересчитывает все пути и обновляет ссылки в соединениях.
- **Вычисляемые пути (Derived Domain Paths)**: Абсолютный путь сущности в биологическом дереве (например, `SensoryCortex.Retina.cross_modal`) является вычисляемым свойством (`path`), динамически строящимся от корня дерева владения.
- **Строгое дерево владения (Ownership Tree)**: Иерархия сущностей представляет собой строгое направленное дерево без циклов и множественного владения.
- **Ссылочный граф поверх дерева (Reference Graph)**: Соединения сокетов (`connections`) являются ссылочными дугами, наложенными поверх иерархического дерева владения. Редактор разрешает строковые пути TOML в UUID сущностей Store и кэширует их отдельно от экспортируемых TOML-данных.

## 3. Контракт DomainRecord

Каждый биологический объект в памяти редактора оборачивается в унифицированную структуру `DomainRecord`:

```typescript
export type UUID = string; // Стандартный RFC4122 UUID v4

export type BiologicalEntityType =
  | 'model'
  | 'department'
  | 'shard'
  | 'layer'
  | 'neuron_type'
  | 'socket'
  | 'port'
  | 'pin'
  | 'connection';

export type EditorEntityType = 
  | 'editor_placement'; // Не экспортируется в биологический TOML

export type DomainEntityType = BiologicalEntityType | EditorEntityType;

export interface DomainRecord<T = any> {
  /** Уникальный UUID записи в Store редактора (первичный ключ) */
  id: UUID;
  
  /** Тип доменной сущности */
  type: DomainEntityType;
  
  /** Динамически вычисляемый путь в дереве владения (например, "SensoryCortex.Retina") */
  path: string;
  
  /** UUID родительской сущности (null для корня - 'model') */
  parentId: UUID | null;
  
  /** Список UUID дочерних сущностей (согласно дереву владения) */
  childrenIds: UUID[];
  
  /** Чистые биологические данные, напрямую сериализуемые в TOML */
  data: T;
  
  /** Метаданные визуализации и интерфейса редактора */
  meta: EntityMeta;
  
  /** Список актуальных диагностических сообщений, привязанных к сущности */
  diagnostics: DiagnosticItem[];
  
  /** Счетчик ревизий сущности для отслеживания изменений (dirty state) */
  revision: number;
}

### Признак экспортируемости (Exportable Flag)
Возможность экспорта записи в биологический TOML является **вычисляемой (computed/derived)** на основе типа сущности:
- `exportable = record.type belongs to BiologicalEntityType`
- Сущности типа `EditorEntityType` (например, `editor_placement`) всегда являются **non-exportable** (флаг равен `false`) и отфильтровываются на этапе сохранения биологического TOML-контракта.


export interface EntityMeta {
  /** Флаг выделения сущности в viewport / дереве проекта */
  selected?: boolean;
  /** Флаг раскрытия ветки дерева в UI */
  expanded?: boolean;
  /** Скрыта ли сущность на 3D-сцене */
  hidden?: boolean;
  /** Пользовательские цветовые переопределения для отладки */
  colorOverride?: string;
}

export interface DiagnosticItem {
  code: string;           // Код ошибки (например, "AXI-VALID-004")
  severity: 'error' | 'warning' | 'info';
  message: string;        // Текстовое описание
  path: string;           // Путь к невалидному свойству внутри record.data
  suggestedFix?: string;  // Подсказка по исправлению
}
```

## 4. Дерево владения (Ownership Tree)

Иерархические связи определяют жесткие границы жизненного цикла сущностей: при удалении родительской записи рекурсивно удаляются все дочерние записи.

```
[model] (ModelConfig)
  ├── [[departments]] (DepartmentEntry) ──► (владеет) ──► [department] (DepartmentConfig)
  └── [[connections]] (ModelConnectionConfig) ──► (ссылается на сокеты, не владеет)

[department] (DepartmentConfig)
  ├── [[shards]] (ShardEntry) ──► (владеет) ──► [shard] (ShardConfig)
  └── [[connections]] (DepartmentConnection) ──► (ссылается на сокеты, не владеет)

[shard] (ShardConfig)
  ├── [[layers]] (LayerConfig) ──► (владеет)
  ├── [[neuron_types]] (NeuronType) ──► (владеет)
  ├── [[sockets]] (SocketConfig) ──► (владеет)
  └── [[ports]] (PortConfig) ──► (владеет) ──► [[pins]] (PinConfig) (владеет)
```

- **Правило владения**:
  - `model` владеет департаментами и глобальными соединениями `connections`.
  - `department` владеет шардами и локальными соединениями `connections`.
  - `shard` владеет анатомическими слоями, типами нейронов, сокетами и внешними портами.
  - `port` владеет своими проекционными пинами.
  - Соединения (`connections`) хранят ссылки на сокеты на обоих концах, но **не владеют** ими. Удаление сокета оставляет соединение в неразрешенном (висящем) состоянии, генерируя ошибку валидации ABI.

## 5. Ссылочный граф (Reference Graph)

Соединения описывают логические пути между сокетами. Для быстрого обхода графа связей редактор строит и кэширует ссылочный граф на основе UUID, а не текстовых строк TOML.

- **Область действия соединений (Scope)**:
  - Соединения связывают **исключительно сокеты** (`connections connect sockets only`).
  - Внешние порты (`ports`) и пины (`pins`) являются устройствами ввода-вывода (external I/O) и **не участвуют** в биологической проводке `[[connections]]` (компилятор Baker настраивает для них UDP-порты напрямую).
- **Кэш связей (Link Cache)**:
  ```typescript
  interface ResolvedConnection {
    connectionId: UUID; // Ссылка на саму DomainRecord типа 'connection'
    sourceSocketId: UUID | null; // Разрешенный UUID сокета-источника (out)
    targetSocketId: UUID | null; // Разрешенный UUID сокета-приемника (in)
    rawFrom: string; // Исходный строковый путь из TOML
    rawTo: string;   // Исходный строковый путь из TOML
  }
  ```
  Если путь не может быть разрешен (например, шард или сокет удален), UUID записываются как `null`, а к сущности соединения привязывается ошибка уровня `AXI-BAKER-001` (неразрешимый путь).

## 6. Индексы Store (Store Indexes)

Для обеспечения отрисовки 3D-сцены с частотой 60 FPS и быстрого поиска сущностей Store поддерживает плоскую таблицу записей и набор предвычисленных индексов:

### 6.1 Типизированные пути во избежание коллизий (Typed Path Strategy)
Использование простого индекса `Map<string, UUID>` по относительному пути (например, `Retina.cross_modal`) может приводить к коллизиям, если, например, сокет и порт имеют одинаковое имя. Для предотвращения этого редактор применяет **типизированные пути** (`byTypedPath`), где ключ включает в себя тип сущности:
- Формат: `byTypedPath: Map<string, UUID>` (ключ: `EntityType:path`, например `socket:SensoryCortex.Retina.cross_modal`).
- Для соединений при этом сохраняется стандартный биологический синтаксис TOML эндпоинтов (например, `SensoryCortex.Retina.cross_modal`).

### 6.2 Индексы разрешения соединений
Для разрешения соединений с учетом области видимости (scope) редактор использует два специализированных индекса вместо одного глобального:
1. **`socketByDepartmentEndpoint`**: `Map<string, UUID>`
   - Ключ: `Department_UUID + ":" + ShardName + "." + SocketName`
   - Позволяет разрешать относительные пути соединений внутри конкретного департамента (`department.toml`).
2. **`socketByModelEndpoint`**: `Map<string, UUID>`
   - Ключ: `DepartmentName + "." + ShardName + "." + SocketName`
   - Позволяет разрешать глобальные пути соединений на уровне модели (`model.toml`).

### 6.3 Базовые индексы
- **`byId`**: Map<UUID, DomainRecord> — главный плоский реестр всех записей в памяти.
- **`childrenByParent`**: Map<UUID, UUID[]> — карта быстрого получения дочерних узлов для построения дерева проекта.
- **`byType`**: Map<DomainEntityType, UUID[]> — группировка сущностей по типам.
- **`diagnosticsByEntity`**: Map<UUID, DiagnosticItem[]> — быстрый доступ к ошибкам конкретной сущности.
- **`dirtyEntities`**: Set<UUID> — реестр измененных сущностей, ожидающих фоновой валидации или сохранения.

## 7. Конвейеры импорта и экспорта

В редакторе операции сохранения состояния сессии и экспорта чистого биологического кода строго разделены.

### 7.1 Сохранение проекта (Save editor project)
Сохраняет текущую сессию редактора, включая визуальные настройки и diagnostics:
- Сохраняет файл проекта `axicad.project.json` (положения камер, viewport, выделения).
- Сохраняет служебный кэш сессии и текущее состояние dirty-флагов.
- Разрешено сохранять проект при любых ошибках валидации (пользователь не теряет незаконченную работу).

### 7.2 Экспорт TOML (Export to TOML)
Генерирует файлы чистого биологического кода для рантайма:
1. **Сбор данных**: Редактор обходит дерево владения, начиная от корневой `model`.
2. **Фильтрация**: Игнорируются все записи, для которых вычисляемое свойство `exportable(record) === false` (например, `editor_placement`).
3. **Очистка данных**: Из каждой допущенной записи извлекается только поле `data`. Все свойства из полей `meta`, `diagnostics`, `path` и UUID `id` полностью отбрасываются.
4. **Сортировка ключей (Stable Ordering)**: Сортировка ключей TOML-таблиц в каноническом порядке для предотвращения лишних diff-ов в Git.
5. **Запись на диск**: Файлы сериализуются в `model.toml`, `department.toml` и `shard.toml` и раскладываются по каталогам.
6. **Блокировка при ошибках**: Экспорт полностью блокируется при наличии ошибок с уровнем серьезности `severity = "error"`.

### 7.3 Компиляция Baker (Baker Compile)
Baker — это внешний CLI-процесс сборщика, который запускается поверх экспортированных TOML-файлов. Перед запуском редактор может выполнить префлайт-экспорт во временную директорию сборщика.

## 8. Стабильность UUID и метаданных

Поскольку файлы TOML не содержат UUID, при полной перезагрузке проекта (например, открытии папки заново) UUID сущностей генерируются заново. Для сохранения визуальных метаданных (размещение шардов, состояние камер) из `axicad.project.json` редактор использует следующую стратегию:

1. **Привязка к путям (Path-based binding)**:
   - Внутри `axicad.project.json` метаданные и размещение шардов привязываются не к UUID, а к вычисляемым типизированным путям (например, `shard:SensoryCortex.Retina`).
2. **Каскадная миграция при переименовании**:
   - Когда пользователь переименовывает департамент или шард в Store, редактор запускает команду-мигратор, которая каскадно обновляет пути в записях Store и соответствующую привязку метаданных в `axicad.project.json`, сохраняя визуальное положение и настройки объекта.

## 9. Конвейер импорта TOML (TOML Import Pipeline)

Процесс загрузки файлов биологического проекта в память редактора состоит из следующих шагов:

```
[ TOML файлы на диске ]
          │
          ▼
1. Разбор синтаксиса TOML (парсинг) ────► Ошибки Tier 1 (Serde)
          │
          ▼
2. Генерация DomainRecords
   - Создание плоских записей
   - Генерация стабильных сессионных UUID
          │
          ▼
3. Построение Ownership Tree
   - Линковка родитель/потомок (parentId/childrenIds)
          │
          ▼
4. Вычисление типизированных путей (byTypedPath)
          │
          ▼
5. Разрешение ссылок соединений
   - Заполнение socketByDepartmentEndpoint и socketByModelEndpoint
          │
          ▼
6. Первичная валидация
   - Запуск проверок Tier 2 (Rust) и Tier 3/4
   - Привязка диагностик к записям
```

## 10. Метаданные редактора (Editor Metadata)

Все визуальные и интерфейсные параметры, необходимые для работы редактора, сохраняются в служебном файле проекта `axicad.project.json`, находящемся в корневом каталоге рядом с `model.toml`.

### Структура `axicad.project.json`
```json
{
  "project_id": "8f2b3c4d-5e6f-7a8b-9c0d-1e2f3a4b5c6d",
  "camera": {
    "position": [150.0, 300.0, 500.0],
    "target": [0.0, 0.0, 0.0],
    "fov": 60.0
  },
  "placements": [
    {
      "entityPath": "shard:SensoryCortex.Retina",
      "position": [10.0, 0.0, 50.0],
      "rotation": [0.0, 0.0, 0.0, 1.0],
      "visible": true
    }
  ],
  "expanded_tree_nodes": [
    "department:SensoryCortex",
    "shard:SensoryCortex.Retina"
  ]
}
```

## 11. Состояние изменений и ревизии (Dirty State & Revisions)

Редактор отслеживает изменения на трех уровнях детализации:

- **Entity Dirty**: Флаг изменения конкретной записи `DomainRecord`. При изменении поля `data` увеличивается счетчик `revision` записи, и ее UUID добавляется в индекс `dirtyEntities`.
- **Document Dirty**: Флаг несохраненности конкретного TOML-документа.
- **Project Dirty**: Глобальный флаг несохраненных изменений всего воркспейса (требует отображения индикатора `*` в заголовке окна редактора).

### Очередь инвалидации валидации (Validation Dirty Queue)
При добавлении сущности в `dirtyEntities` запускается фоновый таймер (debounce ~300мс). После этого измененные сущности передаются в валидатор для обновления диагностик, предотвращая торможение интерфейса при непрерывном вводе данных пользователем.

## 12. Взаимодействие с Undo/Redo

Все изменения Store производятся через выполнение изолированных транзакционных команд (Command Pattern).

- **Атомарность**: Команда инкапсулирует изменение.
- **Патчи отмены (Undo Patches)**: Каждая команда при выполнении генерирует обратную операцию (inverse patch) и сохраняет её в стек Undo.
- **Инвалидация валидации**: После выполнения операций Undo или Redo затронутые сущности принудительно помечаются как `dirty` и отправляются в очередь валидации для обновления диагностических маркеров в реальном времени.

## 13. Интеграция и персистентность диагностики (Diagnostics Integration)

Диагностические ошибки и предупреждения в оперативной памяти редактора привязываются непосредственно к сессионным UUID сущностей. Однако, поскольку UUID генерируются заново при перезагрузке проекта, для сохранения диагностик редактор использует специальный механизм сериализации:

- **Стабильные ключи (Stable Keys)**:
  При сохранении проекта редактора (`axicad.project.json` или вспомогательного кэша) диагностики записываются с привязкой к стабильным текстовым ключам:
  - **Typed Entity Path**: Типизированный путь к сущности (например, `shard:SensoryCortex.Retina`).
  - **TOML Field Path**: Внутренний путь свойства внутри TOML-структуры (например, `neuron_types.Stellate_Exc.membrane.threshold`).
- **Восстановление после перезагрузки (Reattachment)**:
  При повторном открытии проекта и генерации новых UUID редактор считывает сохраненную таблицу диагностик, сопоставляет `typed entity path` с новосозданными индексами и привязывает (reattach) диагностики к новым UUID сущностей.
- **Отображение в UI**:
  При наличии у сущности диагностик с уровнем `severity = "error"` 3D-редактор подсвечивает объект красным цветом на сцене и отображает иконку предупреждения в дереве проекта.
- **Экспорт**:
  Диагностики сохраняются только в рамках сессии/метаданных проекта редактора и полностью отбрасываются при экспорте канонического TOML.

## 14. Исключения из целей спецификации (Non-goals)

- Конкретные детали шейдеров, буферов геометрии Three.js и оптимизации рендеринга.
- Детали реализации сетевых протоколов обмена с сервером AxiCAD.
- Логика расчета физических трасс аксонов (это задача сборщика Baker).
