# Спецификация режима сборки AxiCAD (Composition Workspace Spec)

> Этот документ формально описывает базовую архитектуру и поведение режима сборки (Composition Workspace) 3D-редактора AxiCAD. Документ определяет концепцию уровней композиции (Composition Levels), расчеты габаритных ограничивающих боксов (AABB), правила предотвращения коллизий, конвейеры автовыравнивания и разграничение биологических мутаций от пространственной разметки.

## Status: Draft

---

## 1. Назначение документа (Scope & Non-goals)

Composition Workspace Spec описывает логику проектирования пространственной иерархии биологической сети AxiCAD в 3D-сцене: размещение департаментов, компоновку и локальное смещение входящих в них шардов.

### Вне зоны ответственности (Non-goals)
- Документ не описывает трассировку связей ( Connectome spec) или рост аксонов ( Baker / Growth).
- Документ не задает финальный визуальный стиль UI-элементов, цвета, CSS-стили или точные размеры кнопок в пикселях.
- Документ не вносит изменений в биологический TOML DSL Axicor.

---

## 2. Терминология и иерархия (Composition Levels)

Для предотвращения путаницы в терминологии вводится жесткое разграничение:
- Термин **Layer** (Слой) относится *исключительно* к анатомическим слоям клеток внутри шарда (`[[layers]]` в `shard.toml`).
- Для обозначения пространственных плоскостей и уровней группировки департаментов в 3D-сцене используется термин **Composition Level** (Уровень композиции).

### Иерархия владения на 3D сцене:
```
World / Model (Глобальное пространство симуляции)
  └── Composition Level (Организационный уровень - editor-only)
        └── Department Container (Биологический департамент - biological)
              └── Shard Placement (Биологический шард - biological)
```

---

## 3. Источники правды (Source of Truth)

Режим Composition строго разделяет биологические и пространственные метаданные:

- **Biological Data (TOML)**: Чистые биологические данные проекта. 
  - `model.toml` владеет списком и путями департаментов (`departments`).
  - `department.toml` владеет списком и путями шардов (`shards`).
  - `shard.toml` владеет габаритами воксельной сетки (`dimensions`), анатомическими слоями (`layers`), типами нейронов (`neuron_types`), описанием сокетов (`sockets`) и биологическими настройками (`settings`).
  Запись любых биологических мутаций в Store взводит флаг `toml_documents_dirty = true` для затронутого TOML-документа.
- **Layout Metadata (JSON)**: Настройки `CompositionLevel`, пространственные координаты размещения (`position`, `rotation`), флаги блокировки (`locked`), видимости (`visible`) и свернутости (`collapsed`). Сохраняются исключительно в `axicad.project.json` и взводят флаг `project_file_dirty = true`.
- **Staging Preview**: Временные оверлеи (ghost preview) при перемещении мышью или автовыравнивании не помечают проект грязным до подтверждения (Commit) операции.

## 4. Сущности композиции (Composition Entities)

В режиме Composition вьюпорт оперирует тремя типами пространственных сущностей:

### 4.1 CompositionLevel (Уровень композиции)
Служебная плоскость/зона в 3D-пространстве (editor-only), представляющая собой логический Z-слой со своим базовым `floor Z`.
- `id`: Уникальный строковый или числовой ключ.
- `name`: Название уровня (например, "Sensory Input Level").
- `order` (u32): Порядковый индекс расположения (z-index / vertical band). Первый уровень начинается от $Z=0$.
- `visible` (Boolean): Скрывает/показывает все входящие департаменты.
- `locked` (Boolean): Блокирует перемещение любых объектов на этом уровне.
- `height` (f32): Высота уровня (vertical band) на сцене.

### 4.2 Department Container (Контейнер департамента)
Визуальный габаритный бокс, объединяющий группу шардов. Департамент строго принадлежит одному уровню композиции, размещается в плоскости XY своего уровня и занимает всю доступную высоту этого Z-слоя.
- `typedPath`: Типизированный путь к департаменту (например, `department:SensoryCortex`).
- `assignedCompositionLevel`: Ссылка на ID уровня композиции, на котором расположен департамент.
- `position` (Array of f32): Позиция департамента в глобальных world/model координатах `[X, Y, Z]`.
- `computedAABB` (AABB): Динамически вычисляемый габаритный бокс (см. раздел 8).
- `margin` (f32): Служебный отступ вокруг департамента для предотвращения слипания.

### 4.3 Shard Placement (Размещение шарда)
Физический объемный блок шарда на 3D сцене. В рамках департамента шарды размещаются как одноэтажные объемы внутри Z-слоя.
- `typedPath`: Типизированный путь к шарду (`shard:SensoryCortex.Retina`).
- `position` (Array of f32): Локальная относительная позиция шарда `[X, Y, Z]` внутри родительского департамента.
- `rotation` (Array of f32): Кватернион вращения шарда `[X, Y, Z, W]`.
- `computedAABB` (AABB): Габаритный бокс шарда на основе его биологических воксельных размеров в TOML (`dimensions.w`, `dimensions.d`, `dimensions.h`, где `w` — ширина по оси X, `d` — глубина по оси Y, `h` — высота по оси Z).

> [!NOTE]
> Спецификация файла проекта [project-file-spec-ru.md](project-file-spec-ru.md) должна быть синхронизирована с этой логикой. Ранее описанное поведение `placements.position` для шардов как глобального смещения сцены (scene/world offset) признается недостаточно точным и заменяется на локальное относительное смещение внутри родительского департамента.

## 5. Основные операции

Режим Composition поддерживает следующий набор действий пользователя:

- **Управление уровнями композиции**: Создание, переименование, удаление и изменение порядка (Reorder) уровней.
- **Групповое перемещение департаментов**: Перемещение департамента по сцене перетаскивает все его шарды как единое целое, сохраняя их относительные координаты.
- **Перемещение департамента между уровнями**: Смена привязки `assignedCompositionLevel` с сохранением или автоматической переупаковкой координат шардов.
- **Локальное перемещение шарда**: Изменение относительного смещения шарда внутри родительского департамента.
- **Изменение размеров шарда**: Команда модификации биологических параметров шарда (размеры воксельной сетки `w`, `d`, `h` в TOML) вызывает немедленный пересчет его AABB на 3D сцене.
- **Выравнивание и распределение (Align & Auto-Arrange)**: Автоматическая раскладка выбранных шардов/департаментов по сетке или алгоритму упаковки.

## 6. Правила команд и Dirty State

Все действия в режиме Composition мапятся на транзакционные команды с раздельной пометкой грязных данных:

### 6.1 Команды разметки (Layout-only)
*Модифицируют только `axicad.project.json`, взводят флаг `project_file_dirty = true`*:
- `CreateCompositionLevelCommand` / `UpdateCompositionLevelCommand` / `DeleteCompositionLevelCommand`
- `MoveDepartmentPlacementCommand` (изменяет координаты центра департамента).
- `MoveShardPlacementCommand` (изменяет относительное смещение шарда).
- `UpdateVisibilityCommand` / `UpdateLockStateCommand`
- `ApplyAutoArrangeCommand` (фиксация авторазмещения).

### 6.2 Биологические команды (Biological)
*Модифицируют биологические данные в оперативной памяти Store (на диск файлы напрямую не записываются) и взводят флаг `toml_documents_dirty = true`*:
- `CreateDepartmentCommand` / `CreateShardCommand`: Помимо добавления биологической сущности в Store, они автоматически генерируют дефолтные метаданные пространственного макета (editor layout metadata) или staging placement. Поэтому они **дополнительно** взводят флаг `project_file_dirty = true`.
- `DeleteDepartmentCommand` / `DeleteShardCommand` / `RenameDepartmentCommand` / `RenameShardCommand`: При удалении или изменении путей сущностей их связанные пространственные метаданные в макете автоматически переносятся в карантин (quarantine) или перепривязываются (relatch). Соответственно, они также **дополнительно** взводят флаг `project_file_dirty = true`.
- `UpdateShardDimensionsCommand` (изменение биологических габаритов сетки шарда `w`, `d`, `h`).

> [!NOTE]
> Чисто пространственные команды разметки (Layout-only, раздел 6.1) никогда не затрагивают биологические структуры в Store и не выставляют флаг `toml_documents_dirty = true`.


### 6.3 Жизненный цикл жеста перетаскивания (Drag Gesture Lifecycle)
1. Пользователь зажимает левую кнопку мыши на гизмо объекта и начинает перемещение.
2. Store переходит в режим **Staging**: создается временная запись `Staging Placement preview` в оверлее, оригинальный объект не изменяется, dirty-флаги не взводятся.
3. Нажатие `Escape` в любой момент сбрасывает перемещение, удаляя временный оверлей.
4. Отпускание кнопки мыши фиксирует (Commit) жест: в Store отправляется транзакция с вызовом `MoveDepartmentPlacementCommand` (или `MoveShardPlacementCommand`), сбрасывается staging и взводится `project_file_dirty = true`.

## 7. Модель выделения (Selection Model)

- **Множественный выбор**: Поддерживается выделение нескольких объектов (уровней, департаментов или шардов) через зажатый `Ctrl` или рамкой выбора во вьюпорте.
- **Групповые операции**: При выделении нескольких департаментов/шардов перемещение гизмо сдвигает все выделенные объекты синхронно.
- **Инспектор**: Правая панель отображает свойства выделенного объекта. Если выделено несколько сущностей одного типа, инспектор переходит в режим группового редактирования общих параметров (например, групповое выравнивание по Z или смена цвета оверлея `colorOverride`).

## 8. Контейнеры и трансформации (AABB Calculations)

Ограничивающие боксы (AABB) рассчитываются автоматически по иерархической схеме снизу вверх:

1. **Размер Шарда**:
   `Shard_AABB` вычисляется в микрометрах (µm) на основе воксельных размеров шарда в TOML (`dimensions.w`, `dimensions.d`, `dimensions.h`) и шага вокселя `voxel_size_um` из секции `[simulation]` файла `model.toml`:
   - $\text{Size\_X}_{\text{um}} = \text{dimensions.w} \times \text{voxel\_size\_um}$
   - $\text{Size\_Y}_{\text{um}} = \text{dimensions.d} \times \text{voxel\_size\_um}$
   - $\text{Size\_Z}_{\text{um}} = \text{dimensions.h} \times \text{voxel\_size\_um}$
   Вращение `rotation` (кватернион) применяется к вычисленному локальному боксу для построения финального AABB на сцене.
2. **Позиция Шарда на сцене**:
   Глобальное положение AABB шарда рассчитывается как сумма глобальных координат департамента и локального смещения шарда:
   $$\text{Global\_Position}_{\text{shard}} = \text{department\_layout.position} + \text{placements.position}_{\text{shard}}$$
3. **Department Container AABB**:
   Ограничивающий бокс департамента рассчитывается динамически как минимальный AABB, покрывающий все AABB его дочерних шардов, плюс внешний отступ:
   $$\text{AABB}_{\text{department}} = \text{Union}(\text{AABB}_{\text{child\_shards}}) + \text{Margin}$$
   - **Editor-Only Layout Margin/Padding**: Параметры `margin` и `padding` вокруг контейнера департамента являются исключительно служебными метаданными макета (layout metadata) в редакторе и не сохраняются как биологические отступы в TOML.
4. **Динамический пересчет и фиксированные границы**:
   - При стандартном автоматическом режиме перемещение шарда, выходящее за границы текущего AABB департамента, автоматически расширяет габариты родительского департамента и пересчитывает его `computedAABB`. В этом случае ошибка выхода за границы невозможна.
   - Ошибка **`AXI-EDITOR-004`** (Шард находится за пределами контейнера департамента) может возникать только при использовании *опциональных фиксированных/строгих границ* департамента (fixed/strict department bounds), задаваемых в макете JSON.

## 9. Политика коллизий (Collision Policy)

Режим Composition отслеживает пространственные пересечения объектов:

- **Область проверок**:
  - Пересечение контейнеров департаментов (`Department-vs-Department`) в пределах одного `Composition Level`.
  - Пересечение шардов (`Shard-vs-Shard`) в пределах одного родительского `Department`.
- **Severity**: Пересечение классифицируется как жесткая ошибка **`error`** с диагностическим кодом **`AXI-EDITOR-003`** (Hard composition collision detected). Пространственные коллизии в режиме Composition недопустимы.
- **Блокировки**: Ошибка `AXI-EDITOR-003` принудительно блокирует команды принятия/фиксации размещения (`Apply / commit`) и компиляцию Baker (`Baker compile`).
- **Сохранение черновика**: Наличие коллизий **не блокирует** сохранение служебного файла `axicad.project.json` как черновика, позволяя инженеру сохранить промежуточный прогресс раскладки.

## 10. Авторазмещение и автовыравнивание (Auto-Layout)

- **Контракт интеграции**: Вызов автовыравнивания генерирует `ImportPlan` / `PatchSet` с новыми координатами размещения.
- **Staging**: Новая раскладка отображается во вьюпорте как полупрозрачные призрачные контуры (`Generated Preview / Staging`).
- **Принятие**: Координаты переносятся в `axicad.project.json` только после нажатия кнопки "Принять", выполняющей транзакционную команду `ApplyAutoArrangeCommand`.

---

## 11. Оверлеи вьюпорта в Composition (Viewport Overlays)

В режиме Composition оверлей вьюпорта отображает вспомогательную графику (все элементы несериализуемы в TOML):
- **Level Bands**: Горизонтальные полупрозрачные плоскости, разграничивающие уровни композиции по вертикали.
- **Department AABB Boxes**: Зеленые контуры габаритов департаментов.
- **Shard AABB Boxes**: Белые контуры габаритов шардов.
- **Collision Markers**: Красная подсветка пересекающихся зон AABB.
- **Ghost Preview**: Полупрозрачные контуры объектов при перетаскивании.

## 12. Интеграция с панелями Shell (Panel Usage)

В режиме Composition слоты Shell заполняются следующим образом:
- **`shell.leftPanel`**: Дерево иерархии: Уровни Композиции $\to$ Департаменты $\to$ Шарды.
- **`shell.rightPanel`**: Инспектор свойств выделенного объекта (вкладки: размещения, размеры, видимость, diagnostics).
- **`shell.contextToolbar`**: Кнопки инструментов: "Выбрать", "Переместить", "Групповой сдвиг", "Выровнять по левому краю", "Добавить уровень", "Добавить департамент".
- **`shell.bottomPanel`**: Список коллизий и предупреждений разметки.

## 13. Сохранение (Persistence)

Спецификация [project-file-spec-ru.md](project-file-spec-ru.md) расширяется для сохранения следующих параметров Composition:

- **Канонический путь для шардов**: Для сохранения пространственного положения шардов используется глобальный канонический массив `placements` файла `axicad.project.json`. Отдельный `composition-specific` массив для шардов не заводится, чтобы избежать дублирования.
- **Связи и состояния в `department_layout`**:
  - Поле `assignedCompositionLevel` ссылается на стабильный идентификатор `id` из `composition_levels`.
  - В схему добавляются логические флаги видимости, блокировки и сворачивания (`visible`, `locked`, `collapsed`).
  - Параметр `boundsMode` принимает значения `"computed"` (динамически вычисляемый от потомков) или `"fixed"` (жесткие фиксированные размеры).
  - Опциональный объект `fixedBounds` описывает фиксированные габариты контейнера `{ size: [f32, f32, f32] }` при `boundsMode: "fixed"`.

### Схема расширения `axicad.project.json`:

```json
{
  "composition_levels": [
    {
      "id": "level_1",
      "name": "Sensory Level",
      "order": 1,
      "visible": true,
      "locked": false,
      "height": 100.0
    }
  ],
  "department_layout": [
    {
      "entityPath": "department:SensoryCortex",
      "assignedCompositionLevel": "level_1",
      "position": [0.0, 0.0, 10.0],
      "visible": true,
      "locked": false,
      "collapsed": false,
      "boundsMode": "computed",
      "fixedBounds": {
        "size": [200.0, 200.0, 100.0]
      }
    }
  ],
  "placements": [
    {
      "entityPath": "shard:SensoryCortex.Retina",
      "position": [0.0, 2.0, 0.0],
      "rotation": [0.0, 0.0, 0.0, 1.0],
      "visible": true,
      "locked": false
    }
  ]
}
```

> [!NOTE]
> Спецификация файла проекта [project-file-spec-ru.md](project-file-spec-ru.md) должна быть синхронизирована с этой схемой хранения при реализации.

## 14. Связанные диагностики (Diagnostics)

В режиме Composition генерируются следующие диагностические коды:
- **`AXI-EDITOR-003`** (`error`): Жесткое пространственное пересечение (hard collision) департаментов или шардов на сцене.
- **`AXI-EDITOR-004`** (`error`): Шард находится за пределами контейнера департамента.
- **`AXI-PROJECT-003`** (`error`): Департамент привязан к несуществующему уровню композиции (`assignedCompositionLevel` не разрешен).
- **`AXI-PROJECT-004`** (`error`): Попытка переместить заблокированную (`locked`) сущность.

## 15. Архитектурные решения (Resolved Architecture Decisions)

- **Физический смысл Z (Resolved)**: Composition Levels являются логическими Z-слоями со своим базовым `floor Z`. Первый слой начинается от $Z=0$. Департаменты и их шарды позиционируются относительно высоты своего Z-слоя.
- **Многоуровневые департаменты (Resolved)**: Департамент **не может** занимать несколько уровней композиции. Он строго принадлежит одному Z-слою, занимает XY-область этого уровня и всю доступную высоту уровня. Шарды внутри департамента располагаются как одноэтажные объемы.
