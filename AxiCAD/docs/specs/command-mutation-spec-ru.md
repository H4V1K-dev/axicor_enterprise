# Спецификация командной модели мутаций AxiCAD (Command Mutation Spec)

> Этот документ формально описывает командный слой мутаций (Command Pattern) 3D-редактора AxiCAD. Документ определяет правила внесения изменений в Store, управление транзакциями, механизм Undo/Redo, инвалидацию вычисляемых индексов и каскадное обновление зависимостей при удалении или переименовании сущностей.

## Status: Draft

---

## 1. Назначение документа

Command Mutation Spec описывает архитектуру выполнения, отмены и повтора операций изменения состояния (Store) в редакторе AxiCAD.

### Вне зоны ответственности (Non-goals)
- Документ не описывает графический интерфейс кнопок Undo/Redo и панелей UI.
- Документ не вводит новые биологические TOML-структуры данных.
- Документ не описывает процессы генерации бинарных файлов сборщиком Baker.

---

## 2. Базовые принципы (Core Principles)

- **Командный доступ (Strict Command Mutability)**: Внесение любых изменений в оперативную память Store редактора разрешено исключительно посредством создания и выполнения объектов команд. Прямое изменение записей в обход команд запрещено.
- **Атомарность (Atomicity)**: Каждая команда выполняется как транзакционная единица. Если в процессе выполнения происходит сбой, все затронутые срезы состояния (state slices) Store должны быть полностью возвращены в исходное состояние (Rollback).
- **Инвалидация dirty-состояния**: Команда явно указывает характер своего воздействия на файлы проекта (раздельное выставление флагов `project_file_dirty` и `toml_documents_dirty`).
- **Связность индексов (Index Consistency)**: Команда берет на себя ответственность за инвалидацию и перестроение всех зависимых и вычисляемых индексов в Store.
- **Семантическая эквивалентность при Undo/Redo (Semantic Equivalence)**: Операции отмены (`undo`) и повтора (`redo`) должны восстанавливать состояние Store с обеспечением семантической эквивалентности данных. Счетчик ревизий `revision`, очередь валидации `validationQueue` и диагностики `diagnostics` могут принудительно инвалидироваться и полностью перевычисляться после выполнения `undo` или `redo`.

## 3. Интерфейс Команды (Command Interface)

Каждое действие пользователя, изменяющее данные, оборачивается в объект с TypeScript-контрактом `Command`:

```typescript
import { UUID, DomainRecord } from './editor-store-spec-ru';

export interface Command {
  /** Уникальный UUID экземпляра команды */
  readonly id: UUID;
  
  /** Человекопонятное имя операции для отображения в стеке Undo/Redo */
  readonly label: string;
  
  /** Опциональный типизированный путь области видимости операции (например, "shard:SensoryCortex.Retina") */
  readonly scopeTypedPath?: string;
  
  /** Системное время выполнения команды */
  readonly timestamp: number;
  
  /** Опциональный ключ для группировки частых операций (например, "drag-placement-shard-Retina") */
  readonly mergeKey?: string;

  /**
   * Выполнение команды. Возвращает true в случае успеха
   */
  execute(ctx: MutationContext): boolean;

  /**
   * Отмена изменений, внесенных командой (генерация inverse patch)
   */
  undo(ctx: MutationContext): boolean;

  /**
   * Повторное выполнение отмененной команды
   */
  redo(ctx: MutationContext): boolean;

  /** Список UUID сущностей, затронутых этой командой (для точечной перевалидации) */
  getAffectedEntityIds(): UUID[];

  /** Список типизированных путей, затронутых командой (для инвалидации индексов) */
  getAffectedTypedPaths(): string[];
}
```

## 4. Контекст мутаций (Mutation Context)

При выполнении команда получает объект `MutationContext`, предоставляющий контролируемый доступ к системным API редактора:

```typescript
export interface MutationContext {
  /** Ссылка на плоский реестр сущностей Store (byId) */
  readonly store: Map<UUID, DomainRecord>;
  
  /** Глобальные индексы (byType, byTypedPath, childrenByParent) */
  readonly indexes: StoreIndexes;
  
  /** Очередь инвалидации валидатора (для постановки dirtyEntities на перепроверку) */
  readonly validationQueue: {
    enqueue(entityIds: UUID[]): void;
  };
  
  /** Текущие флаги dirty_state проекта */
  readonly projectDirtyState: {
    markProjectDirty(): void;
    markTomlDocumentDirty(tomlPath: string): void;
  };
  
  /** Утилиты генерации стабильных сессионных UUID и меток времени */
  readonly utils: {
    generateUUID(): UUID;
    getCurrentTimestamp(): number;
  };
}
```

## 5. Категории команд

Система мутаций AxiCAD оперирует фиксированным набором базовых команд:

1. **CreateEntityCommand**: Добавляет новую запись `DomainRecord` в Store, обновляет `childrenIds` родителя, перестраивает индекс `childrenByParent` и `byTypedPath`.
2. **DeleteEntityCommand**: Удаляет запись. Запускает каскадное удаление (cascade delete) потомков и переносит осиротевшие метаданные размещения в карантин.
3. **RenameEntityCommand**: Изменяет биологическое свойство `data.name`. Инициирует пересчет `path` и `byTypedPath` для самой сущности и всех её потомков.
4. **UpdateTomlFieldCommand**: Точечно изменяет значение поля внутри `data` (например, порог мембраны). Инкрементирует ревизию сущности.
5. **MoveReparentEntityCommand**: Переносит сущность к новому родителю (изменяет `parentId`, обновляет списки `childrenIds`).
6. **CreateConnectionCommand / DeleteConnectionCommand**: Добавляет/удаляет запись `connection` в соответствующий массив.
7. **UpdateEditorPlacementCommand**: Изменяет пространственные координаты (`position`, `rotation`) размещения шарда в файле проекта `axicad.project.json`.
8. **UpdateSelectionCommand**: Изменяет состояние выделения UI/проекта.

## 6. Атомарность и Транзакции

Выполнение мутаций внутри Store инкапсулируется в транзакции. Схема жизненного цикла транзакции:

1. **Begin (Создание снимка)**:
   Для обеспечения атомарности создается резервная копия (snapshot) всех затронутых срезов состояния (state slices):
   - Записи Store (`store`).
   - Индексы (`byTypedPath`, `childrenByParent`, `socketByDepartmentEndpoint`, `socketByModelEndpoint`).
   - Флаги `projectDirtyState` (включая `toml_documents_dirty`).
   - Очередь `validationQueue` и побочные эффекты диагностик.
   - Метаданные `migrations.history` и карантин `orphaned_metadata` / `stale_entries`.
2. **Apply**: Команда вносит изменения в поля `data` или `meta` соответствующих `DomainRecord`.
3. **Rebuild Indexes**: Запускается инкрементальное обновление затронутых индексов (`byTypedPath`, `socketByDepartmentEndpoint`, `socketByModelEndpoint`).
4. **Enqueue Validation**: UUID измененных сущностей помещаются в `dirtyEntities` для фонового дебаунс-запуска валидации.
5. **Commit / Rollback**: 
   - Если все шаги execute прошли без ошибок $\to$ транзакция фиксируется (Commit), а команда помещается в стек Undo.
   - Если произошел сбой (например, сбой при reparent, создающий циклическую ссылку) $\to$ Store полностью восстанавливается из резервного снимка по всем срезам состояния (Rollback), а команда отклоняется с выводом ошибки.

## 7. Модель Undo / Redo

История изменений хранится в виде двух независимых стеков: стек отмены (`Undo Stack`) и стек повтора (`Redo Stack`).

- **Поведение при новой мутации**: Выполнение любой новой команды в редакторе (не Undo/Redo) полностью очищает `Redo Stack`.
- **Слияние команд (Merge Policy)**: 
  При выполнении частых непрерывных действий (например, перемещение мыши при перетаскивании шарда, изменение ползунка размера в UI) редактор не создает сотни отдельных команд. Если у текущей команды совпадает `mergeKey` с командой на вершине Undo-стека, и разница во времени `timestamp` $\le 500$мс, новая команда объединяется с предыдущей (обновляются конечные параметры выполнения в execute, а undo сохраняет изначальную точку старта).

## 8. Интеграция с Dirty State

Команды явно разделяют свое влияние на флаги изменений:

- **Изменение TOML (`toml_documents_dirty`)**:
  Если команда модифицирует биологические данные `data` в `BiologicalEntityType` сущности (например, `UpdateTomlFieldCommand` или `CreateConnectionCommand`), она обязана добавить относительный путь целевого TOML-файла в массив `toml_documents_dirty` через `projectDirtyState`.
- **Изменение проекта (`project_file_dirty`)**:
  Если команда модифицирует исключительно визуальные метаданные `meta`, размещение шардов (`UpdateEditorPlacementCommand`) или состояние вьюпорта, она выставляет флаг `project_file_dirty = true`. Это исключает избыточный экспорт TOML при обычном перемещении камер или выделении объектов.
- **Выделение элементов (Selection)**:
  Команда `UpdateSelectionCommand` изменяет только глобальный стейт выделения UI/проекта. Она выставляет флаг `project_file_dirty = true`, но никогда не помечает `toml_documents_dirty` и не ставит сущности в очередь биологической валидации (за исключением явных случаев, когда выделение требует пересчета диагностик отображения в UI).

## 9. Инвалидация вычисляемых индексов (Derived Indexes)

При изменении Store индексы обновляются инкрементально для предотвращения падения производительности:

- **`byTypedPath`**: При изменении имени или перемещении сущности старый typed path удаляется из индекса, а новый записывается.
- **`childrenByParent`**: Перестраивается только для UUID старого родителя и UUID нового родителя.
- **`socketByDepartmentEndpoint` / `socketByModelEndpoint`**: Перестраиваются при переименовании шардов, сокетов или департаментов. Все соединения, чьи строковые эндпоинты совпали с изменившимся путем, помечаются как `dirty` для перепроверки Baker-линковщиком.

## 10. Поведение при переименовании (Rename Behavior)

Команда `RenameEntityCommand` запускает каскадный процесс обновления:

1. **Изменение `data.name` у целевой сущности**.
2. **Рекурсивное обновление `path` и `byTypedPath` у сущности и всех её потомков**.
3. **Сохранение топологии сети**:
   - При инициированном редактором переименовании (`editor-initiated rename`) автоматически обновляются строковые пути `connections.from` и `connections.to` во всех затронутых соединениях для сохранения топологии сети.
   - Все файлы конфигурации TOML, содержащие эти измененные соединения, помечаются как `toml_documents_dirty`.
4. **Обработка внешних нестыковок (import mismatch)**:
   - Если переименование произошло на диске извне, или обнаружена нестыковка импорта, редактор не перезаписывает пути автоматически. Вместо этого генерируется диагностическая ошибка разрешения путей, требующая явного согласия пользователя на обновление или удаление связей.

---

## 11. Поведение при удалении (Delete Behavior)

Удаление биологической сущности подчиняется каскадным правилам:

- **Рекурсивное удаление (Cascade Delete)**: Удаление департамента или шарда автоматически рекурсивно удаляет из Store все входящие в них дочерние сущности (слои, сокеты, пины) для предотвращения утечек памяти в Store.
- **Обработка висящих ссылок (Dangling References)**: 
  - Соединения (`connections`), которые ссылались на удаленные сокеты, остаются в Store, но их эндпоинты не разрешаются (`null`). 
  - Валидатор Tier 4 вешает на эти соединения ошибку `AXI-BAKER-001`.
  - Все файлы конфигурации TOML, содержащие эти поврежденные соединения, помечаются как `toml_documents_dirty`.
- **Карантин метаданных**: Размещение удаленного шарда в `axicad.project.json` переносится в секцию карантина `orphaned_metadata`.

## 12. Интеграция диагностики (Diagnostics Integration)

Команды не занимаются расчетом биологических или топологических ошибок самостоятельно (это задача выделенных валидаторов).

- **Роль команды**: Команда лишь фиксирует факт изменений, инкрементирует `revision` сущности и помещает UUID измененных и связанных объектов в очередь `dirtyEntities` валидатора.
- **Исключение**: Прямая запись ошибок в `diagnostics` команды допускается только для мгновенных структурных сбоев в контексте транзакции (например, при попытке циклического reparent-перемещения шарда самого в себя).
