# Спецификация валидации Axicor/AxiCAD (Validation Spec)

> Этот документ формально определяет многоуровневую систему валидации конфигураций Axicor и интерактивного стейта редактора AxiCAD. Документ классифицирует типы проверок, зоны ответственности компонентов, жизненный цикл валидации и модель вывода диагностических сообщений.

## Status: Draft

---

## 1. Назначение документа

Validation Spec служит для фиксации и распределения всех правил проверки биологических сетей Axicor. 

### Вне зоны ответственности (Non-goals)
- Документ не описывает новые структуры данных или схемы TOML (они определены в `toml-schema-spec-ru.md`).
- Документ не детализирует разметку UI-компонентов редактора AxiCAD.
- Документ не описывает внутреннее поведение симулятора во время выполнения (runtime simulation behavior).

---

## 2. Уровни валидации (Validation Tiers)

Вся система проверок разделена на 4 изолированных уровня валидации, каждый из которых запускается на своем этапе жизненного цикла и выполняется назначенным компонентом:

```
[ TOML File / UI State ]
           │
           ▼
┌───────────────────────────────┐
│ Tier 1: Serde/Rust contract   │ ──► Ошибки структуры и типов (Rust Serde)
└──────────┬────────────────────┘
           │
           ▼
┌───────────────────────────────┐
│ Tier 2: Current Rust validator│ ──► Локальные семантические инварианты (dev-rust-api)
└──────────┬────────────────────┘
           │
           ▼
┌───────────────────────────────┐
│ Tier 3: AxiCAD editor check   │ ──► Интерактивные правила и UX (3D-редактор)
└──────────┬────────────────────┘
           │
           ▼
┌───────────────────────────────┐
│ Tier 4: Baker resolver / ABI  │ ──► Глобальная связность и аппаратные лимиты (baker-cli)
└───────────────────────────────┘
```

---

## 3. Tier 1: Serde/Rust contract checks

Проверки первого уровня выполняются автоматически библиотекой `serde` в Rust при попытке десериализации файлов конфигурации TOML. Нарушение любого из правил этого уровня приводит к **фатальной ошибке десериализации** (разбор файла прекращается).

- **Обязательность полей (Required Fields)**: Наличие всех полей в структурах `ModelConfig`, `WorldConfig`, `SimulationParams`, `DepartmentConfig`, `ShardConfig`, `LayerConfig`, `NeuronType`, `ShardSettings`.
- **Типовая корректность (Type Correctness)**:
  - Координаты и размеры (`w`, `d`, `h`, `width`, `height`, `growth_steps`, `local_u` и т.д.) должны соответствовать числовым типам (signed/unsigned integer, float).
  - Свойства сида и путей должны быть строго строковыми.
- **Строгие строчные перечисления (Enums)**:
  - Направление связи `direction` в сокетах (`SocketConfig`) и портах (`PortConfig`) принимает строго значения `"in"` или `"out"`.
  - Тип высотной привязки аксонов `entry_z` принимает строго значения `"Top"`, `"Mid"`, `"Bottom"`.
- **Запрет неизвестных полей (Deny Unknown Fields)**: Любое недокументированное свойство в TOML-файле вызывает немедленный сбой.
- **Диапазоны числовых типов**: Недопущение переполнения лимитов базовых типов данных (`u8` для `max_dendrites` и рефрактерных периодов, `u32`/`u64` для тиков).

## 4. Tier 2: Current Rust validator checks

Семантические инварианты, проверяемые методами `validate_model`, `validate_department` и `validate_shard` в вычислительном ядре симулятора на Rust (`dev-rust-api/config`). Вызываются после успешной десериализации TOML-структур.

### 4.1 Проверки model.toml (validate_model)
- **Параметры тиков**: Длительность `tick_duration_us` должна быть строго $> 0$.
- **Размер вокселя**: Значение `voxel_size_um` должно быть строго $> 0.0$.
- **Скорость сигнала**: Скорость `signal_speed_m_s` должна быть строго $> 0.0$.
- **Длина сегмента**: Шаг `segment_length_voxels` должен быть строго $> 0$.
- **Дискретная скорость (`INV-CONFIG-003`)**: Отношение скорости сигнала за тик к физическому размеру сегмента аксона ($v_{seg} = \frac{speed\_um\_tick}{segment\_length\_um}$) должно быть строго целым числом (с погрешностью $\le 10^{-5}$).
- **Предельные шаги роста (`INV-CONFIG-005`)**: Глобальное число шагов роста аксонов `axon_growth_max_steps` должно быть строго $\le 255$.
- **Лимит дендритов**: Настройка `max_dendrites` должна быть строго равна `128` (аппаратная константа сомы).
- **Сид**: Строка `master_seed` не должна быть пустой.
- **Размеры мира**: Габариты `width_um`, `depth_um`, `height_um` должны быть строго $> 0.0$.

### 4.2 Проверки department.toml (validate_department)
- **Имена шардов**: Свойство `name` каждого шарда не должно быть пустым.
- **Конфигурации шардов**: Путь `config` к файлу конфигурации шарда не должен быть пустым.
- **Уникальность шардов**: Не допускаются дублирующиеся имена шардов в списке `[[shards]]`.
- **Соединения**: Поля `from` и `to` не должны быть пустыми и должны строго соответствовать точечному формату `ShardName.SocketName` (состоять ровно из 2 текстовых сегментов).

### 4.3 Проверки shard.toml (validate_shard)
- **Размеры шарда**: Размеры должны быть в диапазонах: `0 < w <= 1023`, `0 < d <= 1023`, `0 < h <= 255` (ограничение PackedPosition).
- **Лимит типов нейронов (`INV-CONFIG-001`)**: Список `[[neuron_types]]` должен содержать **не более 16** типов клеток.
- **Имена типов нейронов**: Имя `name` типа нейрона не должно быть пустым. Не допускаются дублирующиеся типы клеток.
- *Примечание*: Rust-валидатор не проверяет список `neuron_types` на обязательную непустоту напрямую. Однако валидный состав слоев `layers.composition` фактически требует наличия описанных типов клеток через ссылки `type_name`.
- **Параметры биологии нейронов**:
  - Рефрактерный период `refractory_period` должен быть $> 0$.
  - Длина распространения спайка `signal_propagation_length` должна быть $> 0$.
  - **Длина спайка и рефрактерность (`INV-CONFIG-004`)**: Длина спайка должна быть строго больше или равна рефрактерному периоду: `signal_propagation_length >= refractory_period`.
  - **Кривая инерции**: Вектор `inertia_curve` в `gsop` должен содержать строго **8 элементов**.
- **Анатомические слои**:
  - Относительная высота слоя `height_pct` должна быть строго $> 0.0$.
  - Плотность слоя `density` должна быть строго $\ge 0.0$ (`INV-CONFIG-002`).
  - Доля распределения клеток в слое `share` должна быть строго $\ge 0.0$.
  - Имя `type_name` в составе слоя должно ссылаться на реально существующий `neuron_types` этого шарда.
  - **Слои (высота)**: Сумма высот всех слоев `layers.height_pct` должна быть строго равна `1.0` (±1e-4).
  - **Слои (состав)**: Сумма долей клеток `composition.share` внутри каждого слоя должна быть строго равна `1.0` (±1e-4).
- **Сокеты и порты**:
  - Имена сокетов и портов не должны быть пустыми (`!is_empty()`).
  - **Ghost VRAM**: Если у шарда есть хотя бы один входящий сокет (`direction = "in"`), то лимит VRAM `settings.ghost_capacity` должен быть строго $> 0$.
  - Имена пинов внешних портов не должны быть пустыми.
  - **Границы проецирования пинов**: Координаты пина должны лежать в диапазоне `0.0..1.0`. Общая область проекции пина не должна выходить за границы грани шарда: `local_u + u_width <= 1.0 + 1e-5` и `local_v + v_height <= 1.0 + 1e-5`.

## 5. Tier 3: AxiCAD editor validator checks

Проверки, выполняемые интерактивным слоем 3D-редактора AxiCAD. Позволяют предупредить пользователя о логических или пространственных нестыковках до экспорта проекта в TOML или запуска Baker.

- **Валидация имен**:
  - Проверка соответствия имен шардов, департаментов, сокетов, портов и типов нейронов регулярному выражению `^[a-zA-Z0-9_-]+$`.
- **Уникальность департаментов**:
  - Контроль уникальности имен департаментов `name` в списке `[[departments]]` файла `model.toml` (не проверяется Rust-валидатором).
- **Связанность с диском (config paths)**:
  - Проверка, что относительные пути к TOML-файлам в полях `config` указывают на существующие файлы на диске.
- **Интерактивные предупреждения (Warnings)**:
  - Предупреждения о зазорах или погрешностях округления (gaps/rounding artifacts) после перевода относительной высоты слоев `height_pct` в дискретные воксельные диапазоны высот по оси Z.
  - Проверка пространственных коллизий (физические AABB-коробки шардов не должны пересекаться в глобальном 3D-пространстве сцены редактора). *Примечание*: Данная проверка относится исключительно к метаданным размещения редактора (`axicad.project.json`) и не является частью биологического TOML DSL.
- **Жизненный цикл изменений (Unsaved changes)**:
  - Контроль несохраненного состояния (`dirty state`) проекта.
- **Подсказки и исправления**:
  - Вывод дружелюбной графической разметки ошибок на 3D-сцене (например, подсветка непересекающихся или висящих в воздухе шардов).

## 6. Tier 4: Baker resolver / ABI checks

Глобальные проверки компилятора топологии (`baker-cli`) на этапе сборки и компоновки (AOT Linker). Эти проверки оценивают корректность связей всей сети и аппаратные ограничения бинарного рантайма GPU.

- **Разрешение путей (Endpoint Resolution)**:
  - Успешное разрешение каждого пути соединения в `connections` в `department.toml` (`ShardName.SocketName`) и в `model.toml` (`DepartmentName.ShardName.SocketName`) в реально существующие сокеты.
- **Существование сокетов**:
  - Проверка физического наличия сокетов источника и цели в соответствующих файлах `shard.toml`.
- **Направление сокетов (Socket direction)**:
  - `source endpoint must be out` (источник соединения должен быть строго исходящим сокетом).
  - `target endpoint must be in` (цель соединения должна быть строго входящим сокетом).
  - `reverse endpoint direction is invalid` (некорректное встречное направление сокетов на концах соединения).
  - *Примечание*: Данное правило не запрещает формирование петель обратной связи (feedback loops) на уровне биологической топологии графа сети.
- **Совпадение габаритов сокетов**:
  - Ширина (`width`) и высота (`height`) сокета источника должны строго совпадать с аналогичными размерами сокета цели.
- **Локальные шаги роста**:
  - Проверка, что параметр `growth_steps` для локального сокета (`SocketConfig`) и пина (`PinConfig`) не превышает `255` (8-битный лимит дескриптора роста).
- **Граничные ограничения адресации PackedPosition**:
  - Проверка, что абсолютные воксельные координаты размещенных Baker-ом нейронов не выходят за физические пределы GPU-адресации.
- **Проверка объема памяти VRAM (Ghost Capacity)**:
  - Емкость `settings.ghost_capacity` шарда-приемника должна быть больше или равна сумме площадей (`width * height`) всех входящих в его сокеты соединений.
- **Компиляция пинов внешних портов**:
  - Проверка уникальности и расчет хэшей имен пинов (`pin_hash = fnv1a32(pin_name)`).
  - Валидация значений `empty_pixel` (разрешены только `"skip"` или `"zero"`).

## 7. Модель вывода ошибок (Error Model)

Диагностика системы валидации должна выводиться в унифицированном машиночитаемом и человекопонятном формате.

### 7.1 Уровни серьезности (Severity)
- **`error`**: Фатальная ошибка. Блокирует симуляцию, экспорт TOML или сборку Baker.
- **`warning`**: Нефатальное предупреждение. Симуляция может быть запущена, но возможны биологические или визуальные аномалии (например, зазоры в слоях шарда).
- **`info`**: Информационное сообщение (подсказки по оптимизации VRAM или шагов роста).

### 7.2 Формат кодов ошибок
Код имеет структуру: `AXI-{TIER}-{NUMBER}`, где `TIER` указывает на источник ошибки:
- `AXI-SERDE-XXX`: Ошибки синтаксиса и десериализации Serde.
- `AXI-VALID-XXX`: Нарушения семантических правил Rust-валидатора.
- `AXI-EDITOR-XXX`: Интерактивные предупреждения и ошибки 3D-редактора.
- `AXI-BAKER-XXX`: Топологические и ABI-ограничения Baker.

### 7.3 Структура диагностического сообщения (JSON-контракт)
```json
{
  "code": "AXI-VALID-004",
  "severity": "error",
  "message": "INV-CONFIG-004: signal_propagation_length (5) must be >= refractory_period (8)",
  "path": "neuron_types.Stellate_Exc.signal.signal_propagation_length",
  "suggested_fix": "Увеличьте значение 'signal_propagation_length' до 8 или более, чтобы покрыть рефрактерный период клетки."
}
```

## 8. Жизненный цикл валидации (Validation Lifecycle)

Выполнение проверок распределено по этапам жизненного цикла проектирования:

1. **При загрузке файлов (On File Load)**:
   - Выполняются проверки **Tier 1** (Serde) и **Tier 2** (Rust validator).
   - Режим только для чтения (Read-only mode) активируется только в том случае, если файлы TOML физически повреждены и их невозможно безопасно распарсить или автоматически восстановить.
2. **В процессе редактирования (During Editing)**:
   - Редактор AxiCAD интерактивно запускает проверки **Tier 3** (Editor checks) в фоновом режиме.
   - Показываются предупреждения и маркеры ошибок на 3D-сцене (AABB коллизии, неверные форматы имен).
3. **При сохранении проекта редактора (Save editor project)**:
   - Сохранение проекта разрешено при любых семантических и топологических ошибках, при этом сохраняются метаданные проекта, текущие диагностики и dirty state, гарантируя сохранность работы пользователя.
4. **При экспорте в TOML или запуске Baker (Export to TOML / Baker compile)**:
   - Операции экспорта биологических файлов или компиляции полностью блокируются при наличии ошибок с уровнем серьезности **`error`** (severity = error).
5. **После Baker-компиляции (After Baker Compile)**:
   - Анализ бинарных логов Baker-а. Если компиляция завершилась сбоем, ошибки парсятся и мапятся обратно на элементы 3D-сцены редактора AxiCAD.
