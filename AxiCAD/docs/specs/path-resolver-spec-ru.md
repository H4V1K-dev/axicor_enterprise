# Спецификация путей и резолвера ссылок AxiCAD (Path Resolver Spec)

> Этот документ формально описывает все виды путей, адресации и ссылок, применяемых в 3D-редакторе AxiCAD и биологическом TOML DSL Axicor. Документ регламентирует правила нормализации, жизненный цикл разрешения (Resolution) ссылок в Store и обработку возникающих ошибок.

## Status: Draft

---

## 1. Назначение документа

Path Resolver Spec регламентирует механизмы адресации биологических сущностей и связывания их с визуальными метаданными среды проектирования. Документ служит архитектурным контрактом для реализации навигации, линковки соединений и каскадных обновлений при модификации структуры проекта.

### Вне зоны ответственности (Non-goals)
- Документ не приводит программную реализацию парсеров путей на TypeScript или Rust.
- Документ не описывает графический интерфейс панелей навигации и дерева UI.

## 2. Типы путей (Path Types)

Система AxiCAD оперирует четырьмя уровнями адресации:

### 2.1 Filesystem Config Paths (Файловые пути)
- **Описание**: Относительные пути, прописанные в полях `config` TOML-документов для связывания файлов.
- **Правила разрешения (Resolution Rules)**:
  - Путь `config` в `model.toml` резолвится относительно директории расположения `model.toml`.
  - Путь `config` в `department.toml` резолвится относительно директории расположения этого конкретного `department.toml`.
  - После успешного разрешения (resolution) путь нормализуется в **канонический путь относительно корня проекта** (`canonical project-root-relative path`). Именно этот нормализованный путь используется в качестве ключа в индексах `configPathToEntity` / `entityToConfigPath`.
  - **Ограничения безопасности**: Использование абсолютных путей на диске и выход за пределы корня проекта через относительные переходы `..` строго запрещены (вызывают ошибку разбора Tier 1).

### 2.2 Typed Entity Paths (Типизированные пути Store)
- **Описание**: Стабильные сессионные пути редактора, однозначно идентифицирующие сущность в Store. Не сохраняются в TOML, но служат ключами связывания метаданных в `axicad.project.json`.
- **Синтаксис**: `<entity_type>:<domain_path>`
- **Примеры**:
  - `model:model` (корневая модель)
  - `department:SensoryCortex` (департамент)
  - `shard:SensoryCortex.Retina` (шард)
  - `socket:SensoryCortex.Retina.cross_modal` (сокет)

### 2.3 Connection Endpoints (Строковые адреса связей)
- **Описание**: Строковые ссылки, указываемые в полях `from` и `to` массивов `[[connections]]` биологического описания TOML.
- **Синтаксис**: Зависит от контекста файла (см. раздел 3).
- **Примеры**: `Retina.cross_modal` (внутридепартаментный), `SensoryCortex.Retina.cross_modal` (междепартаментный).

### 2.4 Session UUID (Идентификаторы сессии)
- **Описание**: Временные уникальные UUID, генерируемые в оперативной памяти Store при старте сессии редактора. Служат для обеспечения реактивных связей в рантайме. Никогда не сохраняются на диске ни в TOML, ни в JSON проекта.

## 3. Границы владения (Ownership Boundaries)

Структура биологических связей подчиняется строгой иерархии:

```
┌────────────────────────────────────────────────────────┐
│                      model.toml                        │
│  - Владеет: [[departments]]                            │
│  - Владеет: [[connections]] (Междепартаментные связи)  │
└──────────────────────────┬─────────────────────────────┘
                           │
                           ▼
┌────────────────────────────────────────────────────────┐
│                   department.toml                      │
│ - Владеет: [[shards]]                                  │
│ - Владеет: [[connections]] (Внутридепартаментные связи)│
└──────────────────────────┬─────────────────────────────┘
                           │
                           ▼
┌────────────────────────────────────────────────────────┐
│                      shard.toml                        │
│  - Владеет: [[layers]], [[neuron_types]]               │
│  - Владеет: [[sockets]], [[ports]]                     │
└────────────────────────────────────────────────────────┘
```

- **Междепартаментные связи (Model-Level)**: Ссылки, соединяющие сокеты шардов, находящихся в *разных* департаментах. Описываются исключительно в корневом `model.toml`.
- **Внутридепартаментные связи (Department-Level)**: Ссылки, соединяющие сокеты шардов в пределах *одного* департамента. Описываются исключительно в соответствующем `department.toml`.
- **Внутреннее владение**: Шард (`shard.toml`) безраздельно владеет своими сокетами (`sockets`) и портами (`ports`).
- **Роль портов**: Порты (`ports`) являются внешними интерфейсами ввода/вывода (I/O) шарда. Согласно TOML-схеме, порты не участвуют в биологических соединениях `connections` напрямую (они служат внешними точками подключения к окружению).

## 4. Грамматика эндпоинтов (Grammar Endpoints)

Строковые адреса в соединениях `connections` парсятся в зависимости от их уровня размещения:

### 4.1 Внутридепартаментные соединения (Department-Level)
Поскольку контекст ограничен текущим департаментом, указание имени департамента избыточно.
- **Грамматический шаблон**: `<ShardName>.<SocketName>`
- **Пример**: `from = "Retina.output"`, `to = "Auditory.cross_feed"`

### 4.2 Междепартаментные соединения (Model-Level)
Поскольку связи соединяют сущности из разных веток иерархии, требуется указание имени департамента.
- **Грамматический шаблон**: `<DepartmentName>.<ShardName>.<SocketName>`
- **Пример**: `from = "SensoryCortex.Retina.output"`, `to = "Thalamus.RelayShard.input"`

> [!NOTE]
> **Открытое архитектурное решение (Open Decision)**:
> Точная расширенная грамматика эндпоинтов (например, поддержка масок `*` для многоадресного вещания соединений или относительных переходов типа `super.Shard`) в настоящее время не финализирована. В текущей версии поддерживаются только жесткие статические пути без использования спецсимволов.

## 5. Конвейер разрешения путей (Resolver Pipeline)

Разрешение (Resolution) строковой ссылки до конкретного UUID объекта в памяти выполняется в 6 шагов:

```
┌───────────────────┐
│ 1. Parse          │ Parse raw string (выделение токенов пути)
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│ 2. Normalize      │ Очистка от лишних пробелов, проверка регистра
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│ 3. Scope Context  │ Определение контекста (Model-level или Department-local)
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│ 4. Index Lookup   │ Поиск UUID сокета в socketByDepartment или socketByModel
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│ 5. Bind UUID      │ Привязка UUID в runtime-кэш DomainRecord в Store
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│ 6. Diagnostics    │ При неудаче: генерация AXI-BAKER-001 (Dangling connection)
└───────────────────┘
```

### 5.1 runtime-привязка UUID (Bind UUID)
- В каноническую биологическую TOML-схему соединений **запрещено добавлять** постоянные UUID-поля (такие как `targetUuid`).
- Связывание с физическим UUID сокета в Store описывается исключительно как **вычисляемый runtime-кэш резолвера** (derived resolver cache) внутри DomainRecord соединения в Store (например, через runtime-свойства `resolvedFromSocketId` и `resolvedToSocketId`).
- Эти свойства хранятся в оперативной памяти в сессии редактора и никогда не экспортируются обратно в TOML при сохранении проекта.

## 6. Вычисляемые индексы (Derived Indexes)

Резолвер опирается на инкрементальные индексы Store для обеспечения высокой производительности поиска (сложность $O(1)$ вместо $O(N)$):

- **`byTypedPath: Map<string, UUID>`**: Сопоставляет уникальный типизированный путь (например, `shard:SensoryCortex.Retina`) с UUID записи.
- **`socketByDepartmentEndpoint: Map<string, UUID>`**: 
  - **Ключ**: `Department_UUID + ":" + ShardName + "." + SocketName`
  - **Назначение**: Используется для разрешения внутридепартаментных соединений (department-local connections) из `department.toml`.
- **`socketByModelEndpoint: Map<string, UUID>`**: 
  - **Ключ**: `DepartmentName + "." + ShardName + "." + SocketName`
  - **Назначение**: Используется для разрешения междепартаментных соединений (model-level connections) из `model.toml`.
- **`configPathToEntity / entityToConfigPath`**: Карты взаимного сопоставления файловых путей и UUID департаментов/шардов.

### Жизненный цикл инвалидации
Индексы инвалидируются и перестраиваются в транзакции при следующих мутациях:
- **Create (Создание)**: При создании сущности через команду в соответствующие индексы добавляются новые typed paths, config path mappings и socket endpoint keys для всех затронутых сущностей.
- **Rename (Переименование)**: Удаляются старые пути, записываются новые, перестраиваются все дочерние пути в дереве.
- **Delete (Удаление)**: Рекурсивное удаление сущностей из всех индексов.
- **Move (Перемещение)**: Пересчет родительского префикса.
- **Import / Accepted PatchSet (Импорт)**: После подтверждения и применения `ImportPlan` или `PatchSet` через command transaction индексы Store обновляются точно так же, как при наборе команд Create/Rename/Delete внутри транзакции.
- **External Reload (Внешнее изменение)**: Полная очистка и перестройка всех индексов с нуля.

## 7. Обработка ошибок разрешения путей (Error Behavior)

Все ошибки разрешения путей собираются в кэш диагностик. Сами по себе они не помечают TOML как dirty.

| Тип ошибки | Серьезность | Описание | Поведение системы |
|---|---|---|---|
| `Missing config file` | `error` | TOML-файл департамента/шарда отсутствует по пути `config`. | Вывод диагностики. Запуск политики **Degraded/Read-only** (проект загружается с отключением редактирования битых путей, в соответствии с `import-export-serialization-spec-ru.md`). |
| `Duplicate typed path` | `error` | Два элемента получили одинаковое имя в одной ветке. | Блокирует применение изменений через команды/импорт (`command/import apply`) или экспорт TOML / компиляцию Baker. **Разрешено** сохранение служебного файла `axicad.project.json`. |
| `Dangling connection` | `error` | Строковый эндпоинт соединения не найден ни в одном индексе. | Диагностика ошибки **AXI-BAKER-001**. Блокирует компиляцию Baker и канонический экспорт TOML на диск, но не выставляет `dirty` состояние TOML автоматически. |
| `Ambiguous endpoint` | `error` | Адрес указывает на несколько конфликтующих сокетов. | Блокировка Baker-компиляции и канонического экспорта TOML. Генерируется ошибка разрешения. |
| `Stale placement metadata` | `warning` | Метаданные размещения в JSON проекта ссылаются на отсутствующий путь. | Перенос метаданных в зону карантина `orphaned_metadata` без вызова блокировок. |

## 8. Взаимодействие с Rename / Delete / Import

- **Переименование в редакторе (Editor-Initiated Rename)**:
  Команда `RenameEntityCommand` обновляет имя, автоматически перевычисляет `byTypedPath` и обновляет строковые пути во всех затронутых соединениях `connections.from/to`. TOML-документы этих соединений помечаются как `toml_documents_dirty`.
- **Внешнее нестыковка на диске (External Mismatch)**:
  При обнаружении нестыковки путей при внешнем изменении файлов на диске редактор не перезаписывает данные автоматически. Генерируется ошибка разрешения, и пользователю предлагается `Suggested Repair PatchSet` (набор патчей исправления).
- **Импорт фрагментов (Import Plan)**:
  Любые импортируемые рецепты сначала формируют `ImportPlan` / `PatchSet`. Резолвер проверяет применимость всех путей PatchSet к текущему Store до выполнения транзакции записи.
