# Каталог диагностик и спецификация ошибок AxiCAD (Diagnostics & Error Catalog Spec)

> Этот документ формально описывает единый контракт диагностических сообщений, правила привязки ошибок к Store и файловой системе, матрицу блокирующего поведения для различных операций, жизненный цикл валидации и начальный реестр кодов ошибок AxiCAD/Axicor.

## Status: Draft

---

## 1. Назначение документа

Diagnostics & Error Catalog Spec регламентирует структуру данных ошибок (`DiagnosticItem`), правила их накопления и отображения в 3D-редакторе AxiCAD, влияние диагностик на экспорт/сборку проекта, а также определяет начальный классификатор кодов ошибок для всей экосистемы.

### Вне зоны ответственности (Non-goals)
- Документ не описывает дизайн UI-компонентов отображения ошибок (списки, всплывающие подсказки).
- Документ не описывает синтаксические парсеры логов.

---

## 2. Схема `DiagnosticItem` (Diagnostic Contract)

Каждая диагностическая запись в Store и файле проекта описывается следующим TypeScript-like контрактом:

```typescript
import { UUID } from './editor-store-spec-ru';

export type DiagnosticSeverity = 'error' | 'warning' | 'info';

export type DiagnosticSource = 
  | 'serde'              // Ошибки парсинга TOML (Tier 1)
  | 'rust-validator'     // Проверки Rust API ядра (Tier 2)
  | 'editor-validator'   // Локальные проверки UI/Вьюпорта (Tier 3)
  | 'resolver'           // Разрешение строковых эндпоинтов (Tier 4)
  | 'baker'              // Проверки ABI линковщика Baker (Tier 4)
  | 'import-adapter'     // Ошибки конвертеров legacy/фрагментов
  | 'project-file';      // Ошибки axicad.project.json

export interface DiagnosticItem {
  /** Уникальный UUID экземпляра диагностики */
  readonly id: UUID;

  /** Унифицированный строковый код ошибки (например, "AXI-BAKER-001") */
  readonly code: string;

  /** Уровень критичности */
  readonly severity: DiagnosticSeverity;

  /** Компонент-источник ошибки */
  readonly source: DiagnosticSource;

  /** Локализованный текст описания проблемы на русском языке */
  readonly message: string;

  /** Временный runtime UUID затронутой сущности в Store (не сериализуется) */
  readonly entityId?: UUID;

  /** Стабильный типизированный путь к сущности (сериализуется в JSON проекта) */
  readonly typedPath?: string;

  /** Относительный путь к файлу конфигурации на диске (для ошибок Tier 1/2) */
  readonly configPath?: string;

  /** Относительный путь свойства внутри TOML (например, "world.inv_seg_speed") */
  readonly tomlPath?: string;

  /** Текстовое предложение по автоматическому или ручному исправлению */
  readonly suggestedFix?: string;

  /** UUID ассоциированного PatchSet для автоисправления (при наличии) */
  readonly repairPatchSetId?: UUID;

  /** Список операций, блокируемых данной диагностикой */
  readonly blockingOperations?: Array<
    | 'save-project'
    | 'export-toml'
    | 'baker-compile'
    | 'apply-patchset'
    | 'run-preview'
    | 'run-simulation'
  >;

  /** Системная метка времени генерации диагностики */
  readonly createdAt: number;
}
```

> [!NOTE]
> Параметры `severity` и `source` дают базовую классификацию диагностик, однако финальная блокировка конкретной операции определяется набором флагов в `blockingOperations` конкретного инстанса ошибки или глобальной derived policy редактора (например, блокировка экспорта TOML при любых биологических ошибках).

## 3. Правила привязки (Binding Rules)

Для сохранения связи ошибки с объектом при перезагрузках и мутациях применяются следующие правила адресации:

- **Внутренний UUID (`entityId`)**: Используется исключительно в оперативной памяти Store для быстрого маппинга диагностики на элементы дерева UI.
- **Типизированный путь (`typedPath`)**: Используется для персистентного кэширования диагностик в файле `axicad.project.json`. Позволяет восстановить ошибку для конкретного элемента при перезапуске редактора до прогона тяжелых валидаторов.
- **Путь к файлу (`configPath`)**: Для ошибок синтаксиса (Tier 1) или недоступности файлов, когда сущность в Store не может быть создана, ошибка привязывается к нормализованному относительному пути файла (например, `departments/SensoryCortex.toml`).
- **Переименования и удаления (Rename/Delete)**: 
  - Активные диагностики в Store при выполнении `delete`/`rename` инвалидируются или принудительно пересчитываются.
  - Сохраненный в `axicad.project.json` кэш диагностик (`diagnostics_cache`), если типизированный путь (`typedPath`) устарел и отсутствует в Store при загрузке: либо автоматически сопоставляется (`relatch`) по истории миграций на новый путь, либо переносится в секцию `orphaned/stale diagnostics cache` в Store редактора для ручного разбора пользователем.

## 4. Семантика уровней серьезности (Severity Semantics)

- **`error` (Ошибка)**: Указывает на критическое нарушение структуры или биологических контрактов. Блокирует экспорт TOML, компиляцию Baker и запуск симуляции. **Важно**: не блокирует сохранение служебного файла `axicad.project.json`, чтобы пользователь мог прервать сессию, не теряя настроек камеры и placements.
- **`warning` (Предупреждение)**: Указывает на неоптимальные настройки (например, выход за рамки рекомендованной плотности клеток) или висящие метаданные placements в карантине. Отображается в UI, но не блокирует операции экспорта и сборки.
- **`info` (Информация)**: Полезные системные сведения (например, отчет о сжатии истории миграций).

## 5. Матрица блокировок (Blocking Matrix)

Влияние диагностик на ключевые операции редактора подчиняется правилам разграничения ответственности данных:

| Операция | Блокируется при `severity` | Блокируется источниками (`source`) | Описание поведения |
|---|---|---|---|
| **Save Project JSON** (`axicad.project.json`) | Никогда | Нет | Сохранение настроек вьюпорта и placements разрешено в любой момент, независимо от наличия ошибок в биологии. |
| **Export TOML** (Запись биологии) | `error` | `serde`, `rust-validator`, `resolver` | Блокируется при любых ошибках, влияющих на canonical biological TOML и возможность его корректного экспорта на диск. |
| **Baker compile** (Сборка ABI) | `error` | `serde`, `rust-validator`, `resolver`, `baker` | Блокируется ошибками, влияющими на biological graph, адресацию сокетов или Baker ABI. Служебные ошибки редактора (`editor-validator`, `project-file`) не блокируют сборку Baker, если биологический контракт не нарушен. |
| **Apply PatchSet** (Применение патчей) | `error` | `serde`, `import-adapter`, `resolver` | Блокируется при наличии фатальных ошибок валидации внутри импортируемого `PatchSet`. |
| **Run Preview / Simulation** (Запуск) | `error` | `serde`, `rust-validator`, `resolver`, `baker` | Симуляция и генерация превью блокируются только биологическими и рантайм-ошибками. Вспомогательные `warning` (например, stale placements в карантине) запуск не блокируют. |

## 6. Пространства имен кодов (Namespaces)

Каждый код ошибки состоит из префикса `AXI-` и буквенного идентификатора категории:

- **`AXI-SERDE-*`**: Ошибки синтаксиса и десериализации TOML-файлов (Tier 1).
- **`AXI-SCHEMA-*`**: Нарушения биологической схемы, лимитов и типов данных Axicor TOML (Tier 2).
- **`AXI-EDITOR-*`**: Нарушения правил редактора, геометрии и вьюпорта (Tier 3).
- **`AXI-RESOLVER-*`**: Ошибки разрешения строковых путей и эндпоинтов в Store (Tier 4).
- **`AXI-BAKER-*`**: Ошибки линковки ABI и кросс-департаментных сопряжений Baker (Tier 4).
- **`AXI-IMPORT-*`**: Ошибки импорта фрагментов, легаси-адаптеров и сгенерированных скриптов.
- **`AXI-PROJECT-*`**: Ошибки структуры и загрузки служебного файла `axicad.project.json`.

> [!NOTE]
> **Замечание о миграции кодов (Migration Note)**:
> Ранее использовавшиеся в документации (например, в [validation-spec-ru.md](validation-spec-ru.md)) коды ошибок семейства `AXI-VALID-*` признаются устаревшими заглушками (legacy placeholders). Новая диагностическая система использует коды `AXI-SCHEMA-*` для нарушений TOML-схемы и ядра Rust API (Tier 2).

## 8. Модель предложений исправлений (Suggested Repairs)

Для ускорения исправления ошибок `DiagnosticItem` может содержать ссылки на готовые патчи:

- **Неавтоматическое применение (Explicit Apply)**: Редактор никогда не применяет исправления автоматически. При наличии `repairPatchSetId` в UI отображается кнопка "Исправить автоматически".
- **Транзакционный накат**: После нажатия кнопки исправления, ассоциированный `PatchSet` передается в Store и применяется строго через транзакцию команд (`command transaction`).
- **Сброс Dirty**: Изменения выставляют соответствующие dirty-флаги согласно правилам мутаций (изменения биологии пачкают TOML, изменения вьюпорта пачкают только JSON).
