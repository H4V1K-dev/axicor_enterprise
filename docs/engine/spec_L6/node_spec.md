## §1. Идентификация

| Поле | Значение |
|---|---|
| **Имя крейта** | `node` |
| **Имя бинарного файла** | `axi-node` |
| **Слой** | Слой 6 — Node Process (`L6` / Процесс Ноды) |
| **Тип** | Binary (`bin`) |
| **no_std** | Нет (`false`) — требуется `std` для интеграции с OS, парсинга CLI и I/O |
| **Описание** | Консольное приложение для запуска локальной симуляции одного шарда из контейнера `.axic` с использованием библиотек `boot` и `runtime` (Stage A). |

---

## §2. Стек и Окружение

### §2.1. Внутренние зависимости (inbound)

| Крейт | Что используется | Зачем |
|---|---|---|
| `boot` (Слой 6) | `bootstrap_local_shard_engine`, `LocalShardComputeInput`, `BootError` | Загрузка `.axic` и инициализация `ShardEngine` |
| `runtime` (Слой 6) | `LocalRuntime`, `LocalRuntimeConfig`, `RuntimeError` | Управление циклом симуляции локального шарда |
| `types` (Слой 0) | Базовые общие типы и константы | Использование при композиции |

### §2.2. Запрещенные зависимости (Forbidden Dependencies)

Крейту `node` категорически запрещено импортировать напрямую следующие компоненты:
1. Крейты физических бэкендов вычислений: `compute-cuda`, `compute-hip`, `compute-cpu`.
2. Библиотеки транспортного и сетевого протоколов: `transport`, `protocol`, `wire`.
3. Модули топологии и компиляции: `baker`, `topology`, `weaver-daemon`.
4. API разбора и валидации форматов TOML/JSON/архивов (`config`, `layout`, `vfs`).

### §2.3. Внешние зависимости

| Crate | Версия | Сфера использования |
|---|---|---|
| `clap` | `=4.5.4` | Парсинг CLI-аргументов и декларативная CLI-схема |
| `serde` | `=1.0.228` | Сериализация JSON-отчетов |
| `serde_json` | `=1.0.117` | Форматирование JSON-вывода |

## §3. Границы Владения (Ownership Boundaries)

- `node` отвечает исключительно за разбор аргументов CLI, координацию выполнения тиков батчами, форматирование результатов симуляции (JSON / текст / CSV) и трансляцию ошибок в Exit Codes ОС.
- `node` не имеет права самостоятельно выполнять симуляцию или работать с аппаратным контекстом (делегируется `runtime` и `compute`).

## §4. Грамматика Командной Строки Stage A

Подкоманда запуска локальной симуляции:
```text
axi-node run-local --archive <PATH> --ticks <U64> --batch-ticks <U32> --max-spikes-per-tick <U32> [--backend cpu|cuda|auto] [--json] [--csv-dir <PATH>]
```

### §4.1. Параметры командной строки:
- `--archive <PATH>`: Обязательный путь к бинарному архиву `.axic`.
- `--ticks <U64>`: Обязательное общее число тиков симуляции для прогона.
- `--batch-ticks <U32>`: Обязательное количество тиков в одном батче.
- `--max-spikes-per-tick <U32>`: Обязательный лимит сгенерированных спайков на один тик (для аллокации буферов).
- `--backend <cpu|cuda|auto>`: Опциональный предпочтительный вычислительный бэкенд. По умолчанию `cpu`.
- `--json`: Вывод итогового отчета в формате JSON в поток `stdout`.
- `--csv-dir <PATH>`: Опциональный путь к папке на диске для записи CSV-отчетов с детализацией по батчам и спайкам.

## §5. Поведение и Логика Выполнения Stage A

1. **Загрузка и Старт**:
   - Нода разбирает аргументы и вызывает `boot::bootstrap_local_shard_engine` с параметрами:
     - `archive_path` = `--archive`
     - `backend_preference` = мапинг из `--backend` (`cpu` -> `Cpu`, `cuda` -> `Cuda`, `auto` -> `Auto`)
     - `virtual_offset` = 0
     - `total_ghosts` = 0
   - При ошибке загрузки процесс завершается с кодом **10** (BootFailed).

2. **Запуск Рантайма**:
   - Конструируется `runtime::LocalRuntimeConfig` с `virtual_offset = 0` и `max_spikes_per_tick = max_spikes_per_tick`.
   - Инициализируется `runtime::LocalRuntime` с инстансом `ShardEngine` от `boot`.
   - При ошибке инициализации рантайма процесс выходит с кодом **20** (RuntimeInitFailed).

3. **Цикл Выполнения (Batch Loop)**:
   - Общее число тиков `N` разбивается на батчи размером `B` тиков.
   - Для каждого батча вызывается `runtime.run_empty_batch(batch_size)`.
   - Если `N` не делится нацело на `B`, последний батч выполняется размером `N % B`.
   - При возникновении ошибок в цикле выполнения процесс выходит с кодом **21** (RuntimeFaulted).

4. **Вывод Результатов (Summary)**:
   - При успешном завершении вызывается `runtime.shutdown()`.
   - Если флаг `--json` отсутствует, отчет выводится в человекочитаемом виде в `stderr` (или `stdout`).
   - Если флаг `--json` передан, в `stdout` пишется один стабильный JSON-объект с полями:
     - `total_ticks`
     - `batches`
     - `total_generated_spikes`
     - `total_output_spikes_written`
     - `total_dropped_spikes`
     - `mean_backend_execution_time_us`
     - `wall_time_us`
     - `final_runtime_state`
     - `backend_kind`
   - Если передан `--csv-dir`, в указанную директорию экспортируются CSV-файлы:
     - `node_batches.csv` (детализация по батчам)
     - `node_outputs.csv` (детализация по тикам)
     - `node_output_spikes.csv` (сырые выходные спайки)
     - `node_summary.csv` (итоговый отчет в формате ключ-значение)

## §6. Коды Возврата ОС (Exit Codes)

| Код | Семантика | Описание |
|---|---|---|
| **0** | Success | Симуляция успешно завершена. |
| **2** | CLI Error | Ошибка разбора или валидации CLI аргументов. |
| **10** | BootFailed | Сбой при открытии или валидации `.axic` архива. |
| **20** | RuntimeInitFailed | Ошибка создания `LocalRuntime` или запуска бэкенда. |
| **21** | RuntimeFaulted | Критическая ошибка во время выполнения тиков. |
| **70** | InternalError | Непредвиденная системная паника. |

## §8. Тестовая Матрица Спеки Stage A

1. **Справка CLI**: Вызов с `--help` выводит usage и возвращает 0.
2. **Пропущенные Аргументы**: Отсутствие обязательных параметров возвращает код 2.
3. **Несуществующий архив**: Невалидный путь к `.axic` возвращает код 10.
4. **Поврежденный архив**: Передача битого файла вместо `.axic` возвращает код 10.
5. **Успешный запуск**: Запуск на CPU-бэкенде возвращает код 0.
6. **Невалидные параметры тиков**: Отрицательные, нулевые тики или невалидные размеры батчей отклоняются с кодом 2 или 20.
7. **Дробление тиков**: Проверка, что при `N % B > 0` последний батч корректно выполняется меньшим размером.
8. **JSON формат**: Проверка структуры и полей JSON-вывода при флаге `--json`.
9. **CSV экспорт**: Убедиться, что при `--csv-dir` файлы создаются и наполняются корректными строками.
10. **Контроль Зависимостей**: Статическая проверка отсутствия зависимостей от `ipc`, `net`, `wire`, `protocol`, `transport`.
