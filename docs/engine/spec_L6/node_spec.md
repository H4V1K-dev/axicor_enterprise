## §1. Идентификация

| Поле | Значение |
|---|---|
| Название | `node` |
| Слой | Слой 6 — Runtime |
| Тип | Binary (bin) |
| no_std | **Нет** (требует разбора CLI-аргументов, перехвата ОС-сигналов и системного логирования) |
| Описание | Тонкий исполняемый демон (Thin Daemon Binary), интегрирующий симулятор с операционной системой хоста. Выполняет разбор CLI-аргументов, ограничивает асинхронный пул, выполняет жесткую привязку к физическим ядрам процессора (CPU Affinity) и обеспечивает безопасный останов (Graceful Shutdown) при получении системных сигналов. |

## §3. Инварианты

Крейт `node` отвечает за целостность процесса на уровне операционной системы.

### §3.1. Структурные инварианты

Секция не применима к данному крейту: Крейт является исполняемым файлом (bin) и не содержит фиксированных макетов или структур в памяти.

### §3.2. Семантические инварианты

- **INV-NODE-001**: *Graceful Shutdown on OS Signal (Безопасный останов)*.
  - *Обоснование*: Бинарь `node` обязан перехватывать сигналы операционной системы SIGINT (Ctrl+C) и SIGTERM и инициировать метод `NodeRuntime::shutdown()`, чтобы гарантировать выполнение C-ABI деаллокаций и предотвратить зависание видеокарты.
  - *Следствие нарушения*: Висящие VRAM-аллокации при убийстве процесса ноды, блокирующие последующие запуски, потеря логов и чекпоинтов.
  - *Где проверяется*: Интеграционный тест с отправкой сигнала SIGTERM.

- **INV-NODE-002**: *CLI Argument Validation (Входной файрвол)*.
  - *Обоснование*: Любые невалидные параметры CLI (отсутствие пути к архиву, неверный формат сетевых адресов) отсекаются парсером `clap` до запуска `boot` пайплайна и каких-либо системных выделений.
  - *Следствие нарушения*: Запуск загрузчика с невалидными путями, ведущий к панике или утечкам ресурсов на ранней стадии.
  - *Где проверяется*: Юнит-тесты на парсинг неверных CLI аргументов.

### §3.3. Межкрейтовые инварианты

- **INV-CROSS-011**: *Graceful State Transit (Node Teardown Sync)*.
  - *Участники*: `runtime`, `node`.
  - *Кто владелец проверки*: `node`.
  - *Обоснование*: Исполняемый файл `node` при получении сигнала прерывания блокирует поток сигнала и вызывает `NodeRuntime::shutdown()`, дожидаясь возврата результата, прежде чем завершить процесс с кодом 0.
  - *Следствие нарушения*: Преждевременный выход из процесса `node` обрывает поток рантайма на середине очистки VRAM.
  - *Где проверяется*: Интеграционный тест Graceful Shutdown.

## §5. Доменная Логика

Крейт `node` представляет собой входную точку и управляющую консоль (daemon) всего узла симуляции (Слой 6). Его задача — объединить инициализирующий загрузчик (`boot`) и рабочий рантайм (`runtime`) в единый исполняемый процесс, управляемый операционной системой.

Выделение `node` в отдельный тонкий бинарник позволяет полностью отделить логику запуска процесса от самого движка симуляции. Бинарник физически лишен бизнес-логики и сводит свою доменную ответственность к четырём системным шагам:

1. **CLI & Environment (Окружение ОС):** Парсинг аргументов командной строки (путь к `.axic` архиву, флаги переопределения бэкенда) и инициализация глобального системного логгера `tracing`.
2. **Изоляция асинхронного пула (Tokio Containment):** Поднятие пула потоков `tokio` с жестко заданным лимитом воркеров. Асинхронный рантайм резервируется исключительно для фоновых задач (Control Plane, телеметрия, TCP). Ему физически не дается доступ к горячему HFT-циклу, чтобы планировщик ОС не отбирал такты.
3. **Аппаратная привязка (CPU Affinity):** `node` намертво привязывает выделенный системный поток HFT-цикла к конкретному физическому ядру процессора через системные вызовы ОС. Это чистая механическая симпатия — мы исключаем вмешательство планировщика ОС (Context Thrashing) в расчеты шарда.
4. **Перехват сигналов (Graceful Teardown):** `node` блокируется и слушает сигналы прерывания ОС (SIGINT/SIGTERM). При их получении он обязан перевести `NodeRuntime` в состояние `Shutdown` и дождаться полного завершения всех потоков, защищая драйвер GPU от гонки деинициализации (Teardown Race).

## §6. Алгоритмы и Формулы

### §6.1. Алгоритм оркестрации жизненного цикла процесса

**Вход**: Аргументы командной строки `std::env::args()`.
**Выход**: Код завершения процесса (0 при успехе, 1 при сбое).
**Детерминизм**: Да.

**Логика:**
Бинарник пошагово собирает приложение, делегируя тяжелую логику в изолированные библиотеки. Главный нюанс — настройка перехвата сигналов ОС до запуска бесконечного цикла `runtime.run()`.

**Псевдокод:**
```rust
fn main() {
    // 1. Инициализация логирования
    init_tracing();

    // 2. Парсинг CLI
    let args = parse_cli_args();

    // 3. Запуск boot пайплайна
    let shard_engine = match boot::BootPipeline::execute(&args.archive_path) {
        Ok(engine) => engine,
        Err(err) => {
            error!("Boot failed: {:?}", err);
            std::process::exit(1);
        }
    };

    // 4. Инициализация NodeRuntime
    let mut runtime = NodeRuntime::new(shard_engine, /* ... */);

    // 5. Настройка обработчиков сигналов (перехват SIGINT/SIGTERM)
    setup_signal_handlers(&runtime);

    // 6. Запуск рантайма (блокирует главный поток до получения сигнала)
    if let Err(err) = runtime.run() {
        error!("Runtime error: {:?}", err);
        std::process::exit(1);
    }

    std::process::exit(0);
}
```

**Численный пример:**

| Вход | Ожидаемый выход | Комментарий |
|------|-----------------|-------------|
| `node --archive valid.axic` | Код 0 (после успешного shutdown) | Корректный запуск и завершение. |
| `node --archive invalid.axic` | Код 1 | Ошибка на фазе boot, завершение процесса. |

## §8. Граничные Случаи и Особые Сценарии

### §8.1. Граничные значения

| # | Ситуация | Ожидаемое поведение |
|---|---|---|
| E-146 | Получение сигнала SIGINT/SIGTERM во время HFT-цикла | `node` перехватывает сигнал, останавливает выполнение и вызывает `runtime.shutdown()`, корректно освобождая ресурсы. |
| E-147 | Ошибка запуска с несуществующим файлом архива | CLI-парсер или загрузчик выдает ошибку, бинарный файл завершается с кодом `1`. |
