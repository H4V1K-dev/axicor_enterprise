### §2.5. Запрещенные операции и зависимости

В крейте `wire` физически и архитектурно запрещены:
- Динамическая память (`alloc`, `Vec`, `String`, `Box`, `BTreeMap`).
- Сетевые сокеты ОС (`std::net`), асинхронный рантайм (`tokio`), каналы (`crossbeam`).
- Реализация L7-фрагментации, сборки пакетов и повторных отправок (принадлежат `protocol`).
- Операции с сокетами и неблокирующий I/O (принадлежат `transport`).
- Таблицы маршрутизации, RCU-указатели и бизнес-логика I/O сервисов (принадлежат `net`).
- Управление жизненным циклом mmap/SHM (принадлежат `vfs`/`ipc`).
- Макеты дисковых файлов дампов `.state`, `.axons`, `.paths` (принадлежат `layout`).
- Генерация топологии, bake артефакты и бизнес-логика Python SDK клиентов.

## §4. Политикой Эндианности и Безопасность Буферов (Endianness & Buffer Safety)

### §4.1. Порядок Байтов (Little-Endian Policy)
Все числовые поля сетевых пакетов в `wire` передаются и сохраняются строго в формате **Little-Endian (LE)**.

#### Магические константы (Magic Constants)
Во избежание путаницы с порядком байтов при объявлении 32-битных сигнатур, все магические константы задаются через байтовые литералы с явным указанием `from_le_bytes`:
```rust
pub const MAGIC_GSIO: u32 = u32::from_le_bytes(*b"GSIO");
pub const MAGIC_GSOO: u32 = u32::from_le_bytes(*b"GSOO");
pub const MAGIC_ROUT: u32 = u32::from_le_bytes(*b"ROUT");
pub const MAGIC_DOPA: u32 = u32::from_le_bytes(*b"DOPA");
pub const MAGIC_TELE: u32 = u32::from_le_bytes(*b"TELE");
```

### §4.2. Безопасное Чтение Невыровненных Буферов (Unaligned Buffer Safety)
Сетевой стек ОС или сокеты UDP могут возвращать байтовые массивы `&[u8]`, адрес начала которых в памяти не выровнен по границе 4 или 8 байт. Прямой cast такого буфера через `bytemuck::from_bytes` на некоторых архитектурах (ARM Cortex, MCU) вызывает аппаратное прерывание (Alignment Trap).

#### Правила безопасного чтении в `wire`:
1. Прямой zero-copy cast через `bytemuck` разрешен только тогда, когда гарантировано выравнивание адреса буфера.
2. Для произвольных сырых сетевых буферов `wire` предоставляет безопасные функции-хелперы (`try_read_header`), выполняющие чтение через `core::ptr::read_unaligned` или копирование в выровненную локальную переменную на стеке.
3. Любое обращение к сетевому буферу без выравнивания **никогда не вызывает панику**.

## §6. Реестр DTO Структур и Физические Размеры

Все публичные структуры DTO в `wire` имеют гарантированный C-ABI макет (`#[repr(C)]`), реализованные трейты `Pod` и `Zeroable`, а также явное выравнивание полей.

### 1. `SpikeEventV2` (Координата спайка в батче)
Представляет собой единичный спайк в сетевом потоке.
- **Атрибуты**: `#[repr(C)]`
- **Размер**: 8 байт.
- **Выравнивание**: 4 байта.

| Поле | Тип | Смещение | Размер | Описание |
|---|---|---|---|---|
| `ghost_id` | `u32` | `0` | 4B | Глобальный ID целевого ghost-аксона |
| `tick_offset` | `u32` | `4` | 4B | Внутрибатчевое смещение тика возникновения |

---

### 2. `SpikeBatchHeaderV2` (Legacy заголовок батча спайков)
Заголовок сетевого пакета спайков.
- **Атрибуты**: `#[repr(C)]`
- **Размер**: 16 байт.
- **Выравнивание**: 4 байта.

| Поле | Тип | Смещение | Размер | Описание |
|---|---|---|---|---|
| `src_zone_hash` | `u32` | `0` | 4B | Хэш зоны-источника спайков |
| `dst_zone_hash` | `u32` | `4` | 4B | Хэш зоны-назначения спайков |
| `epoch` | `u32` | `8` | 4B | Номер эпохи симуляции |
| `chunk_idx` | `u16` | `12` | 2B | Индекс фрагмента L7 |
| `total_chunks` | `u16` | `14` | 2B | Общее количество фрагментов L7 |

**Специальные режимы семантики полей**:
- `total_chunks == 0 && chunk_idx == 0`: Пустой heartbeat / ACK-like барьерный пакет поддержания связи (не содержит полезной нагрузки).
- `chunk_idx == 0xFFFF && total_chunks == 0`: Подтверждение приёма пакета (ACK).

> [!WARNING]
> Первое слово `src_zone_hash` не является константным magic. При диспетчеризации пакетов по первому `u32` возможно совпадение значения хэша с magic-сигнатурами других семейств.

## §7. Представление Ошибок Парсинга (`WireError`)

Крейт `wire` определяет концептуальное перечисление ошибок уровня бинарного парсинга пакетов.

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum WireError {
    BufferTooSmall,
    AlignmentMismatch,
    InvalidMagic,
    UnsupportedVersion,
    PayloadSizeMismatch,
    InvalidLength,
    IntegerOverflow,
    InvalidPacketKind,
}
```

> [!IMPORTANT]
> Ошибки индексации фрагментов L7 (`FragmentIndexOutOfBounds`), устаревания эпохи (`EpochTooOld`), дублирования чанков и переполнения очереди сборки принадлежат строго крейту `protocol` и не входят в `WireError`.

## §8. Вспомогательный API (`no-alloc Helpers`)

Крейт предоставляет комплект высокопроизводительных функций без динамических аллокаций для парсинга и валидации буферов.

### 1. Вычисление размеров
```rust
pub const fn wire_size_of<T>() -> usize {
    core::mem::size_of::<T>()
}
```

### 2. Безопасное чтение заголовка из невыровненного буфера
```rust
pub fn try_read_header<T: bytemuck::Pod>(bytes: &[u8]) -> Result<T, WireError> {
    if bytes.len() < core::mem::size_of::<T>() {
        return Err(WireError::BufferTooSmall);
    }
    unsafe {
        Ok(core::ptr::read_unaligned(bytes.as_ptr() as *const T))
    }
}
```

### 3. Извлечение среза полезной нагрузки (Payload Slice)
```rust
pub fn payload_slice<'a>(
    packet: &'a [u8],
    header_size: usize,
    payload_size: usize,
) -> Result<&'a [u8], WireError> {
    let total_required = header_size.checked_add(payload_size).ok_or(WireError::IntegerOverflow)?;
    if packet.len() < total_required {
        return Err(WireError::PayloadSizeMismatch);
    }
    Ok(&packet[header_size..total_required])
}
```

### 4. Вспомогательные валидаторы длины
- `validate_external_io_len(header: &ExternalIoHeader, packet_len: usize) -> Result<(), WireError>`
- `spike_events_payload_len(event_count: usize) -> Result<usize, WireError>` (вычисление через `checked_mul(8)`).

## §9. Политика Кодогенерации SDK (SDK Generation Policy)

Жесткое требование автоматического вызова `build.rs` при каждой сборке крейта заменяется на безопасное архитектурное правило:

1. **Единый Исполняемый Источник Правды**: Rust-структуры DTO в `wire` являются единственным источником истины для бинарных макетов пакетов.
2. **Опциональная Внешняя Кодогенерация**: Генерация Python SDK (`contract_generated.py`) или метаданных схем выполняется через внешний инструмент кодогенерации или интеграционные тесты расхождения схем (Schema Drift Tests).
3. **Запрет Ручного Дублирования**: Ручное копирование констант и смещений полей в сторонние SDK без автогенерации из метаданных `wire` запрещено.

## §10. Требуемые Инварианты

- **INV-WIRE-001**: Все публичные структуры сетевых DTO обязаны иметь атрибут `#[repr(C)]`.
- **INV-WIRE-002**: Все структуры DTO пакетов реализуют маркерные трейты `bytemuck::Pod` и `bytemuck::Zeroable`.
- **INV-WIRE-003**: Запрещено неявное неинициализированное выравнивание. Все поля явных падов (`_padding`, `_pad`) обнуляются при кодировании.
- **INV-WIRE-004**: Для каждой структуры DTO в крейте записаны статические проверки размера (`const_assert_eq!`).
- **INV-WIRE-005**: Запрещено публичное объявление DTO со стандартным макетом Rust (`repr(Rust)`).
- **INV-WIRE-006**: Запрещено использование `repr(packed)` без отдельного обоснования во избежание Alignment Trap на контроллерах.
- **INV-WIRE-007**: Вся арифметика расчёта длин буферов выполнена через `checked_*` операции.
- **INV-WIRE-008**: Неизвестная сигнатура magic приводит к безопасному возврату ошибки или отбрасыванию пакета, но никогда не вызывает панику.
- **INV-WIRE-009**: Сырые сетевые буферы никогда не вызывают панику при отсутствии выравнивания.

## §11. Golden Tests / Обязательная Матрица Тестирования

Крейт `wire` содержит полный комплект unit-тестов для верификации бинарных контрактов:

1. **Размеры и Выравнивание DTO (`test_dto_sizes_and_alignments`)**: Проверка exact `size_of` и `align_of` для каждого DTO из реестра.
2. **Проверка Магических Констант Little-Endian (`test_magic_constants_le`)**: Сравнение `u32::from_le_bytes` с ожиданиями байтовых байт-строк.
3. **Точный Размер `ExternalIoHeader` (`test_external_io_header_size`)**: Подтверждение размера ровно 20 байт.
4. **Отклонение Несоответствия Полезной Нагрузки (`test_payload_size_mismatch_rejected`)**: Проверка работы `payload_slice` при коротких буферах.
5. **Безопасное Чтение Невыровненных Буферов (`test_unaligned_buffer_safe_read`)**: Проверка чтения `try_read_header` из массива `[u8]` со сдвигом 1 байт.
6. **Кратность Массива `SpikeEventV2` (`test_spike_events_array_alignment`)**: Проверка кратности длины массива спайков 8 байтам.
7. **Семантика `SpikeBatchHeaderV2` (`test_spike_batch_header_semantics`)**: Проверка спец-режимов Heartbeat (`0/0`) и ACK (`0xFFFF/0`).
8. **Размер и Смещение `RouteUpdate` (`test_route_update_layout`)**: Проверка размера 24B и смещения `cluster_secret` на 16B.
9. **Обнуление Паддинга (`test_zeroed_padding_on_construct`)**: Проверка заполнения полей падов нулями.
10. **Zero Allocations Test (`test_helpers_zero_allocations`)**: Гарантия отсутствия вызовов аллокатора при работе хелперов.

## §12. Open Questions / Review Debt (Открытые Вопросы и Противоречия)

В процессе анализа бинарных сетевых макетов выявлены следующие открытые вопросы для согласования:

1. **Отсутствие Magic-поля в `SpikeBatchHeaderV2`**:
   - *Контекст*: Первое слово `src_zone_hash` не является сигнатурой magic. При диспетчеризации пакетов по первому `u32` возможен конфликт с другими magic.
   - *Вопрос*: Требуется ли введение новой версии заголовка спайков с явным полем magic/version?

2. **Границы Владения Файловыми Заголовками `.gxi`, `.gxo`, `.ghosts`**:
   - *Контекст*: В легаси-коде эти заголовки находились в `ipc.rs`.
   - *Вопрос*: К какому крейту следует отнести владение этими заголовками — `layout`, `baker` или `ipc`?

3. **Размер Структуры `AxonHandoverPrune` (12B vs 16B)**:
   - *Контекст*: Текущий размер структуры равен 12 байтам.
   - *Вопрос*: Требуется ли выровнять структуру явным падом до 16 байт для кратности 8/16 байтам?

4. **Размещение `ShardStateHeader` (`SNAP`)**:
   - *Контекст*: Структура используется при сетевой репликации VRAM, но соприкасается с макетами дампов.
   - *Вопрос*: Оставляем ли мы `ShardStateHeader` в `wire` или относим к `layout`/`ipc`?

5. **Фиксация Версии `bytemuck` (`=1.25.0` vs `=1.20.0`)**:
   - *Контекст*: В легаси документации встречались разные версии.
   - *Вопрос*: Подтверждается ли единая фиксация версии `=1.25.0` в Cargo.toml?

6. **Поддержка Архитектур Big-Endian**:
   - *Контекст*: Все пакеты сериализуются в Little-Endian.
   - *Вопрос*: Фиксируем ли мы `compile_error!` при сборке под Big-Endian платформы или вводим явные функции конвертации полей?
