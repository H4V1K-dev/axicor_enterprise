## §1. Идентификация

| Поле | Значение |
|---|---|
| Название | `layout` |
| Слой | Слой 1 — Контракты Данных и Раскладка Памяти (Memory Layouts) |
| Тип | Library (`lib`) |
| no_std | Строго обязателен (`true`) |
| Описание | Единый источник истины для C-ABI контрактов физической памяти, SoA (Structure of Arrays) макетов, формул смещения плоских массивов (State Offsets) и выравнивания структур в движке `AxiEngine`. Крейт содержит исключительно POD (Plain Old Data) структуры с атрибутами `#[repr(C)]` и гарантирует побитовую совместимость (Zero-Copy DMA) между CPU хостом, VRAM видеокарты (CUDA/HIP) и дисковыми файлами дампов (`.state`, `.axons`, `.paths`). Не содержит логики аллокации, FFI-вызовов, математических уравнений физики и системных вызовов ОС. |

## §2. Стек и Окружение

### §2.1. Внутренние зависимости (inbound)

| Крейт | Что используется | Зачем |
|---|---|---|
| `types` | `PackedPosition`, `PackedTarget`, `SomaFlags`, `Voltage`, `Weight`, `AxonHead`, сентинели (`AXON_SENTINEL`, `EMPTY_PIXEL`) | Использование базовых 32-битных и 8-битных примитивов для построения C-ABI структур без дублирования типов и ABI-контрактов. |

### §2.2. Зависимые компоненты (outbound consumers)

Вычислительные ядра и бэкенды (`compute`, `compute-cuda`, `compute-hip`, `compute-cpu`), компилятор топологии (`baker`), а также хранилища и VFS (`ipc`, `vfs`) обязаны использовать структуры данных и формулы расчета смещений исключительно из `layout`. Самостоятельное определение макетов SoA или смещений полей в других крейтах **строжайше запрещено**.

### §2.3. Внешние зависимости

| Crate | Версия | Сфера использования и ограничения |
|---|---|---|
| `bytemuck` | `=1.25.0`, features=`["derive"]` | Runtime `no_std` и `no_alloc` совместим. Предоставляет маркерные трейты `Pod` и `Zeroable` для абсолютно безопасного (Zero-Cost) каста сырых байтовых массивов (из VRAM или mmap) в Rust-структуры. |
| `static_assertions` | `=1.1.0` *(Dev Dependency)* | Используется **исключительно** как dev-зависимость в юнит-тестах для compile-time проверок размеров и выравнивания типов (`const_assert_eq!`). В продакшн-бинарники не попадает. |

### §2.4. Feature Flags

| Feature | Default | Что включает |
|---|---|---|
| `default` | `[]` | По умолчанию крейт собирается в строго изолированном `no_std` и `no_alloc` окружении. |
| `std` | `[]` | Опциональный флаг исключительно для запуска юнит-тестов в среде разработки. |

### §2.5. Запрещенные операции и зависимости

В крейте `layout` физически и архитектурно запрещены:
- Зависимость от `std` и `alloc` в продакшн-сборке (0 динамических аллокаций памяти, запрет `Vec`, `Box`, `String`).
- Операции с файловой системой (`std::fs`), сетевым стеком (`std::net`) и системными вызовами mmap/SHM (принадлежат `vfs`/`ipc`).
- Аллокация GPU/CPU памяти, управление жизненным циклом буферов и FFI-вызовы CUDA/HIP API (принадлежат `compute`).
- Парсинг конфигураций TOML/JSON и Serde-структуры (принадлежат `config`).
- Реализация математических формул GLIF/GSOP и деривация физических параметров (принадлежат `physics`).
- Повторное объявление функций упаковки/распаковки таргетов (таких как `pack_dendrite_target` / `unpack_axon_id`), принадлежащих `types`.

## §3. Ownership Boundaries (Границы Владения)

Для исключения дублирования контрактов и четкого разграничения слоев, спецификация закрепляет следующие зоны ответственности:

| Крейт / Модуль | Монопольная Зона Владения (Single Source of Truth) | Строгие Запреты (Что категорически запрещено в крейте) |
|---|---|---|
| **`layout`** (Слой 1) | **Макеты физической памяти и C-ABI выравнивание**: структуры `VariantParameters`, `BurstHeads8`, `ShardVramPtrs`, порядок плоскостей SoA-дампа, формулы расчёта смещений `StateOffsets`, заголовки файлов `.state`, `.axons`, `.paths`, константы `MAX_DENDRITES = 128`, `MAX_SEGMENTS_PER_AXON = 256`, `CACHE_LINE_BYTES = 64`. | Запрещены формулы физики, упаковка битовых полей, FFI-вызовы GPU, аллокации памяти (`Vec`), системные вызовы mmap и парсинг TOML. |
| **`types`** (Слой 0) | **Примитивы и ABI-упаковка**: `PackedPosition`, `PackedTarget`, `SomaFlags`, сентинели (`AXON_SENTINEL`, `EMPTY_PIXEL`), функции `pack()/unpack()`. | Запрещены SoA-массивы, выравнивание плоскостей и заголовки файлов. |
| **`physics`** (Слой 0) | **Чистая математика**: формулы GLIF, GSOP, Active Tail, Stochastic Heartbeat, деривация `v_seg`. | Запрещено определение констант лимитов памяти (`MAX_DENDRITES`) и структур VRAM. |
| **`compute`** (Слой 3) | **Оркестрация и память вычислителей**: аллокация VRAM/RAM, управление pinned буферами, FFI-вызовы `cudaMemcpyAsync`, исполнение ядер. | Запрещено переопределение C-ABI структур макетов памяти. |
| **`ipc` / `vfs`** (Слой 2) | **Системные вызовы и хранилище**: mmap-монтирование архивов `.axic`, создание песочниц tmpfs, управление разделяемой памятью SHM. | Запрещена бизнес-логика расчёта внутренних смещений плоскостей SoA. |
| **`config`** (Слой 1) | **Парсинг конфигурации TOML**: Serde-структуры DSL и поверхностная валидация. | Запрещены C-ABI структуры с `#[repr(C, align)]` и бинарные макеты. |

### Устранение легаси-дублирования
В легаси-реализации `layout.rs` присутствовали функции `pack_dendrite_target` и `unpack_axon_id`, которые оперировали сырым типом `u32` и дублировали логику упаковки битов. В новой архитектуре `AxiEngine` крейт `layout` **не содержит** функций упаковки/распаковки. Крейт оперирует исключительно типом `types::PackedTarget`, принимая его как готовую C-ABI сущность.

## §4. Фундаментальные Константы

Все константы выравнивания и ограничений памяти в `layout` являются основанием для расчёта физических смещений на диске и в VRAM.

| Константа | Тип | Значение | Описание и Назначение |
|---|---|---|---|
| `MAX_DENDRITES` | `usize` | `128` | Максимальное количество дендритных слотов (колонок) на один нейрон. |
| `MAX_SEGMENTS_PER_AXON` | `usize` | `256` | Максимальное количество 3D-сегментов в геометрии одного аксона. |
| `CACHE_LINE_BYTES` | `usize` | `64` | Размер кэш-линии процессора / GPU (64 байта). Основной квант выравнивания плоскостей. |
| `CUDA_WARP_LANES` | `usize` | `32` | Размер варпа NVIDIA CUDA (32 потока). |
| `HIP_WAVE_LANES` | `usize` | `64` | Размер волнового фронта AMD ROCm/HIP (64 потока). |
| `PADDED_N_ALIGNMENT` | `usize` | `64` | Нейтральный квант выравнивания количества нейронов `padded_n` (кратно 64 для совместимости с CUDA и HIP). |
| `VARIANT_LUT_LEN` | `usize` | `16` | Количество доступных вариаций (профилей) параметров нейронов в таблицы LUT. |

> [!IMPORTANT]
> Константы сентинелей `AXON_SENTINEL` (`0x80000000`) и `EMPTY_PIXEL` (`0xFFFF_FFFF`) импортируются строго из крейта `types` и не дублируются в `layout`.

## §5. C-ABI Структуры Данных

### §5.1. `VariantParameters` (Профиль типов нейронов)

Структура содержит физические и биологические параметры типа нейрона.
- **Атрибуты**: `#[repr(C, align(64))]`
- **Размер**: строго 64 байта.
- **Выравнивание**: строго 64 байта (L1/L2 кэш-линия).

#### Стандартная раскладка полей и смещений (`VariantParameters`)

| Поле | Тип | Смещение | Размер | Описание |
|---|---|---|---|---|
| `threshold` | `i32` | `0` | 4B | Базовый порог генерации спайка |
| `rest_potential` | `i32` | `4` | 4B | Потенциал покоя мембраны |
| `leak_shift` | `u32` | `8` | 4B | Битовый сдвиг экспоненциальной утечки |
| `homeostasis_penalty` | `i32` | `12` | 4B | Пенальти порога при спайке |
| `spontaneous_firing_period_ticks` | `u32` | `16` | 4B | Период спонтанного спайкирования (Stochastic Heartbeat) |
| `initial_synapse_weight` | `u16` | `20` | 2B | Начальный вес нового синапса |
| `gsop_potentiation` | `u16` | `22` | 2B | Базовый импульс потенциации GSOP |
| `gsop_depression` | `u16` | `24` | 2B | Базовый импульс депрессии GSOP |
| `homeostasis_decay` | `u16` | `26` | 2B | Коэффициент затухания гомеостаза |
| `refractory_period` | `u8` | `28` | 1B | Длительность рефрактерного периода сомы |
| `fatigue_capacity` | `u8` | `29` | 1B | Емкость синаптической усталости (STP / Leaky Integrator capacity) |
| `signal_propagation_length` | `u8` | `30` | 1B | Длина активного хвоста спайка |
| `is_inhibitory` | `u8` | `31` | 1B | Флаг тормозного нейрона (1 = GABA, 0 = Glu) |
| `inertia_curve` | `[u8; 8]`| `32` | 8B | Таблица коэффициентов инерции GSOP |
| `ahp_amplitude` | `u16` | `40` | 2B | Амплитуда следовой гиперполяризации |

### §5.2. `BurstHeads8` (Буфер голов спайков)

Кольцевой буфер для хранения 8 последних координат голов импульсов аксона.
- **Атрибуты**: `#[repr(C, align(32))]`
- **Размер**: строго 32 байта.
- **Выравнивание**: строго 32 байта (считывание всех 8 голов за 1 транзакцию L1 кэша).

```text
+------+------+------+------+------+------+------+------+
|  h0  |  h1  |  h2  |  h3  |  h4  |  h5  |  h6  |  h7  |  (32 байта)
+------+------+------+------+------+------+------+------+
```

- **Поля**: `h0, h1, h2, h3, h4, h5, h6, h7` типа `types::AxonHead` (или `u32`).
- **Инициализация**: Пустое состояние создается конструктором `BurstHeads8::empty(types::AXON_SENTINEL)`, где все 8 полей заполняются значением `AXON_SENTINEL` (`0x80000000`).

> [!WARNING]
> В легаси-комментариях встречалась устаревшая запись `AXON_SENTINEL = 0xFFFFFFFF`. Это является ошибкой. Единственным правильным значением сентинеля неактивности из `types v2.2` является `0x80000000`.

### §5.3. `ShardVramPtrs` (C-ABI DTO указателей VRAM)

Официальный C-ABI контракт передаточного объекта (DTO) указателей для взаимодействия между Rust-оркестратором и вычислительными ядрами (CUDA/HIP/CPU).
- **Атрибуты**: `#[repr(C)]`
- **Действующие `derive`**: `Clone`, `Copy`, `Debug` (и `Zeroable` при безопасно проходимой не-unsafe сборке).
- **Строгие ограничения безопасности памяти**: Структура содержит сырые указатели (`*mut T`). **Категорически запрещено** реализовывать/реализовывать через derive трейт `bytemuck::Pod` и включать ненадёжный feature-флаг `unsound_ptr_pod_impl`. Запрещён каст структуры в сырые байты через bytemuck. Тестирование размеров, выравнивания и смещений полей выполняется строго через `static_assertions` и `core::mem::offset_of!` в юнит-тестах.
- **Правило**: Порядок полей является строго фиксированным. Любое изменение порядка ломает побайтовую передачу через FFI.

#### Порядок полей `ShardVramPtrs`

| № | Поле | Тип указателя | Размер элементов в плоскости |
|---|---|---|---|
| 1 | `soma_voltage` | `*mut i32` | `padded_n * 4B` |
| 2 | `soma_flags` | `*mut u8` | `padded_n * 1B` |
| 3 | `threshold_offset` | `*mut i32` | `padded_n * 4B` |
| 4 | `timers` | `*mut u8` | `padded_n * 1B` |
| 5 | `soma_to_axon` | `*mut u32` | `padded_n * 4B` |
| 6 | `dendrite_targets` | `*mut u32` *(PackedTarget)* | `MAX_DENDRITES * padded_n * 4B` |
| 7 | `dendrite_weights` | `*mut i32` | `MAX_DENDRITES * padded_n * 4B` |
| 8 | `dendrite_timers` | `*mut u8` | `MAX_DENDRITES * padded_n * 1B` |
| 9 | `axon_heads` | `*mut BurstHeads8` | `total_axons * 32B` |

## §6. Физическая Раскладка Памяти (SoA State Blobs)

### §6.1. Стандартный порядок плоскостей `.state` дампа

Файл состояния шарда `.state` представляет собой плоский бинарный блоб, состоящий из 16-байтового заголовка `StateFileHeader` и 8 последовательных SoA-плоскостей (State Planes).

**Порядок плоскостей в `.state` блобе**:
1. `soma_voltage`: `[padded_n]` элементов `i32`
2. `soma_flags`: `[padded_n]` элементов `u8` (структура `SomaFlags`)
3. `threshold_offset`: `[padded_n]` элементов `i32`
4. `timers`: `[padded_n]` элементов `u8`
5. `soma_to_axon`: `[padded_n]` элементов `u32`
6. `dendrite_targets`: `[MAX_DENDRITES * padded_n]` элементов `PackedTarget` (`u32`)
7. `dendrite_weights`: `[MAX_DENDRITES * padded_n]` элементов `i32` (Mass Domain)
8. `dendrite_timers`: `[MAX_DENDRITES * padded_n]` элементов `u8`

> [!IMPORTANT]
> Массив голов аксонов `axon_heads` вынесен в отдельный файл `.axons` (блоб размером `total_axons * 32B`) и **не входит** в состав `.state` файла.

---

### §6.2. Формулы расчёта смещений и выравнивание (`StateOffsets`)

Каждая SoA-плоскость в `.state` блобе начинается со смещения, кратного 64 байтам (`CACHE_LINE_BYTES`).

#### Структура `StateOffsets` DTO
Результат функции вычисления смещений `compute_state_offsets` возвращается в виде чистой runtime DTO-структуры `StateOffsets`:
- **Статус**: Runtime Rust DTO. Не является файловым или сетевым C-ABI бинарным представлением (не требует `#[repr(C)]` или bytemuck).
- **Тип полей**: Все поля имеют тип `usize`.
- **Состав полей**: `off_voltage`, `off_flags`, `off_thresh`, `off_timers`, `off_s2a`, `off_targets`, `off_weights`, `off_dtimers`, `total_state_size`.

#### Формула выравнивания
$$\text{align64}(x) = (x + 63) \ \& \ \sim 63$$

#### Формула расчёта смещений плоскостей (`compute_state_offsets`)
Пусть `header_size = 16`.
1. `off_voltage = align64(header_size)` $= 64$
2. `off_flags = align64(off_voltage + padded_n * 4)`
3. `off_thresh = align64(off_flags + padded_n * 1)`
4. `off_timers = align64(off_thresh + padded_n * 4)`
5. `off_s2a = align64(off_timers + padded_n * 1)`
6. `off_targets = align64(off_s2a + padded_n * 4)`
7. `off_weights = align64(off_targets + MAX_DENDRITES * padded_n * 4)`
8. `off_dtimers = align64(off_weights + MAX_DENDRITES * padded_n * 4)`
9. `total_state_size = align64(off_dtimers + MAX_DENDRITES * padded_n * 1)`

#### Конкретный пример для `padded_n = 64` (Единый Стандарт Per-Plane 64B Alignment)
В архитектуре `AxiEngine` утверждён единый межплатформенный стандарт индивидуально выровненных плоскостей (Per-Plane 64B Alignment). Альтернативный вариант плотного блока сомы без межплоскостного выравнивания (896 байт) объявляется **устаревшим и аннулированным**.

Для `padded_n = 64` точные смещения плоскостей в `.state` блобе вычисляются строго следующим образом:
- `off_voltage` = 64 (первая плоскость сразу после 16B заголовка, выравненная до 64B)
- `off_flags` = align64(64 + 64 * 4) = 320
- `off_thresh` = align64(320 + 64 * 1) = 384
- `off_timers` = align64(384 + 64 * 4) = 640
- `off_s2a` = align64(640 + 64 * 1) = 704
- `off_targets` = align64(704 + 64 * 4) = **960 байт**
- `off_weights` = align64(960 + 128 * 64 * 4) = 33 728 байт
- `off_dtimers` = align64(33 728 + 128 * 64 * 4) = 66 496 байт
- `total_state_size` = align64(66 496 + 128 * 64 * 1) = **74 688 байт**

### §6.3. Разъяснение инварианта 1166 байт

Логическая полезная нагрузка на один нейрон равна:
- Сома: $4 + 1 + 4 + 1 + 4 = 14$ байт.
- Дендриты (128 слотов): $128 \times (4 + 4 + 1) = 1152$ байта.
- **Итого**: $14 + 1152 = 1166$ байт.

> [!NOTE]
> Значение 1166 байт является чисто логической суммой полей. Физический размер файла на диске всегда больше из-за 64-байтового выравнивания каждой SoA-плоскости и выравнивания `padded_n`. Прямое чтение файла как массива AoS-структур по 1166 байт **запрещено**.

## §7. Форматы Заголовков Файлов

Все заголовки файлов бинарных дампов имеют фиксированный размер **16 байт** и выравнивание **16 байт** (`#[repr(C, align(16))]`).

### §7.1. Заголовок состояния сомы и дендритов (`StateFileHeader`)
- **Атрибуты**: `#[repr(C, align(16))]`
- **Размер**: 16 байт; **Выравнивание**: 16 байт.
- **Поля**:
  - `magic: [u8; 4]` — идентификатор файла (`*b"AXST"` i.e. `[0x41, 0x58, 0x53, 0x54]`).
  - `version: u32` — версия формата (текущая = 1).
  - `padded_n: u32` — количество нейронов с учетом выравнивания.
  - `total_axons: u32` — общее количество аксонов в шарде.

### §7.2. Заголовок аксонов (`AxonsFileHeader`)
- **Атрибуты**: `#[repr(C, align(16))]`
- **Размер**: 16 байт; **Выравнивание**: 16 байт.
- **Поля**:
  - `magic: [u8; 4]` — идентификатор файла (`*b"AXAX"` i.e. `[0x41, 0x58, 0x41, 0x58]`).
  - `version: u32` — версия формата (текущая = 1).
  - `total_axons: u32` — количество аксонов.
  - `_padding: u32` — 4 байта выравнивания до 16 байт.

### §7.3. Заголовок трассировки путей (`PathsFileHeader`)
- **Атрибуты**: `#[repr(C, align(16))]`
- **Размер**: 16 байт; **Выравнивание**: 16 байт.
- **Поля**:
  - `magic: [u8; 4]` — идентификатор файла (`*b"AXPT"` i.e. `[0x41, 0x58, 0x50, 0x54]`).
  - `version: u32` — версия формата (текущая = 1).
  - `total_axons: u32` — количество аксонов.
  - `max_segments: u32` — максимальное количество сегментов (256).

#### Раскладка файла `.paths`
1. `PathsFileHeader` (16 байт).
2. Массив фактических длин геометрии аксонов `lengths: [u16; total_axons]` (элементы типа `u16`, так как `MAX_SEGMENTS_PER_AXON = 256` превышает вместимость `u8`).
3. Межплоскостной паддинг до границы 64 байт.
4. Матрица координат `[total_axons * MAX_SEGMENTS_PER_AXON]` элементов `types::PackedPosition` (4 байта на ячейку).

#### Формулы расчёта геометрии `.paths`
- `calculate_paths_matrix_offset(total_axons: usize) -> usize`:
  $$\text{matrix\_offset} = \text{align64}(16 + \text{total\_axons} \times 2)$$
- `calculate_paths_file_size(total_axons: usize) -> usize`:
  $$\text{file\_size} = \text{matrix\_offset} + \text{total\_axons} \times \text{MAX\_SEGMENTS\_PER\_AXON} \times 4$$

При `total_axons == 0`: `matrix_offset = align64(16) = 64`, `file_size = 64` байт.
При `total_axons == 100`: `matrix_offset = align64(16 + 200) = align64(216) = 256`, `file_size = 256 + 100 * 256 * 4 = 102 656` байт.

## §10. Resolved Architectural Decisions (Принятые Решения Pass 2)

Все макеты памяти и межкрейтовые контракты в `layout` успешно закрыты и зафиксированы в процессе системного архитектурного прохода Pass 2.

1. **[RESOLVED] Единый стандарт смещений `.state` блоба и квант выравнивания (REV-LAYOUT-001)**:
   - *Решение*: В качестве единственного стандарта выравнивания утвержден режим **Per-Plane 64B Alignment** (`PADDED_N_ALIGNMENT = 64`). Все SoA-плоскости (включая первую соматическую плоскость `soma_voltage` по смещению 64B сразу после 16B заголовка) выравниваются по границе 64 байт. Альтернативный вариант плотного соматического блока (896B) официально объявляется устаревшим и аннулированным. Точное смещение `dendrite_targets` для `padded_n = 64` составляет строго 960 байт.

2. **[RESOLVED] Монопольное владение `ShardVramPtrs` (REV-LAYOUT-002)**:
   - *Решение*: C-ABI DTO указателей `ShardVramPtrs` переносится под монопольное владение крейта `layout`. Крейт `compute-api` реэкспортирует этот тип из `layout`, концентрируя все бинарные контракты памяти в одном месте.

3. **[RESOLVED] Стандартизация сигнатур заголовков (REV-LAYOUT-003)**:
   - *Решение*: Сигнатуры magic во всех заголовках бинарных файлов приводятся к фиксированному байтовому массиву `[u8; 4]` для исключения ошибок эндианности: `StateFileHeader.magic` = `*b"AXST"`, `AxonsFileHeader.magic` = `*b"AXAX"`, `PathsFileHeader.magic` = `*b"AXPT"`.

4. **[RESOLVED] Синхронизация неактивных слотов `PackedTarget` (REV-LAYOUT-004)**:
   - *Решение*: Плоскость `dendrite_targets` использует единую семантику из `types v2.2`. Неактивные слоты (сырой `0` для zero-init и `EMPTY_PIXEL` для сброшенного синапса) проверяются вычислительными ядрами через предикат `PackedTarget::is_inactive()`.

5. **[RESOLVED] Вынесение отладочной структуры `EphysShm` (REV-LAYOUT-005)**:
   - *Решение*: Диагностический буфер разделяемой памяти `EphysShm` выносится из `layout` под монопольное владение крейтов `ipc` / `test-harness`. Крейт `layout` фокусируется исключительно на производственных макетах физической симуляции.

6. **[RESOLVED] Состав `.state` дампа (REV-LAYOUT-006)**:
   - *Решение*: Зафиксировано, что файл `.state` содержит строго 8 горячих SoA-плоскостей Дневной Фазы. Специфичные оффлайн-структуры Ночной Фазы хранятся отдельно и не захламляют горячий VRAM-макет.

7. **[RESOLVED] Аннулирование устаревшего сентинеля аксона (REV-LAYOUT-007)**:
   - *Решение*: Подтверждено полное аннулирование устаревшей записи `0xFFFFFFFF` из легаси-комментариев. Единственным стандартом сентинеля неактивности аксона зафиксирован `AXON_SENTINEL = 0x8000_0000` из `types v2.2`.
