## §1. Идентификация

| Поле | Значение |
|------|----------|
| **Имя крейта** | `compute-cpu` |
| **Слой** | Слой 3 — Вычислительные Бэкенды (Compute Hardware Backends) |
| **Тип** | Library (`lib`) |
| **no_std** | Нет (`false`) — требуется стандартная библиотека Rust, системный аллокатор кучи и многопоточный планировщик Rayon |
| **Описание** | CPU-реализация вычислительного бэкенда, имплементирующая трейт `ComputeBackend` из `compute-api`. Крейт хранит ресурсы хоста за дескрипторами `VramHandle`, выполняет автономное исполнение батчей горячего цикла (`run_day_batch`) на процессорах хоста и служит проверочной реализацией (reference/check implementation) для accelerator-бэкендов (CUDA/HIP) и дифференциального тестирования. Единым источником истины по математике является `physics`, а по макетам памяти — `layout`. |

## §2. Стек и Окружение

### §2.1. Внутренние зависимости (inbound)

| Крейт | Что используется | Зачем |
|-------|-----------------|-------|
| `types` (Слой 0) | Атомарные скалярные типы | Использование фундаментальных типов (`Tick`, `Voltage`, `SomaFlags`, `AXON_SENTINEL`) в вычислительных циклах. |
| `layout` (Слой 1) | Расчет смещений SoA-плоскостей | Нарезка загруженного блоба состояния хоста на безопасные внутренние слайсы с использованием `CACHE_LINE_BYTES` и `PADDED_N_ALIGNMENT`. |
| `physics` (Слой 0) | Чистая математика GLIF, GSOP, Active Tail | Выполнение физики интеграции мембранного потенциала и пластичности с гарантией побитовой идентичности. |
| `compute-api` (Слой 3) | `ComputeBackend`, `VramHandle`, DTO вызовов и `ComputeApiError` | Реализация базового контракта вычислений HAL и типизированной иерархии ошибок. |

> [!IMPORTANT]
> Крейт `compute-cpu` **не содержит** дублирующей физической математики и макетов памяти. Все формулы подключаются напрямую из `physics`, а смещения — из `layout`.

### §2.2. Зависимые Компоненты и Характеристики Бэкенда (`BackendCapabilities`)

| Крейт / Компонент | Роль в системе и взаимодействие |
|-------------------|---------------------------------|
| `compute` (Слой 3) | Фасад вычислений подсоединяет `CpuBackend` через feature flag `cpu` для обеспечения работы симулятора на CPU. |
| `test-harness` (Слой 3) | Использует `CpuBackend` в качестве проверочной реализации (reference implementation) при дифференциальном тестировании CUDA/HIP ядер. |

При вызове метода `capabilities()` структура `CpuBackend` возвращает фиксированные характеристики детерминированного CPU-вычислителя:
```rust
BackendCapabilities {
    lane_count: 1,                 // Скалярное/SIMD исполнение на ядро хоста
    supports_async: false,         // Синхронное исполнение батча хостом
    supports_ephys: false,         // Ограничено до фиксации DTO отладчика
    max_batch_ticks: 1000,         // Лимит тиков за один вызов run_day_batch
    alignment_bytes: 64,           // Требование выравнивания адресов кучи хоста
    pinned_host_required: false,   // Не требуется Pinned DMA память хоста
}
```

### §2.3. Внешние Зависимости

| Crate | Версия | Сфера использования |
|-------|--------|---------------------|
| `rayon` | `=1.11.0` | Многопоточное детерминированное распараллеливание вычислений по нейронным чанкам на ядрах CPU *(Версия выровнена с Workspace Cargo)*. |

> [!NOTE]
> Зависимости `bytemuck` и `slotmap` рассматриваются как внутренние детали реализации. Внешний публичный API крейта полностью избавлен от `anyhow` и сторонних типов.

### §2.4. Feature Flags

Секция публичных feature flags не используется. Крейт собирается как единая автономная библиотека вычислений на CPU.

## §4. Модель Ресурсов Хоста и Выравнивание (Resource Model & Alignment)

В контексте CPU-бэкенда понятие «VRAM» эквивалентно выделенному выровненному региону оперативной памяти хоста (Host RAM Allocation).

### §4.1. Разграничение Двух Видов Выравнивания
В модели ресурсов строго разграничены два типа выравнивания:
1. **Выравнивание Размерности Сомы (`PADDED_N_ALIGNMENT = 64`)**: Дополнение количества нейронов `n` до кратности 64 (`padded_n`). Определяется формулами крейта `layout`.
2. **Выравнивание Адреса Памяти Хоста (`CACHE_LINE_BYTES = 64`)**: Гарантия того, что базовый физический адрес выделенного буфера в куче хоста кратен 64 байтам. Это исключает эффект ложного разделения кэш-линий (False Sharing) между потоками CPU и обеспечивает эффективную работу SIMD-инструкций.

### §4.2. Межспецификационный Долг Конструктора `VramHandle`
> [!WARNING]
> В текущей версии `compute-api v2.0` дескриптор `VramHandle` содержит приватное поле и конструктор уровня `pub(crate)`. Из-за этого сторонние крейты бэкендов не могут создавать экземпляры `VramHandle` в методе `alloc_shard`. Данное противоречие зафиксировано как межспецификационный долг (cross-spec debt): крейт `compute-api` должен предоставить контролируемый фабричный метод (например, `VramHandle::new_for_backend`), не раскрывая сырое внутреннее числовое значение публично.

### §4.3. Аллокация и Загрузка Данных
1. **`alloc_shard(spec: ShardAllocSpec) -> Result<VramHandle, ComputeApiError>`**: Выделяет монолитный регион памяти хоста с адресом, выровненным по границе `CACHE_LINE_BYTES` (64 байта), под размерность `padded_n`, проверяет аппаратные лимиты и регистрирует ресурс под непрозрачным дескриптором `VramHandle`.
2. **`upload_shard(handle: VramHandle, upload: ShardUpload<'_>) -> Result<(), ComputeApiError>`**: Валидирует размеры байтовых блобов `state_blob` и `axons_blob` с помощью формул из `layout`, после чего копирует данные в выделенный ресурс хоста.
3. **Безопасность Указателей**: Публичный API не раскрывает `ShardVramPtrs`, сырые указатели (`*mut u8`) или слайсы с временем жизни за пределами вызова. Внутреннее безопасное нарезание слайсов выполняется с помощью проверенных хелперов `layout`.

## §5. Модель Выполнения и Порядок Этапов (Execution Model & Stage Order)

Метод `run_day_batch(handle, cmd)` выполняет последовательное и автономное исполнение `cmd.sync_batch_ticks` тиков в горячем цикле хоста.

### §5.1. Порядок Выполнения Этапов Внутри Тика (Tick Stage Order)
Каждый тик симуляции внутри батча выполняется в строго детерминированной последовательности:
1. **Валидация Параметров DTO**: Проверка соответствия длин массивов и емкостей `DayBatchCmd`.
2. **Инъекция Виртуальных Входов (Virtual Inputs)**: Запись сенсорных сигналов из битмаски `cmd.input_bitmask` в целевые нейроны по `cmd.virtual_offset` и `cmd.num_virtual_axons`.
3. **Инъекция Входных Спайков (Incoming Spikes)**: Считывание потикового массива спайковых идентификаторов из `cmd.incoming_spikes` с учетом емкости `cmd.max_spikes_per_tick` и фактического количества элементов на тик из `cmd.spike_counts`.
4. **Продвижение Аксонных Головок (Axon Head Propagation / Active Tail)**: Выполнение смещений спайков вдоль аксонных линий согласно математике `physics`.
5. **Обновление Состояния Нейронов (Neuron State Update)**: Интеграция мембранного потенциала GLIF, расчет гомеостаза, рефрактерных таймеров и генерация спайков через функции `physics`.
6. **Применение Синаптической Пластичности (Plasticity Application / GSOP)**: Обновление синаптических весов с учетом дофаминовой модуляции `cmd.dopamine` и Закона Дейла.
7. **Фиксация Результатов (Условный / Draft Этап)**: Запись соматических выходов производится исключительно при наличии утвержденных DTO в `compute-api`.

> [!NOTE]
> Точный формат передачи спайковых выходов и осциллограмм в `BatchResult` находится в статусе открытого вопроса (§11). До его фиксации CPU-бэкенд не создает сторонних публичных буферов.

## §6. Параллелизм, Защита от Гонок и Детерминизм (Parallelism & Race Protection)

Для распараллеливания вычислений по нейронным чанкам используется планировщик `Rayon`.

1. **Строгий Детерминизм (Bit-to-Bit Identity)**: Результат выполнения батча при одинаковых входных данных должен быть побитово идентичен независимо от количества используемых потоков CPU и порядка выполнения задач планировщиком.
2. **Запрет Зависимых от Порядка Редукций**: Запрещено использование неассоциативных или зависящих от порядка операций объединения данных между параллельными итераторами.
3. **Защита от Гонок Данных при Записи Спайков (Spike Write Race Protection)**:
   - В процессе обновления нейронов происходит чтение дендритов/аксонных головок и генерация новых спайков, записываемых в `axon_heads` через карту отображения `soma_to_axon`.
   - Параллельное исполнение этапа обновления нейронов через Rayon допустимо **строго при условии**, что карта `soma_to_axon` гарантирует соответствие «один-к-одному» (дисюнктные зоны записи между параллельными чанками), либо запись сгенерированных спайков выполняется через промежуточный изолированный буфер накопления (Double-Buffering / Staging Buffer). Прямая конкурирующая запись в чужие чанки памяти запрещена.
4. **Отсутствие Внутренних Блокировок**: Поскольку метод трейта принимает `&mut self`, доступ к реестру ресурсов последователен. Использование дополнительных мьютексов (`Mutex`/`RwLock`) внутри горячего цикла вычислений запрещено.

## §7. Политика Обработки Ошибок (Error Policy)

Крейт `compute-cpu` гарантирует полное отсутствие паник (`panic!`) при любых некорректных входных DTO рантайма. Все сбои детерминированно транслируются в `ComputeApiError`:

- Несоответствие выравнивания или размеров геометрии $\to$ `ComputeApiError::InvalidShape` / `AlignmentViolation`.
- Несоответствие размера загружаемого блоба $\to$ `ComputeApiError::SizeMismatch`.
- Передача невалидного, чужого или освобожденного дескриптора $\to$ `ComputeApiError::InvalidHandle` / `ForeignHandle` / `AlreadyFreed`.
- Превышение емкости спайков за тик $\to$ `ComputeApiError::CapacityExceeded`.
- Нехватка оперативной памяти хоста $\to$ `ComputeApiError::OutOfMemory`.
- Ошибки в структуре команды батча $\to$ `ComputeApiError::InvalidBatch`.

Метод `free_shard(handle)` выгружает указанный ресурс. Метод `teardown()` атомарно очищает весь реестр ресурсов хоста; повторный вызов `teardown()` является безопасным и идемпотентным.

## §8. Сохранение Совместимости с Легаси-Фактами (Legacy Behavioral Facts)

В спецификации `compute-cpu` зафиксированы следующие проверочные факты поведения ядра симуляции:

1. **Маркер Сентинела Аксона (`AXON_SENTINEL`)**: Проверка неактивности головы аксона выполняется строго по побитовому правилу с использованием константы `types::AXON_SENTINEL` (`0x80000000`) во избежание проскакивания при $v_{\text{seg}} > 1$.
2. **Двойная Проверка Tombstone в Target (`0` vs `EMPTY_PIXEL`)**: Обработка неактивных синаптических таргетов поддерживает единую политику двойной проверки, утверждаемую в крейтах `types`/`physics`.
3. **Семантика Heartbeat / DDS**: Периодический спайк Heartbeat публикует спайк, но **не сбрасывает** мембранный потенциал $V$, **не взводит** рефрактерный таймер и не меняет параметры гомеостаза, если не произошел полноценный GLIF-спайк.
4. **Затухание Гомеостаза**: Затухание гомеостатического пенальти продолжается во время рефрактерного периода.
5. **Сохранение Бит Типа Сомы**: При изменении бит спайка или спайковой вспышки (Burst) в `SomaFlags` биты типа нейрона должны строго сохраняться.
6. **Дофаминовая Модуляция в GSOP**: Изменение весов синапсов при пластичности GSOP учитывает сигналы награды и наказания D1/D2.
7. **Уплотнение Связей (Sort & Prune)**: Процедура сортировки и удаления деградировавших связей сохраняет сильные тормозные веса и упаковывает массив без дыр.

## §9. Требуемые Инварианты

- **INV-COMPUTE-CPU-001**: Структура `CpuBackend` имплементирует трейт `ComputeBackend` из крейта `compute-api`.
- **INV-COMPUTE-CPU-002**: Вызов `kind()` возвращает строго `BackendKind::Cpu`.
- **INV-COMPUTE-CPU-003**: Память хоста, выделяемая под шард, имеет физический адрес, выровненный по границе 64 байт (`CACHE_LINE_BYTES`), а количество нейронов кратно `PADDED_N_ALIGNMENT`.
- **INV-COMPUTE-CPU-004**: Побитовая идентичность результатов вычислений сохраняется при однопоточном и многопоточном многозадачном исполнении Rayon.
- **INV-COMPUTE-CPU-005**: Параллельное обновление нейронов гарантирует отсутствие гонок данных при записи спайков в `axon_heads` за счет дисюнктной карты `soma_to_axon` или буфера накопления.
- **INV-COMPUTE-CPU-006**: Публичный API бэкенда не раскрывает сырые указатели (`*mut u8`) и C-ABI структуры указателей.
- **INV-COMPUTE-CPU-007**: Вычисления не дублируют формулы интеграции мембранного потенциала и пластичности из крейта `physics`.
- **INV-COMPUTE-CPU-008**: Любой некорректный входной DTO приводит к возврату `Result::Err(ComputeApiError)`, паники запрещены.

## §10. Golden Tests / Обязательная Матрица Тестирования

Крейт `compute-cpu` обязан быть покрыт набором автоматических юнит- и интеграционных тестов:

1. **Реализация Трейта HAL (`test_cpu_implements_compute_backend`)**: Проверка полной имплементации `ComputeBackend`.
2. **Идентификация Бэкенда (`test_cpu_backend_kind`)**: Проверка возврата `BackendKind::Cpu`.
3. **Детекция Характеристик (`test_cpu_backend_capabilities`)**: Проверка совпадения возвращаемой структуры `BackendCapabilities` с описанными значениями §2.2.
4. **Аллокация и Выравнивание (`test_cpu_alloc_shard_alignment`)**: Проверка кратности адреса аллоцированного буфера 64 байтам.
5. **Браковка Неверных Размеров Блобов (`test_cpu_upload_rejects_bad_sizes`)**: Проверка возврата `SizeMismatch` при вызове `upload_shard` с неверными размерами.
6. **Защита от Невалидных Дескрипторов (`test_cpu_rejects_invalid_handles`)**: Проверка вызовов с недействительными, чужими или освобожденными `VramHandle`.
7. **Валидация Команды Батча (`test_cpu_run_day_batch_validation`)**: Проверка валидации длин массивов и лимитов в `DayBatchCmd`.
8. **Гарантия Отсутствия Паник (`test_cpu_panic_free_on_invalid_dto`)**: Проверка возврата `Result::Err` при любых битых параметрах.
9. **Побитовый Детерминизм Вычислений (`test_cpu_bit_to_bit_determinism`)**: Проверка идентичности выходов при повторных запусках на одинаковых входах.
10. **Эквивалентность Однопоточного и Многопоточного Режимов (`test_cpu_rayon_thread_count_equivalence`)**: Сравнение результатов вычислений при 1 потоке и N потоках Rayon.
11. **Проверка Защиты от Гонки при Записи Спайков (`test_cpu_spike_write_race_safety`)**: Тестирование изолированной записи в `axon_heads` при параллельном обновлении.
12. **Изоляция Публичного API от Сырых Указателей (`test_cpu_no_raw_pointers_in_api`)**: Компиляционная проверка сигнатур.
13. **Отсутствие Зависимости от anyhow (`test_cpu_no_anyhow_in_api`)**: Гарантия отсутствия `anyhow` в публичных типах.
14. **Проверка Легаси-Инвариантов Симуляции (`test_cpu_legacy_behavioral_facts`)**: Тесты на отработку `types::AXON_SENTINEL`, затухание гомеостаза в рефрактере и сохранение типов сомы.

## §11. Open Questions / Review Debt (Открытые Вопросы и Противоречия)

В процессе анализа спецификации CPU-бэкенда выявлены следующие открытые вопросы для согласования:

1. **Модель Фабричного Конструктора `VramHandle` в `compute-api`**:
