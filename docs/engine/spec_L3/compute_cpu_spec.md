## §1. Идентификация

| Поле | Значение |
|---|---|
| **Имя крейта** | `compute-cpu` |
| **Слой** | Слой 3 — Вычислительные Бэкенды (Compute Hardware Backends) |
| **Тип** | Library (`lib`) |
| **no_std** | Нет (`false`) — требуется стандартная библиотека Rust, системный аллокатор кучи и изолированный многопоточный планировщик Rayon |
| **Описание** | CPU-реализация вычислительного бэкенда, имплементирующая трейт `ComputeBackend` из `compute-api`. Крейт хранит ресурсы хоста за дескрипторами `VramHandle`, выполняет автономное исполнение батчей горячего цикла (`run_day_batch`) на процессорах хоста и служит проверочной реализацией (reference/check implementation) для accelerator-бэкендов (CUDA/HIP) и дифференциального тестирования. Единым источником истины по математике является `physics`, а по макетам памяти — `layout`. |

## §2. Стек и Окружение

### §2.1. Внутренние зависимости (inbound)

| Крейт | Что используется | Зачем |
|---|---|---|
| `types` (Слой 0) | Атомарные скалярные типы, `PackedTarget::is_inactive()` | Использование типов (`Tick`, `Voltage`, `SomaFlags`, `AXON_SENTINEL`) и проверка неактивных синапсов. |
| `layout` (Слой 1) | Расчет смещений SoA-плоскостей, `VariantParameters`, `VARIANT_LUT_LEN` | Нарезка загруженного блоба состояния хоста на безопасные внутренние слайсы с использованием `CACHE_LINE_BYTES`, `PADDED_N_ALIGNMENT` и таблицы вариантов. |
| `physics` (Слой 0) | Чистая математика GLIF, GSOP, Active Tail, Stochastic Heartbeat | Выполнение физики интеграции мембранного потенциала и пластичности с гарантией побитовой идентичности. |
| `compute-api` (Слой 3) | `ComputeBackend`, `VramHandle`, DTO вызовов, DTO обслуживания и `ComputeApiError` | Реализация базового контракта вычислений HAL и типизированной иерархии ошибок. |

> [!IMPORTANT]
> Крейт `compute-cpu` **не содержит** дублирующей физической математики и макетов памяти. Все формулы подключаются напрямую из `physics`, а смещения — из `layout`.

### §2.2. Зависимые Компоненты и Характеристики Бэкенда (`BackendCapabilities`)

| Крейт / Компонент | Роль в системе и взаимодействие |
|---|---|
| `compute` (Слой 3) | Фасад вычислений подсоединяет `CpuBackend` через feature flag `cpu` для обеспечения работы симулятора на CPU. |
| `test-harness` (Слой 3) | Использует `CpuBackend` в качестве проверочной реализации (reference implementation) при дифференциальном тестировании CUDA/HIP ядер. |

При вызове метода `capabilities()` структура `CpuBackend` возвращает фиксированные характеристики детерминированного CPU-вычислителя:
```rust
BackendCapabilities {
    lane_count: 1,                 // Скалярное/SIMD исполнение на ядро хоста
    supports_async: false,         // Синхронное исполнение батча хостом
    supports_ephys: false,         // Ограничено выгрузкой debug_snapshot full-state
    max_batch_ticks: 1000,         // Лимит тиков за один вызов run_day_batch
    alignment_bytes: 64,           // Требование выравнивания адресов кучи хоста (CACHE_LINE_BYTES)
    pinned_host_required: false,   // Не требуется Pinned DMA память хоста
}
```

### §2.3. Внешние Зависимости

| Crate | Версия | Сфера использования |
|---|---|---|
| `rayon` | `=1.11.0` | Многопоточное детерминированное распараллеливание вычислений по нейронным чанкам через изолированный `rayon::ThreadPool`. |
| `bytemuck` | `=1.25.0` | Безопасная интерпретация слайсов байт в структуры `layout`. |

> [!NOTE]
> Зависимость `slotmap` исключена из проекта. Вместо неё используется внутренний легкий регистр на базе `Vec` с поколениями (`generation counter`).

## §4. Модель Ресурсов Хоста, Управление Памятью и Указатели

В контексте CPU-бэкенда понятие «VRAM» эквивалентно выделенному выровненному региону оперативной памяти хоста (Host RAM Allocation).

### §4.1. Разграничение Выравниваний (Alignment Distinction)
1. **Выравнивание Размерности Сомы (`PADDED_N_ALIGNMENT = 64`)**: Обязательное дополнение количества нейронов `n` в шарде до кратности 64 (`padded_n`). Все SoA-плоскости внутри блоба состояния рассчитываются строго по `padded_n`.
2. **Физический Адрес Кучи (`CACHE_LINE_BYTES = 64`)**: Обязательное выравнивание физического адреса выделенного монолитного буфера памяти хоста по границе 64 байт для предотвращения False Sharing в L1/L2 кэшах CPU при многопоточном исполнении.

### §4.2. Использование `unsafe` в Приватном Модуле
Рукописный `unsafe`-код строго запрещен в публичном API крейта `compute-cpu`. Допускается локализованное использование `unsafe` строго внутри маленького приватного модуля аллокации и представления выровненной памяти (`aligned_mem`), отвечающего за выделение и преобразование 64B-aligned буферов хоста. Каждый `unsafe`-блок обязателен к сопровождению исчерпывающей секцией `# Safety` с указанием инвариантов выравнивания и валидности слайсов.

### §4.3. Аллокация и Загрузка Данных
1. **`alloc_shard(spec: ShardAllocSpec) -> Result<VramHandle, ComputeApiError>`**: Выделяет монолитный регион памяти хоста с базовым адресом, выровненным по границе 64 байт (`CACHE_LINE_BYTES`), регистрирует ресурс во внутреннем реестре с generation counter и конструирует `VramHandle` через фабрику `VramHandle::from_raw_parts(BackendKind::Cpu, id, generation)`.
2. **`upload_shard(handle: VramHandle, upload: ShardUpload<'_>) -> Result<(), ComputeApiError>`**: Валидирует размеры байтовых блобов `state_blob` и `axons_blob` с помощью `validate_upload`, копирует байтовые массивы состояния и аксонов в выделенный ресурс хоста, а также производит полное копирование таблицы `upload.variant_table` во внутреннее backend-owned хранилище ресурса шарда (`[VariantParameters; VARIANT_LUT_LEN]`). Удерживать заимствованную ссылку (`borrowed reference`) из `ShardUpload` после выхода из вызова `upload_shard` категорически запрещено.
3. **`export_maintenance_state(handle, out) / import_maintenance_state(handle, src)`**:
   - Выполняют перенос данных (состояние и аксоны) между RAM-буферами бэкенда и структурами DTO.
   - Копирование производится синхронно с помощью `memcpy`.
   - Входные параметры и размеры буферов строго валидируются через функции `validate_maintenance_export` / `validate_maintenance_import` на стороне HAL-контракта.

## §5. Модель Выполнения и Порядок Этапов (Execution Model & Stage Order)

Метод `run_day_batch(handle, cmd)` выполняет последовательное блокирующее исполнение `cmd.sync_batch_ticks` тиков в горячем цикле хоста.

### §5.1. Полный Порядок Выполнения Этапов Внутри Тика (7-Stage Execution Graph)
1. **Валидация Параметров DTO**: Проверка длин массивов `incoming_spike_counts`, `output_spike_counts` и емкостей `DayBatchCmd` через `validate_day_batch_cmd`.
2. **Инъекция Виртуальных Входов (Virtual Inputs)**: Запись сенсорных сигналов из битмаски `cmd.input_bitmask` в целевые нейроны по `cmd.virtual_offset` и `cmd.num_virtual_axons`.
3. **Инъекция Входных Спайков (Incoming Spikes)**: Считывание потикового массива спайковых идентификаторов из `cmd.incoming_spikes` с учетом емкости `cmd.max_spikes_per_tick` и элементов из `cmd.incoming_spike_counts`.
4. **Продвижение Аксонных Головок (Active Tail Signal Propagation)**: Выполнение смещений спайков вдоль аксонных линий через `propagate_head` для всех 8 голов согласно математике `physics` с проверкой маркерного значения `types::AXON_SENTINEL`.
5. **Обновление Состояния Нейронов, Усталость и Соматический Сброс (Neuron State Update & Fatigue Integration)**:
   - **Восстановление и интеграция усталости**: Для каждого дендритного слота: если `PackedTarget::is_inactive()`, `fatigue = 0`. Для живых слотов `fatigue` восстанавливается на `-1` за тик. Если хвост аксона активен на сегменте, импульс ослабляется по коэффициенту емкости `w * (cap - fat) / cap`, входной ток суммируется, а `fatigue` увеличивается на `+50`.
   - **Интеграция потенциала GLIF и Stochastic HeartBeat**: Вычисляется напряжение $V_{\text{new}}$ и предикаты спайка (`is_glif` и `is_heartbeat`).
   - **Полный соматический сброс**: При любом спайке (`final_spike = is_glif || is_heartbeat`): напряжение сомы сбрасывается к $V_{\text{reset}} = V_{\text{rest}} - A_{\text{ahp}}$, таймер рефрактерности взводится в `refractory_period`, а порог увеличивается на `homeostasis_penalty`. Сгенерированные выходящие спайки фиксируются в `cmd.output_spikes`.
6. **Применение Синаптической Пластичности (GSOP All-to-All STDP Pass)**: Для спайкировавших сом выполняется расчет All-to-All STDP по 8 аксональным головкам с учетом линейного Spatial Cooling, Soft Peak и вычитания `fatigue_penalty`.
7. **Эмиссия Локальных Головок Спайка (Local Spike Axon Head Emission)**: Инициализация новорожденных голов спайка `initial_axon_head(v_seg)` и их запись в `axon_heads` после выполнения этапа пластичности.

## §6. Минимальный Объем Реализации Stage 1 (Stage 1 Implementation Scope)

Для первого этапа реализации (Stage 1) крейта `compute-cpu` фиксируются следующие конкретные границы и артефакты:

1. **Жизненный Цикл HAL**: Полная корректная имплементация `alloc_shard`, `upload_shard`, `export_maintenance_state`, `import_maintenance_state`, `debug_snapshot`, `free_shard`, `teardown`.
2. **Локальный Горячий Цикл**: Исполнение `run_day_batch` поверх выровненных буферов хоста `state_blob`, `axons_blob` и сохраненного массива `variant_table`.
3. **Прямое Использование `physics`**: Потиковое применение функций интеграции GLIF, Stochastic Heartbeat, продвижения аксонов, Active Tail и пластичности GSOP.
4. **Единое Правило Проверки Синапсов**: Предикат `PackedTarget::is_inactive()` применяется как единственный early-exit механизм при обходе матриц дендритов.
5. **Контракт Выходных Спайков**: Заполнение выходящих массивов `cmd.output_spikes` и `cmd.output_spike_counts` строго по контракту `compute-api`.
6. **Dev-Only Проверочный Пример**: Добавление исполняемого примерa `examples/compute_cpu_probe.rs` под флагом `dev-only` для локальной интеграционной проверки загрузки и прогона тестового батча.

## §7. Параллелизм, Изолированный Rayon и Детерминизм

1. **Собственный `rayon::ThreadPool`**: Структура `CpuBackend` владеет экземпляром изолированного пула потоков `rayon::ThreadPool` (конфигурируемого при создании через `CpuBackendConfig`), исключая конфликты и взаимоблокировки с глобальным пулом потоков процесса.
2. **Строгий Побитовый Детерминизм**: Результат выполнения батча при одинаковых входных данных побитово идентичен независимо от количества потоков Rayon (1 поток vs N потоков).
3. **Защита от Гонок Данных (Race Protection)**: Запись спайков в `axon_heads` при параллельной обработке сом выполняется либо через фиксированную дисюнктную карту адресов `soma_to_axon` либо через буферы накопления, исключая Data Races без накладных расходов атомиков.

---

## §8. Политика Обработки Ошибок и Отсутствия Паник (No Panic Policy)

Все сбои детерминированно транслируются в `ComputeApiError`. Прямые паники (`unwrap`, `expect`, `panic!`) внутри производственного кода бэкенда категорически запрещены. Метод `free_shard(handle)` освобождает конкретный шард, а `teardown()` атомарно очищает весь реестр ресурсов хоста.

## §9. Требуемые Инварианты

- **INV-COMPUTE-CPU-001**: Структура `CpuBackend` имплементирует трейт `ComputeBackend` из `compute-api`.
- **INV-COMPUTE-CPU-002**: Вызов `kind()` возвращает строго `BackendKind::Cpu`.
- **INV-COMPUTE-CPU-003**: Память хоста, выделяемая под шард, имеет физический адрес, выровненный по границе 64 байт (`CACHE_LINE_BYTES`).
- **INV-COMPUTE-CPU-004**: Побитовая идентичность результатов вычислений сохраняется при однопоточном и многопоточном исполнении Rayon.
- **INV-COMPUTE-CPU-005**: Проверка неактивных таргетов синапсов выполняется строго предикатом `PackedTarget::is_inactive()`.
- **INV-COMPUTE-CPU-006**: Публичные сигнатуры и DTO не содержат сырых указателей (`*mut u8`) и внешних типов ошибок (`anyhow`).

## §10. Golden Tests / Обязательная Матрица Тестирования

Крейт `compute-cpu` обязан быть покрыт набором автоматических юнит- и интеграционных тестов:

1. **Реализация Трейта HAL (`test_cpu_implements_compute_backend`)**: Проверка полной имплементации `ComputeBackend`.
2. **Идентификация Бэкенда (`test_cpu_backend_kind`)**: Проверка возврата `BackendKind::Cpu`.
3. **Детекция Характеристик (`test_cpu_backend_capabilities`)**: Проверка совпадения возвращаемой структуры `BackendCapabilities` с описанными значениями §2.2.
4. **Аллокация и Выравнивание (`test_cpu_alloc_shard_alignment`)**: Проверка кратности адреса аллоцированного буфера 64 байтам и создания `VramHandle`.
5. **Браковка Неверных Размеров Блобов (`test_cpu_upload_rejects_bad_sizes`)**: Проверка возврата `SizeMismatch` при вызове `upload_shard` с неверными размерами.
6. **Защита от Невалидных Дескрипторов (`test_cpu_rejects_invalid_handles`)**: Проверка вызовов с недействительными, чужими или освобожденными `VramHandle`.
7. **Валидация Команды Батча (`test_cpu_run_day_batch_validation`)**: Проверка валидации длин массивов и лимитов в `DayBatchCmd`.
8. **Гарантия Отсутствия Паник (`test_cpu_panic_free_on_invalid_dto`)**: Проверка возврата `Result::Err` при любых битых параметрах.
9. **Побитовый Детерминизм Вычислений (`test_cpu_bit_to_bit_determinism`)**: Проверка идентичности выходов при повторных запусках на одинаковых входах.
10. **Эквивалентность Однопоточного и Многопоточного Режимов (`test_cpu_rayon_thread_count_equivalence`)**: Сравнение результатов вычислений при 1 потоке и N потоках Rayon.
11. **Проверка Защиты от Гонки при Записи Спайков (`test_cpu_spike_write_race_safety`)**: Тестирование изолированной записи в `axon_heads` при параллельном обновлении.
12. **Изоляция Публичного API от Сырых Указателей (`test_cpu_no_raw_pointers_in_api`)**: Компиляционная проверка сигнатур.
13. **Отсутствие Зависимости от anyhow (`test_cpu_no_anyhow_in_api`)**: Гарантия отсутствия `anyhow` в публичных типах.
14. **Проверка Легаси-Инвариантов Симуляции и Изоляции Таргетов (`test_cpu_legacy_behavioral_facts`)**: Тесты на отработку `types::AXON_SENTINEL`, затухание гомеостаза в рефрактере и проверку таргетов через `PackedTarget::is_inactive()`.
15. **Сквозной экспорт и импорт обслуживания (`test_cpu_maintenance_export_import_roundtrip`)**: Верификация того, что данные после раундтрипа экспорта-импорта остаются побитово идентичными.
16. **Валидация буферов обслуживания бэкендом (`test_cpu_maintenance_buffers_validation`)**: Проверка, что бэкенд отклоняет экспорт/импорт с неверными размерами буферов.

## §11. Resolved Architectural Decisions (Принятые Решения Pass 2.2)

1. **[RESOLVED] Границы Маршрутизации Данных Отладчика Ephys (REV-COMPUTE-CPU-004 / Pass 2.2)**:
   - *Решение*: В `compute-cpu` реализуется только метод `debug_snapshot` для выгрузки полного бинарного состояния `state_blob` и `axons_blob`. Снимок осциллограмм и внешняя IPC-маршрутизация не входят в сферу ответственности `compute-cpu`.
2. **[RESOLVED] Зона Владения Операциями Сортировки и Уплотнения Связей (REV-COMPUTE-CPU-005 / Pass 2.2)**:
   - *Решение*: Операции ночной фазы (`sort_and_prune`), сортировки синапсов и удаления деградировавших связей вынесены на уровень верхних слоев (`compute`/`runtime`/`topology`) и не являются методами `ComputeBackend`.
3. **[RESOLVED] Фиксация Внутренних Зависимостей (REV-COMPUTE-CPU-006 / Pass 2.2)**:
   - *Решение*: Использование `rayon = "=1.11.0"` и `bytemuck = "=1.25.0"` разрешено. Зависимость `slotmap` исключена из состава бэкенда; вместо неё используется внутренний регистр на базе `Vec` с генерационными счетчиками.
