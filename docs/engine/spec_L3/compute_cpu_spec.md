## §1. Идентификация

| Поле | Значение |
|---|---|
| **Имя крейта** | `compute-cpu` |
| **Слой** | Слой 3 — Вычислительные Бэкенды (Compute Hardware Backends) |
| **Тип** | Library (`lib`) |
| **no_std** | Нет (`false`) — требуется стандартная библиотека Rust, системный аллокатор кучи и многопоточный планировщик Rayon |
| **Описание** | CPU-реализация вычислительного бэкенда, имплементирующая трейт `ComputeBackend` из `compute-api`. Крейт хранит ресурсы хоста за дескрипторами `VramHandle`, выполняет автономное исполнение батчей горячего цикла (`run_day_batch`) на процессорах хоста и служит проверочной реализацией (reference/check implementation) для accelerator-бэкендов (CUDA/HIP) и дифференциального тестирования. Единым источником истины по математике является `physics`, а по макетам памяти — `layout`. |

## §2. Стек и Окружение

### §2.1. Внутренние зависимости (inbound)

| Крейт | Что используется | Зачем |
|---|---|---|
| `types` (Слой 0) | Атомарные скалярные типы, `PackedTarget::is_inactive()` | Использование типов (`Tick`, `Voltage`, `SomaFlags`, `AXON_SENTINEL`) и проверка неактивных синапсов. |
| `layout` (Слой 1) | Расчет смещений SoA-плоскостей | Нарезка загруженного блоба состояния хоста на безопасные внутренние слайсы с использованием `CACHE_LINE_BYTES` и `PADDED_N_ALIGNMENT`. |
| `physics` (Слой 0) | Чистая математика GLIF, GSOP, Active Tail | Выполнение физики интеграции мембранного потенциала и пластичности с гарантией побитовой идентичности. |
| `compute-api` (Слой 3) | `ComputeBackend`, `VramHandle`, DTO вызовов и `ComputeApiError` | Реализация базового контракта вычислений HAL и типизированной иерархии ошибок. |

> [!IMPORTANT]
> Крейт `compute-cpu` **не содержит** дублирующей физической математики и макетов памяти. Все формулы подключаются напрямую из `physics`, а смещения — из `layout`.

### §2.2. Зависимые Компоненты и Характеристики Бэкенда (`BackendCapabilities`)

| Крейт / Компонент | Роль в системе и взаимодействие |
|---|---|
| `compute` (Слой 3) | Фасад вычислений подсоединяет `CpuBackend` через feature flag `cpu` для обеспечения работы симулятора на CPU. |
| `test-harness` (Слой 3) | Использует `CpuBackend` в качестве проверочной реализации (reference implementation) при дифференциальном тестировании CUDA/HIP ядер. |

При вызове метода `capabilities()` структура `CpuBackend` возвращает фиксированные характеристики детерминированного CPU-вычислителя:
```rust
BackendCapabilities {
    lane_count: 1,                 // Скалярное/SIMD исполнение на ядро хоста
    supports_async: false,         // Синхронное исполнение батча хостом
    supports_ephys: false,         // Ограничено до фиксации DTO отладчика
    max_batch_ticks: 1000,         // Лимит тиков за один вызов run_day_batch
    alignment_bytes: 64,           // Требование выравнивания адресов кучи хоста (CACHE_LINE_BYTES)
    pinned_host_required: false,   // Не требуется Pinned DMA память хоста
}
```

### §2.3. Внешние Зависимости

| Crate | Версия | Сфера использования |
|---|---|---|
| `rayon` | `=1.11.0` | Многопоточное детерминированное распараллеливание вычислений по нейронным чанкам на ядрах CPU. |
| `slotmap` | `=1.0.7` | Внутренний детерминированный реестр аллокаций памяти хоста за дескрипторами `VramHandle`. |
| `bytemuck` | `=1.25.0` | Безопасная интерпретация слайсов байт в структуры `layout`. |

## §4. Модель Ресурсов Хоста и Выравнивание (Resource Model & Alignment)

В контексте CPU-бэкенда понятие «VRAM» эквивалентно выделенному выровненному региону оперативной памяти хоста (Host RAM Allocation).

### §4.1. Разграничение Выравниваний (Alignment Distinction)
1. **Выравнивание Размерности Сомы (`PADDED_N_ALIGNMENT = 64`)**: Обязательное дополнение количества нейронов `n` в шарде до кратности 64 (`padded_n`). Все SoA-плоскости внутри блоба состояния рассчитываются строго по `padded_n`.
2. **Физический Адрес Кучи (`CACHE_LINE_BYTES = 64`)**: Обязательное выравнивание физического адреса выделенного монолитного буфера памяти хоста по границе 64 байт для предотвращения False Sharing в L1/L2 кэшах CPU при многопоточном исполнении.

### §4.2. Аллокация и Загрузка Данных
1. **`alloc_shard(spec: ShardAllocSpec) -> Result<VramHandle, ComputeApiError>`**: Выделяет монолитный регион памяти хоста с базовым адресом, выровненным по границе 64 байт (`CACHE_LINE_BYTES`), регистрирует ресурс во внутреннем реестре `slotmap` и конструирует `VramHandle` через фабрику `VramHandle::from_raw_parts(BackendKind::Cpu, id, generation)`.
2. **`upload_shard(handle: VramHandle, upload: ShardUpload<'_>) -> Result<(), ComputeApiError>`**: Валидирует размеры байтовых блобов `state_blob` и `axons_blob` с помощью формул из `layout`, после чего осуществляет копирование данных в выделенный ресурс хоста.

## §5. Модель Выполнения и Порядок Этапов (Execution Model & Stage Order)

Метод `run_day_batch(handle, cmd)` выполняет последовательное блокирующее исполнение `cmd.sync_batch_ticks` тиков в горячем цикле хоста.

### §5.1. Полный Порядок Выполнения Этапов Внутри Тика (Tick Stage Order)
1. **Валидация Параметров DTO**: Проверка длин массивов `incoming_spike_counts`, `output_spike_counts` и емкостей `DayBatchCmd`.
2. **Инъекция Виртуальных Входов (Virtual Inputs)**: Запись сенсорных сигналов из битмаски `cmd.input_bitmask` в целевые нейроны по `cmd.virtual_offset` и `cmd.num_virtual_axons`.
3. **Инъекция Входных Спайков (Incoming Spikes)**: Считывание потикового массива спайковых идентификаторов из `cmd.incoming_spikes` с учетом емкости `cmd.max_spikes_per_tick` и элементов из `cmd.incoming_spike_counts`.
4. **Продвижение Аксонных Головок (Active Tail)**: Выполнение смещений спайков вдоль аксонных линий согласно математике `physics` с проверкой маркерного значения `types::AXON_SENTINEL`.
5. **Обновление Состояния Нейронов (Neuron State Update)**: Многопоточная интеграция мембранного потенциала GLIF через Rayon, расчет гомеостаза, рефрактерных таймеров и генерация спайков с использованием функций из `physics`. Проверка неактивных синаптических таргетов выполняется строго через предикат `PackedTarget::is_inactive()` из `types v2.2` (`0` И `EMPTY_PIXEL`). Сгенерированные выходящие спайки записываются в `cmd.output_spikes`.
6. **Применение Синаптической Пластичности (GSOP)**: Обновление синаптических весов с учетом дофаминовой модуляции `cmd.dopamine` и Закона Дейла.

## §6. Параллелизм, Защита от Гонок и Детерминизм (Parallelism & Race Protection)

1. **Строгий Побитовый Детерминизм**: Результат выполнения батча при одинаковых входных данных побитово идентичен независимо от количества потоков Rayon (1 поток vs N потоков).
2. **Защита от Гонок Данных (Race Protection)**: Запись спайков в `axon_heads` при параллельной обработке сом выполняется либо через фиксированную дисюнктную карту адресов `soma_to_axon`, либо через буферы накопления, исключая Data Races без накладных расходов атомиков.

---

## §7. Политика Обработки Ошибок и Отсутствия Паник (No Panic Policy)

Все сбои детерминированно транслируются в `ComputeApiError`. Прямые паники (`unwrap`, `expect`, `panic!`) внутри производственного кода бэкенда категорически запрещены. Метод `free_shard(handle)` освобождает конкретный шард, а `teardown()` атомарно очищает весь реестр ресурсов хоста.

---

## §8. Требуемые Инварианты

- **INV-COMPUTE-CPU-001**: Структура `CpuBackend` имплементирует трейт `ComputeBackend` из `compute-api`.
- **INV-COMPUTE-CPU-002**: Вызов `kind()` возвращает строго `BackendKind::Cpu`.
- **INV-COMPUTE-CPU-003**: Память хоста, выделяемая под шард, имеет физический адрес, выровненный по границе 64 байт (`CACHE_LINE_BYTES`).
- **INV-COMPUTE-CPU-004**: Побитовая идентичность результатов вычислений сохраняется при однопоточном и многопоточном исполнении Rayon.
- **INV-COMPUTE-CPU-005**: Проверка неактивных таргетов синапсов выполняется строго предикатом `PackedTarget::is_inactive()`.
- **INV-COMPUTE-CPU-006**: Публичные сигнатуры и DTO не содержат сырых указателей (`*mut u8`) и внешних типов ошибок (`anyhow`).

## §9. Golden Tests / Обязательная Матрица Тестирования

Крейт `compute-cpu` обязан быть покрыт набором автоматических юнит- и интеграционных тестов:

1. **Реализация Трейта HAL (`test_cpu_implements_compute_backend`)**: Проверка полной имплементации `ComputeBackend`.
2. **Идентификация Бэкенда (`test_cpu_backend_kind`)**: Проверка возврата `BackendKind::Cpu`.
3. **Детекция Характеристик (`test_cpu_backend_capabilities`)**: Проверка совпадения возвращаемой структуры `BackendCapabilities` с описанными значениями §2.2.
4. **Аллокация и Выравнивание (`test_cpu_alloc_shard_alignment`)**: Проверка кратности адреса аллоцированного буфера 64 байтам и создания `VramHandle`.
5. **Браковка Неверных Размеров Блобов (`test_cpu_upload_rejects_bad_sizes`)**: Проверка возврата `SizeMismatch` при вызове `upload_shard` с неверными размерами.
6. **Защита от Невалидных Дескрипторов (`test_cpu_rejects_invalid_handles`)**: Проверка вызовов с недействительными, чужими или освобожденными `VramHandle`.
7. **Валидация Команды Батча (`test_cpu_run_day_batch_validation`)**: Проверка валидации длин массивов и лимитов в `DayBatchCmd`.
8. **Гарантия Отсутствия Паник (`test_cpu_panic_free_on_invalid_dto`)**: Проверка возврата `Result::Err` при любых битых параметрах.
9. **Побитовый Детерминизм Вычислений (`test_cpu_bit_to_bit_determinism`)**: Проверка идентичности выходов при повторных запусках на одинаковых входах.
10. **Эквивалентность Однопоточного и Многопоточного Режимов (`test_cpu_rayon_thread_count_equivalence`)**: Сравнение результатов вычислений при 1 потоке и N потоках Rayon.
11. **Проверка Защиты от Гонки при Записи Спайков (`test_cpu_spike_write_race_safety`)**: Тестирование изолированной записи в `axon_heads` при параллельном обновлении.
12. **Изоляция Публичного API от Сырых Указателей (`test_cpu_no_raw_pointers_in_api`)**: Компиляционная проверка сигнатур.
13. **Отсутствие Зависимости от anyhow (`test_cpu_no_anyhow_in_api`)**: Гарантия отсутствия `anyhow` в публичных типах.
14. **Проверка Легаси-Инвариантов Симуляции и Изоляции Таргетов (`test_cpu_legacy_behavioral_facts`)**: Тесты на отработку `types::AXON_SENTINEL`, затухание гомеостаза в рефрактере и проверку таргетов через `PackedTarget::is_inactive()`.

## §10. Open Questions / Review Debt (Открытые Вопросы и Противоречия)

1. **Маршрутизация Данных Отладчика Ephys**:
   - *Контекст*: Отладочный съем осциллограмм требует доступа к SoA-массивам мембранных потенциалов.
   - *Вопрос*: Являются ли отладочные методы частью `ComputeBackend` или выносятся в отдельный сервис?

2. **Владелец Операций Сортировки и Уплотнения Связей (Sort & Prune)**:
   - *Контекст*: Операция `sort_and_prune` удаляет деградировавшие синапсы во время Ночной Фазы.
   - *Вопрос*: Относится ли `sort_and_prune` к методам `ComputeBackend` или выполняется на уровне `compute`/`runtime`?

3. **Необходимость Внутренних Зависимостей `slotmap` и `bytemuck`**:
   - *Контекст*: Текущая спецификация использует эти библиотеки как внутренние детали реализации.
   - *Вопрос*: Фиксируются ли эти крейты в качестве обязательных внутренних зависимостей `compute-cpu`?

4. **Управление Пулом Потоков Rayon (Global vs Custom Threadpool)**:
   - *Контекст*: По умолчанию Rayon использует глобальный пул потоков процесса.
   - *Вопрос*: Должен ли `CpuBackend` создавать и хранить собственный изолированный `rayon::ThreadPool` для предотвращения конфликтов с другими модулями движка?
