|---|---|
| Название | `compute-hip` |
| Слой | Слой 3 — Compute |
| Тип | Library (`lib`) |

## §3. Инварианты

### §3.1. Структурные инварианты

- **INV-COMPUTE-HIP-001**: *Уникальность контекста HipBackend*.
  - *Обоснование*: Структура `HipBackend` обязана содержать потокобезопасный реестр выделенных дескрипторов VRAM (например, `SlotMap` под `RwLock`). Перекрестные операции с чужими дескрипторами VRAM физически невозможны.
  - *Следствие нарушения*: Попытка обращения к видеопамяти другого шарда (Data Leakage/Memory Corruption), UB.
  - *Где проверяется*: Юнит-тест `test_hip_invalid_handle_checks`.

- **INV-COMPUTE-HIP-002**: *Коалесцированный доступ к VRAM (Coalesced Memory Access)*.
  - *Обоснование*: Все HIP-ядра симуляции Day Phase обязаны обращаться к глобальной памяти GPU (VRAM) непрерывными блоками по 64 потока (вейвфронт) по адресам SoA-массивов, выровненным по границам 64 байт (кэш-линия L2).
  - *Следствие нарушения*: Падение пропускной способности шины VRAM (до 3-5% от пика), срыв latency бюджета выполнения ядра Day Phase.
  - *Где проверяется*: Профилирование с помощью AMD Radeon GPU Profiler (RGP).

### §3.2. Семантические инварианты

- **INV-COMPUTE-HIP-003**: *Математическая консистентность через core_math.h*.
  - *Обоснование*: Вся математика законов интеграции мембранного потенциала ALIF и пластичности весов GSOP обязана подключаться в HIP-ядрах исключительно из общего C++ заголовка `core_math.h`. Прямое дублирование формул в коде ядер запрещено во избежание расхождений.
  - *Следствие нарушения*: Потеря побитового соответствия симуляции (Butterfly Effect) между CPU и GPU.
  - *Где проверяется*: Код-ревью исходников `.hip` / `.cpp` на наличие `#include "core_math.h"`.

- **INV-COMPUTE-HIP-004**: *Изоляция стримов выполнения (Stream Context Isolation)*.
  - *Обоснование*: Каждый экземпляр выделенной памяти шарда (`VramHandle`) жестко привязывается к своему независимому неблокирующему HIP стриму (`hipStream_t`). Асинхронные ядра разных шардов запускаются параллельно в своих стримах.
  - *Следствие нарушения*: Взаимная сериализация вычислений шардов, сильное падение TPS при многошардовой конфигурации на одном GPU.
  - *Где проверяется*: Юнит-тест `test_hip_stream_concurrency`.

- **INV-COMPUTE-HIP-005**: *Безопасность FFI-запуска ядер (FFI Execution Guard)*.
  - *Обоснование*: Любой запуск HIP-ядра обязан оборачиваться FFI-функцией-враппером, возвращающей `hipError_t`. Бэкенд проверяет статус завершения и в случае ошибки мапит его в `ComputeApiError::VendorError` или `DeviceLost`, перехватывая сбой на стороне Rust.
  - *Следствие нарушения*: Тихий сбой выполнения ядер, приводящий к чтению устаревших или поврежденных данных в `download_output`.
  - *Где проверяется*: compile-time, линтинг FFI сигнатур, юнит-тесты.

- **INV-COMPUTE-HIP-006**: *Абсолютное использование Pinned RAM для DMA*.
  - *Обоснование*: Выгрузка моторных команд (`OutputFrame`) и телеметрии (`TelemetryFrame`) из VRAM на хост обязана производиться в предварительно выделенную Page-Locked память (Pinned Memory, `hipHostMalloc`). Использование обычной страничной памяти (Pageable Memory) запрещено во избежание сбоев PCIe контроллера и возникновения page-faults в ядре ОС.
  - *Следствие нарушения*: Падение производительности DMA-трансляции (в 2-4 раза), блокировка потоков оркестратора, сбой `DmaTransferFailed` под нагрузкой.
  - *Где проверяется*: Интеграционные тесты производительности и профилирование PCIe шины.

- **INV-COMPUTE-HIP-007**: *Аппаратные границы отладчика (Ephys Pointer Guard)*.
  - *Обоснование*: Структура `EphysCmd` передает сырые device-указатели `tids_d`, `uvs_d` и `trace_d`. Компилятор Rust не может отследить их границы. Бэкенд аппаратно обязан блокировать любую запись осциллограммы, если `current_tick >= max_ticks`, и прерывать выполнение, если `count > MAX_EPHYS_TARGETS` (16).
  - *Следствие нарушения*: Запись GPU-ядрами за пределы выделенного SHM/VRAM буфера (Out-of-Bounds Write), искажение данных симуляции, SIGSEGV драйвера.
  - *Где проверяется*: Интеграционный тест `test_hip_ephys_out_of_bounds` и аппаратные проверки (C-ABI) внутри `launch_debug_record_v`.

- **INV-COMPUTE-HIP-008**: *Аппаратный размер вейвфронта (AMD Wavefront Size)*.
  - *Обоснование*: Размеры блоков потоков и алгоритмы Wavefront-Aggregated Atomics жестко завязаны на размер вейвфронта AMD (64 потока, Wave64). Специфические инструкции `__ballot` и `__shfl` ожидают 64-битную маску активности вейвфронта (тип `unsigned long long` в HIP).
  - *Следствие нарушения*: Неопределенное поведение (UB) при сборе телеметрии, сбой адресации в VRAM, некорректный расчет `active_soma_ids`.
  - *Где проверяется*: compile-time static asserts в HIP/C++ коде, юнит-тест `test_hip_device_presence` (сверка `warpSize`/`wavefrontSize` через свойства устройства).

- **INV-COMPUTE-HIP-009**: *Неизменяемость памяти хоста при асинхронном DMA (Host Memory Immutability)*.
  - *Обоснование*: Поскольку методы загрузки и запуска батча используют `hipMemcpyAsync`, хост-память (буферы спайков, масок) должна оставаться неизменной и валидной на протяжении всего HFT-цикла до момента явной синхронизации (Stream Sync).
  - *Следствие нарушения*: Use-After-Free на GPU, повреждение входящих данных симуляции, Silent Data Corruption.
  - *Где проверяется*: Статический анализ времени жизни ссылок Rust (borrow checker для `DayBatchCmd<'a>`), а также интеграционные тесты.

### §3.3. Межкрейтовые инварианты

- **INV-CROSS-007**: *Битовое соответствие ShardVramPtrs*.
  - *Участники*: `layout`, `compute-cuda`, `compute-hip`.
  - *Кто владелец проверки*: `layout`.
  - *Обоснование*: См. [spec_layout.md §3.3]. Порядок, типы и выравнивание полей в Rust-структуре `ShardVramPtrs` должны байт-в-байт соответствовать структуре `ShardVramPtrs` в C++/HIP коде.
  - *Следствие нарушения*: Segmentation fault при запуске HIP-ядер, повреждение памяти GPU.
  - *Где проверяется*: FFI-тесты выравнивания в `test-harness`.

- **INV-CROSS-011**: *Кроссплатформенный детерминизм (Bit-to-Bit Identity)*.
  - *Участники*: `compute-api`, `compute-cuda`, `compute-hip`, `compute-cpu`, `test-harness`.
  - *Кто владелец проверки*: `test-harness`.
  - *Обоснование*: См. [spec_compute_api.md §3.3]. Все реализации `GpuBackend` (включая HIP) обязаны возвращать побитово идентичные результаты для одинаковых входов и начального состояния.
  - *Следствие нарушения*: Рассинхронизация распределенного кластера при восстановлении шарда из теневых копий.
  - *Где проверяется*: Интеграционные тесты `test-harness`.

## §4. Публичный API

Крейт экспортирует структуру `HipBackend`, реализующую абстрактный трейт `GpuBackend`. Вся низкоуровневая интеграция с FFI (`*mut c_void`, `ShardVramPtrs`) жестко скрыта внутри бэкенда и не протекает в публичный API.

### §4.1. Типы

```rust
pub struct HipBackend {
    pub device_id: i32,
    /// Внутренний реестр аллоцированных шардов.
    /// Защищен RwLock для обеспечения Send + Sync интерфейса GpuBackend.
    resources: std::sync::RwLock<slotmap::SlotMap<VramHandle, ShardHipResources>>,
}

/// Внутренняя структура (не экспортируется из крейта).
/// Хранит сырые C-ABI указатели и контекст конкретного шарда.
pub(crate) struct ShardHipResources {
    pub vram_ptrs: axicor_layout::ShardVramPtrs,
    /// Непрозрачный указатель `hipStream_t`. Гарантирует изоляцию батчей.
    pub stream: *mut std::ffi::c_void,
    pub layout: axicor_compute_api::ShardLayout,
}
```

- **Семантика**: `HipBackend` — это реализация абстрактного GPU-вычислителя HAL для карт AMD. Выступает в роли диспетчера, мапящего интерфейсные вызовы `run_day_batch` на конкретный `hipStream_t` и Flat-буфер.
- **Жизненный цикл**: Инициализируется один раз при старте оркестратора ноды (`Arc<HipBackend>`). Явно освобождает ресурсы при уничтожении через `free(handle)` или при `Drop` самого бэкенда (с перехватом ошибок HIP).
- **Ограничения на значения**: `device_id` обязан лежать в диапазоне `0..hipGetDeviceCount()`.

### §4.2. Трейты

Крейт предоставляет ровно одну реализацию: `impl GpuBackend for HipBackend`.

- **Контракт**: Реализация обязана гарантировать потокобезопасность (`Send + Sync`).
- **Стримы**: Методы загрузки данных (`upload_state`), батчинга (`run_day_batch`) и выгрузки (`download_output`) выполняются асинхронно (не блокируя хост) путем постановки операций в очередь драйвера через `stream` из `ShardHipResources`.
- **Обработка ошибок**: Любой запуск ядра оборачивается FFI-враппером, возвращающим `hipError_t`. Бэкенд перехватывает этот статус и конвертирует его в `ComputeApiError` (например, `hipErrorOutOfMemory` -> `OutOfMemory`).

### §4.3. Функции

Крейт не содержит публичных свободных функций, кроме конструктора бэкенда:

```rust
impl HipBackend {
    /// Инициализирует контекст HIP для указанного устройства
    pub fn new(device_id: i32) -> Result<Self, ComputeApiError> {
        // ...
    }
}
```

- **Семантика**: Конструктор обязан вызвать `hipSetDevice(device_id)` и проверить доступность GPU. При ошибке инициализации драйвера или отсутствии девайса немедленно возвращает `ComputeApiError::DeviceLost` или `VendorError`.

### §4.4. Константы и Магические Числа

Секция пуста. Крейту `compute-hip` запрещено объявлять свои магические числа для доменной логики. Все константы и лимиты унаследованы из [spec_compute_api.md §4.4] (например, `MAX_EPHYS_TARGETS`) и [spec_layout.md §4.4] (аппаратное выравнивание L2-кэша в 64 байта и `BurstHeads8` в 32 байта).

## §5. Доменная Логика

Реализация абстрактного интерфейса `GpuBackend` на базе платформы AMD HIP (ROCm) для параллельного выполнения фаз симуляции на графических процессорах AMD.

Выделение HIP-бэкенда в отдельный крейт Слоя 3 изолирует платформозависимый код (FFI-биндинги HIP Runtime, управление потоками вычислений, аллокацию видеопамяти через драйвер вендора) от общей логики симулятора и других вычислительных платформ.

Крейт решает задачу высокопроизводительного SIMT-вычисления динамики нейронов и пластичности синапсов. Он распределяет вычисления по потокам волновых фронтов AMD Wavefront с оптимизированным доступом к SoA-структурам в видеопамяти, обеспечивая выполнение горячего цикла симуляции в рамках жесткого временного бюджета.

## §8. Граничные Случаи и Особые Сценарии

### §8.1. Граничные значения

| # | Ситуация | Ожидаемое поведение |
|---|---|---|
| E-048 | **VRAM OOM**: Вызов `hipMalloc` возвращает `hipErrorOutOfMemory`. | См. `[spec_compute_api.md §8.1]`. Бэкенд возвращает `Err(ComputeApiError::OutOfMemory)`. |
| E-049 | **Device Lost / TDR Timeout**: GPU отвалился от PCIe во время работы батча. | См. `[spec_compute_api.md §8.1]`. Бэкенд возвращает `Err(ComputeApiError::DeviceLost)`. |
| E-050 | **Slice Mismatch**: Переданы некорректные слайсы в `DayBatchCmd`. | См. `[spec_compute_api.md §8.1]`. Бэкенд возвращает `Err(ComputeApiError::InvalidLayout)`. |
| E-051 | **Ghost Capacity Exceeded**: Превышение лимита SoA-таблицы связей. | См. `[spec_compute_api.md §8.1]`. Бэкенд возвращает `Err(ComputeApiError::CapacityExceeded)`. |
| E-052 | **Ephys Overflow**: Превышение `MAX_EPHYS_TARGETS` в `EphysCmd`. | См. `[spec_compute_api.md §8.1]`. Бэкенд возвращает `Err(ComputeApiError::InvalidLayout)`. |
| E-057 | **ROCm Architecture Mismatch**: Подключен GPU с неподдерживаемой архитектурой (требуется CDNA/RDNA, gfx90a или новее). | Метод `new` возвращает `Err(ComputeApiError::VendorError(hipErrorNoDevice))`, так как бэкенд аппаратно требует Wave64-совместимых инструкций для агрегированной телеметрии. |
| E-058 | **ROCm Driver Version Mismatch**: Системный драйвер AMD/ROCm старее, чем требует скомпилированный рантайм `hipcc`. | Метод `new` возвращает `Err(ComputeApiError::DeviceLost)` или `VendorError`. |
| E-059 | **FFI Launch Parameter Error**: Ошибка конфигурации сетки (Grid/Block size) при запуске ядер. | Метод `run_day_batch` перехватывает ошибку `hipErrorInvalidConfiguration` и возвращает `Err(ComputeApiError::InvalidLayout)`. |
| E-060 | **Pinned RAM Exhaustion**: Сбой `hipHostMalloc` из-за жестких лимитов ОС. | Ошибка выделения Page-Locked хост-памяти. Бэкенд возвращает `Err(ComputeApiError::OutOfMemory)`. |

### §8.2. Состояния гонки и конкурентность

| # | Сценарий | Защита |
|---|---|---|
| R-015 | **Гонка деинициализации GPU**: ОС уничтожает контекст до вызова `Drop`. | См. `[spec_compute_api.md §8.2]`. Использование явного метода `free()` до завершения процесса. |
| R-016 | **Конкурентный HFT-цикл**: Параллельные запуски для одного `VramHandle`. | См. `[spec_compute_api.md §8.2]`. Маппинг handle на независимый `hipStream_t`. |
| R-019 | **HIP Context Thread Bound Race**: Многопоточные вызовы API HIP без явной привязки контекста девайса. | На входе в каждый FFI-метод бэкенда вызывается `hipSetDevice(device_id)`, что привязывает контекст GPU к текущему OS-потоку (Thread-Local Context Bind). |
| R-020 | **Asynchronous DMA Use-After-Free**: Мутация слайсов в `DayBatchCmd` после вызова `run_day_batch`, но до завершения GPU-вычислений. | Метод `run_day_batch` ставит `hipMemcpyAsync` в очередь и мгновенно возвращает управление. Оркестратор обязан гарантировать, что память хоста не мутирует до вызова глобального барьера `gpu_device_synchronize()`. |

### §8.3. Деградация и Recovery

| # | Отказ | Поведение | Восстановление |
|---|---|---|---|
| D-009 | **Отказ GPU-вычислителя**: Потеря связи с картой. | См. `[spec_compute_api.md §8.3]`. | Fallback на `compute-cpu`. |
| D-010 | **Отказ DMA-транзакции**: Ошибка PCIe. | См. `[spec_compute_api.md §8.3]`. | Retry транзакции, при повторном сбое — переход в `DeviceLost`. |
| D-013 | **HIP Stream Execution Serialization**: Конкуренция нескольких `hipStream_t` за SM на одном GPU. | Время выполнения ядер в стримах может деградировать при высокой плотности параллельных запусков. | Планировщик оркестратора балансирует запуски (через BSP Barrier) или разносит шарды по разным GPU-картам. |
| D-014 | **Wavefront Divergence Penalty (Эпилептический шторм)**: Близко к 100% нейронов шарда спайкуют одновременно. | Ядра `launch_update_neurons` и `launch_apply_gsop` генерируют лавину кэш-промахов L2 из-за хаотичного доступа к дендритам. | Аппаратное самовосстановление: механизмы гомеостаза (ALIF) за 2-3 тика задерут пороги, и шард вернется в штатный температурный режим. |

## §9. Ошибки

### §9.1. Перечисление ошибок

Крейт не объявляет своих собственных публичных типов ошибок, а транслирует системные сбои FFI в абстрактный `ComputeApiError` из `[spec_compute_api.md §9.1]`.

### §9.2. Стратегия обработки

Бэкенд перехватывает возвращаемые статусы FFI-вызовов и мапит коды `hipError_t` следующим образом:

| Код HIP (`hipError_t`) | Целевая ошибка `ComputeApiError` |
|---|---|
| `hipErrorOutOfMemory` | `OutOfMemory` |
| `hipErrorNoDevice` / `hipErrorSharedObjectInitFailed` | `DeviceLost` |
| `hipErrorInvalidValue` / `hipErrorInvalidConfiguration` | `InvalidLayout` |
| `hipErrorInvalidResourceHandle` | `InvalidHandle` |
| `hipErrorLaunchOutOfResources` | `InvalidLayout` |
| Любой другой код | `VendorError(code as i32)` |

### §9.3. Паники

| Условие | Почему паника, а не Err |
|---|---|
| Ошибка `hipFree` внутри деструктора `Drop` | Исключение: для предотвращения двойной паники в процессе аварийного завершения работы, любая ошибка освобождения ресурсов в `Drop` перехватывается, пишется в лог и аппаратно гасится. |

## §10. Зависимости и Интеграция

Крейт `compute-hip` является "листовой" реализацией HAL. Он не предоставляет публичных абстракций, а лишь исполняет контракты нижележащих слоев, оставаясь полностью скрытым за фасадом `compute`.

### §10.1. Что крейт потребляет (inbound)

| Крейт / Инструмент | Что используем | Какой контракт ожидаем |
|---|---|---|
| `compute-api` | `GpuBackend`, `VramHandle`, `DayBatchCmd`, `ComputeApiError` | Крейт обязан реализовать трейт фасада и детерминированно перехватывать все `hipError_t`, транслируя их в абстрактные ошибки. |
| `layout` | `ShardVramPtrs`, `BurstHeads8`, `VariantParameters` | 100% C-ABI совместимость макетов. Выравнивание указателей по 64 байта и голов аксонов по 32 байта для Coalesced Access. |
| `physics` | `core_math.h` | Подключение общей C++ математики (ALIF, GSOP, Active Tail) через `#include` в HIP-ядрах для гарантии побитового кроссплатформенного детерминизма. |
| `cc` (build-dep) | Вызов компилятора `hipcc` | Статическая сборка `bindings.hip` и `physics.hip` в архив `libaxicor_hip.a` на этапе `build.rs` с флагами архитектуры (например, `--offload-arch=gfx90a`). |

### §10.2. Кто потребляет крейт (outbound / обратные зависимости)

Ни один крейт верхнего уровня (оркестратор, сеть) не имеет права зависеть от `compute-hip` напрямую.

| Крейт-потребитель | Что использует | Какой контракт мы обязаны сохранить |
|---|---|---|
| `compute` | `HipBackend::new(device_id)` | Инициализация контекста драйвера и инкапсуляция бэкенда в `Box<dyn GpuBackend>` (через фича-флаг `hip`). Никакие сырые указатели не должны вытекать наружу. |
| `test-harness` | `HipBackend` | Доступ к выгрузке сырой памяти VRAM для дифференциального тестирования. Обязательство побитового соответствия результатов с `compute-cpu` при одинаковом стартовом layout. |

## §12. Бюджеты и Ограничения

### §12.1. Память

| Ресурс | Бюджет | Как считается |
|---|---|---|
| Накладные расходы на VRAM (Overhead) | < 150 MB | Базовое потребление контекста драйвера AMD и структур `hipStream_t`. |

| Операция | Бюджет (p99) | Условия |
|---|---|---|

| Асинхронное копирование D2H (Download) | < 100 µs | 1000 выходов, шина PCIe Gen 4x16, использование Pinned RAM. |

### §12.3. Compile-time

| Ограничение | Значение |
|---|---|
| Время статической сборки HIP-ядер (`hipcc`) | < 30 секунд |
| Полное время сборки крейта с FFI-биндингами | < 45 секунд (в release-профиле) |
