## §1. Идентификация

| Поле | Значение |
|------|----------|
| **Имя крейта** | `compute-hip` |
| **Слой** | Слой 3 — Вычислительные Бэкенды (Compute Hardware Backends) |
| **Тип** | Library (`lib`) |
| **no_std** | Нет (`false`) — зависит от AMD ROCm HIP Runtime API, FFI-связывания и системного аллокатора кучи |
| **Описание** | AMD ROCm/HIP вычислительный бэкенд, реализующий трейт `ComputeBackend` из `compute-api`. Крейт управляет FFI-вызовами HIP Runtime API, VRAM-ресурсами ускорителя AMD, неблокирующими стримами (`hipStream_t`), запуском HIP-ядер и маппингом ошибок вендора в `ComputeApiError`. Публичные типы, ошибки, логи и тесты `compute-hip` не должны использовать CUDA-термины, кроме явных сравнительных ссылок на `compute-cuda`. Крейт не владеет DTO, макетами памяти, физическими уравнениями или рантайм-планировщиком. |

## §2. Стек и Окружение

### §2.1. Внутренние зависимости (inbound)

| Крейт | Что используется | Зачем |
|-------|-----------------|-------|
| `types` (Слой 0) | Атомарные типы и константы | Использование типов (`Tick`, `Voltage`, `SomaFlags`, `AXON_SENTINEL`) и битовых масок. |
| `layout` (Слой 1) | Расчет смещений SoA-плоскостей | Нарезка монолитных буферов VRAM с использованием `CACHE_LINE_BYTES` и `PADDED_N_ALIGNMENT`. |
| `physics` (Слой 0) | Математика GLIF, GSOP, Active Tail | Ожидание побитовой идентичности вычислений на GPU с эталоном `physics`. |
| `compute-api` (Слой 3) | `ComputeBackend`, `VramHandle`, DTO вызовов и `ComputeApiError` | Реализация абстрактного контракта вычислений HAL и типизированной иерархии ошибок. |

> [!IMPORTANT]
> Крейт `compute-hip` содержит C++/HIP ядра, являющиеся аппаратными зеркалами математических уравнений. Крейт **не является источником истины** по физике и макетам памяти: все C++/HIP ядра и ABI-заголовки должны кодогенерироваться или механически верифицироваться из оригинальных крейтов `physics` и `layout`.

### §2.2. Зависимые Компоненты и Характеристики Бэкенда (`BackendCapabilities`)

| Крейт / Компонент | Роль в системе и взаимодействие |
|-------------------|---------------------------------|
| `compute` (Слой 3) | Фасад вычислений подсоединяет `HipBackend` через feature flag `hip` для высокопроизводительного исполнения на графических процессорах AMD. |
| `test-harness` (Слой 3) | Выполняет дифференциальное тестирование результатов `HipBackend` против обязательного CPU-эталона (`CpuBackend`) и опционального `CudaBackend` (при его наличии в системе сборки). |

При вызове метода `capabilities()` структура `HipBackend` возвращает характеристики HIP-вычислителя:
```rust
BackendCapabilities {
    lane_count: 64,                // Строго целевой размер вейвфронта AMD (Wave64) в спецификации v2
    supports_async: true,          // Асинхронное исполнение ядер через HIP streams
    supports_ephys: false,         // Ограничено до фиксации DTO отладчика
    max_batch_ticks: TBD,          // Выводится из конфигурации симулятора и лимитов устройства
    alignment_bytes: 64,           // Выравнивание SoA-плоскостей VRAM под кэш-линию
    pinned_host_required: true,    // Требование Page-Locked памяти для скоростного D2H пути выгрузки результатов
}
```

> [!NOTE]
> Требование `pinned_host_required = true` относится к скоростному пути выгрузки результатов и телеметрии (DMA D2H), когда соответствующие буферы и DTO опубликованы, и не является блокирующим для базового выполнения ядер в памяти VRAM. Поддержка Wave32 выделена как работа на будущее (future work, §11).

### §2.3. Внешние Зависимости и Сборка

| Crate | Версия | Сфера использования |
|-------|--------|---------------------|
| `cc` | `=1.2.56` *(build-dep)* | Компиляция HIP-ядер (`.hip`) с помощью компилятора `hipcc` из состава ROCm SDK *(Входит в рабочий набор сборки)*. |

Секция публичных feature flags не используется. Крейт собирается как выделенная библиотека вычислений для ускорителей AMD ROCm.

## §4. Модель Ресурсов VRAM и Межспецификационные Долги

### §4.1. Физическая Стратегия Аллокации VRAM
При вызове `alloc_shard(spec: ShardAllocSpec)` бэкенд вычисляет необходимые объемы VRAM на основе формул `layout`. Для исключения фрагментации VRAM и высоких накладных расходов запрещено вызывать `hipMalloc` под каждый отдельный SoA-массив. По аналогии с CUDA, аллокация выполняется **двумя ограниченными физическими блоками**:
1. **Монолитный Блок Сомы и Синапсов**: Единая плоская VRAM-аллокация под все SoA-плоскости нейронов и синапсов пропорционально `padded_n`.
2. **Блок Аксонных Головок**: Отдельная VRAM-аллокация под массив `axon_heads`, так как `total_axons != padded_n`.

### §4.2. Межспецификационные Долги (Cross-Spec Debts)
> [!WARNING]
> 1. **Фабричный Конструктор `VramHandle`**: Приватное поле `VramHandle` в `compute-api v2.0` блокирует создание дескрипторов на стороне бэкендов. Ожидается введение фабричного метода `VramHandle::new_for_backend`.
> 2. **Верификация HIP/C++ Зеркал**: Крейты `physics` и `layout` на Rust являются единственным источником истины для формул и констант макетов. C++ заголовки зеркал должны кодогенерироваться или механически верифицироваться против Rust-источников. Проектирование полной автоматической кодогенерации всей структуры ядер бэкенда вынесено в открытые вопросы (§11).
> 3. **Единая Стратегия Кодогенерации CUDA/HIP**: Выделен межбэкендный долг по созданию единой генерации C++ ядер под оба GPU-ускорителя для предотвращения расхождения математики между CUDA и HIP.
> 4. **API Загрузки Constant Memory**: DTO `ShardUpload` не содержит вариантов нейронов. Вопрос передачи таблицы вариантов (`16 * 64 = 1024 байт`) в Constant Memory бэкенда вынесен как открытый вопрос (§11).

### §4.3. Аллокация и Загрузка Данных
1. **`alloc_shard(spec: ShardAllocSpec) -> Result<VramHandle, ComputeApiError>`**: Осуществляет проверку лимитов и выделяет 2 выровненных физических блока VRAM, инициализирует отдельный неблокирующий `hipStream_t` и регистрирует ресурс за дескриптором `VramHandle`.
2. **`upload_shard(handle: VramHandle, upload: ShardUpload<'_>) -> Result<(), ComputeApiError>`**: Валидирует размеры байтовых блобов `state_blob` и `axons_blob` с помощью формул `layout` и выполняет H2D передачу в VRAM через `hipMemcpyAsync`. Будучи синхронным API уровня трейта, метод возвращает управление только после достижения гарантированно видимого для хоста/устройства состояния передачи (Host/Device Transfer-Visible State) или с ошибкой.
3. **Безопасность Указателей**: Публичный API Rust не раскрывает сырые указатели (`*mut u8`), `ShardVramPtrs`, `hipStream_t` или C-ABI структуры. Все указатели изолированы внутри структуры `HipBackend`.

## §5. Модель Выполнения и Порядок Этапов (Execution Model & Stage Order)

Метод `run_day_batch(handle, cmd)` выполняет запуск батча тиков на выделенном HIP-стриме шарда.

### §5.1. Порядок Выполнения Этапов Внутри Тика (Tick Stage Order)
Выполнение каждого тика на GPU строго повторяет эталонный порядок `compute-cpu` и `compute-cuda`:
1. **Валидация DTO**: Проверка длин массивов и емкостей команды `DayBatchCmd`.
2. **Пакетная H2D Передача Входных Буферов (Batch-Level Input Transfer)**: Загрузка массивов входных данных на GPU выполняется один раз на уровне батча, после чего HIP-ядра тиков обращаются к этим VRAM-буферам через потиковое индексирование и срез (slicing).
3. **Инъекция Виртуальных Входов (Virtual Inputs)**: Запуск ядра инъекции сенсорных сигналов из битмаски `cmd.input_bitmask` по `cmd.virtual_offset` и `cmd.num_virtual_axons`.
4. **Инъекция Входных Спайков (Incoming Spikes)**: Запуск ядра распаковки потикового массива спайковых ID из `cmd.incoming_spikes` с учетом емкости `cmd.max_spikes_per_tick` и количества элементов из `cmd.spike_counts`.
5. **Продвижение Аксонных Головок (Active Tail)**: Запуск ядра продвижения спайков вдоль аксонов с учетом скаляра `cmd.v_seg` (жесткое кодирование `v_seg = 1` запрещено).
6. **Обновление Состояния Нейронов (Neuron State Update)**: Запуск ядра интеграции мембранного потенциала GLIF через верифицированное C++ зеркало физики, расчет гомеостаза, рефрактерных таймеров и генерация спайков.
7. **Применение Синаптической Пластичности (GSOP)**: Запуск ядра обновления весов с учетом дофаминовой модуляции `cmd.dopamine` и Закона Дейла.
8. **Фиксация Результатов (Условный / Draft Этап)**: Асинхронное D2H копирование результатов в Pinned-буферы хоста при наличии утвержденных DTO в `compute-api`.

## §6. Параллелизм, Защита от Гонки и Валидация Wave64 (Parallelism, Race Protection & Wave64)

1. **Строгий Детерминизм (Bit-to-Bit Identity)**: Результат вычислений на AMD GPU при одинаковых входных данных обязан быть побитово идентичен результатам `compute-cpu` и `compute-cuda`.
2. **Обязательная Валидация Режима Wave64**: Поскольку спецификация v2 жестко фиксирует использование 64-поточных вейвфронтов (`lane_count = 64`), крейт `compute-hip` обязан доказать выполнение этого условия до запуска ядер путем статически компилируемой проверки (`#if defined(__AMDGCN_WAVEFRONT_SIZE) && __AMDGCN_WAVEFRONT_SIZE == 64`) и динамической runtime-валидации (`wavefrontSize == 64` через `hipGetDeviceProperties`). Запуск ядер на архитектурах в режиме Wave32 без явного отказа запрещен.
3. **Защита от Гонок Данных при Записи Спайков в VRAM**:
   - При обновлении сомы ядра генерируют новые спайки и записывают их в `axon_heads` через карту `soma_to_axon`.
   - Параллельное выполнение ядра обновления допустимо **строго при условии**, что карта `soma_to_axon` гарантирует соответствие «один-к-одному» (дисюнктные адреса записи во VRAM), либо запись спайков выполняется через промежуточный буфер накопления в VRAM. Конкурирующая неатомарная запись запрещена.
4. **Детерминированные Атомики**: Использование атомарных операций в видеопамяти допускается только при условии, что окончательный результат не зависит от неассоциативного порядка потоков.

## §7. Трансляция Ошибок Вендора и Политика Безопасности (Error Mapping & Safety Policy)

Публичный API бэкенда гарантирует полное отсутствие паник (`panic!`, `assert!`, `unwrap`, `expect`). Все ошибки HIP Runtime API перехватываются врапперами запусков ядер и детерминированно мапятся в `ComputeApiError` по семействам ошибок:

| Семейство Ошибок HIP Runtime API | Итоговый Код `ComputeApiError` |
|----------------------------------|--------------------------------|
| **Allocation Errors** (например, сбои выделения памяти VRAM через `hipMalloc`) | `ComputeApiError::OutOfMemory` |
| **Copy / DMA Errors** (например, ошибки асинхронного копирования H2D/D2H `hipMemcpyAsync` или невалидные указатели) | `ComputeApiError::DmaFailed` |
| **Kernel Launch Errors** (например, превышение ресурсов блока или сбои запуска ядер `hipLaunchKernelGGL`) | `ComputeApiError::KernelLaunchFailed` |
| **Stream Synchronization Errors** (например, ошибки синхронизации стрима `hipStreamSynchronize`) | `ComputeApiError::SynchronizeFailed` |
| **Device Lost / Reset Errors** (например, сброс GPU или отключение устройства AMD во время работы) | `ComputeApiError::DeviceLost` |

> [!NOTE]
> Ошибка `BackendUnavailable` относится к фазе автодетекции/пробинга оборудования на уровне фасада `compute` (Layer 3 facade) при отсутствии `hipcc` или инициализированного драйвера. При аппаратном сбое или потере устройства во время выполнения рантайма возвращается `DeviceLost`.

При передаче невалидных DTO прерывание происходит до вызова HIP FFI с возвратом `InvalidShape`, `SizeMismatch` или `InvalidHandle`. Метод `teardown()` атомарно освобождает VRAM, уничтожает `hipStream_t` и переводит дескриптор в `None`; повторный вызов идемпотентен.

## §8. Сохранение Совместимости с Легаси-Фактами (Legacy Behavioral Facts)

1. **Маркер Сентинела Аксона (`AXON_SENTINEL`)**: HIP-ядра используют побитовое правило с константой `types::AXON_SENTINEL` (`0x80000000`), корректно обрабатывая проскакивание при $v_{\text{seg}} > 1$.
2. **Обработка Tombstone в Target (`0` vs `EMPTY_PIXEL`)**: Поддержка единой политики двойной проверки целевых синапсов, утверждаемой в `types`/`physics`.
3. **Семантика Heartbeat / DDS**: Спайк Heartbeat не сбрасывает потенциал $V$, не взводит рефрактерный таймер и не меняет параметры гомеостаза, если не произошел GLIF-спайк.
4. **Затухание Гомеостаза и Близость Сомы**: Гомеостатический пенальти затухает во время рефрактера; биты типа нейрона в `SomaFlags` строго сохраняются.
5. **Дофаминовая Модуляция в GSOP**: Изменение весов учитывает сигналы D1/D2 с соблюдением знаковой политики `physics`.

## §9. Требуемые Инварианты

- **INV-COMPUTE-HIP-001**: Структура `HipBackend` имплементирует трейт `ComputeBackend` из `compute-api`.
- **INV-COMPUTE-HIP-002**: Вызов `kind()` возвращает строго `BackendKind::Hip`.
- **INV-COMPUTE-HIP-003**: Выделение VRAM под шард выполняется строго двумя монолитными блоками памяти.
- **INV-COMPUTE-HIP-004**: Побитовая идентичность вычислений сохраняется с `compute-cpu` и `compute-cuda`.
- **INV-COMPUTE-HIP-005**: Параллельная запись спайков ядрами GPU исключает гонки данных во VRAM за счет дисюнктной карты `soma_to_axon` или буфера накопления.
- **INV-COMPUTE-HIP-006**: Публичный API бэкенда на Rust не раскрывает сырые указатели (`*mut u8`), `hipStream_t` и C-ABI структуры указателей.
- **INV-COMPUTE-HIP-007**: Все вызовы HIP API обернуты в проверки семейств ошибок с маппингом в `ComputeApiError`, паники запрещены.

## §10. Golden Tests / Обязательная Матрица Тестирования

Крейт `compute-hip` обязан быть покрыт набором автоматических интеграционных тестов и статических проверок:

1. **Имплементация HAL Трейта (`test_hip_implements_compute_backend`)**: Проверка реализации `ComputeBackend`.
2. **Идентификация Бэкенда (`test_hip_backend_kind`)**: Проверка возврата `BackendKind::Hip`.
3. **Проверка Характеристик (`test_hip_backend_capabilities`)**: Проверка точности структуры `BackendCapabilities` (§2.2).
4. **Маппинг Пробинга vs Потери Устройства (`test_hip_probe_vs_execution_error_mapping`)**: Отсутствие `hipcc` или GPU при пробинге мапится фасадом в `BackendUnavailable`, а аппаратный сбой во время вычислений — в `DeviceLost`.
5. **Браковка Неверных Размеров до FFI (`test_hip_upload_rejects_bad_sizes`)**: Проверка возврата `SizeMismatch` до обращения к HIP API.
6. **Защита от Невалидных Дескрипторов (`test_hip_rejects_invalid_handles`)**: Проверка обработки битых `VramHandle`.
7. **Изоляция Терминологии CUDA (`test_hip_no_cuda_naming_in_public_api`)**: Публичный API, логи и ошибки `compute-hip` не содержат CUDA-терминологии (за исключением документов/тестов явного сравнения с `compute-cuda`).
8. **Падение Сборки при Рассогласовании ABI Зеркал (`test_hip_abi_mirror_drift_prevention`)**: Тест компиляции/верификации C++ заголовков, падающий при любом расхождении полей, смещений или выравнивания в `ShardVramPtrs`, `BurstHeads8` и `VariantParameters`.
9. **Статус API Загрузки Constant Memory (`test_hip_constant_upload_api_pending`)**: Статический тест, фиксирующий отсутствие публичного API загрузки Constant Memory до утверждения соответствующего DTO в `compute-api`.

## §11. Open Questions / Review Debt (Открытые Вопросы и Противоречия)

1. **Поддержка Режима Wave32 на Архитектурах AMD RDNA (Future Work)**:
   - *Контекст*: Спецификация v2 жестко фиксирует использование Wave64.
   - *Вопрос*: Каким образом в будущих версиях спецификации будет организована адаптация масок вейвфронта для архитектур RDNA с режимом Wave32?

2. **Единая Стратегия Автоматической Кодогенерации C++ Зеркал CUDA/HIP из Rust**:
   - *Контекст*: Во избежание рассинхронизации математики ядра CUDA и HIP не должны содержать дублирующих ручных формул.
   - *Вопрос*: Каким образом будет организован общий пайплайн кодогенерации C++ ядер из источников истины в `physics` и `layout`?

3. **API и DTO Загрузки Таблицы Вариантов в Constant Memory**:
   - *Контекст*: `ShardUpload` не содержит вариантов нейронов.
   - *Вопрос*: Через какой интерфейс (отдельный метод HAL, расширение `ShardUpload` или операция фасада `compute`) таблица вариантов должна передаваться бэкенду?

4. **Модель Фабричного Конструктора `VramHandle` в `compute-api`**:
   - *Контекст*: Приватность `VramHandle` блокирует создание дескрипторов бэкендами.
   - *Вопрос*: Каким образом бэкенд вычислений будет получать экземпляры `VramHandle` из `compute-api`?

5. **Формат и Владелец Pinned-Буферов Результатов**:
   - *Контекст*: Требование `pinned_host_required = true` необходимо для скорейшего DMA D2H.
   - *Вопрос*: Кто создает и держит Pinned-буферы для результатов — `compute-hip` или IPC/runtime swapchain?

6. **Аффинность Потоков ОС и Маркер `Send` для HIP Контекста**:
   - *Контекст*: HIP контексты и стримы привязаны к создавшему их OS-потоку.
   - *Вопрос*: Является ли `HipBackend` маркерным `Send`, или инициализация контекста должна происходить строго внутри целевого OS-потока шарда?

7. **Владение Операциями Синхронизации Ghost-Аксонов и Сортировки**:
   - *Контекст*: Операции `sort_and_prune` и межшардовые патчи затрагивают память ускорителя.
   - *Вопрос*: Относятся ли методы уплотнения синапсов к `ComputeBackend`, или они выносятся в отдельный сервисный слой?
