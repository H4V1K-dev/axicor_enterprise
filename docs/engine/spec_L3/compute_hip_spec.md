## §1. Идентификация

| Поле | Значение |
|---|---|
| **Имя крейта** | `compute-hip` |
| **Слой** | Слой 3 — Вычислительные Бэкенды (Compute Hardware Backends) |
| **Тип** | Library (`lib`) |
| **no_std** | Нет (`false`) — зависит от AMD ROCm HIP Runtime API, FFI-связывания и системного аллокатора кучи |
| **Описание** | AMD ROCm/HIP вычислительный бэкенд, реализующий трейт `ComputeBackend` из `compute-api`. Крейт управляет FFI-вызовами HIP Runtime API, VRAM-ресурсами ускорителя AMD, неблокирующими стримами (`hipStream_t`), внутренними Pinned Host staging буферами, запуском HIP-ядер и маппингом ошибок вендора в `ComputeApiError`. Публичные типы, ошибки, логи и тесты `compute-hip` не должны использовать CUDA-термины. Крейт не владеет DTO, макетами памяти, физическими уравнениями или рантайм-планировщиком. |

## §2. Стек и Окружение

### §2.1. Внутренние зависимости (inbound)

| Крейт | Что используется | Зачем |
|---|---|---|
| `types` (Слой 0) | Атомарные типы и константы, `PackedTarget::is_inactive()` | Использование типов (`Tick`, `Voltage`, `SomaFlags`, `AXON_SENTINEL`) и проверка неактивных синапсов. |
| `layout` (Слой 1) | Расчет смещений SoA-плоскостей | Нарезка монолитных буферов VRAM с использованием `CACHE_LINE_BYTES` и `PADDED_N_ALIGNMENT`. |
| `physics` (Слой 0) | Математика GLIF, GSOP, Active Tail | Ожидание побитовой идентичности вычислений на GPU с эталоном `physics`. |
| `compute-api` (Слой 3) | `ComputeBackend`, `VramHandle`, DTO вызовов, DTO обслуживания и `ComputeApiError` | Реализация абстрактного контракта вычислений HAL и типизированной иерархии ошибок. |

> [!IMPORTANT]
> Крейт `compute-hip` содержит C++/HIP ядра, являющиеся аппаратными зеркалами математических уравнений. Крейт **не является источником истины** по физике и макетам памяти: все C++/HIP ядра и ABI-заголовки верифицируются из оригинальных крейтов `physics` и `layout`.

### §2.2. Зависимые Компоненты и Характеристики Бэкенда (`BackendCapabilities`)

| Крейт / Компонент | Роль в системе и взаимодействие |
|---|---|
| `compute` (Слой 3) | Фасад вычислений подсоединяет `HipBackend` через feature flag `hip` для высокопроизводительного исполнения на графических процессорах AMD. |
| `test-harness` (Слой 3) | Выполняет дифференциальное тестирование результатов `HipBackend` против `CpuBackend` и `CudaBackend`. |

При вызове метода `capabilities()` структура `HipBackend` возвращает характеристики HIP-вычислителя:
```rust
BackendCapabilities {
    lane_count: 64,                // Размер вейвфронта AMD (Wave64)
    supports_async: true,          // Внутренняя поддержка неблокирующих стримов HIP
    supports_ephys: false,         // Ограничено до фиксации DTO отладчика
    max_batch_ticks: 1000,         // Лимит тиков за один вызов run_day_batch
    alignment_bytes: 64,           // Выравнивание SoA-плоскостей VRAM под кэш-линию
    pinned_host_required: true,    // Требование Page-Locked памяти для скоростного DMA
}
```

## §4. Модель Ресурсов VRAM и Внутренние Pinned Буферы

### §4.1. Физическая Стратегия Ограниченных Аллокаций VRAM и Pinned Staging
При вызове `alloc_shard(spec: ShardAllocSpec)` бэкенд вычисляет необходимые объемы VRAM на основе формул `layout` и создает `VramHandle` через `VramHandle::from_raw_parts(BackendKind::Hip, id, generation)`. Аллокация VRAM под шард выполняется строго **двумя монолитными блоками памяти** (Блок 1: Соматические плоскости и синапсы; Блок 2: Аксонные головки `BurstHeads8`), предотвращая фрагментацию памяти видеокарты.

При вызове `upload_shard(handle, upload)` бэкенд валидирует блобы данных, осуществляет загрузку состояния и аксонов во VRAM, а также выполняет синхронный H2D-перенос таблицы вариантов `upload.variant_table` в GPU Constant Memory (`__constant__`). Крейт `compute-hip` самостоятельно владеет внутренними Pinned Host staging буферами (`hipHostMalloc`), копируя входящие данные из заимствованных срезов `upload.state_blob` / `cmd.incoming_spikes` и выгружая результаты в `cmd.output_spikes`.

При вызове методов обслуживания `export_maintenance_state` и `import_maintenance_state` бэкенд осуществляет перенос данных (состояние и аксоны) между VRAM устройства AMD и переданными хост-буферами с помощью синхронного DMA-копирования (H2D/D2H) через стрим. На текущем этапе (в рамках Stage-заглушки) данные методы могут возвращать ошибку `UnsupportedFeature`, однако полный интерфейсный контракт описан и изоморфен CPU/CUDA-пути. 

Любая биологическая мутация (прунинг, спраутинг, сортировка) в VRAM бэкенда запрещена. Бэкенд является чистым исполнителем и мостом памяти.

## §5. Модель Выполнения и Порядок Этапов (Execution Model & Stage Order)

Метод `run_day_batch(handle, cmd)` выполняет запуск батча тиков на выделенном HIP-стриме шарда и блокирует вызывающий поток до завершения тиков батча и синхронизации результатирующих буферов в `cmd.output_spikes` и `cmd.output_spike_counts`.

### §5.1. Порядок Выполнения Этапов Внутри Тика (Tick Stage Order)
1. **Валидация DTO**: Проверка длин массивов и емкостей команды `DayBatchCmd`.
2. **Пакетная H2D Передача Входных Буферов**: Загрузка массивов входных данных на GPU через внутренние Pinned Host буферы.
3. **Инъекция Виртуальных Входов**: Запуск ядра инъекции сенсорных сигналов из битмаски `cmd.input_bitmask`.
4. **Инъекция Входных Спайков**: Запуск ядра распаковки массива спайковых ID из `cmd.incoming_spikes`.
5. **Продвижение Аксонных Головок (Active Tail)**: Запуск ядра продвижения спайков с учетом скаляра `cmd.v_seg` и маркера `AXON_SENTINEL`.
6. **Обновление Состояния Нейронов**: Запуск ядра интеграции потенциала GLIF, гомеостаза и генерации спайков. HIP-ядра проверяют неактивные таргеты строго по предикату `PackedTarget::is_inactive()` (`0` И `EMPTY_PIXEL`). Сгенерированные спайки записываются во VRAM буфер выходов.
7. **Применение Синаптической Пластичности (GSOP)**: Запуск ядра обновления весов.
8. **D2H Передача Выходов**: Копирование выходящих спайковых ID из VRAM в Pinned Host буфер и далее в `cmd.output_spikes`, а также заполнение `cmd.output_spike_counts`.

## §6. Параллелизм, Защита от Гонки и Валидация Wave64

1. **Строгий Побитовый Детерминизм**: Результат вычислений побитово идентичен `compute-cpu` и `compute-cuda`.
2. **Обязательная Валидация Режима Wave64**: Крейт `compute-hip` проверяет условие `wavefrontSize == 64` до запуска ядер.

## §7. Трансляция Ошибок Вендора и Политика Безопасности

Все вызовы HIP API обернуты in проверки семейств ошибок с трансляцией в `ComputeApiError`, прямой вызов паник категорически запрещен:

| Семейство Ошибок HIP Runtime API | Итоговый Код `ComputeApiError` |
|---|---|
| **Allocation Errors** (`hipMalloc`) | `ComputeApiError::OutOfMemory` |
| **Copy / DMA Errors** (`hipMemcpyAsync`) | `ComputeApiError::DmaFailed` |
| **Kernel Launch Errors** | `ComputeApiError::KernelLaunchFailed` |
| **Stream Synchronization Errors** | `ComputeApiError::SynchronizeFailed` |
| **Device Lost / Reset Errors** | `ComputeApiError::DeviceLost` |

## §8. Требуемые Инварианты

- **INV-COMPUTE-HIP-001**: Структура `HipBackend` имплементирует трейт `ComputeBackend` из `compute-api`.
- **INV-COMPUTE-HIP-002**: Вызов `kind()` возвращает строго `BackendKind::Hip`.
- **INV-COMPUTE-HIP-003**: Выделение VRAM под шард выполняется строго двумя монолитными блоками памяти.
- **INV-COMPUTE-HIP-004**: Побитовая идентичность вычислений сохраняется с `compute-cpu` и `compute-cuda`.
- **INV-COMPUTE-HIP-005**: Параллельная запись спайков ядрами GPU исключает гонки данных во VRAM за счет дисюнктной карты `soma_to_axon` или буфера накопления.
- **INV-COMPUTE-HIP-006**: Публичный API бэкенда на Rust не раскрывает сырые указатели (`*mut u8`), `hipStream_t` и C-ABI структуры указателей.
- **INV-COMPUTE-HIP-007**: Все вызовы HIP API обернуты в проверки семейств ошибок с маппингом в `ComputeApiError`, паники запрещены.
- **INV-COMPUTE-HIP-008**: HIP-ядра проверяют неактивные синаптические таргеты строго через `PackedTarget::is_inactive()`.

## §9. Golden Tests / Обязательная Матрица Тестирования

Крейт `compute-hip` обязан быть покрыт набором автоматических интеграционных тестов и статических проверок:

1. **Имплементация HAL Трейта (`test_hip_implements_compute_backend`)**: Проверка реализации `ComputeBackend`.
2. **Идентификация Бэкенда (`test_hip_backend_kind`)**: Проверка возврата `BackendKind::Hip`.
3. **Проверка Характеристик (`test_hip_backend_capabilities`)**: Проверка точности структуры `BackendCapabilities` (§2.2).
4. **Маппинг Пробинга vs Потери Устройства (`test_hip_probe_vs_execution_error_mapping`)**: Отсутствие `hipcc` или GPU при пробинге мапится фасадом в `BackendUnavailable`, а аппаратный сбой во время вычислений — в `DeviceLost`.
5. **Браковка Неверных Размеров до FFI (`test_hip_upload_rejects_bad_sizes`)**: Проверка возврата `SizeMismatch` до обращения к HIP API.
6. **Защита от Невалидных Дескрипторов (`test_hip_rejects_invalid_handles`)**: Проверка обработки битых `VramHandle`.
7. **Изоляция Терминологии CUDA (`test_hip_no_cuda_naming_in_public_api`)**: Публичный API, логи и ошибки `compute-hip` не содержат CUDA-терминологии (за исключением документов/тестов явного сравнения с `compute-cuda`).
8. **Падение Сборки при Рассогласовании ABI Зеркал (`test_hip_abi_mirror_drift_prevention`)**: Тест компиляции/верификации C++ заголовков, падающий при любом расхождении полей, смещений или выравнивания в `ShardVramPtrs`, `BurstHeads8` и `VariantParameters`.
9. **Загрузка Таблицы Вариантов в Constant Memory (`test_hip_constant_upload_api`)**: Верификация H2D загрузки таблицы вариантов `upload.variant_table` в GPU Constant Memory (`__constant__`) при вызове `upload_shard`.
10. **Ограниченность Физических Аллокаций (`test_hip_bounded_allocations`)**: Верификация вызова ровно 2 физических аллокаций VRAM на шард через Mock/Stub FFI.
11. **Совпадение Порядка Этапов с CPU и CUDA (`test_hip_stage_order_matches_reference`)**: Дифференциальный тест последовательности вычислений на эталонном фикстурном шарде.
12. **Обязательная Валидация Wave64 (`test_hip_wave64_validation_assertion`)**: Проверка отказа бэкенда от работы при несоблюдении условия `wavefrontSize == 64`.
13. **Идемпотентность Teardown и Уничтожение Стримов (`test_hip_idempotent_teardown`)**: Проверка уничтожения `hipStream_t` и безопасного повторного вызова `teardown()`.
14. **Проверка контракта обслуживания бэкендом (`test_hip_maintenance_contract_stub`)**: Проверка наличия методов обслуживания в трейте `ComputeBackend` и возврата `UnsupportedFeature`.

## §10. Open Questions / Review Debt (Открытые Вопросы и Противоречия)

1. **Поддержка Режима Wave32 на Архитектурах AMD RDNA (Future Work)**:
   - *Контекст*: Спецификация v2 жестко фиксирует использование Wave64.
   - *Вопрос*: Каким образом в будущих версиях спецификации будет организована адаптация масок вейвфронта для архитектур RDNA с режимом Wave32?

2. **Единая Стратегия Автоматической Кодогенерации C++ Зеркал CUDA/HIP из Rust**:
   - *Контекст*: Во избежание рассинхронизации математики ядра CUDA и HIP не должны содержать дублирующих ручных формул.
   - *Вопрос*: Каким образом будет организован общий пайплайн кодогенерации C++ ядер из источников истины в `physics` и `layout`?

3. **Аффинность Потоков ОС и Маркер `Send` для HIP Контекста**:
   - *Контекст*: HIP контексты и стримы привязаны к создавшему их OS-потоку.
   - *Вопрос*: Является ли `HipBackend` маркерным `Send`, или инициализация контекста должна происходить строго внутри целевого OS-потока шарда?

4. **Владение Операциями Синхронизации Ghost-Аксонов и Сортировки**:
   - *Контекст*: Операции `sort_and_prune` и межшардовые патчи затрагивают память ускорителя.
   - *Вопрос*: Относятся ли методы уплотнения синапсов к `ComputeBackend`, или они выносятся в отдельный сервисный слой?

## §11. Resolved Architectural Decisions (Принятые Решения Pass 2.2)

1. **[RESOLVED] API и DTO Загрузки Таблицы Вариантов в Constant Memory (REV-COMPUTE-HIP-003 / Pass 2.2)**:
   - *Решение*: В DTO `ShardUpload` добавлено фиксированное заимствованное поле `variant_table: &'a [VariantParameters; VARIANT_LUT_LEN]`. Таблица вариантов синхронно передается на GPU во время `upload_shard` и размещается в HIP Constant Memory (`__constant__`).
2. **[RESOLVED] Поддержка экспорта и импорта обслуживания (isomorphic DMA)**:
   - *Решение*: Описан контракт обслуживания шарда через DMA (H2D/D2H) для isomorphic parity с CPU/CUDA бэкендами. Night policy (sort_and_prune) явно запрещена внутри HIP-ядер бэкенда.
