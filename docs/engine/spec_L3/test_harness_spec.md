## §1. Идентификация

| Поле | Значение |
|---|---|
| **Имя крейта** | `test-harness` |
| **Слой** | Слой 3 — Вычислительные Бэкенды (Dev-Support / Conformance Suite) |
| **Тип** | Dev-Support Library (`lib`) |
| **no_std** | Нет (`false`) — требуется стандартная библиотека Rust, системный аллокатор кучи хоста и тестовый раннер |
| **Описание** | Вспомогательный тестовый крейт Слоя 3, предназначенный для проверки соответствия вычислительных бэкендов (`ComputeBackend`), дифференциального прогона тестовых фикстур побитового сравнения, валидации ABI-зеркал и отслеживания расхождений между реализациями (CPU, CUDA, HIP, Mock). Крейт не является частью боевого рантайма и не владеет физикой, макетами памяти или HAL-контрактами. |

## §4. Архитектура и Модель Тестирования (Testing Model)

Тестовый комплекс `test-harness` разделяет уровни проверки на изолированные категории:

1. **Тестирование Соответствия Контракту (Trait-Level Conformance)**: Прямые вызовы методов трейта `ComputeBackend` на конкретных бэкендах для проверки строгого соблюдения контракта HAL.
2. **Дифференциальные Прогоны (Differential Execution Runs)**: Параллельный запуск идентичных фикстур на проверочной реализации `CpuBackend` и проверяемом бэкенде (CUDA, HIP, Mock) с пошаговой сверкой результатов.
3. **Изоляция Тестов Фасада (Facade Lifecycle Isolation)**: Проверка автоматов состояний `ShardEngine` вынесена в отдельную категорию под фичей `facade`, предотвращая смешивание проверок трейта и фасада.
4. **Матрица Аппаратной Доступности**:
   - `CpuBackend` является обязательным базовым эталоном.
   - Ускорители CUDA и HIP тестируются только при включении соответствующих feature-флагов и наличии физического оборудования/SDK в системе сборки.
   - Перекрестное сравнение CUDA vs HIP является опциональным и не блокирует отдельные запуски CUDA или HIP тестов.

## §5. Модель Фикстур (Fixture Model)

Все тестовые сценарии строятся на базе детерминированного генератора случайных чисел с фиксированным `MasterSeed`.

### §5.1. Категории Минимальных Фикстур
1. **Empty Shard**: Нулевой шард для проверки базовой аллокации и пустой загрузки.
2. **Single Soma / Single Axon**: Минимальный точечный шард из одного нейрона и одного аксона для отладки пошаговой интеграции GLIF.
3. **Small Recurrent Graph**: Малый рекуррентный граф с обратными связями под проверку пластичности GSOP.
4. **Boundary Padded N**: Шард с граничными значениями `padded_n` (например, 64, 128) для проверки выравнивания памяти и краевых условий чанков.
5. **Spike-Heavy Case**: Сценарий с высокой лавинообразной спайковой активностью для проверки лимитов емкости буферов.
6. **No-Input Stability Case**: Длительный прогон без внешних сигналов для проверки стабильности нулевого уровня и затухания гомеостаза.

### §5.2. Контракт Сборки Фикстур
Фикстуры конструируют входные данные исключительно через официальные DTO `ShardAllocSpec`, `ShardUpload` и `DayBatchCmd`. **Строго запрещено** объявлять собственные макеты памяти или вручную рассчитывать смещения: все размеры и правила выравнивания запрашиваются напрямую из крейта `layout`.

## §6. Пошаговая Политика Сравнения и Снэпшоты (Comparison Policy & Snapshot Debt)

### §6.1. Детализация Отчета о Расхождении
При обнаружении любого побитового или скалярного расхождения между эталоном и проверяемым бэкендом, тестовый раннер прерывает выполнение и формирует детальный отчет о первом несоответствии (First-Mismatch Report):
- Пара сравниваемых бэкендов (например, `CPU vs CUDA`).
- Идентификатор и параметры фикстуры.
- Номер тика симуляции, на котором возникло расхождение.
- Наименование поля структуры или смещение байта в буфере.
- Ожидаемое значение (Expected от CPU) и фактическое значение (Actual от проверяемого бэкенда).

### §6.2. Классификация Ошибок Сравнения
Раннер четко разграничивает 5 категорий сбоев в тестовой матрице:
1. **`ResultMismatch`**: Несовпадение возвращенных результатов вычислений (`BatchResult`).
2. **`SnapshotMismatch`**: Несовпадение побитовых срезов состояния VRAM/RAM памяти.
3. **`ErrorMappingMismatch`**: Различие в кодах ошибок при подаче битых входных DTO.
4. **`AbiMirrorMismatch`**: Отклонение размера, полей или выравнивания C++ зеркал от Rust-структур `layout`.
5. **`TestSkipped`**: Информационный пропуск теста из-за отсутствия соответствующей аппаратной фичи или драйвера вендора.

### §6.3. Открытый Долг Снэпшотов Состояния (State Snapshot Debt)
> [!WARNING]
> В текущей версии `compute-api v2.0` отсутствует публичный метод выгрузки полного состояния VRAM для отладочных целей. В связи с этим требование 100% побитового сравнения всех плоскостей SoA-памяти зафиксировано как целевой инвариант, но его полная автоматизированная реализация находится в статусе открытого вопроса (§11) до согласования test-only snapshot API или расширения `BatchResult`.

## §7. Иерархия Ошибок и Отчетность (`HarnessError`)

Все ошибки тестового комплекса оборачиваются в типизированную структуру или enum `HarnessError` с использованием `anyhow` для форматирования текстовых отчетов:

```rust
#[derive(Debug)]
pub enum HarnessErrorKind {
    ResultMismatch { tick: u64, field: &'static str, expected: String, actual: String },
    SnapshotMismatch { offset: usize, expected: u8, actual: u8 },
    ErrorMappingMismatch { expected: ComputeApiError, actual: ComputeApiError },
    AbiMirrorMismatch { struct_name: &'static str, reason: &'static str },
    BackendInitializationFailed(ComputeApiError),
}
```

## §8. Требуемые Инварианты

- **INV-HARNESS-001**: `test-harness` ни при каких условиях не дублирует и не объявляет собственную физическую математику (единственный источник — `physics`).
- **INV-HARNESS-002**: `test-harness` ни при каких условиях не задает правила выравнивания или смещения полей SoA-массивов в памяти (единственный источник — `layout`).
- **INV-HARNESS-003**: `test-harness` не имеет прямых зависимостей от вендорских SDK (CUDA/HIP API) при выключенных соответствующих feature-флагах.
- **INV-HARNESS-004**: Отсутствие опционального аппаратного ускорителя явно фиксируется в отчетах как пропуск (`TestSkipped`), а не маскируется под успешное прохождение теста.
- **INV-HARNESS-005**: Набор фикстур и CPU-прогонов доступен по умолчанию во всех тестовых профилях без необходимости наличия GPU.
- **INV-HARNESS-006**: Все отчеты о прогонах являются детерминированными и воспроизводимыми по комбинации `MasterSeed` + `FixtureID` + `BackendKind`.

## §9. Golden Tests / Обязательная Матрица Тестирования

Крейт `test-harness` обязан содержать и выполнять следующий набор автоматических тестов:

1. **Прогон Базовых Фикстур на CPU (`test_cpu_reference_fixture_runs`)**: Верификация работоспособности эталонного CPU-пути на всех фикстурах §5.1.
2. **Соответствие Трейту HAL (`test_compute_backend_trait_conformance`)**: Комплексная проверка соответствия реализации контракту `ComputeBackend`.
3. **Тестирование Mock-Бэкенда (`test_mock_backend_conformance`)**: Валидация корректности вызовов на Mock-бэкенде.
4. **Дифференциальный Тест CUDA vs CPU (`test_cuda_matches_cpu_fixtures_when_available`)**: Побитовая сверка CUDA с CPU при включенной фиче `cuda` и наличии оборудования.
5. **Дифференциальный Тест HIP vs CPU (`test_hip_matches_cpu_fixtures_when_available`)**: Побитовая сверка HIP с CPU при включенной фиче `hip` и наличии оборудования.
6. **Отчет о Пропуске Опциональной Матрицы (`test_backend_unavailable_is_reported_as_skip_for_optional_matrix`)**: Проверка корректной генерации статуса `TestSkipped` при отсутствии оборудования.
7. **Единообразие Ошибок Валидации (`test_invalid_dto_errors_are_consistent_across_backends`)**: Проверка совпадения транслируемых ошибок `ComputeApiError` на всех бэкендах при подаче одинаковых битых DTO.
8. **Проверка Порядка Этапов Батча (`test_batch_stage_order_fixture`)**: Тест строгого соблюдения последовательности 8 этапов вычислений.
9. **Отсутствие Вендорских Типов в Публичном API Тестов (`test_no_raw_vendor_types_in_public_test_api`)**: Компиляционная проверка отсутствия сырых указателей и тип-дескрипторов вендоров в типах тестов.
10. **Валидация Размеров Блобов через Layout (`test_layout_blob_size_validation_fixture`)**: Проверка корректности расчета объемов памяти фикстур через функции `layout`.
11. **Контроль Дрифта ABI-Зеркал (`test_abi_mirror_drift_detection`)**: Автоматическая проверка совпадения размеров и выравниваний `ShardVramPtrs`, `BurstHeads8` и `VariantParameters`.
12. **Детализация Отчета о Первом Расхождении (`test_first_mismatch_report_contains_tick_field_offset_expected_actual`)**: Верификация наличия всех обязательных полей в отчете об ошибке сравнения.
13. **Тестирование Фасада под Фичей (`test_facade_lifecycle_under_facade_feature`)**: Проверка автоматов состояний `ShardEngine` под фичей `facade`.
14. **Учет Долга Снэпшотов Состояния (`test_snapshot_api_debt_is_tracked_until_compute_api_resolves_it`)**: Статический тест, отслеживающий статус открытого вопроса по debug snapshot API.

## §10. Open Questions / Review Debt (Открытые Вопросы и Противоречия)

В процессе анализа спецификации тестового комплекса выявлены следующие открытые вопросы для согласования:

1. **Механизм Отладочного Снижения Состояния (Debug Full-State Snapshot API)**:
   - *Контекст*: Публичный API `compute-api` не предоставляет выгрузку полных SoA-массивов VRAM.
   - *Вопрос*: Требуется ли введение отладочного методологического трейта (например, `DebugSnapshotExt`), доступного только под фичей `test-harness`, или формат выгрузки `BatchResult` будет расширен?

2. **Окончательная Форма Полезной Нагрузки `BatchResult`**:
   - *Контекст*: Структура результатов батча находится на этапе согласования.
   - *Вопрос*: Какие именно поля (число спайков, маски выходов, чексуммы) включаются в `BatchResult` для базового потикового сравнения?

3. **Локализация Пайплайна Генерации и Верификации ABI-Зеркал**:
   - *Контекст*: Тесты дрифта ABI-зеркал проверяют совпадение типов между Rust и C++.
   - *Вопрос*: Где именно должен жить генератор C++ заголовков — в крейте `layout`, внутри бэкендов или в виде автономной build-helper утилиты?

4. **Организация Автоматического Запуска Аппаратных Тестов в CI/CD**:
   - *Контекст*: Тестирование CUDA и HIP требует наличия физических GPU и установленных SDK на раннерах.
   - *Вопрос*: Каким образом маркируются тесты для запуска на специализированных CI-раннерах с GPU ускорителями?

5. **Разграничение Дифференциальных и Свойственных/Фаззинг Тестов (Property/Fuzzing Tests)**:
   - *Контекст*: Помимо детерминированных фикстур, эффективен случайный фаззинг входных буферов.
   - *Вопрос*: Должны ли property-based тесты (на базе `proptest` / `quickcheck`) жить внутри `test-harness` или выноситься в отдельный крейт `compute-fuzz`?

6. **Управление Pinned-Буферами Хоста для Снэпшотов**:
   - *Контекст*: Для скоростного снятия дампов VRAM требуется Page-Locked память.
   - *Вопрос*: Кто выделяет и утилизирует Pinned-буферы при отладочном снятии дампов состояний в `test-harness`?
