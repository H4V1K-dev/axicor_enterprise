## §1. Идентификация

| Поле | Значение |
|---|---|
| **Имя крейта** | `test-harness` |
| **Слой** | Слой 3 — Вычислительные Бэкенды (Dev-Support / Conformance Suite) |
| **Тип** | Dev-Support Library (`lib`) |
| **no_std** | Нет (`false`) — требуется стандартная библиотека Rust, системный аллокатор кучи хоста и тестовый раннер |
| **Описание** | Вспомогательный тестовый комплекс Слоя 3, предназначенный для проверки соответствия вычислительных бэкендов (`ComputeBackend`), дифференциального прогона тестовых фикстур побитового сравнения, валидации ABI-зеркал и отслеживания расхождений между реализациями (CPU, CUDA, HIP, Mock). Крейт не является частью боевого рантайма и не владеет физикой, макетами памяти или HAL-контрактами. |

## §4. Архитектура и Модель Тестирования (Testing Model)

Тестовый комплекс `test-harness` разделяет уровни проверки на изолированные категории:
1. **Тестирование Соответствия Контракту (Trait-Level Conformance)**: Прямые вызовы методов трейта `ComputeBackend` на конкретных бэкендах.
2. **Дифференциальные Прогоны (Differential Execution Runs)**: Параллельный запуск идентичных фикстур на проверочной реализации `CpuBackend` и проверяемом бэкенде (CUDA, HIP, Mock) с потиковой сверкой выходящих спайков и дампов состояния через `debug_snapshot`.
3. **Разделение Результатов Теста**: `Passed`, `Failed(HarnessErrorKind)`, `Skipped { backend, reason }`.

---

## §5. Модель Фикстур и Детерминизм

Все тестовые сценарии строятся на базе детерминированного генератора случайных чисел с фиксированным `MasterSeed`. Фикстуры конструируют входные данные исключительно через официальные DTO `ShardAllocSpec`, `ShardUpload` и `DayBatchCmd`.

---

## §6. Пошаговая Политика Сравнения и Снэпшоты Состояния

### §6.1. Детализация Отчета о Расхождении (First-Mismatch Report)
При обнаружении любого побитового или скалярного расхождения между эталоном и проверяемым бэкендом, тестовый раннер прерывает выполнение и формирует детальный отчет о первом несоответствии, включающий номер тика (`tick`), имя поля (`field`), физическое смещение в буфере (`offset`), ожидаемое (`expected`) и фактическое (`actual`) значение.

### §6.2. Отладочный Снимок Состояния (`debug_snapshot`)
В `compute-api v2.1` метод `debug_snapshot(&mut self, handle: VramHandle, snapshot: ShardSnapshotMut<'_>) -> Result<(), ComputeApiError>` зафиксирован в составе основного трейта `ComputeBackend`. Тестовый комплекс использует этот метод для выгрузки полного состояния VRAM/RAM в подготовленные Pinned/Host буферы и 100% побитового сравнения плоскостей SoA-памяти между CPU и GPU бэкендами.

## §7. Нормализация Ошибок и Отчетность (`HarnessError`)

```rust
#[derive(Debug)]
pub enum HarnessOutcome {
    Passed,
    Failed(HarnessErrorKind),
    Skipped { backend: BackendKind, reason: String },
}

#[derive(Debug)]
pub enum HarnessErrorKind {
    ResultMismatch { tick: u64, field: &'static str, expected: String, actual: String },
    SnapshotMismatch { offset: usize, expected: u8, actual: u8 },
    ErrorMappingMismatch { expected: ComputeApiError, actual: ComputeApiError },
    AbiMirrorMismatch { struct_name: &'static str, reason: &'static str },
    BackendUnavailable { backend: BackendKind, reason: String },
    FeatureNotCompiled { feature: &'static str },
    BackendError(ComputeApiError),
    FacadeError(String),
}
```

## §8. Требуемые Инварианты

- **INV-HARNESS-001**: `test-harness` ни при каких условиях не дублирует и не объявляет собственную физическую математику (единственный источник — `physics`).
- **INV-HARNESS-002**: `test-harness` ни при каких условиях не задает правила выравнивания или смещения полей SoA-массивов в памяти (единственный источник — `layout`).
- **INV-HARNESS-003**: Отсутствие опционального аппаратного ускорителя или драйвера явно фиксируется в отчетах как пропуск (`HarnessOutcome::Skipped`).
- **INV-HARNESS-004**: Побитовая проверка сгенерированных спайковых ID выполняется строго в каждом тике батча.

## §9. Golden Tests / Обязательная Матрица Тестирования

Крейт `test-harness` обязан реализовывать 14 базовых тестовых сценариев:

1. **Прогон Базовых Фикстур на CPU (`test_cpu_reference_fixture_runs`)**: Верификация работоспособности эталонного CPU-пути на всех фикстурах §5.1.
2. **Соответствие Трейту HAL (`test_compute_backend_trait_conformance`)**: Комплексная проверка соответствия реализации контракту `ComputeBackend`.
3. **Тестирование Mock-Бэкенда (`test_mock_backend_conformance`)**: Валидация корректности вызовов на Mock-бэкенде.
4. **Дифференциальный Тест CUDA vs CPU (`test_cuda_matches_cpu_fixtures_when_available`)**: Побитовая сверка CUDA с CPU при включенной фиче `cuda` и наличии оборудования.
5. **Дифференциальный Тест HIP vs CPU (`test_hip_matches_cpu_fixtures_when_available`)**: Побитовая сверка HIP с CPU при включенной фиче `hip` и наличии оборудования.
6. **Отчет о Пропуске Опциональной Матрицы (`test_backend_unavailable_is_reported_as_skip_for_optional_matrix`)**: Проверка корректной генерации статуса `HarnessOutcome::Skipped` при отсутствии оборудования.
7. **Единообразие Ошибок Валидации Хоста (`test_invalid_dto_errors_are_consistent_across_backends`)**: Проверка совпадения транслируемых ошибок валидации хоста (`ComputeApiError`) до вызова вендорских API при подаче одинаковых битых DTO.
8. **Проверка Порядка Этапов Батча (`test_batch_stage_order_fixture`)**: Верификация строгого соблюдения последовательности 8 этапов вычислений через Mock/Trace бэкенд либо специально подобранные фикстуры, чувствительные к порядку этапов.
9. **Отсутствие Вендорских Типов в Публичном API Тестов (`test_no_raw_vendor_types_in_public_test_api`)**: Компиляционная проверка отсутствия сырых указателей и тип-дескрипторов вендоров в типах тестов.
10. **Валидация Размеров Блобов через Layout (`test_layout_blob_size_validation_fixture`)**: Проверка корректности расчета объемов памяти фикстур через функции `layout`.
11. **Контроль Дрифта ABI-Зеркал (`test_abi_mirror_drift_detection`)**: Автоматическая проверка совпадения размеров и выравниваний `ShardVramPtrs`, `BurstHeads8` и `VariantParameters`.
12. **Детализация Отчета о Первом Расхождении (`test_first_mismatch_report_contains_tick_field_offset_expected_actual`)**: Верификация наличия всех обязательных полей в отчете об ошибке сравнения.
13. **Тестирование Фасада под Фичей (`test_facade_lifecycle_under_facade_feature`)**: Проверка автоматов состояний `ShardEngine` под фичей `facade`.
14. **Снятие и Потиковое Сравнение Снэпшотов (`test_debug_snapshot_comparison`)**: Валидация вызова `debug_snapshot` для потикового снимка и 100% побитового сравнения SoA-плоскостей состояния VRAM между CPU и GPU.

## §10. Open Questions / Review Debt (Открытые Вопросы и Противоречия)

1. **Локализация Пайплайна Генерации и Верификации ABI-Зеркал (REV-PHYS-009)**:
   - *Контекст*: Тесты дрифта ABI-зеркал проверяют совпадение типов между Rust и C++.
   - *Вопрос*: Где именно должен жить генератор C++ заголовков — в крейте `layout`, внутри бэкендов или в виде автономной build-helper утилиты?

2. **Организация Автоматического Запуска Аппаратных Тестов в CI/CD (REV-TEST-004)**:
   - *Контекст*: Тестирование CUDA и HIP требует наличия физических GPU и установленных SDK на раннерах.
   - *Вопрос*: Каким образом маркируются тесты для запуска на специализированных CI-раннерах с GPU ускорителями?

3. **Разграничение Дифференциальных и Свойственных/Фаззинг Тестов (REV-TEST-005)**:
   - *Контекст*: Помимо детерминированных фикстур, эффективен случайный фаззинг входных буферов.
   - *Вопрос*: Должны ли property-based тесты (на базе `proptest` / `quickcheck`) жить внутри `test-harness` или выноситься в отдельный крейт `compute-fuzz`?
