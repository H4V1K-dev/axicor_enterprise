## §1. Идентификация

| Поле | Значение |
|---|---|
| **Имя крейта** | `test-harness` |
| **Слой** | Слой 3 — Вычислительные Бэкенды (Dev-Support / Conformance Suite) |
| **Тип** | Dev-Support Library (`lib`) |
| **no_std** | Нет (`false`) — требуется стандартная библиотека Rust, системный аллокатор кучи хоста и тестовый раннер |
| **Описание** | Вспомогательный тестовый комплекс Слоя 3, предназначенный для проверки соответствия вычислительных бэкендов (`ComputeBackend`), дифференциального прогона тестовых фикстур побитового сравнения, валидации ABI-зеркал и отслеживания расхождений между реализациями (CPU, CUDA, HIP, Mock). Крейт не является частью рантайма и не владеет физикой, макетами памяти или HAL-контрактами. В дифференциальном тестировании эталоном (Reference Implementation) выступает CpuBackend, с выходом и снимками которого сравниваются тестируемые бэкенды. Harness не дублирует и не пересчитывает физические законы или формулы разметки памяти самостоятельно. |

## §4. Архитектура и Модель Тестирования (Testing Model)

Тестовый комплекс `test-harness` разделяет уровни проверки на изолированные категории:
1. **Тестирование Соответствия Контракту (Trait-Level Conformance)**: Прямые вызовы методов трейта `ComputeBackend` на конкретных бэкендах.
2. **Дифференциальные Прогоны (Differential Execution Runs)**: Параллельный запуск идентичных фикстур на проверочной реализации `CpuBackend` (служащим эталонным исполнением) и проверяемом бэкенде (CUDA, HIP, Mock) с потиковой сверкой выходящих спайков и дампов состояния через `debug_snapshot`. Harness не выполняет независимый расчет физики или смещений.
3. **Разделение Результатов Теста**: `Passed`, `Failed(HarnessErrorKind)`, `Skipped { backend, reason }`.

## §5. Модель Фикстур и Детерминизм

Все тестовые сценарии строятся на базе детерминированного генератора случайных чисел с фиксированным `MasterSeed`.

### §5.1. Конструирование входных данных
Фикстуры конструируют входные данные исключительно через официальные DTO Слоя 3:
- `ShardAllocSpec`
- `ShardUpload`
- `DayBatchCmd`

### §5.2. Состав ShardUpload
`ShardUpload` обязательно и всегда содержит:
- `state_blob`
- `axons_blob`
- `variant_table: &'a [VariantParameters; VARIANT_LUT_LEN]`

### §5.3. Принципы генерации и инварианты фикстур
- **Источники геометрии**: Размеры и смещения (offsets) `state_blob` и `axons_blob` запрашиваются исключительно из функций и констант крейта `layout`.
- **Источники базовых констант**: Константные значения, seed, хеши и атомарные типы берутся напрямую из крейта `types`.
- **Запрет дублирования логики**: Физические функции и формулы симуляции из крейта `physics` не дублируются и не копируются в `test-harness`. Для проверки бэкендов в качестве эталона используется запускаемый `CpuBackend`, а `test-harness` только сравнивает результаты других бэкендов с ним.

## §6. Пошаговая Политика Сравнения и Снэпшоты Состояния

### §6.1. Политика сравнения результатов выполнения батча
Дифференциальное сравнение результатов выполнения `DayBatchCmd` (возвращаемых в `BatchResult`) разделяется на детерминированные и недетерминированные (телеметрия) компоненты:
1. **Детерминированные поля `BatchResult`**: Сравниваются на полное совпадение:
   - `ticks_executed`
   - `generated_spikes_count`
   - `output_spikes_written`
   - `dropped_spikes_count`
2. **Недетерминированные поля (телеметрия)**: Поле `execution_time_us` является wall-clock телеметрией. Оно **не должно** участвовать в побитовом сравнении.
3. **Выходные спайки**: Выходные спайки `output_spikes` и счетчики спайков `output_spike_counts` сравниваются строго потиково.

### §6.2. Детализация Отчета о Расхождении (First-Mismatch Report)
При обнаружении любого побитового или скалярного расхождения между эталоном (`CpuBackend`) и проверяемым бэкендом, тестовый раннер немедленно прерывает выполнение и формирует детальный отчет о первом несоответствии (`HarnessErrorKind`), выражающий все обязательные поля:
- Имя фикстуры (`fixture_name`)
- Номер тика (`tick`)
- Имя поля или плоскости (`field_or_plane`)
- Смещение (`offset`): байтовое смещение внутри плоскости/блоба (для снимков)
- Ожидаемое значение (`expected`)
- Фактическое значение (`actual`)

Для расхождений снимков (`SnapshotMismatch`) обязательно должны быть понятны плоскость (`plane`) и байтовое смещение (`offset`). Для расхождений результатов батча (`ResultMismatch`) обязательно должны быть понятны имя фикстуры (`fixture_name`), тик (`tick`) и имя поля (`field`).

### §6.3. Отладочный Снимок Состояния (`debug_snapshot`)
В `compute-api v2.2` метод `debug_snapshot(&mut self, handle: VramHandle, snapshot: ShardSnapshotMut<'_>) -> Result<(), ComputeApiError>` зафиксирован в составе основного трейта `ComputeBackend`. 
Тестовый комплекс использует этот метод для выгрузки полного состояния VRAM/RAM и осуществляет 100% побайтовое сравнение по плоскостям SoA-памяти в `state_blob` и `axons_blob` между эталонным CPU-бэкендом и проверяемым бэкендом.

**Правила аллокации буферов для снимков:**
- `test-harness` выделяет для `ShardSnapshotMut` стандартные host-буферы (обычная host-память).
- Бэкенд самостоятельно отвечает за выделение, использование и очистку любых внутренних pinned/staging буферов, если они требуются для выполнения копирования из VRAM в переданные host-срезы.

## §7. Нормализация Ошибок и Отчетность (`HarnessError`)

```rust
#[derive(Debug)]
pub enum HarnessOutcome {
    Passed,
    Failed(HarnessErrorKind),
    Skipped { backend: BackendKind, reason: String },
}

#[derive(Debug)]
pub enum HarnessErrorKind {
    ResultMismatch {
        fixture_name: String,
        tick: u64,
        field: &'static str,
        expected: String,
        actual: String,
    },
    SnapshotMismatch {
        fixture_name: String,
        tick: u64,
        plane: &'static str,
        offset: usize, // byte offset within the plane/blob
        expected: u8,
        actual: u8,
    },
    ErrorMappingMismatch { expected: ComputeApiError, actual: ComputeApiError },
    AbiMirrorMismatch { struct_name: &'static str, reason: &'static str },
    BackendUnavailable { backend: BackendKind, reason: String },
    FeatureNotCompiled { feature: &'static str },
    BackendError(ComputeApiError),
    FacadeError(String),
}
```

## §8. Требуемые Инварианты

- **INV-HARNESS-001**: `test-harness` ни при каких условиях не дублирует и не объявляет собственную физическую математику (единственный источник — `physics`).
- **INV-HARNESS-002**: `test-harness` ни при каких условиях не задает правила выравнивания или смещения полей SoA-массивов в памяти (единственный источник — `layout`).
- **INV-HARNESS-003**: Отсутствие опционального аппаратного ускорителя или драйвера явно фиксируется в отчетах как пропуск (`HarnessOutcome::Skipped`).
- **INV-HARNESS-004**: Побитовая проверка сгенерированных спайковых ID выполняется строго в каждом тике батча.

## §9. Golden Tests / Обязательная Матрица Тестирования

Крейт `test-harness` обязан реализовывать 14 базовых тестовых сценариев:

1. **Прогон Базовых Фикстур на CPU (`test_cpu_reference_fixture_runs`)**: Верификация работоспособности эталонного CPU-пути (`CpuBackend`) на детерминированных conformance фикстурах и сборка эталонных результатов.
2. **Соответствие Трейту HAL (`test_compute_backend_trait_conformance`)**: Комплексная проверка соответствия реализации контракту `ComputeBackend`.
3. **Тестирование Mock-Бэкенда (`test_mock_backend_conformance`)**: Валидация корректности вызовов на Mock-бэкенде.
4. **Дифференциальный Тест CUDA vs CPU (`test_cuda_matches_cpu_fixtures_when_available`)**: Потиковая сверка CUDA с CPU при включенной фиче `cuda` и наличии оборудования. Сравниваются детерминированные поля `BatchResult`, выходные спайки, а также побайтово состояние SoA-плоскостей `state_blob` и `axons_blob` через `debug_snapshot`.
5. **Дифференциальный Тест HIP vs CPU (`test_hip_matches_cpu_fixtures_when_available`)**: Потиковая сверка HIP с CPU при включенной фиче `hip` и наличии оборудования. Сравниваются детерминированные поля `BatchResult`, выходные спайки, а также побайтово состояние SoA-плоскостей `state_blob` и `axons_blob` через `debug_snapshot`.
6. **Отчет о Пропуске Опциональной Матрицы (`test_backend_unavailable_is_reported_as_skip_for_optional_matrix`)**: Проверка корректной генерации статуса `HarnessOutcome::Skipped` при отсутствии оборудования для опциональных бэкендов.
7. **Единообразие Ошибок Валидации Хоста (`test_invalid_dto_errors_are_consistent_across_backends`)**: Проверка совпадения транслируемых ошибок валидации хоста (`ComputeApiError`) до вызова вендорских API при подаче одинаковых битых DTO.
8. **Проверка Порядка Этапов Батча (`test_batch_stage_order_fixture`)**: Верификация строгого соблюдения последовательности 8 этапов вычислений через Mock/Trace бэкенд либо специально подобранные фикстуры, чувствительные к порядку этапов.
9. **Отсутствие Вендорских Типов в Публичном API Тестов (`test_no_raw_vendor_types_in_public_test_api`)**: Компиляционная проверка отсутствия сырых vendor-типов (указателей, дескрипторов CUDA/HIP) в типах и сигнатурах публичного API тестов.
10. **Валидация Размеров Блобов через Layout (`test_layout_blob_size_validation_fixture`)**: Fixture builder валидирует размеры и смещения state/axon буферов, сверяясь с формулами `layout`. Проверяется обязательное наличие `variant_table` в каждом `ShardUpload`.
11. **Контроль Дрифта ABI-Зеркал (`test_abi_mirror_drift_detection`)**: Автоматическая проверка совпадения размеров и выравниваний `ShardVramPtrs`, `BurstHeads8` и `VariantParameters`.
12. **Детализация Отчета о Первом Расхождении (`test_first_mismatch_report_contains_fixture_tick_field_or_plane_offset_expected_actual`)**: Верификация того, что отчет о первом расхождении (`HarnessErrorKind`) выражает все обязательные поля: имя фикстуры (`fixture_name`), тик (`tick`), поле/плоскость (`field_or_plane`/`field`/`plane`), смещение (`offset`), ожидаемое (`expected`) и фактическое (`actual`) значения.
13. **Тестирование Фасада под Фичей (`test_facade_lifecycle_under_facade_feature`)**: Проверка автоматов состояний `ShardEngine` под фичей `facade`.
14. **Снятие и Побайтовое Сравнение Снэпшотов (`test_debug_snapshot_comparison`)**: Валидация вызова `debug_snapshot` для побайтового сравнения SoA-плоскостей `state_blob` и `axons_blob` в выделенных harness-ом host-буферах.

## §11. Resolved Architectural Decisions (Принятые Решения Pass 2)

1. **Снимок состояния VRAM (REV-TEST-001)**: В `ComputeBackend` зафиксирован метод `debug_snapshot` с DTO `ShardSnapshotMut` в `compute-api v2.2`.
2. **Форма BatchResult (REV-TEST-002)**: Duplicate of REV-COMPUTE-API-004. Решено через разделение: выходящие спайки пишутся напрямую в предоставленные `DayBatchCmd.output_spikes` host-буферы, а `BatchResult` возвращает детерминированные счетчики и телеметрию времени выполнения.
3. **Аппаратные CUDA/HIP тесты в CI/CD (REV-TEST-004)**: Решено для Stage 1: аппаратные CUDA/HIP тесты подключаются через условную компиляцию (feature-gated). При отсутствии GPU бэкенд возвращает статус `HarnessOutcome::Skipped`. Организация выделенных GPU CI runners вынесена в будущие задачи инфраструктуры и не блокирует реализацию.
4. **Разграничение дифференциальных и свойственных/фаззинг тестов (REV-TEST-005)**: Решено: `test-harness` содержит детерминированные conformance фикстуры. Свойственные/fuzz тесты выносятся в будущий отдельный крейт `compute-fuzz` или отдельный опциональный слой и не входят в Stage 1.
5. **Управление Pinned-Буферами (REV-TEST-006)**: Duplicate of REV-COMPUTE-API-002. Решено: бэкенды самостоятельно управляют внутренними pinned/staging буферами, а `test-harness` передает срезы обычной host-памяти через `ShardSnapshotMut`.
6. **Устранение RNG Seed Debt**: Горячий цикл детерминирован, передача RNG seed в `DayBatchCmd` не требуется.
