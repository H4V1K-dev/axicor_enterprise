## §1. Идентификация

| Поле | Значение |
|------|----------|
| **Имя крейта** | `compute-cuda` |
| **Слой** | Слой 3 — Вычислительные Бэкенды (Compute Hardware Backends) |
| **Тип** | Library (`lib`) |
| **no_std** | Нет (`false`) — зависит от CUDA Runtime API, FFI-связывания и системного аллокатора кучи |
| **Описание** | NVIDIA CUDA вычислительный бэкенд, реализующий трейт `ComputeBackend` из `compute-api`. Крейт управляет FFI-вызовами CUDA Runtime API, VRAM-ресурсами ускорителя, неблокирующими стримами (`cudaStream_t`), запуском CUDA-ядер и маппингом ошибок вендора в `ComputeApiError`. Крейт не владеет DTO, макетами памяти, физическими уравнениями или рантайм-планировщиком. |

## §2. Стек и Окружение

### §2.1. Внутренние зависимости (inbound)

| Крейт | Что используется | Зачем |
|-------|-----------------|-------|
| `types` (Слой 0) | Атомарные типы и константы | Использование типов (`Tick`, `Voltage`, `SomaFlags`, `AXON_SENTINEL`) и битовых масок. |
| `layout` (Слой 1) | Расчет смещений SoA-плоскостей | Нарезка монолитных буферов VRAM с использованием `CACHE_LINE_BYTES` и `PADDED_N_ALIGNMENT`. |
| `physics` (Слой 0) | Математика GLIF, GSOP, Active Tail | Ожидание побитовой идентичности вычислений на GPU с эталоном `physics`. |
| `compute-api` (Слой 3) | `ComputeBackend`, `VramHandle`, DTO вызовов и `ComputeApiError` | Реализация абстрактного контракта вычислений HAL и типизированной иерархии ошибок. |

> [!IMPORTANT]
> Крейт `compute-cuda` содержит C++/CUDA ядра, являющиеся аппаратными зеркалами математических уравнений. Крейт **не является источником истины** по физике и макетам памяти: все C++/CUDA ядра и ABI-заголовки должны кодогенерироваться или механически верифицироваться из оригинальных крейтов `physics` и `layout`.

### §2.2. Зависимые Компоненты и Характеристики Бэкенда (`BackendCapabilities`)

| Крейт / Компонент | Роль в системе и взаимодействие |
|-------------------|---------------------------------|
| `compute` (Слой 3) | Фасад вычислений подсоединяет `CudaBackend` через feature flag `cuda` для высокопроизводительного исполнения на NVIDIA GPU. |
| `test-harness` (Слой 3) | Выполняет дифференциальное тестирование результатов `CudaBackend` против `CpuBackend`. |

При вызове метода `capabilities()` структура `CudaBackend` возвращает характеристики CUDA-вычислителя (без введения случайных скрытых лимитов):
```rust
BackendCapabilities {
    lane_count: 32,                // Аппаратный размер варпа NVIDIA (Warp Size)
    supports_async: true,          // Асинхронное исполнение ядер через CUDA streams
    supports_ephys: false,         // Ограничено до фиксации DTO отладчика
    max_batch_ticks: TBD,          // Выводится из конфигурации симулятора и лимитов устройства
    alignment_bytes: 64,           // Выравнивание SoA-плоскостей VRAM под L2 кэш
    pinned_host_required: true,    // Требование Page-Locked памяти для скоростного DMA D2H
}
```

### §2.3. Внешние Зависимости

| Crate | Версия | Сфера использования |
|-------|--------|---------------------|
| `cc` | `=1.2.56` *(build-dep)* | Компиляция CUDA-ядер (`.cu`) с помощью компилятора `nvcc` на этапе сборки *(Зафиксировано в качестве рабочей версии)*. |

> [!NOTE]
> Зависимость `bytemuck` рассматривается как внутренняя деталь реализации FFI-преобразований. Публичный API бэкенда избавлен от `anyhow` и сторонних вендорских типов.

### §2.4. Feature Flags

Секция публичных feature flags не используется. Крейт собирается как выделенная библиотека вычислений для ускорителей NVIDIA.

## §4. Модель Ресурсов VRAM и Межспецификационные Долги

### §4.1. Физическая Стратегия Аллокации VRAM
При вызове `alloc_shard(spec: ShardAllocSpec)` бэкенд вычисляет необходимые объемы VRAM на основе формул `layout`. Для исключения фрагментации VRAM и высоких накладных расходов запрещено вызывать `cudaMalloc` под каждый отдельный SoA-массив. Аллокация выполняется **двумя ограниченными физическими блоками**:
1. **Монолитный Блок Сомы и Синапсов**: Единая плоская VRAM-аллокация под все SoA-плоскости нейронов и синапсов пропорционально `padded_n`.
2. **Блок Аксонных Головок**: Отдельная VRAM-аллокация под массив `axon_heads`, так как `total_axons != padded_n`.

### §4.2. Межспецификационные Долги (Cross-Spec Debts)
> [!WARNING]
> 1. **Фабричный Конструктор `VramHandle`**: Приватное поле `VramHandle` в `compute-api v2.0` блокирует создание дескрипторов на стороне бэкендов. Ожидается введение фабричного метода `VramHandle::new_for_backend`.
> 2. **Автоматическая Генерация C++ Зеркал (C++ Header Mirrors)**: Ручное дублирование C-ABI структур (`VariantParameters`, `BurstHeads8`, `ShardVramPtrs`) и физических формул в C++ заголовках (`core_math.h`) категорически запрещено как долгосрочный дизайн. C++ зеркала должны генерироваться или механически верифицироваться из единого источника истины в Rust (`physics`/`layout`).
> 3. **Размер Constant Memory и API Загрузки Вариантов**: Разъяснено, что текущий DTO `ShardUpload` содержит только `state_blob` и `axons_blob` и не несет вариантов нейронов. Вопрос загрузки таблицы вариантов в Constant Memory (`16 * 64 = 1024 байт`) вынесен как открытый вопрос/долг: требуется либо отдельный метод в HAL, либо расширение DTO `ShardUpload`, либо операция фасада `compute`.

### §4.3. Аллокация и Загрузка Данных
1. **`alloc_shard(spec: ShardAllocSpec) -> Result<VramHandle, ComputeApiError>`**: Осуществляет проверку лимитов и выделяет 2 выровненных физических блока VRAM, инициализирует отдельный неблокирующий `cudaStream_t` и регистрирует ресурс за дескриптором `VramHandle`.
2. **`upload_shard(handle: VramHandle, upload: ShardUpload<'_>) -> Result<(), ComputeApiError>`**: Валидирует размеры байтовых блобов `state_blob` и `axons_blob` с помощью формул `layout` и выполняет H2D передачу в VRAM через `cudaMemcpyAsync`. Поскольку методы синхронны, возврат происходит после гарантированного завершения передачи или полной синхронизации стрима.
3. **Безопасность Указателей**: Публичный API Rust не раскрывает сырые указатели (`*mut u8`), `ShardVramPtrs`, `cudaStream_t` или C-ABI структуры. Все указатели изолированы внутри структуры `CudaBackend`.

## §5. Модель Выполнения и Порядок Этапов (Execution Model & Stage Order)

Метод `run_day_batch(handle, cmd)` выполняет запуск батча тиков на выделенном CUDA-стриме шарда.

### §5.1. Порядок Выполнения Этапов Внутри Тика (Tick Stage Order)
Выполнение каждого тика на GPU строго повторяет эталонный порядок `compute-cpu`:
1. **Валидация DTO**: Проверка длин массивов и емкостей команды `DayBatchCmd`.
2. **Пакетная H2D Передача Входных Буферов (Batch-Level Input Transfer)**: Скоростная загрузка массивов входных данных на GPU выполняется один раз на уровне батча (или по мере готовности), после чего CUDA-ядра тиков обращаются к этим VRAM-буферам через потиковое индексирование и срез (slicing).
3. **Инъекция Виртуальных Входов (Virtual Inputs)**: Запуск ядра инъекции сенсорных сигналов из битмаски `cmd.input_bitmask` по `cmd.virtual_offset` и `cmd.num_virtual_axons`.
4. **Инъекция Входных Спайков (Incoming Spikes)**: Запуск ядра распаковки потикового массива спайковых ID из `cmd.incoming_spikes` с учетом емкости `cmd.max_spikes_per_tick` и количества элементов из `cmd.spike_counts`.
5. **Продвижение Аксонных Головок (Active Tail)**: Запуск ядра продвижения спайков вдоль аксонов с учетом скаляра `cmd.v_seg` (жесткое кодирование `v_seg = 1` запрещено).
6. **Обновление Состояния Нейронов (Neuron State Update)**: Запуск ядра интеграции мембранного потенциала GLIF через верифицированное C++ зеркало физики, расчет гомеостаза, рефрактерных таймеров и генерация спайков.
7. **Применение Синаптической Пластичности (GSOP)**: Запуск ядра обновления весов с учетом дофаминовой модуляции `cmd.dopamine` и Закона Дейла.
8. **Фиксация Результатов (Условный / Draft Этап)**: Асинхронное D2H копирование результатов в Pinned-буферы хоста при наличии утвержденных DTO в `compute-api`.

## §6. Параллелизм, Защита от Гонки и Детерминизм (Parallelism & Determinism)

1. **Строгий Детерминизм (Bit-to-Bit Identity)**: Результат вычислений на CUDA GPU при одинаковых входных данных обязан быть побитово идентичен результатам `compute-cpu` и не зависеть от сетки блоков/потоков.
2. **Защита от Гонок Данных при Записи Спайков в VRAM**:
   - При обновлении сомы ядра генерируют новые спайки и записывают их в `axon_heads` через карту `soma_to_axon`.
   - Параллельное выполнение ядра обновления допустимо **строго при условии**, что карта `soma_to_axon` гарантирует соответствие «один-к-одному» (дисюнктные адреса записи во VRAM), либо запись спайков выполняется через промежуточный буфер накопления в VRAM. Конкурирующая неатомарная запись запрещена.
3. **Детерминированные Атомики**: Использование `atomicAdd` в видеопамяти для сбора телеметрии или суммирования выходов допустимо только при условии, что порядок считывания результатов не нарушает побитовую идентичность, либо выходы сортируются/упорядочиваются перед публикацией.

## §7. Трансляция Ошибок Вендора и Политика Безопасности (Error Mapping & Safety Policy)

Публичный API бэкенда гарантирует полное отсутствие паник (`panic!`, `assert!`, `unwrap`, `expect`). Все ошибки CUDA Runtime API перехватываются врапперами запусков ядер и детерминированно мапятся в `ComputeApiError` по семействам ошибок:

| Семейство Ошибок CUDA Runtime API | Итоговый Код `ComputeApiError` |
|-----------------------------------|--------------------------------|
| **Allocation Errors** (например, сбои выделения памяти VRAM) | `ComputeApiError::OutOfMemory` |
| **Copy / DMA Errors** (например, ошибки асинхронного копирования H2D/D2H или невалидные указатели) | `ComputeApiError::DmaFailed` |
| **Kernel Launch Errors** (например, превышение ресурсов блока или сбои запуска ядер) | `ComputeApiError::KernelLaunchFailed` |
| **Stream Synchronization Errors** (например, ошибки синхронизации стрима при выполнении) | `ComputeApiError::SynchronizeFailed` |
| **Device Lost / Reset Errors** (например, аппаратный сброс GPU или отключение устройства во время работы) | `ComputeApiError::DeviceLost` |

> [!NOTE]
> Ошибка `BackendUnavailable` относится к фазе автодетекции/пробинга оборудования на уровне фасада `compute` (Layer 3 facade) и не используется как ошибка выполнения внутри уже созданного бэкенда. При аппаратом сбое во время работы рантайма возвращается `DeviceLost`.

При передаче невалидных DTO прерывание происходит до вызова CUDA FFI с возвратом `InvalidShape`, `SizeMismatch` или `InvalidHandle`. Метод `teardown()` атомарно освобождает VRAM, уничтожает `cudaStream_t` и переводит дескриптор в `None`; повторный вызов идемпотентен.

## §8. Сохранение Совместимости с Легаси-Фактами (Legacy Behavioral Facts)

1. **Маркер Сентинела Аксона (`AXON_SENTINEL`)**: CUDA-ядра используют побитовое правило с константой `types::AXON_SENTINEL` (`0x80000000`), корректно обрабатывая проскакивание при $v_{\text{seg}} > 1$.
2. **Обработка Tombstone в Target (`0` vs `EMPTY_PIXEL`)**: Поддержка единой политики двойной проверки целевых синапсов, утверждаемой в `types`/`physics`.
3. **Семантика Heartbeat / DDS**: Спайк Heartbeat не сбрасывает потенциал $V$, не взводит рефрактерный таймер и не меняет параметры гомеостаза, если не произошел GLIF-спайк.
4. **Затухание Гомеостаза и Близость Сомы**: Гомеостатический пенальти затухает во время рефрактера; биты типа нейрона в `SomaFlags` строго сохраняются.
5. **Дофаминовая Модуляция в GSOP**: Изменение весов учитывает сигналы D1/D2 с соблюдением знаковой политики `physics`.

## §9. Требуемые Инварианты

- **INV-COMPUTE-CUDA-001**: Структура `CudaBackend` имплементирует трейт `ComputeBackend` из `compute-api`.
- **INV-COMPUTE-CUDA-002**: Вызов `kind()` возвращает строго `BackendKind::Cuda`.
- **INV-COMPUTE-CUDA-003**: Выделение VRAM под шард выполняется строго двумя монолитными блоками памяти.
- **INV-COMPUTE-CUDA-004**: Побитовая идентичность вычислений сохраняется с `compute-cpu`.
- **INV-COMPUTE-CUDA-005**: Параллельная запись спайков ядрами GPU исключает гонки данных во VRAM за счет дисюнктной карты `soma_to_axon` или буфера накопления.
- **INV-COMPUTE-CUDA-006**: Публичный API бэкенда на Rust не раскрывает сырые указатели (`*mut u8`), `cudaStream_t` и C-ABI структуры указателей.
- **INV-COMPUTE-CUDA-007**: Все вызовы CUDA API обернуты в проверки семейств ошибок с маппингом в `ComputeApiError`, паники запрещены.

## §10. Golden Tests / Обязательная Матрица Тестирования

Крейт `compute-cuda` обязан быть покрыт набором автоматических интеграционных тестов и статических проверок:

1. **Имплементация HAL Трейта (`test_cuda_implements_compute_backend`)**: Проверка реализации `ComputeBackend`.
2. **Идентификация Бэкенда (`test_cuda_backend_kind`)**: Проверка возврата `BackendKind::Cuda`.
3. **Проверка Характеристик (`test_cuda_backend_capabilities`)**: Проверка точности структуры `BackendCapabilities` (§2.2).
4. **Разграничение Инициализации и Потери Устройства (`test_cuda_device_lost_vs_unavailable`)**: Проверка того, что отсутствие GPU на фазе пробинга обрабатывается фасадом как `BackendUnavailable`, а сбой во время работы рантайма возвращает `DeviceLost`.
5. **Браковка Неверных Размеров до FFI (`test_cuda_upload_rejects_bad_sizes`)**: Проверка возврата `SizeMismatch` до обращения к CUDA API.
6. **Защита от Невалидных Дескрипторов (`test_cuda_rejects_invalid_handles`)**: Проверка обработки битых `VramHandle`.
7. **Изоляция Публичного API (`test_cuda_no_raw_pointers_in_api`)**: Компиляционная проверка сигнатур на отсутствие сырых указателей и вендорских типов.
8. **Падение Сборки при Рассогласовании ABI Зеркал (`test_cuda_abi_mirror_drift_prevention`)**: Тест компиляции/верификации C++ заголовков, падающий при любом расхождении полей, смещений или выравнивания в `ShardVramPtrs`, `BurstHeads8` и `VariantParameters`.
9. **Статус API Загрузки Constant Memory (`test_cuda_constant_upload_api_pending`)**: Статический тест, фиксирующий отсутствие публичного API загрузки Constant Memory до утверждения соответствующего DTO в `compute-api`.
10. **Ограниченность Физических Аллокаций (`test_cuda_bounded_allocations`)**: Верификация вызова ровно 2 физических аллокаций VRAM на шард через Mock/Stub FFI.
11. **Совпадение Порядка Этапов с CPU (`test_cuda_stage_order_matches_cpu`)**: Дифференциальный тест последовательности вычислений на эталонном фикстурном шарде.
12. **Идемпотентность Teardown и Уничтожение Стримов (`test_cuda_idempotent_teardown`)**: Проверка уничтожения `cudaStream_t` и безопасного повторного вызова `teardown()`.

## §11. Open Questions / Review Debt (Открытые Вопросы и Противоречия)

1. **Механизм Кодогенерации и Верификации C++ Зеркал из Rust (`physics`/`layout`)**:
   - *Контекст*: Зафиксирован запрет на ручной дублирующий C++ код.
   - *Вопрос*: Какая утилита или генератор (например, `cbindgen` или пользовательский AOT-скрипт) будет координировать автоматическую сборку C++ зеркал из источников истины?

2. **API и DTO Загрузки Таблицы Вариантов в Constant Memory**:
   - *Контекст*: `ShardUpload` содержит только байтовые блобы состояния и аксонов.
