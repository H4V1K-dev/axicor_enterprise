## §1. Идентификация

| Поле | Значение |
|---|---|
| Название | `compute-cuda` |
| Слой | Слой 3 — Compute |
| Тип | Library (`lib`) |
| no_std | **Нет** (зависит от CUDA Runtime API, динамического FFI-связывания и системного аллокатора хоста) |
| Описание | NVIDIA CUDA вычислительный бэкенд, реализующий абстрактный интерфейс вычислительного ядра и управляющий выполнением фазы симуляции Day Phase на графических ускорителях NVIDIA. |

## §3. Инварианты

### §3.1. Структурные инварианты

- **INV-COMPUTE-CUDA-001**: *Уникальность контекста CudaBackend*.
  - *Обоснование*: Структура `CudaBackend` обязана содержать потокобезопасный реестр выделенных дескрипторов VRAM (например, `SlotMap` под `RwLock`). Перекрестные операции с чужими дескрипторами VRAM физически невозможны.
  - *Следствие нарушения*: Попытка обращения к видеопамяти другого шарда (Data Leakage/Memory Corruption), UB.
  - *Где проверяется*: Юнит-тест `test_cuda_invalid_handle_checks`.

- **INV-COMPUTE-CUDA-002**: *Коалесцированный доступ к VRAM (Coalesced Memory Access)*.
  - *Обоснование*: Все CUDA-ядра симуляции Day Phase обязаны обращаться к глобальной памяти GPU (VRAM) непрерывными блоками по 32 потока (варп) по адресам SoA-массивов, выровненным по границам 64 байт (кэш-линия L2).
  - *Следствие нарушения*: Падение пропускной способности шины VRAM (до 3-5% от пика), срыв latency бюджета выполнения ядра Day Phase.
  - *Где проверяется*: Профилирование с помощью NVIDIA Nsight Compute.

### §3.2. Семантические инварианты

- **INV-COMPUTE-CUDA-003**: *Математическая консистентность через core_math.h*.
  - *Обоснование*: Вся математика законов интеграции мембранного потенциала ALIF и пластичности весов GSOP обязана подключаться в CUDA-ядрах исключительно из общего C++ заголовка `core_math.h`. Прямое дублирование формул в коде ядер запрещено во избежание расхождений.
  - *Следствие нарушения*: Потеря побитового соответствия симуляции (Butterfly Effect) между CPU и GPU.
  - *Где проверяется*: Код-ревью исходников `.cu` на наличие `#include "core_math.h"`.

- **INV-COMPUTE-CUDA-004**: *Изоляция стримов выполнения (Stream Context Isolation)*.
  - *Обоснование*: Каждый экземпляр выделенной памяти шарда (`VramHandle`) жестко привязывается к своему независимому неблокирующему CUDA стриму (`cudaStream_t`). Асинхронные ядра разных шардов запускаются параллельно в своих стримах.
  - *Следствие нарушения*: Взаимная сериализация вычислений шардов, сильное падение TPS при многошардовой конфигурации на одном GPU.
  - *Где проверяется*: Юнит-тест `test_cuda_stream_concurrency`.

- **INV-COMPUTE-CUDA-005**: *Безопасность FFI-запуска ядер (FFI Execution Guard)*.
  - *Обоснование*: Любой запуск CUDA-ядра обязан оборачиваться FFI-функцией-враппером, возвращающей `cudaError_t`. Бэкенд проверяет статус завершения и в случае ошибки мапит его в `ComputeApiError::VendorError` или `DeviceLost`, перехватывая сбой на стороне Rust.
  - *Следствие нарушения*: Тихий сбой выполнения ядер, приводящий к чтению устаревших или поврежденных данных в `download_output`.
  - *Где проверяется*: compile-time, линтинг FFI сигнатур, юнит-тесты.

- **INV-COMPUTE-CUDA-006**: *Абсолютное использование Pinned RAM для DMA*.
  - *Обоснование*: Выгрузка моторных команд (`OutputFrame`) и телеметрии (`TelemetryFrame`) из VRAM на хост обязана производиться в предварительно выделенную Page-Locked память (Pinned Memory, `cudaMallocHost`). Использование обычной страничной памяти (Pageable Memory) запрещено во избежание сбоев PCIe контроллера и возникновения page-faults в ядре ОС.
  - *Следствие нарушения*: Падение производительности DMA-трансляции (в 2-4 раза), блокировка потоков оркестратора, сбой `DmaTransferFailed` под нагрузкой.
  - *Где проверяется*: Интеграционные тесты производительности и профилирование PCIe шины.

- **INV-COMPUTE-CUDA-007**: *Аппаратные границы отладчика (Ephys Pointer Guard)*.
  - *Обоснование*: Структура `EphysCmd` передает сырые device-указатели `tids_d`, `uvs_d` и `trace_d`. Компилятор Rust не может отследить их границы. Бэкенд аппаратно обязан блокировать любую запись осциллограммы, если `current_tick >= max_ticks`, и прерывать выполнение, если `count > MAX_EPHYS_TARGETS` (16).
  - *Следствие нарушения*: Запись GPU-ядрами за пределы выделенного SHM/VRAM буфера (Out-of-Bounds Write), искажение данных симуляции, SIGSEGV драйвера.
  - *Где проверяется*: Интеграционный тест `test_cuda_ephys_out_of_bounds` и аппаратные проверки (C-ABI) внутри `launch_debug_record_v`.

- **INV-COMPUTE-CUDA-008**: *Аппаратный размер варпа (NVIDIA Warp Size)*.
  - *Обоснование*: Размеры блоков потоков и алгоритмы Warp-Aggregated Atomics жестко завязаны на размер варпа NVIDIA (32 потока). Специфические инструкции `__ballot_sync(0xFFFFFFFF, ...)` и `__shfl_sync` ожидают 32-битную маску активности варпа.
  - *Следствие нарушения*: Неопределенное поведение (UB) при сборе телеметрии, сбой адресации в VRAM, некорректный расчет `active_soma_ids`.
  - *Где проверяется*: compile-time static asserts в CUDA/C++ коде, юнит-тест `test_cuda_device_presence` (сверка `warpSize` через свойства устройства).

- **INV-COMPUTE-CUDA-009**: *Неизменяемость памяти хоста при асинхронном DMA (Host Memory Immutability)*.
  - *Обоснование*: Поскольку методы загрузки и запуска батча используют `cudaMemcpyAsync`, хост-память (буферы спайков, масок) должна оставаться неизменной и валидной на протяжении всего HFT-цикла до момента явной синхронизации (Stream Sync).
  - *Следствие нарушения*: Use-After-Free на GPU, повреждение входящих данных симуляции, Silent Data Corruption.
  - *Где проверяется*: Статический анализ времени жизни ссылок Rust (borrow checker для `DayBatchCmd<'a>`), а также интеграционные тесты.

### §3.3. Межкрейтовые инварианты

- **INV-CROSS-007**: *Битовое соответствие ShardVramPtrs*.
  - *Участники*: `layout`, `compute-cuda`, `compute-hip`.
  - *Кто владелец проверки*: `layout`.
  - *Обоснование*: См. [spec_layout.md §3.3]. Порядок, типы и выравнивание полей в Rust-структуре `ShardVramPtrs` должны байт-в-байт соответствовать структуре `ShardVramPtrs` в C++/CUDA коде.
  - *Следствие нарушения*: Segmentation fault при запуске CUDA-ядер, повреждение памяти GPU.
  - *Где проверяется*: FFI-тесты выравнивания в `test-harness`.

- **INV-CROSS-011**: *Кроссплатформенный детерминизм (Bit-to-Bit Identity)*.
  - *Участники*: `compute-api`, `compute-cuda`, `compute-hip`, `compute-cpu`, `test-harness`.
  - *Кто владелец проверки*: `test-harness`.
  - *Обоснование*: См. [spec_compute_api.md §3.3]. Все реализации `GpuBackend` (включая CUDA) обязаны возвращать побитово идентичные результаты для одинаковых входов и начального состояния.
  - *Следствие нарушения*: Рассинхронизация распределенного кластера при восстановлении шарда из теневых копий.
  - *Где проверяется*: Интеграционные тесты `test-harness`.

## §4. Публичный API

Крейт экспортирует структуру `CudaBackend`, реализующую абстрактный трейт `GpuBackend`. Вся низкоуровневая интеграция с FFI (`*mut c_void`, `ShardVramPtrs`) жестко скрыта внутри бэкенда и не протекает в публичный API.

### §4.1. Типы

```rust
pub struct CudaBackend {
    pub device_id: i32,
    /// Внутренний реестр аллоцированных шардов.
    /// Защищен RwLock для обеспечения Send + Sync интерфейса GpuBackend.
    resources: std::sync::RwLock<slotmap::SlotMap<VramHandle, ShardCudaResources>>,
}

/// Внутренняя структура (не экспортируется из крейта).
/// Хранит сырые C-ABI указатели и контекст конкретного шарда.
pub(crate) struct ShardCudaResources {
    pub vram_ptrs: axicor_layout::ShardVramPtrs,
    /// Непрозрачный указатель `cudaStream_t`. Гарантирует изоляцию батчей.
    pub stream: *mut std::ffi::c_void,
    pub layout: axicor_compute_api::ShardLayout,
}
```

- **Семантика**: `CudaBackend` — это реализация абстрактного GPU-вычислителя HAL для карт NVIDIA. Выступает в роли диспетчера, мапящего интерфейсные вызовы `run_day_batch` на конкретный `cudaStream_t` и Flat-буфер.
- **Жизненный цикл**: Инициализируется один раз при старте оркестратора ноды (`Arc<CudaBackend>`). Явно освобождает ресурсы при уничтожении через `free(handle)` или при `Drop` самого бэкенда (с перехватом ошибок CUDA).
- **Ограничения на значения**: `device_id` обязан лежать в диапазоне `0..cudaGetDeviceCount()`.

### §4.2. Трейты

Крейт предоставляет ровно одну реализацию: `impl GpuBackend for CudaBackend`.

- **Контракт**: Реализация обязана гарантировать потокобезопасность (`Send + Sync`).
- **Стримы**: Методы загрузки данных (`upload_state`), батчинга (`run_day_batch`) и выгрузки (`download_output`) выполняются асинхронно (не блокируя хост) путем постановки операций в очередь драйвера через `stream` из `ShardCudaResources`.
- **Обработка ошибок**: Любой запуск ядра оборачивается FFI-враппером, возвращающим `cudaError_t`. Бэкенд перехватывает этот статус и конвертирует его в `ComputeApiError` (например, `cudaErrorMemoryAllocation` -> `OutOfMemory`).

### §4.3. Функции

Крейт не содержит публичных свободных функций, кроме конструктора бэкенда:

```rust
impl CudaBackend {
    /// Инициализирует контекст CUDA для указанного устройства
    pub fn new(device_id: i32) -> Result<Self, ComputeApiError> {
        // ...
    }
}
```

- **Семантика**: Конструктор обязан вызвать `cudaSetDevice(device_id)` и проверить доступность GPU. При ошибке инициализации драйвера или отсутствии девайса немедленно возвращает `ComputeApiError::DeviceLost` или `VendorError`.

### §4.4. Константы и Магические Числа

Секция пуста. Крейту `compute-cuda` запрещено объявлять свои магические числа для доменной логики. Все константы и лимиты унаследованы из [spec_compute_api.md §4.4] (например, `MAX_EPHYS_TARGETS`) и [spec_layout.md §4.4] (аппаратное выравнивание L2-кэша в 64 байта и `BurstHeads8` в 32 байта).

## §5. Доменная Логика

Реализация абстрактного интерфейса `GpuBackend` на базе платформы NVIDIA CUDA для параллельного выполнения фаз симуляции на графических процессорах NVIDIA.

Выделение CUDA-бэкенда в отдельный крейт Слоя 3 изолирует платформозависимый код (FFI-биндинги CUDA Runtime, управление потоками вычислений, аллокацию видеопамяти через драйвер вендора) от общей логики симулятора и других вычислительных платформ.

Крейт решает задачу высокопроизводительного SIMT-вычисления динамики нейронов и пластичности синапсов. Он распределяет вычисления по потокам варпов NVIDIA Warp с оптимизированным доступом к SoA-структурам в видеопамяти, обеспечивая выполнение горячего цикла симуляции в рамках жесткого временного бюджета.

## §8. Граничные Случаи и Особые Сценарии

### §8.1. Граничные значения

| # | Ситуация | Ожидаемое поведение |
|---|---|---|
| E-048 | **VRAM OOM**: Вызов `cudaMalloc` возвращает `cudaErrorMemoryAllocation`. | См. `[spec_compute_api.md §8.1]`. Бэкенд возвращает `Err(ComputeApiError::OutOfMemory)`. |
| E-049 | **Device Lost / TDR Timeout**: GPU отвалился от PCIe во время работы батча. | См. `[spec_compute_api.md §8.1]`. Бэкенд возвращает `Err(ComputeApiError::DeviceLost)`. |
| E-050 | **Slice Mismatch**: Переданы некорректные слайсы в `DayBatchCmd`. | См. `[spec_compute_api.md §8.1]`. Бэкенд возвращает `Err(ComputeApiError::InvalidLayout)`. |
| E-051 | **Ghost Capacity Exceeded**: Превышение лимита SoA-таблицы связей. | См. `[spec_compute_api.md §8.1]`. Бэкенд возвращает `Err(ComputeApiError::CapacityExceeded)`. |
| E-052 | **Ephys Overflow**: Превышение `MAX_EPHYS_TARGETS` в `EphysCmd`. | См. `[spec_compute_api.md §8.1]`. Бэкенд возвращает `Err(ComputeApiError::InvalidLayout)`. |
| E-053 | **CUDA Compute Capability Mismatch**: Подключен GPU с Compute Capability < 6.1 (Pascal). | Метод `new` возвращает `Err(ComputeApiError::VendorError(cudaErrorNotSupported))`, так как бэкенд аппаратно требует инструкций `__ballot_sync` и `__popc` для телеметрии. |
| E-054 | **CUDA Driver Version Mismatch**: Системный драйвер NVIDIA старее, чем требует скомпилированный рантайм `nvcc`. | Метод `new` возвращает `Err(ComputeApiError::DeviceLost)` или `VendorError(cudaErrorCallRequiresNewerDriver)`. |
| E-055 | **FFI Launch Parameter Error**: Ошибка конфигурации сетки (Grid/Block size) при запуске ядер. | Метод `run_day_batch` перехватывает ошибку `cudaErrorInvalidConfiguration` и возвращает `Err(ComputeApiError::InvalidLayout)`. |
| E-056 | **Pinned RAM Exhaustion**: Сбой `cudaMallocHost` из-за жестких лимитов ОС (например, `ulimit -l` на Linux). | Ошибка выделения Page-Locked памяти для DMA-буферов на хосте. Бэкенд возвращает `Err(ComputeApiError::OutOfMemory)`, явно транслируя сбой RAM (оперативной памяти), а не VRAM. |

### §8.2. Состояния гонки и конкурентность

| # | Сценарий | Защита |
|---|---|---|
| R-015 | **Гонка деинициализации GPU**: ОС уничтожает контекст до вызова `Drop`. | См. `[spec_compute_api.md §8.2]`. Использование явного метода `free()` до завершения процесса. |
| R-016 | **Конкурентный HFT-цикл**: Параллельные запуски для одного `VramHandle`. | См. `[spec_compute_api.md §8.2]`. Маппинг handle на независимый `cudaStream_t`. |
| R-017 | **CUDA Context Thread Bound Race**: Многопоточные вызовы API CUDA без явной привязки контекста девайса. | На входе в каждый FFI-метод бэкенда вызывается `cudaSetDevice(device_id)`, что привязывает контекст GPU к текущему OS-потоку (Thread-Local Context Bind). |
| R-018 | **Asynchronous DMA Use-After-Free**: Мутация слайсов в `DayBatchCmd` после вызова `run_day_batch`, но до завершения GPU-вычислений. | Метод `run_day_batch` ставит `cudaMemcpyAsync` в очередь и мгновенно возвращает управление. C-ABI аппаратно слеп к лайфтаймам Rust. Оркестратор обязан гарантировать, что память хоста не мутирует до вызова глобального барьера `gpu_device_synchronize()`. |

### §8.3. Деградация и Recovery

| # | Отказ | Поведение | Восстановление |
|---|---|---|---|
| D-009 | **Отказ GPU-вычислителя**: Потеря связи с картой. | См. `[spec_compute_api.md §8.3]`. | Fallback на `compute-cpu`. |
| D-010 | **Отказ DMA-транзакции**: Ошибка PCIe. | См. `[spec_compute_api.md §8.3]`. | Retry транзакции, при повторном сбое — переход в `DeviceLost`. |
| D-011 | **CUDA Stream Execution Serialization**: Конкуренция нескольких `cudaStream_t` за SM на одном GPU. | Время выполнения ядер в стримах может деградировать при высокой плотности параллельных запусков. | Планировщик оркестратора балансирует запуски (через BSP Barrier) или разносит шарды по разным GPU-картам. |
| D-012 | **Warp Divergence Penalty (Эпилептический шторм)**: Близко к 100% нейронов шарда спайкуют одновременно. | Ядра `launch_update_neurons` и `launch_apply_gsop` генерируют лавину кэш-промахов L2 из-за хаотичного доступа к дендритам. TPS может упасть в 5-10 раз. | Аппаратное самовосстановление: механизмы гомеостаза (ALIF) за 2-3 тика задерут пороги, и шард вернется в штатный температурный режим. |

## §9. Ошибки

### §9.1. Перечисление ошибок

Крейт не объявляет своих собственных публичных типов ошибок, а транслирует системные сбои FFI в абстрактный `ComputeApiError` из `[spec_compute_api.md §9.1]`.

### §9.2. Стратегия обработки

Бэкенд перехватывает возвращаемые статусы FFI-вызовов и мапит коды `cudaError_t` следующим образом:

| Код CUDA (`cudaError_t`) | Целевая ошибка `ComputeApiError` |
|---|---|
| `cudaErrorMemoryAllocation` | `OutOfMemory` |
| `cudaErrorDeviceUninitialized` / `cudaErrorDeviceLost` | `DeviceLost` |
| `cudaErrorInvalidValue` / `cudaErrorInvalidConfiguration` | `InvalidLayout` |
| `cudaErrorInvalidResourceHandle` | `InvalidHandle` |
| `cudaErrorLaunchOutOfResources` | `InvalidLayout` |
| Любой другой код | `VendorError(code as i32)` |

### §9.3. Паники

| Условие | Почему паника, а не Err |
|---|---|
| Ошибка `cudaFree` внутри деструктора `Drop` | Исключение: для предотвращения двойной паники (double-panic abort) в процессе аварийного завершения работы, любая ошибка освобождения ресурсов в `Drop` перехватывается, пишется в лог и аппаратно гасится. |

## §10. Зависимости и Интеграция

Крейт `compute-cuda` является "листовой" реализацией HAL. Он не предоставляет публичных абстракций, а лишь исполняет контракты нижележащих слоев, оставаясь полностью скрытым за фасадом `compute`.

### §10.1. Что крейт потребляет (inbound)

| Крейт / Инструмент | Что используем | Какой контракт ожидаем |
|---|---|---|
| `compute-api` | `GpuBackend`, `VramHandle`, `DayBatchCmd`, `ComputeApiError` | Крейт обязан реализовать трейт фасада и детерминированно перехватывать все `cudaError_t`, транслируя их в абстрактные ошибки. |
| `layout` | `ShardVramPtrs`, `BurstHeads8`, `VariantParameters` | 100% C-ABI совместимость макетов. Выравнивание указателей по 64 байта и голов аксонов по 32 байта для Coalesced Access. |
| `physics` | `core_math.h` | Подключение общей C++ математики (ALIF, GSOP, Active Tail) через `#include` в CUDA-ядрах для гарантии побитового кроссплатформенного детерминизма. |
| `cc` (build-dep) | Вызов компилятора `nvcc` | Статическая сборка `bindings.cu` и `physics.cu` в архив `libaxicor_cuda.a` на этапе `build.rs` с флагами архитектуры (например, `-arch=sm_61`). |

### §10.2. Кто потребляет крейт (outbound / обратные зависимости)

Ни один крейт верхнего уровня (оркестратор, сеть) не имеет права зависеть от `compute-cuda` напрямую.

| Крейт-потребитель | Что использует | Какой контракт мы обязаны сохранить |
|---|---|---|
| `compute` | `CudaBackend::new(device_id)` | Инициализация контекста драйвера и инкапсуляция бэкенда в `Box<dyn GpuBackend>` (через фича-флаг `cuda`). Никакие сырые указатели не должны вытекать наружу. |
| `test-harness` | `CudaBackend` | Доступ к выгрузке сырой памяти VRAM для дифференциального тестирования. Обязательство побитового соответствия результатов с `compute-cpu` при одинаковом стартовом layout. |

## §12. Бюджеты и Ограничения

### §12.1. Память

| Ресурс | Бюджет | Как считается |
|---|---|---|
| Накладные расходы на VRAM (Overhead) | < 150 MB | Базовое потребление контекста драйвера NVIDIA и структур `cudaStream_t`. |
| Память под один шард | `layout.total_size` | Рассчитывается строго по C-ABI формулам из `[spec_layout.md §6.3]`. |

### §12.2. Латентность

| Операция | Бюджет (p99) | Условия |
|---|---|---|
| Выполнение ядер Day Phase (1 тик) | < 500 µs | 1M нейронов, 128M синапсов, NVIDIA RTX 4090. |
| Асинхронное копирование D2H (Download) | < 100 µs | 1000 выходов, шина PCIe Gen 4x16, использование Pinned RAM. |

### §12.3. Compile-time

| Ограничение | Значение |
|---|---|
| Время статической сборки CUDA-ядер (`nvcc`) | < 30 секунд |
| Полное время сборки крейта с FFI-биндингами | < 45 секунд (в release-профиле) |
