## §1. Идентификация

| Поле | Значение |
|---|---|
| **Имя крейта** | `vfs` |
| **Слой** | Слой 2 — Инфраструктура Файловой Системы и Архивов (Virtual File System & Archives) |
| **Тип** | Library (`lib`) |
| **no_std** | Нет (`false`) — требуются системные вызовы ОС для mmap, работы с файловой системой `std::fs` и путями `std::path` |
| **Описание** | Контейнерный изолятор формата архивов `.axic`, таблицы оглавления (TOC), нормализации путей, чтения в режиме strictly Read-Only через mmap и предлагаемых атомарных примитивов распаковки для движка `AxiEngine`. Крейт обеспечивает безопасный Zero-Copy доступ к байтовым массивам вложенных файлов. Владелец вложенных артефактов определяется соответствующей спецификацией формата; `vfs` хранит и возвращает исключительно сырые байты (`&[u8]`). |

## §2. Стек и Окружение

### §2.1. Внутренние зависимости (inbound)

| Крейт | Что используется | Зачем |
|---|---|---|
| `types` (Слой 0) | Базовые типы данных | Использование общих стандартов типов при обработке путей и идентификаторов. |

### §2.2. Зависимые компоненты и Направление Зависимостей (outbound consumers)

> [!IMPORTANT]
> Крейты `layout` и `config` **не являются** прямыми потребителями `vfs`, не открывают файлы `.axic` и архитектурно не зависят от `vfs`. Оркестратор `boot` и компилятор `baker` извлекают сырые байты `&[u8]` из `vfs` и передают их в `layout` (для C-ABI приведения) и `config` (для парсинга TOML).

| Крейт / Компонент | Роль в системе и взаимодействие |
|---|---|
| `baker` (Слой 4) | Использует предлагаемые низовые примитивы `AxicPacker` (если `AxicPacker` остаётся в `vfs`) для упаковки скомпилированных файлов в `.axic` контейнер. |
| `boot` (Слой 6) | Модуль загрузки открывает `AxicArchive`, извлекает сырые байты `&[u8]` и передает их в `layout` и `config`, а также координирует атомарную экстракцию мутабельных файлов. |

### §2.3. Внешние зависимости

| Crate | Версия | Сфера использования |
|---|---|---|
| `memmap2` | `=0.9.10` | Кроссплатформенный memory-mapped I/O для безопасного проецирования архива в адресное пространство процессов. |

> [!NOTE]
> Зависимость `anyhow = "=1.0.102"` исключена из публичного API библиотеки `vfs`. Внутренние утилиты CLI и упаковщик используют строго типизированные ошибки `VfsError`.

### §2.4. Feature Flags

Секция публичных feature flags не используется. Крейт собирается в единой конфигурации как для этапа сборки архива (`baker`), так и для этапа загрузки рантайма (`boot`).

## §4. Бинарный Формат Архива `.axic` (Archive Container Format)

Архив `.axic` представляет собой монолитный бинарный контейнер только для чтения (ROM-образ). Все числовые поля архива сериализуются строго в формате **Little-Endian (LE)**.

### §4.1. Заголовок Архива (Archive Header)
Заголовок занимает первые 12 байт файла:

| Смещение | Тип | Имя Поля | Значение / Описание |
|---|---|---|---|
| `0..4` | `[u8; 4]` | `magic` | Сигнатура `*b"AXIC"` (`0x43495841` в LE) |
| `4..8` | `u32 LE` | `version` | Версия бинарного формата архива (текущая версия `1`) |
| `8..12` | `u32 LE` | `file_count` | Количество файлов в оглавлении TOC |

> [!NOTE]
> Пустой архив с `file_count == 0` является валидным файлом контейнера. При открытии такого архива метод `list_files()` возвращает пустой итератор, `get_file()` возвращает `None`, а `require_file()` возвращает ошибку `VfsError::FileNotFound`.

### §4.2. Таблица Оглавления (Table of Contents — TOC)
Таблица оглавления начинается ровно со смещения 12 байт (`0x0C`). Каждый элемент TOC имеет фиксированный размер **272 байта**:

| Смещение | Тип | Имя Поля | Описание |
|---|---|---|---|
| `0..256` | `[u8; 256]` | `path` | Нормализованный UTF-8 путь архива с обязательным нуль-терминатором `\0` |
| `256..264` | `u64 LE` | `offset` | Абсолютное смещение полезной нагрузки файла от начала архива |
| `264..272` | `u64 LE` | `size` | Точный размер полезной нагрузки файла в байтах |

### §4.3. Разграничение Путей Хоста и Путей Архива (Host vs Archive Paths)
В системе строго разграничиваются пути операционной системы хоста (Host Paths) и внутренние пути внутри архива (Archive Paths):

1. **Пути Хоста (Host Paths)**: На платформе Windows пути файловой системы могут содержать обратные слэши `\`. При упаковке `AxicPacker` выполняет обязательную нормализацию путей хоста в пути архива.
2. **Пути Архива в TOC (Archive Paths)**: Пути, сохраненные в TOC, **никогда не содержат** символ `\`. При открытии архива или разборе TOC любой путь с символом `\` категорически бракуется с ошибкой `VfsError::InvalidPath`.
3. **Запросы к API (`get_file` / `require_file`)**: Передаваемые пути запросов либо изначально передаются в нормализованном формате путей архива, либо приводятся к нему через явный хелпер `normalize_archive_path`.
4. **Правила Нормализации Путей Архива**:
   - Только валидный UTF-8.
   - Строго разделитель `/`.
   - Запрет абсолютных путей (без ведущего `/` и букв дисков вроде `C:`).
   - Запрет сегментов `..` и `.`.
   - Запрет пустых сегментов (`//`).
   - Полезная нагрузка пути не более 255 байт (обязателен нуль-терминатор `\0` на или до индекса 255).
   - Дубликаты путей архива строго запрещены.

### §4.4. Политика Выравнивания Полезной Нагрузки (Payload Alignment Policy)
Полезная нагрузка каждого файла внутри архива выравнивается паддингом по границе `ARCHIVE_PAYLOAD_ALIGNMENT = 4096` байт.

> [!IMPORTANT]
> Выравнивание по границе 4096 байт фиксируется как правило упаковки полезной нагрузки архива для оптимизации сетевого I/O, DMA и кэш-линий. Оно не гарантирует возможность независимого подфайлового монтирования (sub-file mmap) на всех операционных системах, так как граница выделения памяти (Allocation Granularity) на платформе Windows составляет 64 КБ (65536 байт). При полном монтировании архива `vfs` отображает весь файл целиком и возвращает безопасные срезы `&[u8]`.

## §5. Правила Упаковки и Создания Архива (Packing Rules)

Если предлагаемый низкоуровневый модуль `AxicPacker` остаётся в `vfs`, он гарантирует побитовую воспроизводимость и безопасность при создании файлов `.axic`:

1. **Детерминированный Порядок Файлов**: Перед записью TOC и полезной нагрузки все нормализованные пути архива сортируются в лексикографическом порядке (Lexicographical Sort).
2. **Защита от Коллизий**: Упаковщик бракует сборку при обнаружении совпадающих путей после нормализации (`DuplicatePath`).
3. **Запрет Неявного Усечения**: Пути длиной более 255 байт вызывают ошибку `PathTooLong`, а не усекаются молча.
4. **Защита от Само-включения (Self-Inclusion Guard)**: Упаковщик проверяет, чтобы целевой выходной файл `.axic` не находился внутри упаковываемой директории.
5. **Заполнение Нулями (Zeroed Padding)**: Все байты выравнивающего паддинга забиваются нулями (`0x00`).
6. **Атомарная Запись (Atomic Output)**: Временный файл архива записывается в той же директории, синхронизируется на диск (`flush`/`sync_all`), после чего атомарно переименовывается в целевое имя.

## §6. Безопасный Read API (Read Architecture)

Класс `AxicArchive` предоставляет только безопасный, иммутабельный и потокобезопасный доступ к данным:

```rust
pub struct AxicArchive {
    // Внутреннее mmap отображение и распарсенная таблица TOC
}
```

### §6.1. Методы Доступа
- **`AxicArchive::open(path: impl AsRef<Path>) -> Result<AxicArchive, VfsError>`**: Открывает файл, создает mmap-отображение и выполняет полную валидацию заголовка и TOC.
- **`get_file(&self, path: &str) -> Option<&[u8]>`**: Принимает путь архива (или автоматически нормализует через `normalize_archive_path`) и возвращает срез байт для опциональных артефактов. Для пустых файлов (`size == 0`) возвращает `Some(&[])`. При отсутствии файла или пустом архиве возвращает `None`.
- **`require_file(&self, path: &str) -> Result<&[u8], VfsError>`**: Возвращает срез байт или ошибку `VfsError::FileNotFound` для обязательных загрузочных файлов.
- **`contains(&self, path: &str) -> bool`**: Проверяет наличие файла в оглавлении.
- **`list_files(&self) -> impl Iterator<Item = &str>`**: Итератор по нормализованным путям архива для диагностики и обнаружения артефактов AxiCAD. На пустом архиве возвращает пустой итератор.

### §6.2. Потокобезопасность (`Send + Sync`)
Тип `AxicArchive` реализует маркерные трейты `Send` и `Sync`. Данная гарантия валидна исключительно благодаря тому, что mmap-отображение является строго Read-Only (`PROT_READ`), а таблица TOC иммутабельна после открытия.

## §7. Механика Безопасной Мутабельной Экстракции (Mutable Extraction)

Артефакты внутри `.axic` находятся в режиме строгого ROM (Read-Only). Для файлов, требующих мутации в процессе Ночной Фазы (`.state`, `.axons`, `.paths`), крейт `vfs` предоставляет атомарные хелперы извлечения.

### §7.1. Правила Безопасной Экстракции
1. **Изоляция Директории**: Запуск процедуры экстракции принимается от вышележащего слоя (`boot`), который задает целевую кэш-директорию.
2. **Защита от Выхода из Коры (Path Traversal Protection)**: При извлечении любого файла `vfs` проверяет, что итоговый абсолютный путь строго находится внутри целевой директории экстракции (`ExtractOutsideRoot`).
3. **Атомарность файлов**: Каждый файл извлекается во временный файл с последующим переименованием (`AtomicRenameFailed`), исключая чтение частично записанных файлов со стороны процессов симулятора.

> [!CAUTION]
> Экспорт файлов манифестов в разделяемую память (SHM) или временные файлы не входит в зону ответственности `vfs`. Крейт `vfs` предоставляет только сырые байты файла, а управление экспортом в SHM принадлежит слоям `boot` или `ipc`.

## §8. Иерархия Ошибок (`VfsError`)

Публичный API крейта `vfs` использует строго типизированное перечисление ошибок без внешних оберток:

```rust
#[derive(Debug)]
pub enum VfsError {
    InvalidMagic,
    UnsupportedVersion,
    HeaderTooSmall,
    TocOutOfBounds,
    TocCountOverflow,
    EntryOutOfBounds,
    OffsetOverflow,
    AlignmentViolation,
    PathTooLong,
    PathNotUtf8,
    PathNotTerminated,
    InvalidPath,
    DuplicatePath,
    FileNotFound,
    Io(std::io::Error),
    MapFailed,
    ExtractOutsideRoot,
    AtomicRenameFailed,
}
```

## §9. Требуемые Инварианты

- **INV-VFS-001**: Первые 4 байта архива равны `*b"AXIC"`.
- **INV-VFS-002**: Все смещения полезной нагрузки выровнены по границе `ARCHIVE_PAYLOAD_ALIGNMENT` (4096 байт).
- **INV-VFS-003**: Отображение mmap открывается строго в режиме Read-Only.
- **INV-VFS-004**: Все пути в TOC хранятся в нормализованном UTF-8 формате без символов `\` с обязательным нуль-терминатором в пределах 256 байт.
- **INV-VFS-005**: Запрещены пути с символами `..`, ведущими слэшами или буквами дисков.
- **INV-VFS-006**: Порядок записей в TOC при упаковке детерминирован и отсортирован лексикографически.
- **INV-VFS-007**: Экстракция файлов за пределы целевой рабочей директории аппаратно прерывается с ошибкой `ExtractOutsideRoot`.
- **INV-VFS-008**: Отсутствие любых динамических трейтов (`dyn Trait`) во внутреннем API доступа.

## §10. Golden Tests / Обязательная Матрица Тестирования

Крейт `vfs` обязан быть покрыт набором автоматических юнит-тестов:

1. **Открытие Валидного Архива (`test_valid_archive_open`)**: Проверка чтения байтов из созданного тестового архива.
2. **Браковка Невалидного Magic и Версии (`test_reject_bad_magic_and_version`)**: Проверка возврата `InvalidMagic` и `UnsupportedVersion`.
3. **Защита от Усеченного Заголовка и TOC (`test_reject_truncated_header`)**: Проверка возврата ошибок при коротком файле.
4. **Проверка Выхода Границ TOC (`test_reject_toc_out_of_bounds`)**: Браковка смещений `offset + size`, выходящих за границы mmap.
5. **Проверка Переполнения Целых Чисел (`test_reject_integer_overflow`)**: Проверка устойчивости к переполнению при `offset + size`.
6. **Браковка Дубликатов Путей (`test_reject_duplicate_normalized_paths`)**: Проверка выявления дубликатов при упаковке.
7. **Защита от Path Traversal (`test_reject_path_traversal`)**: Проверка браковки путей с `../secret`.
8. **Проверка Отсутствия Нуль-Терминатора (`test_reject_non_terminated_path`)**: Браковка путей длиной 256 байт без `\0`.
9. **Проверка Не-UTF8 Путей и Символа `\` (`test_reject_invalid_archive_paths`)**: Браковка бинарного мусора и обратных слэшей в именах файлов TOC.
10. **Обработка Пустых Файлов (`test_zero_size_file_returns_empty_slice`)**: Подтверждение возврата `&[]` для файлов размером 0 байт.
11. **Фиксация Поведения Пустого Архива (`test_empty_archive_handling`)**: Явный тест обработки архива с `file_count == 0` (возврат пустых итераторов и `FileNotFound`).
12. **Детерминизм Упаковки (`test_packer_deterministic_output`)**: Проверка совпадения хэша выходящего архива независимо от порядка чтения ФС.
13. **Проверка Выравнивания Смещений (`test_payload_offset_alignment`)**: Проверка кратности всех смещений 4096 байтам.
14. **Защита от Само-включения (`test_packer_rejects_self_inclusion`)**: Проверка блокировки упаковки целевого `.axic` в самого себя.
15. **Безопасность Экстракции (`test_extraction_directory_containment`)**: Гарантия невозможности записи за пределы рабочей директории.
16. **Атомарность Записи при Экстракции (`test_atomic_extraction_semantics`)**: Проверка создания временных файлов и их переименования.

## §11. Open Questions / Review Debt (Открытые Вопросы и Противоречия)

В процессе анализа спецификации VFS выявлены следующие открытые вопросы для согласования:

1. **Версионирование Бинарного Формата `.axic`**:
   - *Контекст*: В текущей спецификации зафиксирована версия `1`.
   - *Вопрос*: Сохраняется ли версия `1` для первого Rust-переписывания, или требуется инкремент версии?

2. **Объем Отображения mmap (Whole Archive vs Sub-File Views)**:
   - *Контекст*: Сейчас `vfs` отображает весь архив целиком и возвращает срезы `&[u8]`.
   - *Вопрос*: Требуется ли поддержка независимых подфайловых отображений (Sub-File Mmap Views) для каждого файла отдельно?

3. **Адекватность Выравнивания для Платформы Windows**:
   - *Контекст*: Размер `ARCHIVE_PAYLOAD_ALIGNMENT` равен 4096 байтам, но граница выделения памяти (Allocation Granularity) на Windows равна 64 КБ (65536 байт).
   - *Вопрос*: Требуется ли увеличить выравнивание полезной нагрузки до 64 КБ для поддержки независимого sub-file mmap на Windows?

4. **Контрольные Суммы Элементов TOC (SHA256 / CRC32)**:
   - *Контекст*: Сейчас TOC хранит только смещение и размер.
   - *Вопрос*: Требуется ли добавить контрольные суммы (CRC32 или SHA256) для каждого файла в TOC для верификации целостности реестра артефактов?

5. **Политика Сжатия Файлов в Архиве**:
   - *Контекст*: Сжатие файлов делает невозможным прямой Zero-Copy access через mmap без декомпрессии в кучу.
   - *Вопрос*: Запрещается ли сжатие окончательно, или допускается опциональный алгоритм (например, zstd) для некритичных артефактов?

6. **Размещение Низкоуровневого Упаковщика (`AxicPacker`)**:
