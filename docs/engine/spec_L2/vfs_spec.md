| **Слой** | Слой 2 — Инфраструктура Файловой Системы и Архивов (Virtual File System & Archives) |
| **Тип** | Library (`lib`) |

## §1.1. Scope of Stage A (Границы реализации Stage A)

### Входит в Stage A:
1. **Открытие архива в режиме Read-Only**: `AxicArchive::open(path)` на базе проецирования всего файла архива целиком через mmap (из библиотеки `memmap2`).
2. **Публичный интерфейс чтения**: функции `get_file`, `require_file`, `contains`, `list_files`.
3. **Упаковщик низкого уровня**: `AxicPacker` / `pack_entries` для компоновки набора файлов `(archive_path, bytes)` в монолитный файл `.axic`.
4. **Лексикографическая сортировка TOC**: детерминированный порядок записей для побитовой воспроизводимости.
5. **Валидация целостности**: проверка сигнатуры magic, версии бинарного формата, размеров заголовка, TOC, границ полезной нагрузки и соответствия правилам путей (Path Rules).
6. **Zero-Copy доступ**: возврат ссылок на сырые байты (`&[u8]`) файлов без парсинга их содержимого (без знания внутренних структур `.state`, `.axons`, `.paths`).
7. **Поддержка пустых файлов**: файлы нулевого размера являются валидными и возвращают `&[]`.

### НЕ входит в Stage A (Отложено):
1. **Мутабельная экстракция**: распаковка содержимого архива во внешнюю рабочую директорию хоста (Mutable Extraction) и защита от выхода за пределы директории.
2. **Сжатие данных**: компрессия/декомпрессия файлов внутри контейнера запрещена.
3. **Криптографическая проверка**: расчет и сверка контрольных сумм файлов (SHA256, CRC32, etc.) и проверка целостности TOC.
4. **Независимое отображение файлов (sub-file mmap)**: пофайловое отображение в память VRAM/RAM с выравниванием по границе 64 КБ (актуально для Windows).
5. **Экспорт в SHM / Temp**: логика развертывания манифестов в разделяемую память хоста (SHM) или временные директории.
6. **Обход хостовой ФС (Directory Walking)**: сборка путей с хоста на упаковщике.
7. **Рантайм-политики загрузки**: проверка состава необходимых файлов для загрузки шарда (Required Boot Policy).

## §4. Бинарный Формат Архива `.axic` (Archive Container Format)

Архив `.axic` представляет собой монолитный бинарный контейнер только для чтения (ROM-образ). Все числовые поля архива сериализуются строго в формате **Little-Endian (LE)**.

### §4.1. Заголовок Архива (Archive Header)
Заголовок занимает первые 12 байт файла:

| Смещение | Тип | Имя Поля | Значение / Описание |
|---|---|---|---|
| `0..4` | `[u8; 4]` | `magic` | Сигнатура `*b"AXIC"` (`0x43495841` в LE) |
| `4..8` | `u32 LE` | `version` | Версия бинарного формата архива (текущая версия `1`) |
| `8..12` | `u32 LE` | `file_count` | Количество файлов в оглавлении TOC |

TOC (Table of Contents) начинается ровно со смещения 12 байт (`0x0C`).

> [!NOTE]
> Пустой архив с `file_count == 0` является валидным файлом контейнера. При открытии такого архива метод `list_files()` возвращает пустой итератор, `get_file()` возвращает `None`, а `require_file()` возвращает ошибку `VfsError::FileNotFound`.

### §4.2. Таблица Оглавления (Table of Contents — TOC)
Каждый элемент TOC имеет фиксированный размер **272 байта** и начинается сразу за заголовком (смещение 12 байт):

| Смещение | Тип | Имя Поля | Описание |
|---|---|---|---|
| `0..256` | `[u8; 256]` | `path` | Нормализованный UTF-8 путь архива с обязательным нуль-терминатором `\0` |
| `256..264` | `u64 LE` | `offset` | Абсолютное смещение полезной нагрузки файла от начала архива |
| `264..272` | `u64 LE` | `size` | Точный размер полезной нагрузки файла в байтах |

### §4.3. Разграничение Путей Хоста и Путей Архива (Host vs Archive Paths)
В системе строго разграничиваются пути операционной системы хоста (Host Paths) и внутренние пути внутри архива (Archive Paths):

1. **Пути Архива в TOC (Archive Paths)**: Пути, сохраненные в TOC, **никогда не содержат** символ `\`. При открытии архива или разборе TOC любой путь с символом `\` категорически бракуется с ошибкой `VfsError::InvalidPath`.
2. **Правила Путей Архива**:
   - Только валидный UTF-8.
   - Строго разделитель `/`.
   - Запрет абсолютных путей (без ведущего `/` и букв дисков вроде `C:`).
   - Запрет сегментов `..` и `.`.
   - Запрет пустых сегментов (`//`).
   - Полезная нагрузка пути не более 255 байт (обязателен нуль-терминатор `\0` на или до индекса 255).
   - Дубликаты путей архива строго запрещены.
3. **Запросы к API**: Принимаются только пути, соответствующие правилам путей архива. Несоответствующие пути возвращают `None` (или ошибку).

### §4.4. Политика Выравнивания Полезной Нагрузки (Payload Alignment Policy)
- Полезная нагрузка первого файла начинается со смещения `align_up(12 + file_count * 272, 4096)`.
- Смещение каждого последующего файла в полезной нагрузке должно быть выровнено по границе `ARCHIVE_PAYLOAD_ALIGNMENT = 4096` байт относительно начала файла архива.
- Все выравнивающие байты (паддинг) обязаны быть заполнены нулями (`0x00`) со стороны упаковщика.

> [!IMPORTANT]
> Выравнивание по границе 4096 байт фиксируется как правило упаковки полезной нагрузки архива для оптимизации сетевого I/O, DMA и кэш-линий. Оно не гарантирует возможность независимого подфайлового монтирования (sub-file mmap) на всех операционных системах, так как граница выделения памяти (Allocation Granularity) на платформе Windows составляет 64 КБ (65536 байт). При полном монтировании архива `vfs` отображает весь файл целиком и возвращает безопасные срезы `&[u8]`.

## §5. Правила Упаковки и Создания Архива (Packing Rules & API)

Модуль `AxicPacker` в крейте `vfs` предоставляет низкоуровневые API для сборки `.axic` файлов с детерминированным лексикографическим порядком оглавления и выравниванием полезной нагрузки. Stage A упаковщик принимает уже логические archive paths, а не host paths.

### Спецификация API Упаковщика:

```rust
pub struct ArchiveEntry<'a> {
    pub path: &'a str,
    pub bytes: &'a [u8],
}

pub fn pack_entries(entries: &[ArchiveEntry]) -> Result<Vec<u8>, VfsError>;
```

#### Правила упаковки:
1. **Детерминированный Порядок**: Входные записи перед сборкой сортируются лексикографически по их нормализованному пути архива.
2. **Проверка путей**: Каждый путь проверяется на соответствие правилам (см. §4.3). При нарушении возвращается `VfsError::InvalidPath` или `VfsError::PathTooLong`.
3. **Защита от дубликатов**: При обнаружении путей с одинаковыми именами возвращается `VfsError::DuplicatePath`.
4. **Выравнивание полезной нагрузки**: Адрес каждого файла выравнивается по границе 4096 байт. Паддинг заполняется нулями `0x00`.
5. **Пустые файлы**: Разрешены и упаковываются с размером `size = 0`. Смещение (`offset`) пустого файла выравнивается аналогично другим файлам.

## §6. Безопасный Read API (Read Architecture)

Класс `AxicArchive` предоставляет только безопасный, иммутабельный и потокобезопасный доступ к данным:

```rust
pub struct AxicArchive {
    // Внутреннее mmap отображение файла архива и разобранная таблица TOC
}

impl AxicArchive {
    /// Открывает файл архива и проецирует его в память целиком в режиме Read-Only.
    /// Производит валидацию заголовка, TOC и границ.
    pub fn open(path: impl AsRef<std::path::Path>) -> Result<Self, VfsError>;

    /// Возвращает срез байт для файла. Если файл пустой (size == 0), возвращает Some(&[]).
    /// Если файл отсутствует, возвращает None.
    pub fn get_file(&self, path: &str) -> Option<&[u8]>;

    /// То же, что и `get_file`, но возвращает VfsError::FileNotFound в случае отсутствия.
    pub fn require_file(&self, path: &str) -> Result<&[u8], VfsError>;

    /// Возвращает true, если файл присутствует в оглавлении.
    pub fn contains(&self, path: &str) -> bool;

    /// Возвращает итератор по нормализованным путям файлов TOC в архиве.
    pub fn list_files(&self) -> impl Iterator<Item = &str>;
}
```

### §6.1. Потокобезопасность (`Send + Sync`)
Тип `AxicArchive` реализует маркерные трейты `Send` и `Sync`. Данная гарантия валидна исключительно благодаря тому, что mmap-отображение является строго Read-Only (`PROT_READ`), а таблица TOC иммутабельна после открытия.

## §8. Иерархия Ошибок (`VfsError`)

Публичный API крейта `vfs` использует строго типизированное перечисление ошибок без внешних оберток:

```rust
#[derive(Debug)]
pub enum VfsError {
    InvalidMagic,
    UnsupportedVersion,
    HeaderTooSmall,
    TocOutOfBounds,
    TocCountOverflow,
    EntryOutOfBounds,
    OffsetOverflow,
    AlignmentViolation,
    PathTooLong,
    PathNotUtf8,
    PathNotTerminated,
    InvalidPath,
    DuplicatePath,
    FileNotFound,
    Io(std::io::Error),
    MapFailed,
}
```

## §9. Требуемые Инварианты

- **INV-VFS-001**: Первые 4 байта архива равны `*b"AXIC"`.
- **INV-VFS-002**: Все смещения полезной нагрузки выровнены по границе `ARCHIVE_PAYLOAD_ALIGNMENT` (4096 байт).
- **INV-VFS-003**: Отображение mmap открывается строго в режиме Read-Only.
- **INV-VFS-004**: Все пути в TOC хранятся в нормализованном UTF-8 формате без символов `\` с обязательным нуль-терминатором в пределах 256 байт.
- **INV-VFS-005**: Запрещены пути с символами `..`, `.`, ведущими слэшами или буквами дисков.
- **INV-VFS-006**: Порядок записей в TOC при упаковке детерминирован и отсортирован лексикографически.
- **INV-VFS-007**: Отсутствие любых динамических трейтов (`dyn Trait`) во внутреннем API доступа.

## §10. Golden Tests / Обязательная Матрица Тестирования

Крейт `vfs` обязан быть покрыт набором автоматических юнит-тестов:

1. **Открытие Валидного Архива (`test_valid_archive_open`)**: Проверка чтения байтов из созданного тестового архива.
2. **Браковка Невалидного Magic и Версии (`test_reject_bad_magic_and_version`)**: Проверка возврата `InvalidMagic` и `UnsupportedVersion`.
3. **Защита от Усеченного Заголовка и TOC (`test_reject_truncated_header`)**: Проверка возврата ошибок `HeaderTooSmall`/`TocOutOfBounds` при коротком файле.
4. **Проверка Выхода Границ TOC (`test_reject_toc_out_of_bounds`)**: Браковка смещений `offset + size`, выходящих за границы mmap.
5. **Проверка Переполнения Целых Чисел (`test_reject_integer_overflow`)**: Проверка устойчивости к переполнению при `offset + size` (`OffsetOverflow`).
6. **Браковка Дубликатов Путей (`test_reject_duplicate_normalized_paths`)**: Проверка выявления дубликатов при упаковке (`DuplicatePath`).
7. **Защита от Path Traversal (`test_reject_path_traversal`)**: Проверка браковки путей с `../secret` или `./`.
8. **Проверка Отсутствия Нуль-Терминатора (`test_reject_non_terminated_path`)**: Браковка путей длиной 256 байт без `\0` (`PathNotTerminated`).
9. **Проверка Не-UTF8 Путей и Символа `\` (`test_reject_invalid_archive_paths`)**: Браковка бинарного мусора и обратных слэшей в именах файлов TOC (`InvalidPath` / `PathNotUtf8`).
10. **Обработка Пустых Файлов (`test_zero_size_file_returns_empty_slice`)**: Подтверждение возврата `&[]` для файлов размером 0 байт.
11. **Фиксация Поведения Пустого Архива (`test_empty_archive_handling`)**: Явный тест обработки архива с `file_count == 0` (возврат пустых итераторов и `FileNotFound`).
12. **Детерминизм Упаковки (`test_packer_deterministic_output`)**: Проверка совпадения хэша выходящего архива независимо от порядка передачи записей.
13. **Проверка Выравнивания Смещений (`test_payload_offset_alignment`)**: Проверка кратности всех смещений 4096 байтам.
14. **Проверка реализации Send/Sync (`test_archive_send_sync`)**: Проверка статического контракта потокобезопасности `AxicArchive`.

## §11. Resolved / Deferred Review Debt (Разрешенные и Отложенные Вопросы)

В процессе анализа спецификации VFS приняты следующие архитектурные решения по открытым вопросам в рамках Stage A:

1. **Версионирование Бинарного Формата `.axic`**:
   - *Решение*: Сохраняется версия бинарного формата `.axic` version = 1.
2. **Объем Отображения mmap (Whole Archive vs Sub-File Views)**:
   - *Решение*: В Stage A отображается только архив целиком через read-only mmap.
3. **Адекватность Выравнивания для Платформы Windows**:
   - *Решение*: Выравнивание полезной нагрузки фиксируется на уровне 4096 байт для Stage A. Подфайловое независимое отображение (sub-file mmap) на 64 КБ отложено.
4. **Контрольные Суммы Элементов TOC (SHA256 / CRC32)**:
   - *Решение*: Отложено. Контрольные суммы и криптографическая проверка не входят в Stage A.
5. **Политика Сжатия Файлов в Архиве**:
   - *Решение*: Сжатие строго запрещено для Stage A для обеспечения прямого Zero-Copy доступа.
6. **Размещение Низкоуровневого Упаковщика (`AxicPacker`)**:
