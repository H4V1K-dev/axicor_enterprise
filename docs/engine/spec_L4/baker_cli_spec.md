## §1. Идентификация

| Поле | Значение |
|---|---|
| **Имя крейта** | `baker-cli` |
| **Имя бинарного файла** | `axi-baker` |
| **Слой** | Слой 4 — Geometry, Growth & Connectome Generation (`L4`) |
| **Тип** | Executable Binary (`bin`) |
| **no_std** | Нет (`false`) — требуется парсинг CLI аргументов и I/O |
| **Описание** | Консольный интерфейс сборки локального single-shard архива `.axic` из конфигурационного файла TOML с помощью библиотек `config` и `baker` (Stage A). |

---

## §2. Стек и Окружение

### §2.1. Внутренние зависимости (inbound)

| Крейт | Что используется | Зачем |
|---|---|---|
| `config` (Слой 1) | Схемы конфигурации TOML, валидация | Чтение и валидация TOML конфигурации шарда |
| `baker` (Слой 4) | `bake_local_shard_axic`, `BakerError` | Генерация и упаковка локальных артефактов в `.axic` |

### §2.2. Внешние Зависимости

| Crate | Версия | Сфера использования |
|---|---|---|
| `clap` | `=4.5.4` | Парсинг CLI-аргументов, флагов и генерация справки |
| `serde` | `=1.0.228` | Сериализация JSON-отчетов |
| `serde_json` | `=1.0.117` | Форматирование JSON-вывода |
| `tempfile` | `=3.10.1` | Атомарная и безопасная запись выходного файла |

> [!IMPORTANT]
> Прямые зависимости от вычислительных бэкендов (`compute`, `compute-api`), рантайма (`boot`, `runtime`) и межпроцессного взаимодействия (`ipc`, `net`) категорически запрещены.

## §3. Ownership Boundaries (Границы Владения)

- `baker-cli` отвечает исключительно за консольный интерфейс (Exit Codes, I/O потоки, парсинг аргументов, атомарную запись результирующего файла).
- `baker-cli` не имеет права самостоятельно парсить внутренности TOML (делегируется `config`) или упаковывать `.axic` байты (делегируется `baker` / `vfs`).

---

## §4. Грамматика Командной Строки Stage A

Подкоманда сборки локального шарда:
```text
axi-baker bake-local --shard <PATH> --out <PATH> --seed <U64> --voxel-size-um <FLOAT> [--force] [--json]
```

### §4.1. Параметры командной строки:
- `--shard <PATH>`: Обязательный путь к конфигурационному TOML-файлу шарда.
- `--out <PATH>`: Обязательный путь к создаваемому бинарному файлу `.axic`.
- `--seed <U64>`: Обязательное зерно псевдослучайного генератора (MasterSeed).
- `--voxel-size-um <FLOAT>`: Обязательный физический размер вокселя в микрометрах.
- `--force`: Разрешить перезапись файла по пути `--out` (без этого флага при наличии существующего файла сборка должна прерываться с ошибкой).
- `--json`: Вывод результатов сборки в формате JSON в поток `stdout` (без этого флага `stdout` остается пустым, а человекочитаемые отчеты идут в `stderr`).

---

## §5. Политика Потоков Вывода и Атомарности

1. **Разделение stdout/stderr**:
   - `stdout` зарезервирован строго под машиночитаемый вывод. При `--json` туда выводится ровно один валидный JSON-объект с отчетом о сборке. При обычном запуске `stdout` пуст.
   - `stderr` используется для вывода человекочитаемых логов, прогресса и ошибок.
2. **Атомарная Замена Выходного Файла**:
   - Запись результирующего `.axic` файла производится во временный файл в той же директории, что и целевой `--out` (например, `<out_filename>.tmp`), и только после успешной упаковки и закрытия файла производится атомарная замена (rename) исходного файла.
   - Если сборка упала с ошибкой, существующий файл по пути `--out` не изменяется и не затирается.

## §6. Коды Возврата ОС (Exit Codes)

| Код | Семантика | Описание |
|---|---|---|
| **0** | Success | Сборка успешно завершена, `.axic` сохранен. |
| **1** | Validation/Config Error | Ошибка парсинга или валидации TOML-конфигурации. |
| **2** | CLI Argument Error | Ошибка параметров командной строки (неверные флаги). |
| **3** | I/O Error | Ошибка файловой системы (запись, права, перезапись без `--force`). |
| **4** | Internal Error | Внутренний сбой компилятора или паника. |

## §7. Out of Scope (Вне рамок Stage A)

- Генерация sidecar-манифестов (`manifest.toml`).
- Потоковый JSON-lines прогресс (`--json-lines`).
- Подкоманды `inspect`, `meta`, `edge`.
- Интегрированная обработка сигналов прерывания (SIGINT / Ctrl+C).
- Directory walking для пакетной сборки.
- Мультишардовый layout.

---

## §8. Тестовая Матрица Спеки Stage A

1. **Справка CLI**: Вызов с `--help` возвращает 0 и выводит usage.
2. **Пропущенные Аргументы**: Отсутствие обязательных параметров возвращает код 2.
3. **Невалидный TOML**: Передача синтаксически неверного или логически невалидного TOML возвращает код 1.
4. **Успешная Сборка**: Передача корректных параметров генерирует `.axic` файл и возвращает код 0.
5. **Политика Перезаписи без `--force`**: Если файл `--out` уже существует, без флага `--force` сборка возвращает код 3.
6. **Политика Перезаписи с `--force`**: При наличии существующего файла и флага `--force` происходит перезапись с кодом 0.
7. **JSON Режим**: При флаге `--json` в `stdout` пишется валидный JSON-объект результатов.
8. **Читаемость Результата**: Сгенерированный `.axic` архив успешно проходит валидацию и парсинг в `vfs` / `boot`.
9. **Неизменяемость Источников**: Входной TOML-файл конфигурации не подвергается никаким изменениям.
10. **Контроль Зависимостей**: Статическая проверка отсутствия зависимостей от `compute`, `boot`, `runtime`, `node`, `ipc`, `net`.
