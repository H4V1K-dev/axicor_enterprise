## §1. Идентификация

| Поле | Значение |
|---|---|
| **Имя крейта** | `baker-cli` |
| **Слой** | Слой 4 — Geometry, Growth & Connectome Generation (`L4`) |
| **Тип** | Executable Binary (`bin`) |
| **no_std** | Нет (`false`) — требуется обработка аргументов командной строки, взаимодействие со стандартными потоками I/O (`stdout`/`stderr`), обработка сигналов ОС и системный аллокатор |
| **Описание** | Консольный интерфейс и sidecar-точка входа для запуска AOT-компилятора `baker` и опциональной оптимизации `edge-model`. Крейт отвечает за грамматику CLI-аргументов, разделение человекочитаемых и машиночитаемых потоков вывода, коды возврата ОС (Exit Codes), формат протокола событийной sidecar-интеграции для AxiCAD и безопасное формирование структур `baker::BakeRequest`. Крейт не владеет алгоритмами компиляции, геометрией, C-ABI макетами, TOML-схемами, форматом контейнера `.axic` или алгоритмами edge-дистилляции. |

Использование `anyhow` разрешено только в бинарном крейте `baker-cli` и не протаскивается в библиотеки.

## §4. Грамматика Командной Строки и Подкоманды (CLI Subcommands v2)

Крейт предоставляется в виде консольного бинарного файла с подкомандами:

### §4.1. Подкоманда `bake`
Основная команда для компиляции нейроморфного графа:
- `--project-root <PATH>`: Путь к корневому каталогу проекта с диска (взаимоисключающе с `--snapshot-bundle`).
- `--snapshot-bundle <PATH>`: Путь к временному snapshot-бандлу AxiCAD (взаимоисключающе с `--project-root`).
- `--out <PATH>`: Обязательный путь к создаваемому файлу `.axic`.
- `--staging-dir <PATH>`: Опциональный путь к временному каталогу стадийной сборки.
- `--target-profile <PROFILE>`: Профиль сборки (`desktop`, `edge`, `debug`).
- `--report-json <PATH>`: Путь для сохранения итогового машиночитаемого отчета `BakeReport`.
- `--json`: Режим монолитного вывода итогового машиночитаемого JSON-результата в `stdout` (взаимоисключающе с `--json-lines`).
- `--json-lines`: Режим потоковой передачи событий sidecar поочередными однострочными JSON-объектами в `stdout` (взаимоисключающе с `--json`).
- `--job-id <ID>` / `--request-id <ID>`: Идентификатор задачи для тэгирования событий в sidecar-режиме (если не передан, CLI генерирует локальный UUID только для событий, без попадания в байты архива).
- `--snapshot-id <ID>`: Принимает идентификатор снапшота (передается строго в метаданные отчета, но исключается из байтов архива).
- `--force`: Обязательный флаг при перезаписи уже существующего целевого файла `.axic`.

### §4.2. Подкоманда `inspect`
Инспектирование существующего архива `.axic` или манифеста артефактов:
- Читает структуру контейнера и выводит список встроенных файлов, размеры и побитовые хэши SHA-256 (работает строго через инспекционные хелперы библиотеки `baker` без прямой зависимости от `vfs`).

### §4.3. Подкоманда `meta`
Вывод метаданных компилятора для утилит AxiCAD Bridge:
- Печатает версию движка, поддерживаемые версии протоколов, форматы отчетов и список команд. Поддерживает флаг `--meta-json`.

### §4.4. Подкоманда `edge` / `edge-convert`
Опциональная подкоманда делегирования оптимизации графа под встраиваемые платформы (активна строго при вызове `--features edge`):
- Вызывает функции библиотеки `edge-model` без реализации собственной алгоритмической логики дистилляции.

## §5. Политика Потоков Вывода и Обработки Ошибок (stdout / stderr Policy)

1. **Человекочитаемые Логи в `stderr`**: Все текстовые сообщения прогресса, информационные строки и логи системы `tracing` направляются строго в поток `stderr`.
2. **Машиночитаемый JSON в `stdout`**: Поток `stdout` зарезервирован строго под машиночитаемые данные. При флаге `--json` в `stdout` печатается ровно один валидный монолитный JSON-документ результата. Флаги `--json` и `--json-lines` сделаны строго взаимоисключающими.
3. **Строгий Sidecar в `--json-lines`**: В режиме `--json-lines` в поток `stdout` выводятся строго валидные однострочные JSON-объекты событий. Внедрение неструктурированного текста в `stdout` категорически запрещено.
4. **Политика Ошибок Парсинга Clap**: Если ошибка парсинга аргументов возникает после распознавания флагов `--json` или `--json-lines`, CLI пытается вывести структурированную ошибку в `stdout`. Ранние ошибки парсинга (до распознавания формата вывода) выводятся в человекочитаемом виде в `stderr` с кодом возврата 2.
5. **Безопасный Panic Hook**: В машиночитаемых режимах (`--json` / `--json-lines`) CLI устанавливает кастомный panic hook и предпринимает попытку вывести событие `failed` или структурированный JSON-объект ошибки до выхода. Необработанная паника считается внутренним сбоем с кодом возврата 4.

## §6. Протокол Событий Sidecar (`JSON-lines Event Model`)

В режиме `--json-lines` утилита эмитирует строго типизированные события в `stdout`. Матрица обязательных типов событий (`event`):
- `started`: старт компилятора;
- `progress`: процент выполнения фазы;
- `log`: текстовые логи с уровнями серьезности;
- `diagnostic`: отфильтрованные и структурированные ошибки/предупреждения;
- `artifact-ready`: уведомление о готовности конкретного бинарного блоба;
- `completed`: успешное завершение всей сборки с указанием `out_path`;
- `failed`: фатальный сбой или останов по ошибке валидации;
- `cancelled`: контролируемый останов процесса по сигналу отмены.

Все события содержат обязательный общий минимум полей: `event`, `job_id`, `timestamp` (опциональное/волатильное строковое поле времени для UI, не влияющее на байты архива), `phase` (опционально) и `message` (опционально).

```json
{"event":"started","job_id":"job_123","timestamp":"2026-06-29T03:30:00Z","phase":"BuildContext","message":"Compilation started"}
{"event":"progress","job_id":"job_123","phase":"PerShardTopology","percent":45.0}
{"event":"diagnostic","job_id":"job_123","phase":"ParseAndValidate","code":"E_LAYER_CAPACITY","severity":"error","message":"Layer capacity exceeded","source_ref":"shard_0.toml:L12"}
{"event":"artifact-ready","job_id":"job_123","artifact_name":"state_blob","hash":"a1b2...","byte_len":1048576}
{"event":"completed","job_id":"job_123","out_path":"/tmp/model.axic","status":"success"}
```

Категорически запрещено внедрение бинарных дампов в JSON-события. Передаются строго пути, хэши, размеры и строковые метаданные.

## §7. Коды Возврата Операционной Системы (Exit Codes)

Утилита завершает процесс со следующими кодами возврата:

| Код | Семантика | Описание |
|---|---|---|
| **0** | Success | Компиляция или операция успешно завершена. |
| **1** | Validation Error | Ошибка пользователя, валидации TOML или структуры графа. |
| **2** | CLI Argument Error | Ошибка параметров командной строки (неверные флаги). |
| **3** | I/O / Storage Error | Ошибка файловой системы, отсутствия прав или записи `.axic`. |
| **4** | Internal Engine Error | Неожиданный внутренний сбой компилятора или паника. |
| **130** | Controlled Cancellation | Контролируемый останов компиляции по сигналу отмены от пользователя (SIGINT / Ctrl+C). Ожидаемый результат отмены, не является сбоем. |

---

## §8. Обработка Сигналов Отмены (Cancellation & Signals)

1. **Перехват SIGINT (Ctrl+C)**: Утилита регистрирует обработчик сигналов прерывания. До момента внедрения в `baker` полноценного cancellation token API, перехват SIGINT работает как контролируемое непрерывное завершение процесса на уровне ОС с кодом возврата 130. После добавления cancellation token API CLI будет атомарно транслировать токен отмены во внутренний контекст `BakeRequest`.
2. **Изоляция Незавершенных Артефактов**: Частично скомпилированный архив `.axic` никогда не регистрируется как успешный результат. Временные стадийные файлы удаляются или изолируются согласно правилам утилизации.

---

## §9. Правила Безопасности и Перезаписи (Safety & Path Rules)

1. **Соблюдение Path Containment**: Утилита `baker-cli` не имеет права обходить проверки границ путей, выполняемые библиотекой `baker`.
2. **Атомарная Замена при Перезаписи (`--force`)**: Если целевой файл `.axic` по указанному пути `--out` уже существует, утилита требует обязательный флаг `--force`. Запись проводится во временный файл рядом с целевым каталогом (`.axic.tmp`), а замещение целевого файла происходит атомарно строго после полного успеха компиляции.
3. **Неизменяемость Исходников**: Команды утилиты ни при каких условиях не модифицируют файлы исходного проекта.

---

## §10. Трансляция Ошибок и Отчетов (Error / Report Mapping)

Ошибки библиотеки `BakerError` транслируются в текстовое резюме в `stderr` и структуры JSON-диагностик в `stdout` (при активированных флагах `--json` или `--json-lines`):

- Формат сохраняет фазу сборки (`phase`), стабильный код (`code`), серьезность (`severity`) и ссылку на источник (`source_ref`).
- Любая нераспознанная внутренняя ошибка мапится в стабильный код системной диагностики CLI: `AXI-CLI-UNKNOWN`.

## §11. Golden Tests / Обязательная Матрица Тестирования

Крейт `baker-cli` обязан быть покрыт набором интеграционных тестов командной строки:

1. **Рендеринг Справки (`test_baker_cli_help_renders`)**: Проверка успешного вывода `--help` с кодом 0.
2. **Требование Одного Источника Входа (`test_baker_cli_requires_one_input_source`)**: Возврат ошибки при отсутствии параметров `--project-root` и `--snapshot-bundle`.
3. **Браковка Взаимоисключающих Параметров Входа (`test_baker_cli_rejects_project_root_and_snapshot_together`)**: Проверка ошибки при одновременной передаче обоих источников.
4. **Взаимоисключаемость Режимов Вывода (`test_baker_cli_json_and_json_lines_are_mutually_exclusive`)**: Проверка возврата ошибки при одновременном вызове `--json` и `--json-lines`.
5. **Проброс Job ID в События Sidecar (`test_baker_cli_job_id_propagates_to_json_lines`)**: Проверка трансляции пользовательского `--job-id` в однострочные JSON-события.
6. **Feature-Gate Подкоманды Edge (`test_baker_cli_edge_feature_gates_edge_subcommand`)**: Проверка активности подкоманды `edge` только при сборке с `--features edge`.
7. **Код Возврата Отмены 130 (`test_baker_cli_cancelled_exit_code_130`)**: Проверка корректного возврата кода 130 при имитации отмены по SIGINT.
8. **Атомарная Замена Выходного Файла (`test_baker_cli_atomic_output_replace_on_force`)**: Проверка того, что при `--force` старый файл `.axic` не затирается в случае сбоя компиляции.
9. **Политика Ошибок Аргументов в Machine Mode (`test_baker_cli_machine_mode_argument_error_policy`)**: Верификация вывода структурированных JSON-ошибок при аргументных сбоях после распознавания флага вывода.
10. **Чистый stdout при `--json` (`test_baker_cli_json_stdout_is_clean`)**: Проверка того, что в `stdout` попадает строго один валидный JSON без постороннего текста.
11. **Вывод Человеческих Логов в stderr (`test_baker_cli_human_logs_go_to_stderr`)**: Проверка отправки текстовых логов `tracing` строго в `stderr`.
12. **Валидность Событий `--json-lines` (`test_baker_cli_json_lines_are_valid_one_line_events`)**: Проверка поочередной валидности однострочных JSON-объектов.
13. **Код Возврата Успеха (`test_baker_cli_exit_code_success`)**: Верификация кода возврата 0 при успешной компиляции.
14. **Код Возврата Ошибки Валидации (`test_baker_cli_exit_code_validation_error`)**: Верификация кода возврата 1 при ошибках валидации проекта.
15. **Изоляция от Compute и Network (`test_baker_cli_does_not_depend_on_compute_ipc_net`)**: Статический контроль отсутствия зависимостей от `compute`, `ipc`, `wire`.
16. **Вывод Метаданных Компилятора (`test_baker_cli_meta_json_contains_versions_and_commands`)**: Верификация структуры ответа подкоманды `meta --meta-json`.
17. **Исключение Snapshot ID из Байтов Архива (`test_baker_cli_snapshot_id_not_written_to_archive`)**: Проверка того, что `--snapshot-id` не меняет байты сгенерированного `.axic`.
18. **Неизменяемость Исходных TOML (`test_baker_cli_no_source_toml_mutation`)**: Проверка совпадения хэшей исходников после вызова CLI.

## §12. Open Questions / Review Debt (Открытые Вопросы и Противоречия)

1. **Окончательное Имя Бинарного Файла Утилиты**:
   - *Контекст*: В спецификации используется имя `baker-cli`.
   - *Вопрос*: Какое конечное имя бинарника будет утверждено для поставки — `baker-cli`, `axiengine-baker` или единый фасадный исполняемый файл `axiengine`?

2. **Единая Схема Протокола JSON-lines для AxiCAD Bridge**:
   - *Контекст*: Sidecar-режим генерирует однострочные события.
   - *Вопрос*: Требуется ли вынести общие DTO-структуры событий sidecar в отдельный крейт контрактов сопряжения с AxiCAD?

3. **Зависимости Подкоманды Инспектирования (`inspect`)**:
   - *Контекст*: Команда `inspect` читает структуру `.axic` архива.
   - *Вопрос*: Должна ли команда `inspect` обращаться исключительно к хелперам `baker` или может напрямую подключать легкие инспекционные методы `vfs`?

4. **Выделение Edge-Конвертора в Отдельный Бинарник**:
   - *Контекст*: Подкоманда `edge` выполняет конвертацию моделей.
   - *Вопрос*: Целесообразно ли сохранять подкоманду `edge` в `baker-cli` или вынести ее в отдельный исполняемый файл `edge-cli`?

5. **Точный API Отмены (Cancellation Token API) из Библиотеки `baker`**:
   - *Контекст*: CLI обрабатывает сигнал SIGINT как процессный останов до появления токена отмены.
   - *Вопрос*: Каким образом компилятор `baker` предоставит атомарные токены отмены для трансляции из `baker-cli`?

6. **Централизованное Фиксирование Версий Зависимостей (Workspace-Wide Pinning)**:
   - *Контекст*: Версии `clap`, `tracing`, `serde_json` зафиксированы в спецификации.
   - *Вопрос*: Требуется ли централизованный манифест версий зависимостей на уровне всего workspace для исключения дрифта сторонних CLI-крейтов?
