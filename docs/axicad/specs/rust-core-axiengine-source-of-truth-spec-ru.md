# Спецификация канонического вычислительного ядра AxiEngine и контракта источника истины (Rust Core AxiEngine Source of Truth Spec)

> Этот документ формально определяет стратегический архитектурный контракт между визуальной рабочей средой 3D-редактора AxiCAD и будущим каноническим вычислительным ядром на Rust (AxiEngine). Спецификация фиксирует разграничение зон ответственности, правила перепроектирования ядра, матрицу владения алгоритмами, протоколы синхронизации бинарных файлов и модель обмена данными.

## Status: Draft

## 1. Назначение документа (Scope & Non-goals)

Данная спецификация закрепляет статус будущего Rust-ядра **AxiEngine** как единственного исполнимого источника истины (executable truth) для всей биологической математики, алгоритмов раскладки, трассировки и компиляции модели.

### Назначение (Scope)
- Определение границ и зон ответственности между веб-средой редактора (AxiCAD) и вычислительным движком (AxiEngine).
- Закрепление статуса спецификаций (`docs/specs/*.md`) как архитектурного контракта системы.
- Формализация политики перепроектирования (rewrite policy) ядра и разграничения источников между legacy MVP и прототипами.
- Определение матрицы владения алгоритмами, данными и структурами ABI запекания (Baker).
- Описание трехэтапной модели моста интеграции (Bridge Integration Model) и контрактов межпроцессного обмена данными.
- Определение правил синхронизации бинарных файлов, проверки версий и обработки аварийных режимов (Failure Modes).
- Описание принципов создания эталонных фикстур (Golden Fixtures) на основе сценариев legacy MVP.

### Вне зоны ответственности (Non-goals)
- Документ не описывает внутреннюю реализацию каждого конкретного алгоритма на уровне строк кода Rust.
- Документ не фиксирует окончательный неизменяемый состав или внутреннюю файловую структуру крейтов Cargo.
- Документ не является спецификацией пользовательского интерфейса (UI/UX) или визуального слоя Three.js.
- Документ не требует сохранения существующей кодовой базы `dev-rust-api` или прямого переноса ее текущей архитектуры.

## 2. Главный принцип (Main Principle)

Архитектурное разграничение ролей в экосистеме Axicor строго подчиняется следующей фундаментальной формуле:

> **Specs define the contract.**  
> **AxiEngine executes the truth.**  
> **AxiCAD visualizes, edits, and debugs it.**

```
┌───────────────────────────────────────────────────────────────────────────┐
│                           Architectural Specs                             │
│                         (docs/specs/*.md Contract)                        │
└─────────────────────┬───────────────────────────────┬─────────────────────┘
                      │                               │
                      ▼                               ▼
┌──────────────────────────────────────────┐    ┌───────────────────────────┐
│           AxiEngine (Rust Core)          │    │      AxiCAD (Web UI)      │
│          Executable Source of Truth      │    │ Editor, Debugger, Visuals │
├──────────────────────────────────────────┤    ├───────────────────────────┤
│ • Canonical Math & Algorithms            │    │ • Interaction & Gestures  │
│ • TOML Parsing & Serde Boundary          │◀──▶│ • Scene Rendering (3D)    │
│ • Packed ABI & Binary Generation         │    │ • Reactive Editor Store   │
│ • Layout, Routing, Growth, Inference     │    │ • Orchestration & Bridge  │
└──────────────────────────────────────────┘    └───────────────────────────┘
```

## 3. Системные роли (System Roles)

Для исключения неопределенности в вопросах источника правды за каждым компонентом и кодовой базой закрепляется жесткая системная роль:

1. **`docs/specs/*.md` (Architectural Contract)**: Канонические спецификации. Определяют типы данных, инварианты, бинарные маски ABI и логические правила системы. Являются верховным арбитром при любых расхождениях между редактором и движком.
2. **Legacy MVP (`axicor-master`)**: `axicor-master` — внешний локальный legacy MVP reference directory, не отслеживается git-репозиторием AxiCAD и используется только как behavioral archaeology / e2e reference.
3. **`dev-rust-api` (Abandoned Refactor Prototype)**: Заброшенный незавершенный прототип рефакторинга. **Не является источником истины** и не считается legacy MVP. Может служить лишь источником вдохновения для наименований крейтов только после обязательного архитектурного ревью.
4. **`AxiEngine` (Fresh Rust Rewrite Target & Executable Truth)**: Разрабатываемое заново с чистого листа вычислительное Rust-ядро. Является единственным каноническим исполнителем алгоритмов, математики, валидации, запекания (Baker), симуляции роста и рантайма.
5. **`AxiCAD` (UI / Editor / Debugger / Orchestrator)**: Интерактивная веб-среда. Выполняет роли визуализатора 3D-сцены, лабораторного стенда для отладки, редактора пользовательских мутаций и оркестратора процесса сборки/вызова движка.
6. **TOML Configuration (`model.toml`, `department.toml`, `shard.toml`)**: Канонический декларативный формат хранения биологической модели и проекта Axicor.
7. **Project Metadata (`axicad.project.json`)**: Исключительно служебный файл настроек редактора, визуальной раскладки панелей, состояний камеры, флагов блокировки и локального кэша предпросмотра. Не содержит биологической истины.

## 4. Политика перепроектирования Движка (AxiEngine Rewrite Policy)

Разработка нового вычислительного ядра `AxiEngine` ведется в соответствии со следующими принципами:

- **Отсутствие цели переноса кода из прототипа**: Прямой перенос кодовой базы из `dev-rust-api` не является целью разработки. Допускается лишь точечное извлечение идей и структур после их строгой сверки с нормативными `.md` спецификациями.
- **Правило наследования имен крейтов**: Именования крейтов из прототипа могут наследоваться только в том случае, если они полностью совпадают с новой целевой архитектурой и доменными границами.
- **Принцип "Сначала фикстура"**: Любой алгоритм из legacy MVP (`axicor-master`) перед реализацией в `AxiEngine` сначала формализуется спецификацией и набором золотых тестов (Golden Fixtures), и лишь затем реализуется заново на Rust.

## 5. Матрица владения алгоритмами (Algorithm Ownership Matrix)

Таблица строго разделяет зоны ответственности между вычислительным ядром AxiEngine и визуальной средой AxiCAD:

| Домен / Функция | Владелец (Canonical Owner) | Описание и границы ответственности |
|-----------------|----------------------------|------------------------------------|
| **TOML Parsing & Serde** | **AxiEngine** (+ Import/Export) | Канонический парсинг, строгое соблюдение типов и проверку неизвестных полей (`deny_unknown_fields`) выполняет AxiEngine. AxiCAD использует парсер на границе импорта/экспорта. |
| **Packed ABI & Binary Format** | **AxiEngine Only** | Упаковка бинарных масок `PackedPosition`, `PackedTarget`, `SomaFlags` и генерация байт-кода запекания осуществляются строго ядрами Rust. |
| **Geometry Math & Bounds** | **AxiEngine** (Canonical) | Вычисление воксельных сеток, габаритов AABB и геометрических трансформаций канонически принадлежит движку. AxiCAD может кэшировать вычисленные значения для рендеринга. |
| **Layout & Autoplacement** | **AxiEngine** | Алгоритмы автораскладки шардов и департаментов, упаковки объемов и выравнивания выполняются движком. |
| **Constraint & Invariant Check** | **AxiEngine** (Canonical) | Каноническая проверка биологических и геометрических инвариантов. AxiCAD отображает кэшированные объекты `DiagnosticItem` в UI. |
| **Selection & Raycasting** | **AxiCAD** (+ Engine if needed) | Управление жизненным циклом выделения и рейкастинг по Three.js мешам принадлежит AxiCAD. При необходимости точного рейкастинга по вокселям вызываются геом-сервисы движка. |
| **Tool Gestures Lifecycle** | **AxiCAD** (Interaction) | Обработка ввода мыши, контроллеров и состояний инструментов принадлежит AxiCAD. Валидация жеста и генерация предложенных патчей (`proposed patches`) выполняются движком. |
| **Socket Sampling & Tract Routing** | **AxiEngine** | Дискретный семплинг сокетов, трассировка трактов в 3D-сетке и расчёт каналов связей производятся строго движком. |
| **Soma Placement Preview** | **AxiEngine** | Правило-ориентированная генерация массива позиций сом (`SomaPreview`) осуществляется движком. |
| **Growth Simulation & Preview** | **AxiEngine** | Морфогенез, синаптогенез и рост аксонов/дендритов (как локальный, так и сетевой) вычисляются движком. |
| **Inference & Runtime Stepping** | **AxiEngine** | Исполнение нейросетевого рантайма, пошаговая симуляция (stepping) и расчёт потенциалов выполняются движком. |
| **Rendering, Materials & HUD** | **AxiCAD Only** | Отрисовка 3D-сцены, настройка шейдеров, материалов, оверлеев осциллографа и HUD принадлежат исключительно AxiCAD. |
| **Project Layout Metadata** | **AxiCAD Only** | Сохранение настроек вьюпорта, положений окон и метаданных камеры в `axicad.project.json` принадлежит AxiCAD. |

## 6. Модель моста и интеграции (Bridge / Integration Model)

Интеграция между AxiCAD и AxiEngine реализуется по поэтапной схеме (Staged Approach), позволяющей наращивать производительность по мере развития проекта:

```
┌───────────────────────────────────────────────────────────────────────────┐
│                      Bridge Integration Stages                            │
├───────────────────────────────────────────────────────────────────────────┤
│  Stage 1 (MVP)   : Sidecar CLI Process (Subprocess IPC / stdio / json)   │
│  Stage 2 (Local) : Long-lived Local Daemon (gRPC / WebSocket / IPC Pipe)  │
│  Stage 3 (Hot)   : In-process WebAssembly (WASM / FFI for Fast Preview)  │
└───────────────────────────────────────────────────────────────────────────┘
```

- **Stage 1 (Sidecar CLI / Process Bridge)**: Базовый режим для MVP. AxiCAD вызывает бинарный файл AxiEngine как дочерний процесс (sidecar CLI) с передачей команд через аргументы или поток `stdin`/`stdout`.
- **Stage 2 (Long-lived Daemon / Service)**: Промышленный режим для локальной разработки. AxiEngine запускается как фоновый локальный демонический процесс (daemon service), взаимодействующий с AxiCAD по высокопроизводительному протоколу (gRPC, WebSocket или Unix/Named Pipes).
- **Stage 3 (Optional In-process WASM / Shared Library)**: Опциональный режим для мгновенного обратного отклика (hot preview). Вычислительные модули запекания и геометрической проверки компилируются в WebAssembly (WASM) и исполняются прямо в потоке браузера.

### Механика взаимодействия:
AxiCAD формирует и отправляет объект запроса `EngineRequest` (снэпшот модели, команда мутации, запрос на шаг симуляции). AxiEngine обрабатывает запрос и возвращает структурированный ответ `EngineResponse` (список диагностик, фреймы предпросмотра, наборы патчей или события рантайма).

## 7. Конвейер сборки и синхронизации бинарных файлов (Build & Binary Sync Flow)

Визуальная среда AxiCAD выступает оркестратором локального окружения и должна обеспечивать автоматическое управление бинарными файлами движка:

### Концептуальный пайплайн оркестрации:
1. **Обнаружение (Discovery)**: При запуске или по команде пользователя AxiCAD выполняет поиск исполнимого файла `AxiEngine` в системных путях (`PATH`), локальной директории проекта или целевых папках сборки Cargo (`target/release`).
2. **Сборка по требованию (Build on Demand)**: Если бинарный файл отсутствует или устарел, AxiCAD предоставляет интерфейс для инициализации локальной сборки крейтов AxiEngine через инструмент автоматизации.
3. **Валидация совместимости (Handshake & Sync)**: Перед установкой моста связи AxiCAD запрашивает у движка информацию о сборке (`BuildInfo`) и проверяет три обязательных критерия:
   - Совпадение семантической версии движка (`version`);
   - Совпадение версии IPC-протокола (`protocol_version`);
   - Соответствие хэша сборки или схемы данных (`build_hash`).
4. **Блокировка при расхождении (Protocol Mismatch)**: В случае обнаружения несовместимости версий или протокола AxiCAD принудительно отключает мост, блокирует операции запекания и выдает пользователю соответствующее предупреждение.
5. **Диагностика сбоев сборки**: Если процесс компиляции движка завершается с ошибкой, AxiCAD перехватывает вывод компилятора и формирует объекты `DiagnosticItem` в интерфейсе редактора.

## 8. Контракты обмена данными (Data Exchange Contracts)

Ниже представлены концептуальные интерфейсы обмена данными между AxiCAD и AxiEngine (в формате абстрактного TypeScript/Rust DTO без привязки к конкретной библиотеке сериализации):

```typescript
// Базовый версионный рукопожатный контракт
interface ProtocolVersion {
  major: number;
  minor: number;
  patch: number;
}

interface BuildInfo {
  version: string;
  protocolVersion: ProtocolVersion;
  buildHash: string;
  targetTriple: string;
}

// Запрос от AxiCAD к AxiEngine
interface EngineRequest {
  requestId: string;
  commandType: 'validate' | 'compile_baker' | 'generate_layout' | 'step_simulation' | 'sample_sockets';
  modelSnapshotPath?: string;          // Путь к конфигурации модели
  payloadJson?: string;                // Сериализованные параметры команды
}

// Ответ от AxiEngine к AxiCAD
interface EngineResponse {
  requestId: string;
  status: 'success' | 'error' | 'rejected';
  diagnostics: DiagnosticItem[];        // Сформированный список ошибок и предупреждений
  patchSet?: PatchSet;                 // Предложенные изменения в модель
  previewFrame?: PreviewFrame;         // Данные предпросмотра (сом/трактов)
  executionTimeMs: number;
}

// Структура набора изменений (PatchSet)
interface PatchSet {
  id: string;
  targetDocumentPath: string;          // Затронутый TOML файл
  operations: PatchOperation[];
}

interface PatchOperation {
  op: 'add' | 'remove' | 'replace';
  path: string;                        // TOML path (например, "neuron_types.0.membrane.threshold")
  valueJson: string;
}

// Данные фрейма предпросмотра для рендеринга вьюпорта
interface PreviewFrame {
  frameId: number;
  somaPositions?: Float32Array;        // Упакованный массив координат [X, Y, Z, TypeID]
  tractRoutes?: TractRoutePreview[];   // Данные проложенных геом-трактов
}

// Событие отладочной трассировки симуляции
interface TraceEvent {
  tick: number;
  eventType: 'spike' | 'growth_step' | 'homeostasis_update';
  entityTypedPath: string;
  payload: Record<string, unknown>;
}
```

## 10. Правила Предпросмотра и Канона (Preview vs Canonical Rules)

Для предотвращения рассинхронизации данных между визуальным редактором и вычислительным ядром вводятся жесткие правила работы с предпросмотрами:

1. **Единоличное владение математикой предпросмотра**: Любой предпросмотр (preview), затрагивающий модельный граф, физику или геометрию (например, сгенерированные точки сом, траектории роста аксонов, автотрассировка трактов), считается результатом вычислений AxiEngine.
2. **Кэширование без права канона**: AxiCAD может кэшировать полученные данные предпросмотра в оперативной памяти (RAM) или служебных метаданных проекта (`axicad.project.json`) для обеспечения высокой частоты кадров (FPS) вьюпорта. Кэш **никогда** не является каноническим источником правды.
3. **Коммит через командную модель**: Перевод результатов предпросмотра в каноническое состояние модели осуществляется строго путем генерации объекта `PatchSet` и его транзакционного применения через командную модель мутаций (`Command Mutation Flow`).
4. **Суверенитет TOML**: Файл проекта `axicad.project.json` хранит исключительно пользовательские настройки вьюпорта и кэш визуализации. Он не может подменять или переопределять биологические контракты TOML и Rust-типов.

## 11. Политика использования JavaScript/TypeScript в AxiCAD (JS/TS Policy)

Среда AxiCAD разрабатывается на стекe JavaScript/TypeScript, однако роль этого языка строго ограничена клиентскими задачами:

### Жесткие ограничения JS/TS:
- Код на JS/TS **не реализует** и **не дублирует** канонические алгоритмы биологической модели, физики, запекания или геометрии.

### Разрешенные сферы ответственности JS/TS:
1. **Управление реактивным UI**: Состояние панелей, меню, модальных окон и реактивный Store редактора.
2. **Жизненный цикл жестов (Gesture Handling)**: Перехват событий мыши, клавиатуры и контроллеров ввода, сборка намерений пользователя.
3. **Трансформации рендеринга**: Пространственные матричные трансформации объектов Three.js на сцене, управление камерой и светом.
4. **Визуальная интерполяция**: Сглаживание движения объектов и анимации UI между фреймами предпросмотра.
5. **Оперативное кэширование**: Хранение полученных от движка структур данных в памяти веб-приложения.
6. **Форматные адаптеры**: Конвертация данных для отображения в таблицах и виджетах HUD.

### Политика использования прототипов (Non-canonical Prototypes):
Если в процессе разработки AxiCAD создается временный TS-прототип (TS prototype) какого-либо алгоритма для быстрой проверки гипотезы UI, этот код **обязан быть явно помечен** в исходниках и документации как `NON-CANONICAL PROTOTYPE` с указанием срока его удаления и замены на вызовы AxiEngine.

## 12. Тестирование и эталонные фикстуры (Testing & Golden Fixtures)

Качество и детерминированность нового движка AxiEngine обеспечиваются системой золотых тестов (Golden Fixtures):

- **Статус e2e-сценариев legacy MVP**: Сценарии использования и тестовые модели из legacy MVP (`axicor-master`) переводятся в статус эталонных фикстур (Golden Fixtures) и размещаются в крейте `axi-fixtures`.
- **Типы эталонных проверок**:
  - `TOML input -> expected diagnostics`: Проверка точности генерации диагностик при синтаксических и биологических ошибках.
  - `TOML input -> expected packed ABI`: Проверка бинарной совместимости упакованных масок с зафиксированными эталонами.
  - `Placement input -> expected layout/route`: Проверка детерминизма алгоритмов автораскладки и трассировки.
  - `Growth seed -> deterministic trace`: Проверка совпадения траекторий роста при заданном случайном сиде (`master_seed`).
  - `Inference seed/input -> deterministic trace`: Проверка совпадения спайковых откликов и состояний мембран рантайма.
- **Интеграционное тестирование моста**: Интеграционные тесты AxiCAD проверяют корректность работы моста путем сравнения ответов движка с ожиданиями фикстур.

## 13. Обработка сбоев и аварийных режимов (Failure Modes)

Среда AxiCAD должна корректно отрабатывать возможные внештатные ситуации при взаимодействии с движком:

| Сбойный режим (Failure Mode) | Поведение системы и реакция AxiCAD |
|------------------------------|------------------------------------|
| **AxiEngine binary missing** | Мост отключается. В UI выводится предупреждение с предложением указать путь к бинарнику или запустить локальную сборку. Операции запекания блокируются. |
| **Build failed** | Отключается возможность запуска моста. Перехватывается вывод компилятора Cargo, формируются объекты `DiagnosticItem` со статусом ошибки. |
| **Protocol / Version mismatch** | Мост принудительно разрывается. Пользователю выводится сообщение о несовместимости версий AxiCAD и AxiEngine с указанием требуемых версий протокола. |
| **Engine panic / crash** | Редактор перехватывает аварийный разрыв процесса, сохраняет текущий сессионный файл `axicad.project.json` во избежание потери метаданных и предлагает перезапустить движок. |
| **Non-deterministic output** | Регистрируется системная диагностика высокой критичности. Результаты симуляции помечаются как недостоверные. |
| **Unsupported legacy feature** | При попытке задействовать выведенную из строя функцию легаси-кода генерируется информационная диагностика с предложением использовать актуальный аналог. |
| **Dirty Store vs Stale Snapshot** | Если в AxiCAD есть несохраненные мутации в Store, а движок запрошен на выполнение запекания, редактор автоматически формирует свежий временный снэпшот для передачи в движок. |

## 14. Открытые архитектурные решения (Open Decisions)

В рамках проектирования контракта AxiEngine выделены следующие открытые вопросы, требующие утверждения отдельных ADR:

1. **Выбор формата моста в MVP**: Использовать ли в качестве основного моста в MVP легкий sidecar CLI или сразу разворачивать фоновый локальный демон (daemon)?
2. **Протокол бинарного обмена**: Использовать ли текстовый JSON для обмена сообщениями моста на ранних этапах или сразу переходить на бинарную сериализацию (MessagePack / bincode / protobuf)?
3. **Организация репозиториев**: Разрабатывать ли AxiEngine внутри единого монорепозитория рядом с AxiCAD или вынести его в отдельный независимый репозиторий?
4. **Сроки введения WebAssembly**: На каком этапе развития проекта потребуется компиляция модальных модулей запекания и валидации в in-process WASM для мгновенного предпросмотра?
5. **Стратегия версионирования**: Каков точный механизм сквозного версионирования протокола связи, TOML-схемы и хэша сборки бинарных файлов?
6. **Минимальный вертикальный срез (Vertical Slice)**: Какой минимальный набор крейтов AxiEngine (например, `types` + `config` + `baker`) необходим для реализации первого рабочего вертикального среза компиляции?
7. **Методология переноса алгоритмов**: Какую стратегию использовать при переносе сложных биофизических алгоритмов из legacy MVP — ручной перенос с нуля, разработка тестов-фикстур в первую очередь или полуавтоматическая трансляция?
