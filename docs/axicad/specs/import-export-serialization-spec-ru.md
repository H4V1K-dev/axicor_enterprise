# Спецификация импорта, экспорта и сериализации AxiCAD (Import-Export & Serialization Spec)

> Этот документ формально описывает процессы сериализации и десериализации, правила импорта фрагментов и устаревших моделей через адаптеры, гарантии сохранения структуры при циклическом проходе (Round-Trip) и разграничение первичного источника правды (Source of Truth) от генерируемых артефактов.

## Status: Draft

---

## 1. Назначение документа

Import-Export & Serialization Spec документирует механизмы обмена данными между файловой системой (диском), реактивной памятью Store 3D-редактора AxiCAD и внешними генерируемыми артефактами (скриптами Python, отчетами, Baker-конфигурациями).

### Вне зоны ответственности (Non-goals)
- Документ не вводит новые биологические параметры Axicor и не изменяет TOML-схему (см. `toml-schema-spec-ru.md`).
- Документ не описывает структуру бинарных файлов сборщика Baker.
- Документ не навязывает конкретную реализацию парсера Python-кода (помимо ограниченных адаптеров трансляции).

---

## 2. Модель единственного источника правды (Source of Truth Model)

Архитектура AxiCAD строго разделяет типы данных по их авторитетности:

```
┌────────────────────────────────────────────────────────┐
│             ПЕРВИЧНЫЕ ИСТОЧНИКИ (Source of Truth)      │
├───────────────────────────────┬────────────────────────┤
│ Биологическая модель (TOML)   │ Модель редактора (JSON)│
│  - model.toml                 │  - axicad.project.json │
│  - department.toml            │    (камеры, viewport,  │
│  - shard.toml                 │     placements шардов) │
└──────────────┬────────────────┴───────────┬────────────┘
               │                            │
               ▼                            ▼
┌────────────────────────────────────────────────────────┐
│             ВТОРИЧНЫЕ / ГЕНЕРИРУЕМЫЕ АРТЕФАКТЫ         │
├────────────────────────────────────────────────────────┤
│  - Baker compile outputs (бинарные файлы, манифесты)   │
│  - Скрипты Python SDK (axipy)                          │
│  - Отчеты валидации и логи                             │
└────────────────────────────────────────────────────────┘
```

- **Canonical Biological Source**: Файлы `model.toml`, `department.toml` и `shard.toml` являются единственным источником правды для структуры биологической сети.
- **Editor-Only Source**: Файл `axicad.project.json` является единственным источником правды для визуальной сцены, настроек вьюпорта и интерфейса.
- **Derived Binary Artifacts (Heavy Payloads)**: Для тяжелых производных данных (бинарные сборки Baker, записи симуляций, кэш сом) применяется гибридная модель сериализации. Они не заталкиваются в `axicad.project.json`, а сохраняются в виде отдельных бинарных файлов в хранилище производных артефактов (`.local-storage/artifacts/`). Их целостность, хэши и версии схем отслеживаются через `Artifact Registry`.
- **Generated Artifacts**: Все файлы компиляции Baker, отчеты валидации и скрипты Python SDK являются воспроизводимыми результатами генерации. Они могут быть безвозвратно удалены и перестроены в любой момент на основе первичных TOML/JSON источников. Любые ручные изменения в генерируемых артефактах игнорируются, если они не импортируются обратно в Store через выделенные адаптеры.

## 3. Канонический конвейер импорта (Canonical Import Pipeline)

Загрузка проекта в реактивный Store редактора выполняется по строгому многоэтапному алгоритму:

1. **Загрузка корневой модели**: Чтение и парсинг `model.toml`. При ошибках синтаксиса/десериализации Tier 1 загрузка прерывается.
2. **Разрешение путей департаментов**: Чтение относительных путей `config` и загрузка всех `department.toml`.
3. **Разрешение путей шардов**: Загрузка всех связанных `shard.toml` файлов.
4. **Построение Store и Ownership Tree**:
   - Создаются записи `DomainRecord` для каждого элемента.
   - Каждой записи сопоставляется временный сессионный UUID.
   - На основе вложенности TOML-таблиц выстраиваются иерархические связи `parentId` / `childrenIds`.
5. **Вычисление типизированных путей (byTypedPath)**: Для всех сущностей рассчитывается абсолютный путь с префиксом типа (например, `shard:SensoryCortex.Retina`).
6. **Разрешение ссылок соединений**: Строковые эндпоинты в `connections` преобразуются в UUID сокетов через индексы `socketByDepartmentEndpoint` и `socketByModelEndpoint`.
7. **Наложение метаданных проекта (Latching)**: Считывается `axicad.project.json` (при его наличии). Координаты размещения шардов, состояние камер и настройки UI сопоставляются с UUID сущностей по типизированным путям. Устаревшие метаданные отправляются в карантин `orphaned_metadata`.
8. **Первичный прогон валидатора**: Запускаются семантические и топологические проверки Tier 2, 3 и 4, результаты привязываются к UUID сущностей.

## 4. Канонический конвейер экспорта (Canonical Export Pipeline)

Экспорт биологического кода TOML на диск гарантирует очистку файлов от служебных параметров редактора:

1. **Сбор данных**: Обход дерева владения Store снизу вверх (начиная от корня `model`).
2. **Фильтрация**: Игнорируются все записи, для которых вычисляемое свойство `exportable(record) === false` (например, визуальные placements, камеры).
3. **Очистка метаданных**: Из биологических DomainRecord извлекается только чистое поле `data`. Служебные поля `id`, `meta`, `diagnostics` и `path` полностью отбрасываются.
4. **Сортировка полей (Stable Ordering)**: Ключи и структуры упорядочиваются в строго детерминированном каноническом виде (см. раздел 11) для предотвращения шума в Git diff.
5. **Запись файлов**: Файлы записываются в файлы конфигураций по каноническому шаблону каталогов.
6. **Сброс Dirty флагов**: После успешного подтверждения записи на диск из списка `toml_documents_dirty` удаляются **только те** пути к TOML-документам, которые были реально успешно записаны на диск.

### Семантика Dirty State
- **Локальность ошибок**: Диагностические ошибки (diagnostics), найденные при импорте или фоновой валидации, сами по себе не переводят биологические документы в dirty-состояние (`toml_documents_dirty`).
- **Блокировки**: Висящие соединения (dangling connections) блокируют компиляцию Baker и канонический экспорт биологических TOML на диск, генерируя соответствующие ошибки diagnostics.
- **Условие загрязнения**: Биологический документ помечается как dirty исключительно в том случае, если пользовательская команда в редакторе или принятая пользователем команда автоматического исправления/импорта (accepted import/repair command) реально модифицировала полезный биологический payload (`data`).

## 5. Гарантии Round-Trip (Round-Trip Guarantees)

Циклический проход данных (TOML на диске $\to$ Store в памяти $\to$ сохранение TOML на диск) должен обеспечивать следующие гарантии:

- **Семантическая эквивалентность (Semantic Equivalence)**: Загруженные биологические параметры должны сохраняться без каких-либо изменений значений полей.
- **Стабильная сортировка (Stable Ordering)**: Порядок разделов и массивов таблиц должен оставаться неизменным при повторном экспорте.
- **Защита от утечек (No Metadata Leak)**: Ни при каких обстоятельствах UUID, placements или diagnostics не должны записываться в биологические TOML-файлы.
- **Сохранение комментариев и ручного форматирования (Комментарии)**: 
  > [!IMPORTANT]
  > Сохранение комментариев и стилистики ручного форматирования TOML-файлов при перезаписи редактором является **открытым архитектурным решением (open decision)**. В текущей версии экспорт полностью перезаписывает TOML-файлы, генерируя их заново по каноническому шаблону, ручные комментарии внутри файлов стираются.
- **Строгая схема**: Неизвестные поля TOML-файлов не сохраняются и не игнорируются, а вызывают ошибку десериализации Tier 1.

## 6. Импорт фрагментов (Fragment Import)

Импорт отдельных фрагментов сети (например, импорт готового шарда, набора типов клеток или слоев из внешних библиотек рецептов) подчиняется транзакционным правилам:

- **Стейджинг и План импорта (ImportPlan / PatchSet)**: При импорте фрагментов, загрузке через legacy-адаптеры или Python-транслятор редактор не вносит немедленные изменения в Store. Сначала генерируется промежуточный `ImportPlan` (план импорта) или `PatchSet` (набор патчей).
- **Принятие изменений (Accept/Apply)**: Сгенерированный `PatchSet` применяется к реактивному Store редактора исключительно через стандартную транзакцию команд (`command transaction`) и только после явного подтверждения пользователя в UI (кнопка "Применить"). Это обеспечивает полную поддержку Undo/Redo для импортированных данных.
- **Разрешение конфликтов имен**: При обнаружении дубликатов имен сущностей в импортируемом `PatchSet` редактор останавливает транзакцию, выводит диагностическое окно и предлагает пользователю стратегии разрешения (переименовать импортируемый объект, перезаписать существующий или отменить импорт).

## 7. Legacy-адаптеры (Legacy/Import Adapters)

Адаптеры используются для загрузки устаревших версий моделей Axicor (например, MVP-схем или моделей из legacy репозиториев):

- **Нормализация**: Выходные данные адаптера должны быть приведены к текущему каноническому контракту перед загрузкой в Store.
- **Логирование потерь (Lossy Conversions)**: Если старая модель содержала параметры, которые больше не поддерживаются текущей TOML-схемой, адаптер обязан отбросить эти данные, сформировать детальный лог предупреждений о потерянных данных (lossy conversion log) и отобразить его пользователю.
- **Отсутствие обратной совместимости при экспорте**: Редактор экспортирует файлы только в текущей канонической версии схемы. Обратная запись в устаревший формат не поддерживается.

## 8. Экспорт генерируемого кода (Script/Code Exporters)

Редактор предоставляет функционал генерации вспомогательных артефактов:

- **Генерация Python SDK скрипта (Axipy)**: Сборка Python-скрипта, воспроизводящего структуру сети через API Axipy.
- **Направление трансляции**: Генерация кода является односторонним процессом (TOML $\to$ Python). Изменения, внесенные пользователем вручную в сгенерированный Python-скрипт, не импортируются обратно в TOML, за исключением случаев использования специализированных адаптеров импорта.
- **Отчеты валидации (Validation Reports)**: Экспорт результатов проверок Tier 2-4 в формате Markdown или JSON для внешних CI/CD систем.

## 9. Авторазмещение и геометрия (Auto-Placement & Generated Geometry)

Результаты работы алгоритмов автоматического размещения шардов и автотрассировки аксонов Baker-ом интегрируются по следующим правилам:

- **Биологическая независимость**: Координаты авторазмещения не модифицируют файлы конфигураций TOML.
- **Стейджинг геометрии (Generated Preview / Staging)**: Вычисленные алгоритмами авторазмещения положения шардов или пути аксонов, вычисленные автотрассировкой Baker, сначала загружаются в редактор как временное визуальное превью (`Generated Preview / Staging`).
- **Принятие размещений**: Координаты авторазмещения переносятся в постоянные placements служебного файла `axicad.project.json` только после явного вызова команды подтверждения ("Принять изменения") пользователем через `UpdateEditorPlacementCommand`.
- **Временный статус трассировок**: Вычисленные Baker-ом пути прорастания аксонов (`axon traces`) остаются исключительно в preview/cache визуализации редактора и никогда не перезаписывают биологические TOML-файлы.

## 10. Обработка конфликтов при сериализации

| Сценарий конфликта | Серьезность | Поведение системы | Рекомендуемое исправление |
|---|---|---|---|
| Дублирование имен шардов / департаментов | `error` | Блокировка экспорта TOML | Переименовать один из объектов в UI редактора. |
| Отсутствие TOML-файла по пути `config` | `error` | Вывод диагностики, загрузка в Read-only | Восстановить файл на диске или переназначить путь в Store. |
| Висящие эндпоинты соединений (dangling connections) | `error` | Блокировка компиляции Baker и canonical export, диагностика без автоматического dirty | Удалить некорректное соединение или создать сокет с нужным именем. |
| Устаревшие пути placements в `axicad.project.json` | `warning` | Перенос метаданных в orphaned_metadata карантин | Привязать метаданные к новому пути или удалить их. |

## 11. Детерминированный порядок сериализации (Serialization Ordering)

Для минимизации изменений при коммитах в Git при экспорте применяется жесткая сортировка полей и секций:

1. **Сортировка файлов**: Файлы сохраняются в алфавитном порядке имен департаментов и шардов.
2. **Порядок полей в `model.toml`**:
   - `connections = []` / `[[connections]]` (пишутся строго до первого раздела `[world]`).
   - `[world]`.
   - `[simulation]`.
   - `[[departments]]` (список таблиц, отсортированный по алфавиту `name`).
3. **Порядок полей в `department.toml`**:
   - `connections = []` / `[[connections]]` (пишутся строго до списка `[[shards]]`).
   - `[[shards]]` (список таблиц, отсортированный по алфавиту `name`).
4. **Порядок полей в `shard.toml`**:
   - `[dimensions]`.
   - `[settings]`.
   - `[[layers]]` (порядок Z снизу вверх).
   - `[[neuron_types]]` (порядок по бинарному индексу 0..15).
   - `[[sockets]]` / `[[ports]]` (по алфавиту `name`).

## 12. Исключения из целей спецификации (Non-goals)

- Структуры бинарных сеток VRAM и GPU-манифестов.
- Логи выполнения симуляции спайков (Spike Trace logs).
- Разработка полноценного двустороннего транслятора Python-кода в TOML (поддерживается только односторонняя генерация).
