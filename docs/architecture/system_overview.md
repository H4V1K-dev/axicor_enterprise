# Архитектурный план: Подробное проектирование экосистемы Axicor и редактора AxiCAD

Этот документ содержит детальную проектную спецификацию для монорепозитория Axicor и структуры его модулей в рамках разработки локальной песочницы.

---

## 1. Структура монорепозитория разработки

```
axicad-workspace/
├── dev-rust-api/          # Вычислительное ядро симуляции (Rust-крейты: Verlet physics, A* routing, collision)
├── dev-sdk-api/           # Изолированное Python SDK (axipy) для генерации рецептов топологий
├── dev-client-api/        # Локальный Python-сервер (REST API, транслятор путей, компилятор оверрайдов)
├── dev-js-api/            # Модульный 3D-редактор (Vanilla JS, Three.js, конечный автомат режимов CAD)
├── dev-site-api/          # Шаблон статического лендинга/лаунчера
├── examples/              # Исходные рецепты сетей пользователя (*.py файлы рецептов)
└── .local-storage/        # [Git-ignored] Локальный кэш visualizer-а (placement.json, routes.json, оверрайды)
```

## 2. Модуль dev-js-api (Фронтенд Редактора)

Этот модуль содержит основной интерактивный интерфейс. Мы разделяем его на слои для устранения сильной связанности (coupling).

```
dev-js-api/js/
├── main.js                # Точка входа (Orchestrator). Инициализирует Viewport, ModeManager, загружает данные.
├── viewer.js              # Инициализация WebGL рендерера, камер (Persp/Ortho), освещения, OrbitControls.
├── scene_builder.js       # Синхронизатор сцены. Читает Store и обновляет THREE.Mesh (шарды, сокеты, трассы).
├── editor.js              # Инициализация редактора, TransformControls, регистрация режимов.
│
├── store/                 # Слой управления состоянием
│   ├── store.js           # Реактивное Key-Value хранилище состояния (Single Source of Truth).
│   ├── event_bus.js       # Шина событий для слабой связанности (события выделения, ошибок, сохранения).
│   └── history_manager.js # Гибридный Undo/Redo стек с Timeline Branching.
│
├── editor/                # Интерактивный слой (События мыши, Хоткеи, Манипуляции)
│   ├── modes/             # FSM (Конечный автомат) режимов редактирования
│   │   ├── base_mode.js   # Абстрактный класс режима
│   │   ├── select_mode.js # Выделение объектов
│   │   ├── translate_mode.js # Перемещение с Gizmo
│   │   ├── resize_mode.js # Изменение размеров шардов
│   │   ├── add_shard_mode.js # Спавн нового шарда
│   │   ├── add_socket_mode.js # Размещение сокета
│   │   └── add_route_mode.js # Протяжка линии связи
│   ├── mode_manager.js    # Управление активным режимом, переключение, стек истории (Escape)
│   └── collision_manager.js # CollisionManager: Raycasting сцены, Backface Culling, X-ray выделение сокетов
│
├── rendering/             # Слой отображения специализированной 3D графики
│   ├── mesh_factory.js    # Фабрика 3D объектов (загрузка моделей пинов/сокетов)
│   ├── route_renderer.js  # Отрисовка сплайнов связей
│   └── soma_renderer.js   # Отрисовка соматических элементов
│
└── ui/                    # Компоненты UI (HTML/CSS)
    ├── sidebar.js         # Панель инспектора свойств (размеры шарда, позиция сокета)
    ├── toolbar.js         # Панель переключения инструментов (инспектирование, перемещение, создание)
    ├── project_hub.js     # Окно выбора, загрузки и импорта проектов
    ├── settings_panel.js  # Панель настроек редактора (размеры сеток, шаги привязок)
    └── viewcube.js        # WebGL ViewCube (CubeGizmo.glb) для навигации камеры
```

### 2.1. Слой Domain & State (Бизнес-логика и Стейт)
- **Store (`js/store/store.js`)**:
  - Хранит только чистые JSON-данные: `placementData`, `routesData`, `selectedShardKey`, `selectedSocketKey`, `activeMode`, `modalActive`.
  - Запрещено помещать в Store инстансы `THREE.Mesh` или `THREE.Scene`.
  - Компоненты и Рендерер подписываются на ключи: `store.on('placementData', (newData) => { ... })`.
- **Event Bus (`js/store/event_bus.js`)**:
  - Отвечает за глобальные асинхронные уведомления: `emit(EVENTS.VALIDATION_REQ)`, `emit(EVENTS.DATA_RELOADED)`.
- **HistoryManager (`js/store/history_manager.js`)**:
  - Реализует гибридный Undo/Redo стек с Timeline Branching. При отмене действия и начале новой ветки изменений, будущая история отсекается.
  - Каждое действие записывается как экшен с парой функций: `execute()` и `undo()`.

### 2.2. Слой Взаимодействия: Modes & Tools (FSM)
Мы четко разделяем понятия:
- **Tool (Инструмент)**: UI-компонент (кнопка в панели или клавиша Q, W, E, R, T). При клике на инструмент срабатывает переключение логического режима.
- **Mode (Режим)**: Класс-наследник `BaseMode`, определяющий поведение приложения при взаимодействии пользователя со сценой.

#### Класс `BaseMode`:
```javascript
export class BaseMode {
  enter(context) {}       // Инициализация (показать гизмо, изменить курсор)
  exit() {}                // Сброс (скрыть гизмо, очистить временную графику)
  onPointerDown(event, raycaster) {}
  onPointerMove(event, raycaster) {}
  onPointerUp(event, raycaster) {}
  onKeyDown(event) {}
  onKeyUp(event) {}
  onUpdate(dt) {}          // Вызывается в цикле requestAnimationFrame (например, для анимаций)
}
```
- **ModeManager (`js/editor/mode_manager.js`)**:
  - Хранит зарегистрированные режимы и переключает их: `setMode('translate')`.
  - Слушает глобальные клавиатурные события и мышь, преобразует координаты мыши в `THREE.Raycaster` и делегирует вызов активному режиму.
  - Управляет стеком истории режимов: нажатие `Escape` или отпускание `Ctrl` автоматически возвращает к предыдущему активному режиму.

### 2.3. Слой Отображения (Three.js Viewport)
- **`js/viewer.js`**:
  - Создает и конфигурирует холст WebGL.
  - Управляет ортографической и перспективной камерами с плавным переходом (`animateCameraTo`).
  - Содержит чистый цикл рендеринга `animate()`, вызывающий `controls.update()` и `modeManager.update(dt)`.
- **`js/scene_builder.js`**:
  - Выступает в роли реактивного моста. Слушает `store.on('placementData')`.
  - При обновлении данных он не пересоздает сцену с нуля. Он точечно обновляет `position` и `scale` существующих `THREE.Mesh` по их уникальным ключам (из `shardMeshes` и `socketMeshes`).
  - Перерисовывает линии связей (`routes`), запрашивая координаты у алгоритма трассировки.

### 2.4. Слой Коллизий (`js/editor/collision_manager.js`)
- Содержит **CollisionManager**, который фильтрует пересечения луча (`THREE.Raycaster`) с 3D объектами.
- Реализует правила:
  1. **Приоритет выбора**: Точки трасс -> Сокеты -> Шарды -> Линии трасс.
  2. **X-ray выделение**: Если шард выделен и прозрачен, клик сквозь его передние грани выделяет сокеты, находящиеся внутри шарда.
  3. **Backface Culling**: Игнорирует сокеты, нормаль которых развернута от камеры.

## 3. Модули на Python: dev-sdk-api и dev-client-api

Мы разделяем логику работы с данными/SDK и инфраструктуру локального веб-сервера.

### 3.1. dev-sdk-api (Python SDK)
Изолированная библиотека для программной генерации, валидации и модификации топологий AxiCAD.
```
dev-sdk-api/
├── setup.py              # Конфиг публикации на PyPI (как axipy)
└── axipy/
    ├── __init__.py
    ├── builder.py        # Объектные модели данных (Shard, Socket, Connection)
    ├── errors.py         # Локальные правила валидации зазоров и слоев
    └── contract/
        ├── __init__.py
        └── contract_generated.py # Сгенерированная ABI-структура данных
```

### 3.2. dev-client-api (Локальный сервер и алгоритмы)
Локальный REST сервер и компилятор. Работает как связующий мост между фронтендом, SDK и файловой системой.
```
dev-client-api/
├── server/
│   ├── server.py         # HTTP-сервер. Обеспечивает раздачу статики фронтенда и API-эндпоинты.
│   │                     # Выполняет маппинг: /projects/local/ -> .local-storage/
│   │                     #                    /projects/scripts/ -> examples/
│   ├── persistence.py    # Логика загрузки/сохранения overrides и вызова алгоритмов.
│   └── model_loader.py   # Модуль импорта рецептов сети с мокированием классов axipy.
└── algorithms/
    ├── placer.py         # Алгоритм орбитального размещения шардов.
    ├── router.py         # Алгоритм трассировки линий связи.
    └── layout_overrides.json # Файл переопределений по умолчанию.
```

### API Контракты:
- **`GET /api/projects`**: Возвращает список скриптов рецептов (из `examples/`) и локальных сохраненных проектов (из `.local-storage/`).
- **`GET /api/projects/{name}/load`**: Читает файлы `placement.json` и `routes.json` из `.local-storage/{name}/` и отдает их фронтенду.
- **`POST /api/projects/{name}/save`**: Принимает обновленные `placementData` и `routesData` в формате JSON и записывает их в `.local-storage/{name}/layout_overrides.json`.
- **`POST /api/projects/{name}/compile`**: Запускает локальную компиляцию (вызов плейсера и роутера для обновления `placement.json` и `routes.json` на основе скрипта рецепта и оверрайдов).

## 4. Модуль dev-rust-api (Тяжелые Алгоритмы)

Набор производительных алгоритмов, написанных на Rust. Проектируется так, чтобы компилироваться в двух режимах:
1. **Нативная библиотека (или исполняемый файл)** — вызывается бэкендом для пакетной симуляции.
2. **WebAssembly (`wasm-pack`)** — импортируется напрямую во фронтенд для мгновенных интерактивных расчетов в браузере.

```
dev-rust-api/
├── Cargo.toml
└── src/
    ├── lib.rs            # Регистрация WASM-биндингов (wasm-bindgen)
    ├── physics.rs        # Физика кабелей (Axon Verlet Integration):
    │                     # Принимает координаты пинов, выводит сплайн пути с учетом коллизий.
    ├── routing.rs        # Автотрассировка: алгоритм поиска пути в 3D пространстве с обходом препятствий.
    └── validation.rs     # Валидатор: статический анализ зазоров, пересечений и нарушений топологии.
```

## 6. Верификационный план во время разработки

### Автоматические тесты:
- **Rust алгоритмы**: Запуск юнит-тестов физики и трассировки:
  `cd dev-rust-api && cargo test`
- **Python SDK и Сервер**: Проверка правильности работы с топологией и API:
  `cd dev-sdk-api && pytest`
  `cd dev-client-api && pytest`
- **JS-клиент**: Линтинг и проверка сборки редактора:
  `cd dev-js-api && npm run lint && npm run build`
